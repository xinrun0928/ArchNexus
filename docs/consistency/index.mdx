# 一致性

一致性是分布式系统设计的核心问题之一。当数据分布在多个节点上时，如何保证所有节点看到相同的数据？如何在性能和一致性之间取得平衡？这些问题的答案直接决定了系统的可靠性。

## 章节概述

一致性不是一个可以用「强」或「弱」简单回答的问题。在不同的业务场景下，一致性的含义和需求可能截然不同。一个金融支付系统需要强一致性来保证资金安全，而一个社交媒体应用可能只需要最终一致性来换取更高的可用性和性能。

本章节从理论到实践，系统介绍分布式一致性的核心概念和技术方案：

**一致性理论**部分解答了一致性是什么、为什么重要、分布式环境下有哪些挑战等基础问题，并提供了根据业务场景选择一致性模型的方法论。

**强一致性**部分深入探讨了线性一致性、顺序一致性等最强的一致性保证，以及单主复制、共识算法等实现方案。理解强一致性是正确使用它的前提。

**最终一致性**部分关注弱一致性模型，探讨数据如何收敛、冲突如何解决、哪些场景适合使用最终一致性。

**读写分离**部分是数据库性能优化的重要手段，但同时带来了一致性挑战。理解主从复制原理、复制延迟处理是正确实施读写分离的基础。

**缓存一致性**是现代系统绕不开的话题。从 Cache Aside 到 Write Behind，从单级缓存到多级缓存，不同策略有不同的适用场景和一致性问题。

**分布式事务**讨论了在分布式环境下如何保证跨服务的事务特性。TCC、本地消息表、事务消息等方案各有优劣，需要根据业务场景选择。

## 核心原则

**一致性是有代价的**。强一致性意味着更低的性能和可用性。在选择一致性方案之前，首先应该问：这个场景真的需要这么强的一致性吗？

**没有银弹**。不同的方案在一致性强度、性能、复杂度上有显著差异。没有适用于所有场景的最佳方案，只有最适合当前业务场景的方案。

**分层设计**。不同的数据可能有不同的一致性需求。核心数据强一致，非核心数据可以使用最终一致性。同一系统中混合使用多种一致性方案是常见的实践。
