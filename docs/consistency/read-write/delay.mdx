# 从库延迟处理

你的读写分离看起来工作正常，但用户开始抱怨数据「不对」。

问题出在哪？**从库延迟太大，但应用没有正确处理**。

从库延迟是读写分离的固有挑战，需要正确应对。

## 延迟监控

### 延迟指标

```sql
-- 查看复制状态
SHOW SLAVE STATUS\G

-- 关键指标说明
-- Seconds_Behind_Master: 从库落后主库的秒数
-- Slave_IO_Running: I/O 线程是否运行
-- Slave_SQL_Running: SQL 线程是否运行
-- Read_Master_Log_Pos: 已读取的主库 binlog 位置
-- Exec_Master_Log_Pos: 已执行的主库 binlog 位置
-- Relay_Log_Space: relay log 占用的空间
```

### 应用层监控

```java
// 延迟监控服务
@Service
public class ReplicationLagMonitor {
    @Autowired
    private DataSource slaveDataSource;

    // 获取复制延迟（秒）
    public long getReplicationLag() {
        try (Connection conn = slaveDataSource.getConnection();
             Statement stmt = conn.createStatement();
             ResultSet rs = stmt.executeQuery("SHOW SLAVE STATUS")) {

            if (rs.next()) {
                return rs.getLong("Seconds_Behind_Master");
            }
        } catch (SQLException e) {
            log.error("获取复制延迟失败", e);
        }
        return -1;
    }

    // 监控告警
    @Scheduled(fixedRate = 10000)
    public void monitorAndAlert() {
        long lag = getReplicationLag();

        if (lag < 0) {
            alertService.critical("复制线程已停止");
        } else if (lag > 300) {
            alertService.critical("复制延迟严重: " + lag + "秒");
        } else if (lag > 60) {
            alertService.warning("复制延迟较高: " + lag + "秒");
        }
    }
}
```

## 延迟应对策略

### 策略一：延迟阈值路由

```java
// 基于延迟阈值的路由
@Service
public class LagAwareRouter {
    private static final long CRITICAL_LAG = 60;  // 60 秒
    private static final long WARNING_LAG = 10;    // 10 秒

    public DataSource selectReadDataSource() {
        long lag = getReplicationLag();

        // 延迟严重，读主库
        if (lag > CRITICAL_LAG) {
            return masterDataSource;
        }

        // 延迟较低，读从库
        return selectSlave();
    }

    private long getReplicationLag() {
        // 实现延迟获取
        return 0;
    }
}
```

### 策略二：读主库补偿

```java
// 关键读取读主库
@Service
public class CriticalReadService {
    @Autowired
    private DataSource masterDataSource;

    @Autowired
    private DataSource slaveDataSource;

    // 订单相关读取读主库
    public Order getOrder(Long orderId) {
        try {
            return jdbcTemplate.queryForObject(
                "SELECT * FROM orders WHERE id = ?",
                new OrderRowMapper(),
                orderId
            );
        } catch (EmptyResultDataAccessException e) {
            return null;
        }
    }

    // 配合延迟监控，动态调整
    public Order getOrderWithLagAware() {
        long lag = getReplicationLag();

        // 延迟严重，读主库
        if (lag > 60) {
            return readFromMaster(orderId);
        }

        // 延迟正常，读从库
        return readFromSlave(orderId);
    }
}
```

### 策略三：延迟感知连接池

```java
// 延迟感知的读写分离连接池
public class LagAwareRoutingDataSource extends AbstractDataSource {
    private DataSource masterDataSource;
    private List<DataSource> slaveDataSources;

    private volatile long lastKnownLag = 0;
    private final ScheduledExecutorService scheduler = Executors.newScheduledThreadPool(1);

    @PostConstruct
    public void init() {
        // 定期检查延迟
        scheduler.scheduleAtFixedRate(() -> {
            lastKnownLag = checkReplicationLag();
        }, 0, 5, TimeUnit.SECONDS);
    }

    @Override
    public Connection getConnection() {
        if (shouldUseMaster()) {
            return masterDataSource.getConnection();
        }
        return selectSlave().getConnection();
    }

    private boolean shouldUseMaster() {
        return lastKnownLag > 60;
    }
}
```

## 延迟故障处理

### 自动故障转移

```java
// 延迟过高时自动转移
@Service
public class LagFailoverHandler {
    @Autowired
    private AlertService alertService;

    @Scheduled(fixedRate = 5000)
    public void checkAndHandle() {
        long lag = getReplicationLag();

        // 延迟超过阈值
        if (lag > 300) {  // 5 分钟
            alertService.critical("复制延迟严重，执行故障处理");

            // 1. 停止写入到该从库
            disableSlave(slaveId);

            // 2. 告警通知运维
            alertService.notifyOperations(
                "从库 " + slaveId + " 延迟严重，已自动隔离");

            // 3. 记录事件
            auditLog.log("SLAVE_LAG_CRITICAL",
                "slaveId=" + slaveId, "lag=" + lag);
        }
    }
}
```

## 总结

从库延迟是读写分离的固有挑战。

**延迟成因**：

- 大事务
- 从库负载高
- 网络延迟
- 锁冲突

**应对策略**：

- 延迟阈值路由：延迟高时读主库
- 关键读取读主库
- 延迟感知连接池

**减少延迟**：

- 拆分大事务
- 增加从库数量
- 升级从库规格
- 同城同机房部署

**监控要点**：

- 复制延迟秒数
- 复制线程状态
- binlog 位置
