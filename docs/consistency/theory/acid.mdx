# ACID 与一致性

你的订单系统显示下单成功，但库存却没有扣减。

问题出在哪？**事务没有正确维护业务一致性**。

ACID 是数据库事务的四大特性，其中 Consistency（一致性）是最容易被误解的一个。

## ACID 是什么

```
┌─────────────────────────────────────────────────────────────────┐
│                    ACID 四大特性                                   │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  A - Atomicity（原子性）                                        │
│  ───────────────────────────────────                           │
│  事务是最小执行单位，要么全部成功，要么全部失败           │
│                                                                  │
│  C - Consistency（一致性）                                      │
│  ───────────────────────────────────                           │
│  事务执行前后，数据必须满足所有业务规则               │
│                                                                  │
│  I - Isolation（隔离性）                                        │
│  ───────────────────────────────────                           │
│  并发执行的事务互不干扰                                   │
│                                                                  │
│  D - Durability（持久性）                                       │
│  ───────────────────────────────────                           │
│  事务提交后，其结果永久保存                               │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

## ACID 中的一致性

```
┌─────────────────────────────────────────────────────────────────┐
│                    ACID 中的一致性                               │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  ACID 中的一致性 ≠ CAP 中的「一致性」                       │
│                                                                  │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │                                                         │   │
│  │  ACID 一致性：                                         │   │
│  │  「事务执行前后，系统状态必须满足所有约束条件」       │   │
│  │                                                         │   │
│  │  示例约束：                                             │   │
│  │  - 账户余额不能为负                                   │   │
│  │  - 库存数量不能为负                                   │   │
│  │  - 外键引用必须存在                                   │   │
│  │  - 唯一索引不能重复                                   │   │
│  │                                                         │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                  │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │                                                         │   │
│  │  CAP 一致性：                                          │   │
│  │  「分布式系统中，所有节点看到相同的数据」             │   │
│  │                                                         │   │
│  │  这是两个完全不同的概念！                             │   │
│  │                                                         │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

## 原子性 + 隔离性 = 一致性

```
┌─────────────────────────────────────────────────────────────────┐
│                    原子性和隔离性保障一致性                        │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  一致性的破坏往往来自：                                         │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │                                                         │   │
│  │  1. 事务执行中途失败（缺少原子性）                   │   │
│  │     例：扣款成功，库存扣减失败                       │   │
│  │                                                         │   │
│  │  2. 并发修改相互干扰（缺少隔离性）                   │   │
│  │     例：两个请求同时购买，最后库存为负               │   │
│  │                                                         │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                  │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │                                                         │   │
│  │  原子性：确保事务要么全部完成，要么全部不做        │   │
│  │  隔离性：确保并发执行的事务互不干扰               │   │
│  │  持久性：确保提交的结果不会丢失                   │   │
│  │                                                         │   │
│  │  三者共同保障了「事务执行前后，系统状态一致」    │   │
│  │                                                         │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

## 业务一致性实现

### 转账事务

```java
// 转账事务
@Transactional
public void transfer(String fromAccount, String toAccount, BigDecimal amount) {
    // 1. 检查余额（一致性约束）
    Account from = accountRepository.findById(fromAccount);
    if (from.getBalance().compareTo(amount) < 0) {
        throw new InsufficientBalanceException("余额不足");
    }

    // 2. 扣款
    from.setBalance(from.getBalance().subtract(amount));
    accountRepository.save(from);

    // 3. 存款
    Account to = accountRepository.findById(toAccount);
    to.setBalance(to.getBalance().add(amount));
    accountRepository.save(to);

    // 4. 记录日志（可选）
    transferLogRepository.save(new TransferLog(fromAccount, toAccount, amount));
}
```

### 库存扣减

```java
// 库存扣减（防止超卖）
@Transactional
public boolean reduceStock(Long productId, int quantity) {
    // 使用乐观锁或悲观锁
    Product product = productRepository.findByIdWithLock(productId);

    // 检查库存
    if (product.getStock() < quantity) {
        throw new InsufficientStockException("库存不足");
    }

    // 扣减库存
    product.setStock(product.getStock() - quantity);
    productRepository.save(product);

    return true;
}

// 或者使用乐观锁
@Transactional
public boolean reduceStockOptimistic(Long productId, int quantity) {
    int updated = productRepository.updateStockWithOptimisticLock(
        productId, quantity, product.getVersion());

    if (updated == 0) {
        throw new OptimisticLockException("并发冲突，请重试");
    }

    return true;
}
```

### 订单创建

```java
// 订单创建（复合一致性）
@Transactional
public Order createOrder(Long userId, List<OrderItem> items) {
    // 1. 校验用户状态
    User user = userRepository.findById(userId);
    if (!user.isActive()) {
        throw new UserInactiveException("用户状态异常");
    }

    // 2. 校验商品和库存
    BigDecimal totalAmount = BigDecimal.ZERO;
    for (OrderItem item : items) {
        Product product = productRepository.findById(item.getProductId());

        if (product.getStock() < item.getQuantity()) {
            throw new InsufficientStockException("商品库存不足: " + product.getName());
        }

        totalAmount = totalAmount.add(product.getPrice().multiply(new BigDecimal(item.getQuantity())));
    }

    // 3. 计算优惠
    BigDecimal discount = calculateDiscount(user, totalAmount);

    // 4. 创建订单
    Order order = new Order();
    order.setUserId(userId);
    order.setTotalAmount(totalAmount.subtract(discount));
    order.setStatus(OrderStatus.CREATED);
    orderRepository.save(order);

    // 5. 扣减库存
    for (OrderItem item : items) {
        productRepository.reduceStock(item.getProductId(), item.getQuantity());
    }

    // 6. 扣减余额
    user.setBalance(user.getBalance().subtract(totalAmount.subtract(discount)));
    userRepository.save(user);

    return order;
}
```

## 总结

ACID 中的「一致性」是业务层面的概念。

**与 CAP 的区别**：

- ACID 一致性：事务执行前后数据满足业务约束
- CAP 一致性：分布式系统中所有节点看到相同数据

**一致性保障**：

- 原子性：事务要么全做，要么全不做
- 隔离性：并发事务互不干扰
- 持久性：提交结果不丢失

**实现要点**：

- 在事务开始时校验所有约束
- 使用乐观锁或悲观锁防止并发问题
- 保持事务简短，减少锁持有时间
