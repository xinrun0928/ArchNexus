# 分布式一致性挑战

你的单机数据库运行良好，但一旦扩展到多个节点，问题就出现了：数据不一致、故障难处理、性能下降。

问题出在哪？**分布式系统天生就面临一致性的挑战**。

## 分布式一致性的本质问题

```
┌─────────────────────────────────────────────────────────────────┐
│                    分布式一致性的本质问题                           │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  三个不可能的假设（FALLacies of Distributed Computing）：       │
│                                                                  │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │                                                         │   │
│  │  1. 网络是可靠的                                       │   │
│  │     现实：网络会延迟、会丢包、会中断                 │   │
│  │                                                         │   │
│  │  2. 延迟为零                                          │   │
│  │     现实：任何通信都有延迟，跨机房延迟可达 100ms     │   │
│  │                                                         │   │
│  │  3. 带宽无限                                          │   │
│  │     现实：带宽有限，需要考虑数据分片和压缩         │   │
│  │                                                         │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                  │
│  这些假设的不成立，导致了分布式一致性的根本挑战               │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

│  │         节点 A ────── 节点 B                       │   │
│  │           │                  │                        │   │
│  │           │                  │                        │   │
│  │           ▼                  ▼                        │   │
│  │         节点 C ────── 节点 D                       │   │

## 挑战三：时钟不一致

```
┌─────────────────────────────────────────────────────────────────┐
│                    挑战三：时钟不一致                             │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  分布式系统中，每个节点都有自己的时钟：                       │
│                                                                  │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │                                                         │   │
│  │  节点 A：时间 10:00:00                               │   │
│  │  节点 B：时间 10:00:03  ← 时钟漂移 3 秒              │   │
│  │  节点 C：时间 09:59:57  ← 时钟回拨                  │   │
│  │                                                         │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                  │
│  问题：                                                        │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │                                                         │   │
│  │  1. 事件顺序无法靠时间戳判断                         │   │
│  │  2. 「先写先到」可能变成「后写先到」               │   │
│  │  3. 依赖时间的过期检测可能失效                       │   │
│  │                                                         │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                  │
│  解决方案：                                                    │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │                                                         │   │
│  │  - 使用逻辑时钟（Lamport Timestamp）                   │   │
│  │  - 使用向量时钟（Vector Clock）                        │   │
│  │  - 使用中心授时服务（NTP、TrueTime）                 │   │
│  │                                                         │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

│                                                                  │

│                                                                  │

## 总结

分布式一致性面临多重挑战。

**核心挑战**：

- 网络分区不可避免
- 节点可能故障
- 时钟无法精确同步
- 并发修改产生冲突
- CAP 必须权衡
- 复制延迟客观存在

**应对思路**：

- 明确业务一致性需求
- 选择合适的复制策略
- 引入共识算法
- 补偿和重试机制

**关键认知**：

- 强一致性是有代价的
- 不是所有场景都需要强一致性
- 理解 trade-off 才能做出正确决策
