# 读写冲突解决

你的系统使用了最终一致性，但两个用户同时修改了同一份数据，现在不知道该用哪个值。

问题出在哪？**没有正确处理并发冲突**。

冲突解决是最终一致性系统的核心问题。

## 冲突的本质

```
┌─────────────────────────────────────────────────────────────────┐
│                    冲突的本质                                     │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  冲突（Conflict）：                                              │
│                                                                  │
│  「两个或多个操作同时修改同一份数据，导致数据状态不明确」       │
│                                                                  │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │                                                         │   │
│  │  示例：                                               │   │
│  │                                                         │   │
│  │  节点 A：write(key, "Alice")  at T1                │   │
│  │  节点 B：write(key, "Bob")    at T2                │   │
│  │                                                         │   │
│  │  T1 和 T2 在物理上可能是并行的                       │   │
│  │  系统无法判断哪个是「最新」的                         │   │
│  │                                                         │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

## 冲突检测

### 基于版本向量的冲突检测

```java
// 基于向量时钟的冲突检测
public class ConflictDetector {
    // 判断两个版本是否冲突
    public static boolean isConflict(VectorClock v1, VectorClock v2) {
        // 如果两个版本是并发的，则冲突
        return VectorClock.isConcurrent(v1, v2);
    }

    // 向量时钟并发判断
    public static boolean isConcurrent(VectorClock v1, VectorClock v2) {
        boolean v1Greater = false;
        boolean v2Greater = false;

        Set<String> allKeys = new HashSet<>();
        allKeys.addAll(v1.getVersions().keySet());
        allKeys.addAll(v2.getVersions().keySet());

        for (String key : allKeys) {
            long t1 = v1.getOrDefault(key, 0L);
            long t2 = v2.getOrDefault(key, 0L);

            if (t1 > t2) v1Greater = true;
            if (t2 > t1) v2Greater = true;
        }

        // 如果两者互相不小于对方，则并发（冲突）
        return v1Greater && v2Greater;
    }
}
```

### 基于时间戳的冲突检测

```java
// 基于时间戳的冲突检测
public class TimestampConflictDetector {
    // 判断是否冲突
    public boolean isConflict(TimestampedValue v1, TimestampedValue v2) {
        // 如果时间戳相同，可能是冲突
        if (v1.getTimestamp() == v2.getTimestamp()) {
            // 需要额外的判断依据
            return !v1.getNodeId().equals(v2.getNodeId());
        }

        // 否则不冲突（后写者胜）
        return false;
    }
}
```

## 冲突解决策略

### Last-Write-Wins（LWW）

```
┌─────────────────────────────────────────────────────────────────┐
│                    Last-Write-Wins (LWW)                         │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  策略：时间戳最大的写入胜出                                      │
│                                                                  │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │                                                         │   │
│  │  示例：                                               │   │
│  │                                                         │   │
│  │  write(key, "Alice") at T1 → 时间戳 100              │   │
│  │  write(key, "Bob")   at T2 → 时间戳 200             │   │
│  │                                                         │   │
│  │  结果：Bob 胜出                                        │   │
│  │                                                         │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                  │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │                                                         │   │
│  │  优点：                                                │   │
│  │  ✓ 实现简单                                           │   │
│  │  ✓ 结果确定                                           │   │
│  │                                                         │   │
│  │  缺点：                                                │   │
│  │  ✗ 可能丢失数据（先写入的数据被覆盖）                 │   │
│  │  ✗ 依赖时钟同步                                       │   │
│  │  ✗ 并发写入可能产生意外结果                          │   │
│  │                                                         │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

```java
// LWW 实现
public class LWWConflictResolver implements ConflictResolver {
    @Override
    public Object resolve(Object value1, Object value2,
                         Metadata meta1, Metadata meta2) {
        // 比较时间戳
        if (meta1.getTimestamp() > meta2.getTimestamp()) {
            return value1;
        } else {
            return value2;
        }
    }
}
```

### 基于应用逻辑的合并

```java
// 基于应用逻辑的合并
@Service
public class BusinessConflictResolver {
    // 账户余额冲突解决
    public BigDecimal resolveAccountBalance(
            BigDecimal value1, Metadata meta1,
            BigDecimal value2, Metadata meta2) {
        // 选择时间戳较大的
        if (meta1.getTimestamp() > meta2.getTimestamp()) {
            return value1;
        }
        return value2;
    }

    // 库存冲突解决（防止超卖）
    public int resolveInventory(int value1, int value2) {
        // 选择较小的值（保守策略）
        return Math.min(value1, value2);
    }

    // 用户资料冲突解决
    public UserProfile resolveProfile(UserProfile p1, UserProfile p2) {
        UserProfile result = new UserProfile();

        // 每个字段选择最新的
        result.setName(p1.getTimestamp() > p2.getTimestamp()
            ? p1.getName() : p2.getName());

        result.setAvatar(p1.getTimestamp() > p2.getTimestamp()
            ? p1.getAvatar() : p2.getAvatar());

        result.setEmail(p1.getTimestamp() > p2.getTimestamp()
            ? p1.getEmail() : p2.getEmail());

        // 合并标签
        Set<String> tags = new HashSet<>();
        tags.addAll(p1.getTags());
        tags.addAll(p2.getTags());
        result.setTags(tags);

        return result;
    }
}
```

## 冲突检测与解决的实践

```java
// 冲突检测与解决的完整实现
@Service
public class ConflictAwareKVStore {
    private final ConflictResolver resolver;

    // 读取
    public VersionedValue read(String key) {
        // 从所有副本收集数据
        List<VersionedValue> versions = collectFromAllReplicas(key);

        if (versions.isEmpty()) {
            return null;
        }

        // 检测冲突
        if (hasConflict(versions)) {
            return resolveConflict(key, versions);
        }

        // 无冲突，返回最新值
        return getLatestValue(versions);
    }

    // 写入
    public void write(String key, Object value) {
        // 写入所有副本
        for (Replica replica : replicas) {
            replica.write(key, value, generateTimestamp());
        }
    }

    // 检测冲突
    private boolean hasConflict(List<VersionedValue> versions) {
        if (versions.size() < 2) {
            return false;
        }

        for (int i = 0; i < versions.size(); i++) {
            for (int j = i + 1; j < versions.size(); j++) {
                if (ConflictDetector.isConflict(
                        versions.get(i).getClock(),
                        versions.get(j).getClock())) {
                    return true;
                }
            }
        }
        return false;
    }

    // 解决冲突
    private VersionedValue resolveConflict(String key, List<VersionedValue> versions) {
        log.warn("检测到冲突: key={}, versions={}", key, versions.size());

        // 使用配置的解决策略
        VersionedValue result = versions.get(0);
        for (int i = 1; i < versions.size(); i++) {
            result = new VersionedValue(
                resolver.resolve(
                    result.getValue(), versions.get(i).getValue(),
                    result.getMetadata(), versions.get(i).getMetadata()),
                mergeMetadata(result.getMetadata(), versions.get(i).getMetadata()));
        }

        // 将解决后的值写回所有副本
        for (Replica replica : replicas) {
            replica.write(key, result.getValue(), result.getTimestamp());
        }

        return result;
    }
}
```

## 总结

冲突解决是最终一致性系统的核心问题。

**冲突检测**：

- 基于向量时钟：精确判断并发
- 基于时间戳：简单但依赖时钟同步

**解决策略**：

- LWW：后写者胜，简单但可能丢数据
- FWW：先写者胜，保守但可能过时
- 自动合并：适用于特定类型
- 应用逻辑：最灵活但实现复杂

**CRDT**：

- 天然无冲突的数据结构
- 无需协调
- 适用于特定场景

**实践要点**：

- 根据业务选择合适的解决策略
- 监控冲突频率
- 在应用层处理特殊冲突
