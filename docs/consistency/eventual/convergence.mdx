# 最终一致性的收敛

你的系统使用了最终一致性，但用户的数据永远都不一致。

问题出在哪？**没有正确理解数据收敛的条件和机制**。

最终一致性不是「永远不一致」，而是「收敛后才一致」。

### Gossip 实现

```java
// Gossip 协议实现
public class GossipProtocol {
    private final Map<String, Object> data = new ConcurrentHashMap<>();
    private final VersionVector version = new VersionVector();
    private final String nodeId;
    private final ScheduledExecutorService scheduler = Executors.newScheduledThreadPool(1);

    public GossipProtocol(String nodeId) {
        this.nodeId = nodeId;
    }

    // 启动 Gossip
    public void start(List<GossipProtocol> nodes, int intervalMs) {
        scheduler.scheduleAtFixedRate(() -> {
            // 随机选择一个节点交换
            GossipProtocol peer = nodes.stream()
                .filter(n -> !n.nodeId.equals(this.nodeId))
                .findAny()
                .orElse(null);

            if (peer != null) {
                gossipWith(peer);
            }
        }, intervalMs, intervalMs, TimeUnit.MILLISECONDS);
    }

    // 与对等节点交换数据
    private void gossipWith(GossipProtocol peer) {
        // 交换版本向量
        VersionVector peerVersion = peer.getVersion();
        VersionVector myVersion = this.version.getCopy();

        // 确定需要同步的数据
        Map<String, Object> toSend = new HashMap<>();
        for (String key : this.data.keySet()) {
            if (isNewer(key, myVersion, peerVersion)) {
                toSend.put(key, this.data.get(key));
            }
        }

        // 接收对方的数据
        Map<String, Object> received = peer.receiveAndRespond(toSend, myVersion);

        // 应用接收到的数据
        for (Map.Entry<String, Object> entry : received.entrySet()) {
            if (isNewer(entry.getKey(), peerVersion, myVersion)) {
                this.data.put(entry.getKey(), entry.getValue());
            }
        }
    }

    // 判断哪个版本更新
    private boolean isNewer(String key, VersionVector v1, VersionVector v2) {
        // 简单实现：比较版本向量的值
        return v1.get(key) > v2.getOrDefault(key, 0L);
    }
}
```

### 基于向量时钟的收敛

```java
// 向量时钟收敛
public class VectorClockConvergence {
    private final Map<String, VectorClock> clocks = new ConcurrentHashMap<>();

    // 更新数据并增加版本
    public void update(String key, Object value, String nodeId) {
        VectorClock clock = clocks.computeIfAbsent(key, k -> new VectorClock());
        clock.increment(nodeId);
        clocks.put(key, clock);
    }

    // 获取数据及其版本
    public Pair<Object, VectorClock> get(String key) {
        VectorClock clock = clocks.get(key);
        return new Pair<>(data.get(key), clock);
    }

    // 检测冲突
    public boolean hasConflict(String key, VectorClock clock1, VectorClock clock2) {
        return VectorClock.isConcurrent(clock1, clock2);
    }

    // 合并冲突
    public Object merge(String key, VectorClock clock1, Object value1,
                        VectorClock clock2, Object value2) {
        // 使用 Last-Write-Wins 策略
        if (VectorClock.happensBefore(clock1, clock2)) {
            return value2;
        } else if (VectorClock.happensBefore(clock2, clock1)) {
            return value1;
        } else {
            // 冲突：使用自定义合并策略
            return mergeConflict(value1, value2);
        }
    }
}
```

## 总结

数据收敛是最终一致性的核心保证。

**定义**：在没有新更新的情况下，所有副本最终会持有相同数据。

**收敛机制**：

- Gossip 协议：去中心化、抗故障
- 向量时钟：精确跟踪依赖关系
- CRDT：天然收敛的数据结构

**收敛时间**：

- 取决于节点数、网络延迟、冲突频率
- 通常是 O(log N) 轮
- 没有明确的时间保证

**收敛条件**：

- 更新最终被所有节点接收
- 冲突解决策略是确定性的
- 没有永久网络分区

**设计要点**：

- 选择合适的冲突解决策略
- 监控收敛时间
- 在应用层处理不一致窗口
