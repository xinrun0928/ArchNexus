# 知识导览

## 为什么要系统学习后端架构

真正让一个工程师拉开差距的，不是写了多少行代码，而是能否在系统出现问题时快速定位根因，能否在架构选型时权衡利弊，能否在代码 Review 时发现潜在风险。这些能力不是靠加班熬出来的，而是靠系统化的知识积累和刻意训练。

大多数人在学习后端技术的过程中，容易陷入两个极端：要么只关注「怎么用」，CRUD 写得熟练，但不知道底层发生了什么；要么钻进理论里出不来，CAP 分布式一致性聊得头头是道，但写一个接口都费劲。ArchNexus 试图填补这个鸿沟——既讲原理，也讲实践；既有深度，也有广度。

## 知识体系全景图

后端架构知识可以划分为几个核心领域，每个领域都有其独特的思维模式和解决问题的方法论。

**原理层**解决「为什么」的问题。为什么分布式系统必须接受 CAP 权衡？为什么数据库事务要有隔离级别？为什么缓存会带来一致性问题？不懂原理，出了问题是真不知道往哪个方向查。

**模式层**解决「怎么组织」的问题。设计模式是代码层面的组织方式，架构模式是系统层面的组织方式。CQRS 将读写分离做到极致，Saga 模式让分布式事务有了可落地的方案，六边形架构让业务逻辑与技术细节解耦。模式不是银弹，但确实能大幅降低沟通成本和认知负担。

**工程层**解决「怎么落地」的问题。系统设计讲的是如何根据业务需求推导容量规划、如何选择分库分表策略、如何设计消息队列的幂等消费。性能优化讲的是如何测量真实延迟、如何找到瓶颈在 IO 还是 CPU、如何避免过早优化。

**治理层**解决「怎么长期维护」的问题。架构演进不是推翻重来，而是如何在保持业务连续性的前提下逐步优化。可用性保障不是靠堆机器，而是靠超时、熔断、降级、限流这些朴素的手段。安全架构不是加个鉴权就完事了，而是从身份认证到数据保护的全链路设计。

## 核心模块导航

### 架构原理（/principles/）

原理是所有决策的底层依据。如果不懂 CAP 理论，你可能会在选型时做出错误判断——明明业务场景需要强一致性，却因为「分布式系统不能用 CP」这种片面的认知选择了 AP 方案，导致数据错乱而找不到根因。

这一模块重点涵盖：CAP 与 BASE 理论的深层含义及其适用边界； Paxos/Raft 共识算法的工作机制与工程实现；一致性模型从线性一致性到最终一致性的完整光谱；分布式事务中两阶段提交、三阶段提交与 Saga 的取舍逻辑；幂等性设计在不同场景下的实现方案。

### 设计模式（/patterns/）

模式是经验的抽象。23 种 GoF 设计模式不只是面试题，更是在长期实践中被反复验证的代码组织方案。但设计模式不是终点——并发设计模式（生产者-消费者、双检查锁定、ThreadLocal）解决的是线程安全问题；分布式设计模式（事件溯源、CDC、Saga）解决的是跨服务协作问题；架构模式（CQRS、六边形、整洁架构）解决的是系统长期可维护性问题。

这一模块重点涵盖：创建型模式在对象构造中的权衡；结构型模式在代码组织中的价值；行为型模式在职责划分中的最佳实践；并发设计模式在多线程场景下的具体应用；分布式设计模式在跨服务协作中的落地；六边形、CQRS、事件溯源等架构模式的设计意图与适用条件。

### 系统设计（/system-design/）

系统设计考察的是综合判断能力。一个完整的需求分析，需要考虑数据量、访问模式、一致性要求、容灾方案，然后才能推导出存储选型、扩展策略、缓存架构。很多人觉得系统设计是玄学，其实是因为没有建立系统的推导框架。

这一模块重点涵盖：需求分析方法论——如何从业务需求推导技术指标；扩展策略——水平扩展与垂直扩展的选择与组合；负载均衡——算法选型与健康检查；数据分片——分片键设计、跨分片查询、数据迁移；缓存策略——缓存失效、穿透、击穿、雪崩的完整应对；消息队列——选型对比、消费模式、顺序保证、幂等设计。

### 性能原理（/performance/）

性能优化最怕两件事：一是方向错误，在不构成瓶颈的地方反复优化；二是过早优化，在核心路径还没梳理清楚的时候就开始扣细节。真正的性能优化应该是：先测量、再定位、最后优化。

这一模块重点涵盖：性能指标体系——延迟、吞吐量、QPS、TPS 的关系与测量方法；延迟分析——从 DNS 解析到 TLS 握手到数据库查询的完整链路；吞吐量优化——从连接池到线程池到批处理的系统化思路；并发模型——IO 多路复用、异步非阻塞的性能本质；内存管理——JVM 堆布局、GC 算法选择与调优；JIT 编译——热点探测与内联优化对性能的影响。

### 可用性与容错（/availability/）

系统的高可用不是靠硬件堆出来的，而是靠设计出来的。故障不可避免，关键是如何让故障的影响范围最小化、如何快速恢复、如何避免级联扩散。SLA 不是承诺出来的，是设计出来的。

这一模块重点涵盖：可用性理论与 SLA——多少个 9 意味着什么，不同场景的 SLA 应该如何设定；故障模型——雪崩、级联故障、重试风暴的形成机制；熔断——熔断器模式的工作原理与状态机转换；限流——令牌桶、滑动窗口、漏桶算法的适用场景；降级——非核心功能降级的策略与实现；超时与重试——超时设置的科学方法、重试次数与退避策略；健康检查——主动探测与被动检测的结合。

### 数据一致性（/consistency/）

一致性是分布式系统中最复杂的问题之一。强一致性意味着性能代价，最终一致性意味着业务妥协。没有完美的方案，只有适合场景的权衡。

这一模块重点涵盖：一致性光谱——线性一致性、顺序一致性、因果一致性、最终一致性的定义与适用场景；读写分离——复制延迟带来的读写一致性挑战与应对；缓存一致性——Cache Aside、Read Through、Write Through 的选择；分布式事务——Seata 的 AT 模式、TCC 模式、Saga 模式的对比与落地；本地消息表与事务性发件箱——弱一致性下的可靠消息投递。

### 并发编程（/concurrency/）

并发问题是最难调试的问题之一，因为它往往不是必现的，而是依赖于时序和负载。死锁、活锁、饥饿、竞态条件——这些问题在线下压测时可能完全不出现，一到生产环境就开始频繁报警。

这一模块重点涵盖：并发控制理论——临界区、互斥、活锁、死锁的定义与识别；锁机制—— synchronized 与 Lock 的选择、锁粒度与性能的关系；乐观锁与悲观锁——版本号、CAS 操作、ABA 问题的完整方案；CAS 与 ABA 问题—— AtomicStampedReference 等解决方案；事务隔离级别——脏读、不可重复读、幻读的成因与四种隔离级别的权衡；死锁处理——死锁预防、死锁检测、死锁恢复的策略。

### 架构演进（/evolution/）

好的架构不是一步到位的，而是演进出来的。从单体到微服务，从巨石应用到分布式系统，每一步都有其内在逻辑和风险点。很多团队在演进过程中踩坑，不是因为技术选型错了，而是因为时机不对。

这一模块重点涵盖：单体到微服务的演进路径——什么时候该拆、怎么拆、拆了之后怎么管；服务拆分策略——按业务能力拆分与按技术边界拆分的区别；数据迁移方案——双写、CDC、灰度切流的具体实现；架构选型——技术债务与架构腐化的识别与治理；中台化——中台建设的价值与代价，什么时候适合走这条路。

### 安全架构（/security/）

安全不是加个过滤器就完事了。认证、授权、加密、会话管理，每一个环节都有其独特的攻击面和防护策略。安全问题的代价往往是最高的——数据泄露不只是业务损失，还有合规风险和用户信任的崩塌。

这一模块重点涵盖：认证与授权——Session、JWT、OAuth 2.0、OIDC 的演进与适用场景；加密解密——对称加密与非对称加密的组合使用，密钥管理与轮换；会话管理——会话 fixation 攻击、会话劫持的防护；OAuth 2.0 与 OpenID Connect——四种授权模式的选择与安全配置；防御深度——认证层、授权层、审计层的分层设计；常见漏洞——SQL 注入、XSS、CSRF 的原理与防护。

## 学习路径建议

每个人的背景不同，学习路径也应该有所差异。

**如果是初学者**，建议从并发编程和系统设计入手。这两个模块与日常开发最贴近，可以直接指导代码实践。在理解线程、锁、事务这些基础概念后，再去看原理层的内容，会发现很多理论其实早就遇到过，只是当时没意识到。

**如果是已有一定经验的开发者**，建议从架构原理和设计模式入手。这两个模块能帮助你建立更高层次的抽象能力，学会用模式的语言描述问题、讨论方案。在 Code Review 时，能说出「这里可以考虑用事件溯源」比说「这里设计得不太好」要有用得多。

**如果是准备晋升或面试**，建议重点关注系统设计、性能原理、可用性与容错三个模块。这三个模块考察的是综合判断能力，也是面试中最容易拉开差距的地方。

## 核心原则

学习过程中有两个核心原则：

**原理优先于框架**。Spring Boot 很实用，但不懂 Servlet 原理、不懂 Spring IoC 容器的人，遇到问题往往只会翻文档、搜 Stack Overflow。框架会过时，原理不会。

**问题驱动学习**。不要为了学而学。每学一个知识点，尝试回答：这个知识解决的是什么问题？如果没有这个技术，人们会怎么解决这个问题？它的代价是什么？带着问题去学，知识才会真正内化。

最后，架构能力不是看出来的，是练出来的。看文档能建立知识框架，真正遇到问题时的判断力，来自无数次踩坑和复盘。ArchNexus 能帮你建立框架，但真正的成长，还是要靠在真实项目中去磨。
