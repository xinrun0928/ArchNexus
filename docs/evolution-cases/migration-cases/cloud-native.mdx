# 云原生迁移

某传统企业（年收入 50 亿，IT 团队 200 人）在 2019 年决定启动数字化转型，其中最核心的工程改造是「上云」。

说是「上云」，实际上是把原来跑在物理机和 VMware 虚拟机上的应用，迁移到阿里云 ACK（阿里云容器服务 Kubernetes）。这个过程历时 18 个月，投入了 15 个人的团队，最终成功将 80% 的应用迁移到了云上。

项目负责人在回顾这段经历时说：「最难的不是技术，是人和流程。技术人员要学 Kubernetes，业务人员担心迁移期间业务中断，领导层要看到 ROI。我们踩了很多坑，也学到了很多。」

这一节用这个真实案例，完整还原一次传统企业的云原生迁移过程。

## 背景：为什么传统企业要上云

### 当时的 IT 架构

```
机房：自建数据中心，双机热备
服务器：物理机 30 台，VMware 虚拟机 50 台
应用：Java Spring MVC + WebSphere Application Server（WAS）
数据库：Oracle RAC，两台 IBM 小型机
中间件：IBM MQ，WebLogic
监控：Zabbix + 手工巡检
```

### 痛点

```
1. 资源利用率低：物理机平均 CPU 利用率 15%，内存利用率 30%
2. 扩容周期长：申请一台物理机要 2 周，虚拟机要 3 天
3. 发布效率低：WAS 手工部署，一次发布要 4 小时
4. 故障恢复慢：虚拟机 HA 需要 15 分钟，应用启动需要 10 分钟
5. 成本高：每年 IT 基础设施投入约 2000 万
```

### 上云的目标

```
业务目标：
- 资源利用率提升 50%
- 扩容时间从 2 周缩短到 5 分钟
- 发布频率从每月 1 次提升到每周 3 次
- 故障恢复时间从 30 分钟缩短到 5 分钟

财务目标：
- IT 基础设施成本降低 30%
- 减少对硬件厂商的依赖
- 按需付费，避免资源闲置
```

## 第一阶段：评估与规划（2 个月）

### 1. 应用分类

不是所有应用都适合上云，也不是所有应用都应该优先上云。团队按以下维度对应用分类：

| 分类 | 标准 | 数量 | 上云优先级 |
| --- | --- | --- | --- |
| 云就绪（Cloud-Ready） | 无状态、配置外置、支持健康检查 | 30 个 | 最高 |
| 云适配（Cloud-Ready with Changes） | 有状态、需要改造 | 20 个 | 中等 |
| 云不友好（Not Cloud-Ready） | 依赖本地存储、需要特殊硬件 | 15 个 | 最低 |
| 不上云 | 合规要求、物理设备依赖 | 10 个 | 排除 |

云就绪的应用是第一批迁移目标，它们的特点是：应用本身不需要任何改造，只要做好容器化和配置适配就可以迁移。

### 2. 改造清单

云适配的应用需要改造后才能迁移。团队列出了改造清单：

```markdown
# 应用云适配改造清单

## 必改项
- [ ] 移除本地文件存储，改用对象存储（OSS）或云盘
- [ ] 配置外置到 ConfigMap/Secret
- [ ] 实现健康检查端点（/health）
- [ ] 实现优雅停机（处理完正在处理的请求再退出）
- [ ] 日志输出到 stdout/stderr（不要写文件）

## 选改项
- [ ] 增加探针（ readinessProbe / livenessProbe）
- [ ] 配置资源限制（requests / limits）
- [ ] 接入链路追踪
- [ ] 接入 Prometheus 指标暴露

## 改造工作评估
- 预计改造工作量：每人月 × 8 = 8 人月
- 预计完成时间：4 个月
- 优先级：基础设施平台完成后开始
```

### 3. 迁移风险评估

```markdown
# 迁移风险评估

风险 1：网络规划不合理，导致跨可用区调用延迟增加
概率：高
影响：中
缓解措施：架构评审时充分考虑网络拓扑，优先迁移同可用区内的服务

风险 2：应用对云厂商 API 的依赖导��锁定
概率：中
影响：中
缓解措施：使用 Kubernetes 标准 API，接入层抽象云厂商差异

风险 3：迁移期间业务中断
概率：中
影响：高
缓解措施：灰度迁移，先迁移测试环境，再迁移生产流量

风险 4：Kubernetes 学习曲线导致团队适应期长
概率：高
影响：中
缓解措施：提前 3 个月开始培训，组建云原生能力中心
```

## 第二阶段：基础设施建设（4 个月）

### 1. Kubernetes 集群搭建

团队选择了阿里云 ACK，选择托管版（ASK）以减少运维负担：

```yaml
# Kubernetes 集群配置
cluster:
  name: production-cluster
  region: cn-hangzhou
  version: 1.28
  # 网络插件：Terway（阿里云容器网络插件，性能优于 Flannel）
  network:
    plugin: terway
    vpc_cidr: 192.168.0.0/16
    service_cidr: 172.21.0.0/16
  # 多可用区部署
  zones:
    - cn-hangzhou-h
    - cn-hangzhou-m
    - cn-hangzhou-k
```

### 2. 应用容器化

这是迁移中最繁琐的步骤。团队花了 3 个月为 50 个应用编写 Dockerfile：

```dockerfile
# 多阶段构建：构建阶段
FROM maven:3.8 AS builder
WORKDIR /app
COPY pom.xml .
RUN mvn dependency:go-offline
COPY src ./src
RUN mvn package -DskipTests

# 多阶段构建：运行阶段
FROM eclipse-temurin:17-jre-alpine
WORKDIR /app

# 安全：非 root 用户
RUN addgroup -S appgroup && adduser -S appuser -G appgroup
USER appuser

# 健康检查
HEALTHCHECK --interval=30s --timeout=3s --start-period=60s --retries=3 \
    CMD wget --quiet --tries=1 --spider http://localhost:8080/health || exit 1

# 入口
COPY --from=builder /app/target/*.jar app.jar
ENTRYPOINT ["java", "-XX:+UseContainerSupport", "-XX:MaxRAMPercentage=75.0", "-jar", "app.jar"]
```

关键点：

- 使用多阶段构建，减小镜像体积（从 800MB 降到 200MB）
- 使用非 root 用户运行容器
- 添加健康检查，确保 K8s 知道容器是否正常
- 使用 `-XX:+UseContainerSupport` 让 JVM 自动感知容器资源限制

### 3. 配置迁移

原来的 WAS 配置需要迁移到 Kubernetes ConfigMap：

```yaml title="Kubernetes ConfigMap"
apiVersion: v1
kind: ConfigMap
metadata:
  name: order-service-config
data:
  application.yml: |
    server:
      port: 8080
    spring:
      datasource:
        url: jdbc:mysql://mysql:3306/orders
        username: ${DB_USERNAME}
        password: ${DB_PASSWORD}
      redis:
        host: redis
        port: 6379
  log-level: INFO
---
apiVersion: v1
kind: Secret
metadata:
  name: order-service-secrets
type: Opaque
data:
  # echo -n "password" | base64
  db-password: cGFzc3dvcmQ=
  api-key: YXBpLWtleS1oZXJl
```

### 4. GitOps 部署

团队选择了 ArgoCD 实现 GitOps：

```yaml title="ArgoCD Application 配置"
apiVersion: argoproj.io/v1alpha1
kind: Application
metadata:
  name: order-service
  namespace: argocd
spec:
  project: production
  source:
    repoURL: https://github.com/company/k8s-configs
    targetRevision: main
    path: production/order-service
  destination:
    server: https://kubernetes.default.svc
    namespace: production
  syncPolicy:
    automated:
      prune: true          # 自动删除旧资源
      selfHeal: true        # 自动同步到期望状态
    syncOptions:
      - CreateNamespace=true
```

所有 K8s 资源文件存在 Git 仓库中，ArgoCD 自动检测仓库变化并同步到集群。发布变成了简单的 `git push`。

## 第三阶段：灰度迁移（8 个月）

### 1. 第一批迁移：云就绪应用

云就绪的 30 个应用是第一批迁移目标，因为它们不需要代码改造。

迁移流程：

```
1. 在测试环境部署新版本 Kubernetes 工作负载
2. 测试环境验证（功能测试、性能测试）
3. 预生产环境部署（与生产并行，不接收流量）
4. 生产环境灰度：1% 流量到 K8s 版本
5. 逐步扩大灰度：10% → 30% → 50% → 100%
6. 旧环境下线（保留 2 周应急回退）
```

### 2. 遇到的问题及解决方案

**问题一：健康检查导致频繁重启**

```yaml
# 问题：应用启动时间长（30 秒），K8s 探活失败导致容器重启
# 解决方案：调整探活参数

livenessProbe:
  httpGet:
    path: /health
    port: 8080
  initialDelaySeconds: 60    # 启动 60 秒后才开始探活
  periodSeconds: 10
  failureThreshold: 3

readinessProbe:
  httpGet:
    path: /health
    port: 8080
  initialDelaySeconds: 30    # 启动 30 秒后开始探活
  periodSeconds: 5
  failureThreshold: 3
```

**问题二：OOMKilled**

```bash
# 问题：应用被 Kubernetes 杀掉（OOMKilled），日志显示 exit code 137
# 原因：JVM 堆大小配置超过了容器内存限制

# 原配置：
# JVM_ARGS: "-Xmx2g"  # 容器内存限制是 1Gi，JVM 堆就占满了

# 解决方案：使用容器感知参数
# JVM_ARGS: "-XX:+UseContainerSupport -XX:MaxRAMPercentage=75.0"
# 这样 JVM 会自动检测容器内存限制，设置为 75%
```

**问题三：服务间调用超时**

```yaml
# 问题：迁移后部分服务间调用超时
# 原因：Kubernetes 内部 DNS 解析比预期慢 2~5ms
# 解决方案：调整 Feign 超时配置

feign:
  client:
    config:
      default:
        connectTimeout: 2000      # 从 1000ms 增加到 2000ms
        readTimeout: 5000         # 从 3000ms 增加到 5000ms
```

### 3. 第二批迁移：云适配应用

云适配应用需要代码改造，这部分工作更复杂。团队采用了并行运行策略：新版本同时支持物理机部署和 K8s 部署，用相同的代码，通过环境变量区分部署方式。

```java
// 环境适配代码
@Service
public class StorageService {

    @Value("${storage.type:local}")
    private String storageType;

    public String upload(File file) {
        if ("oss".equals(storageType)) {
            // 云上：使用 OSS
            return ossService.upload(file);
        } else {
            // 物理机：使用本地存储
            return localStorageService.upload(file);
        }
    }
}
```

## 第四阶段：收尾与优化（4 个月）

### 1. 遗留应用处理

最终有 5 个应用无法迁移到云上，原因包括：

- **合规问题**：金融监管要求数据不能出省，云上跨可用区部署不满足要求
- **硬件依赖**：依赖专用加密卡，云上无法提供
- **技术债过高**：应用架构过于老旧，改造代价超过重写

对于这些应用，团队采用了混合云方案：在私有云保留这些应用，通过 API 网关对外统一提供服务。

### 2. 成本优化

上云后发现了新的成本问题：有些应用申请的资源远超实际使用。

```yaml
# 资源配置优化
resources:
  requests:
    memory: "256Mi"     # 从 512Mi 降到 256Mi
    cpu: "100m"         # 从 200m 降到 100m
  limits:
    memory: "512Mi"     # 从 1Gi 降到 512Mi
    cpu: "500m"         # 从 1000m 降到 500m
```

通过 VPA（Vertical Pod Autoscaler）自动调整资源配置，三个月内将平均资源使用率从 25% 提升到了 45%。

### 3. 建立 SRE 能力

云原生环境需要新的运维能力。团队引入了 SRE（Site Reliability Engineering）机制：

```
SRE 核心工作：
- 服务可用性目标（SLO）：每个服务定义可用性目标
- 错误预算：允许的服务不可用时间
- 告警阈值：基于 SLO 计算，预警而非报警
- 事后复盘：所有 P1/P2 故障必须复盘
```

## 最终结果

18 个月后，迁移完成了 80%。

| 指标 | 迁移前 | 迁移后 | 变化 |
| --- | --- | --- | --- |
| 资源利用率（CPU） | 15% | 45% | +200% |
| 扩容时间 | 2 周 | 5 分钟 | -99.6% |
| 发布频率 | 每月 1 次 | 每周 3 次 | +300% |
| MTTR | 30 分钟 | 5 分钟 | -83% |
| IT 基础设施成本 | 2000 万/年 | 1400 万/年 | -30% |
| 服务可用性 | 99.5% | 99.95% | +0.45% |

```

- 人员成本：15 人 × 18 个月 = 270 人月
- 培训成本：约 50 万
- 第三方支持：约 100 万（咨询费��认证费）
```

## 总结：云原生迁移的核心认知

回顾这次 18 个月的迁移，有几个关键认知：

**第一，基础设施先行，应用迁移在后**。Kubernetes 集群、CI/CD、GitOps、日志和监控这些基础设施没有就绪之前，不要开始迁移应用。没有这些基础设施，云原生只是换了一个地方跑单体应用。

**第二，评估和规划比执行更重要**。迁移前的应用分类、风险评估、改造清单，直接决定了迁移的可行性和工作量。很多迁移项目延期，都是因为低估了改造工作量和风险。

**第三，灰度迁移是控制风险的关键**。不要一次性全量迁移。从测试环境到预生产环境，从 1% 流量到 100% 流量，每一步都要有验证和回退机制。

**第四，成本控制是迁移后的持续工作**。云上成本是可变的，需要持续监控和优化。VPA、自动扩缩容、Spot 实例等手段，都是降低云成本的有效方法。

**第五，上云不是终点，是起点**。完成迁移只是第一步，建立 SRE 能力、实现持续改进、防止技术债再次积累，才是云原生迁移的长期目标。
