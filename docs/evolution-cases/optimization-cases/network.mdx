# 网络调优

某电商平台的订单服务调用商品服务获取商品信息，正常情况下延迟约 5ms。但有一天，监控发现商品服务的平均延迟突然从 5ms 飙升到 200ms+，订单服务的整体响应时间也随之恶化。

DBA 查了数据库，说数据库没问题。商品服务团队查了自己的代码，说没有耗时操作。运维查了服务器 CPU 和内存，都正常。

最后发现问题出在**订单服务和商品服务之间的 HTTP 连接池配置上**——连接池太小，在高并发下大量请求在等待连接，而不是在处理请求。这个问题的表象是「商品服务慢」，而真正的根因在网络连接层。

这一节用这个真实案例，完整还原一次网络性能问题的排查和解决过程。

## 问题描述

### 现象

```
订单服务 → 商品服务调用链路
正常情况：延迟约 5ms
异常情况：延迟 200ms~500ms，批量请求时有大量超时
影响：订单服务 p99 延迟从 50ms 上升到 500ms+
```

### 监控数据

```
商品服务：
- CPU 使用率：30%（正常）
- 内存使用率：40%（正常）
- 数据库查询时间：3ms（正常）
- 响应时间：200ms+（异常）

订单服务：
- 调用商品服务时，大量请求处于 WAITING 状态
- 等待时间：195ms（200ms 延迟减去 5ms 处理时间）
```

**等待时间 195ms**——这说明问题不是商品服务处理慢，而是**订单服务在等待获取连接**。

## 第一轮排查：确认等待时间

### 在应用层记录等待时间

```java
// Feign 拦截器：记录请求等待时间
@Component
public class FeignMetricsInterceptor implements RequestInterceptor {

    @Override
    public void apply(RequestTemplate template) {
        // 在请求开始时记录时间
        template.header("X-Request-Start-Time", String.valueOf(System.currentTimeMillis()));
    }
}

// 响应日志拦截器
@Component
public class FeignResponseInterceptor implements ResponseInterceptor {

    @Override
    public Response intercept(Response response) {
        long startTime = Long.parseLong(
            response.request().headers().get("X-Request-Start-Time").iterator().next());
        long duration = System.currentTimeMillis() - startTime;
        long networkDelay = duration - response.request().requestTemplate().url().length();

        if (duration > 50) {
            log.warn("慢 Feign 调用：{}ms，目标：{}",
                duration, response.request().url());
        }

        return response;
    }
}
```

监控数据证实了猜测：**商品服务的实际处理时间是 5ms，但订单服务等待连接的时间是 195ms**。

## 第二轮排查：连接池分析

### 查看连接池状态

```bash
# 查看当前 TCP 连接状态
ss -s
# Total: 150 (kernel 180)
# TCP:   45 (estab 20, closed 5, orphaned 0, synrecv 0, timewait 10)

# 查看与商品服务的连接数
ss -tnp | grep :8081 | wc -l
# 50 个连接，全部是 ESTABLISHED 状态

# 查看连接池配置
# 订单服务的 HTTP 连接池配置：
# maxTotal = 50
# maxPerRoute = 50
# 当前并发请求数：300
```

问题清楚了：**连接池最大 50 个连接，但高峰期并发请求数达到 300**。每个请求要等前面的请求释放连接，平均等待时间 = 50 个连接的释放周期。假设每个请求处理时间 5ms，50 个连接的释放周期就是 50 × 5ms = 250ms。

### 查看连接池配置

```java
// 订单服务当前的 HTTP 客户端配置
@Configuration
public class HttpClientConfig {

    @Bean
    public CloseableHttpClient httpClient() {
        PoolingHttpClientConnectionManager cm = new PoolingHttpClientConnectionManager();
        cm.setMaxTotal(50);          // 最大连接数 50
        cm.setDefaultMaxPerRoute(50);  // 每个路由最大 50

        RequestConfig config = RequestConfig.custom()
            .setConnectTimeout(3000, TimeUnit.MILLISECONDS)
            .setSocketTimeout(5000, TimeUnit.MILLISECONDS)
            .setConnectionRequestTimeout(2000, TimeUnit.MILLISECONDS)  // [!code highlight] 等待连接超时 2 秒
            .build();

        return HttpClients.custom()
            .setConnectionManager(cm)
            .setDefaultRequestConfig(config)
            .build();
    }
}
```

`connectionRequestTimeout = 2000ms`——当等待连接超过 2 秒时，请求会直接抛出 `ConnectionPoolTimeoutException`。这就是为什么高峰期有大量超时。

## 第一轮优化：扩大连接池

```java
// 优化后的 HTTP 客户端配置
@Configuration
public class HttpClientConfig {

    @Bean
    public CloseableHttpClient httpClient() {
        PoolingHttpClientConnectionManager cm = new PoolingHttpClientConnectionManager();

        // 根据预估 QPS 计算连接数
        // 假设商品服务单次调用 5ms，最大可支持 QPS = 连接数 / 5ms
        // 如果目标 QPS = 1000，则需要连接数 = 1000 × 5ms = 50
        // 考虑一定的冗余，设置为 200
        cm.setMaxTotal(200);           // 最大连接数 200
        cm.setDefaultMaxPerRoute(100); // 每个路由最大 100

        // 连接保活
        cm.setValidateAfterInactivity(2000);  // 空闲 2 秒后验证连接有效性

        RequestConfig config = RequestConfig.custom()
            .setConnectTimeout(1000, TimeUnit.MILLISECONDS)
            .setSocketTimeout(3000, TimeUnit.MILLISECONDS)
            .setConnectionRequestTimeout(1000, TimeUnit.MILLISECONDS)  // 等待连接超时 1 秒
            .build();

        return HttpClients.custom()
            .setConnectionManager(cm)
            .setDefaultRequestConfig(config)
            .setRetryHandler(new DefaultHttpRequestRetryHandler(3, true))
            .build();
    }
}
```

**优化后结果**：延迟从 200ms 降到了 15ms。但在大促峰值时，仍然偶发 50ms 级别的延迟。

## 第二轮排查：超时配置问题

### 问题复现

大促期间，商品服务的处理时间从 5ms 上升到了 15ms（因为数据库负载增加了）。这看似不大的变化，却触发了新的问题：

```
商品服务正常处理时间：5ms
连接池大小：200
最大支持 QPS：200 / 5ms = 40000

商品服务慢时处理时间：15ms
连接池大小：200
最大支持 QPS：200 / 15ms = 13333

大促峰值 QPS：15000
结果：15000 > 13333，连接不够用
```

### 问题的本质

`SocketTimeout = 3000ms`——当商品服务处理变慢时（从 5ms 到 15ms），每个连接能处理的请求数减少了三倍。但连接池大小没变，高峰期就会出现连接不够用的情况。

## 第二轮优化：超时 + 重试 + 降级

### 优化一：更合理的超时配置

```java
// 超时配置原则：
// connectTimeout：建立连接的时间，一般 1 秒足够
// socketTimeout：单个请求的处理时间，根据服务 SLA 设置
// connectionRequestTimeout：等待从连接池获取连接的时间

RequestConfig config = RequestConfig.custom()
    .setConnectTimeout(1, TimeUnit.SECONDS)
    .setSocketTimeout(5, TimeUnit.SECONDS)       // 商品服务 SLA 是 5 秒
    .setConnectionRequestTimeout(500, TimeUnit.MILLISECONDS)  // 500ms 内获取不到连接就放弃
    .build();
```

### 优化二：熔断 + 降级

```java
// 使用 Resilience4j 实现熔断
@Service
public class ProductClientService {

    private CircuitBreaker circuitBreaker;

    @Autowired
    private ProductFeignClient productFeignClient;

    @PostConstruct
    public void init() {
        circuitBreaker = CircuitBreaker.of("productService",
            CircuitBreakerConfig.custom()
                .failureRateThreshold(50)           // 50% 失败率触发熔断
                .slowCallRateThreshold(80)         // 80% 慢调用触发熔断
                .slowCallDurationThreshold(3, TimeUnit.SECONDS)  // 超过 3 秒为慢调用
                .slidingWindowSize(10)             // 滑动窗口 10 次调用
                .slidingWindowType(SlidingWindowType.COUNT_BASED)
                .minimumNumberOfCalls(5)           // 至少 5 次调用才判断
                .waitDurationInOpenState(30, TimeUnit.SECONDS)  // 熔断 30 秒后半开
                .permittedNumberOfCallsInHalfOpenState(3)  // 半开状态允许 3 次调用
                .build());
    }

    public Product getProduct(Long productId) {
        try {
            // 尝试从缓存获取
            Product cached = productCache.get(productId);
            if (cached != null) {
                return cached;
            }

            // 调用商品服务，熔断保护
            Supplier<Product> supplier = CircuitBreaker
                .decorateSupplier(circuitBreaker,
                    () -> productFeignClient.getProduct(productId));

            Product product = supplier.get();
            productCache.put(productId, product);
            return product;

        } catch (CallNotPermittedException e) {
            // 熔断触发，返回降级数据
            log.warn("商品服务熔断，返回降级数据: {}", productId);
            return getDegradedProduct(productId);

        } catch (Exception e) {
            // 其他异常，返回降级数据
            return getDegradedProduct(productId);
        }
    }

    private Product getDegradedProduct(Long productId) {
        // 返回基础商品信息，不包含实时库存等动态数据
        Product degraded = new Product();
        degraded.setId(productId);
        degraded.setName("商品（暂时不可用）");
        degraded.setPrice(BigDecimal.ZERO);
        return degraded;
    }
}
```

### 优化三：连接池自适应

```java
// 根据负载动态调整连接池大小
@Aspect
@Component
public class ConnectionPoolAutoTuningAspect {

    private final PoolingHttpClientConnectionManager connectionManager;

    @Scheduled(fixedRate = 60000)  // 每分钟检查一次
    public void autoTuneConnectionPool() {
        PoolStats stats = connectionManager.getTotalStats();

        int leased = stats.getLeased();     // 已借出
        int pending = stats.getPending();   // 等待中
        int available = stats.getAvailable(); // 可用
        int max = stats.getMax();

        double utilization = (double) leased / max;

        if (utilization > 0.8 && pending > 0) {
            // 利用率超过 80%，且有等待请求，扩容
            int newMax = Math.min(max * 2, 500);
            connectionManager.setMaxTotal(newMax);
            log.info("连接池扩容：{} -> {}", max, newMax);
        }

        if (utilization < 0.3 && max > 50) {
            // 利用率低于 30%，缩容
            int newMax = Math.max(max / 2, 50);
            connectionManager.setMaxTotal(newMax);
            log.info("连接池缩容：{} -> {}", max, newMax);
        }
    }
}
```

## 第三轮优化：DNS 缓存与 HTTPDNS

还有一个隐藏的网络问题是 **DNS 解析延迟**。

### 问题

每次调用商品服务时，Feign 需要将域名解析为 IP 地址。默认情况下，DNS 缓存时间由 JVM 参数 `-Dnetworkaddress.cache.ttl` 控制，默认为 30 秒。在服务节点变更或负载均衡调整时，DNS 缓存可能导致请求打到已经下线的节点上。

### 解决方案：HTTPDNS

```java
// 使用阿里云 HTTPDNS 替代默认 DNS
@Configuration
public class HttDnsConfig {

    @Bean
    public DnsResolver dnsResolver() {
        // HTTPDNS 绕过系统 DNS，避免 DNS 缓存问题
        return new DnsResolver() {
            @Override
            public InetAddress[] resolve(String host) throws UnknownHostException {
                try {
                    // 调用 HTTPDNS API 获取 IP 列表
                    HttpGet request = new HttpGet(
                        "http://203.107.1.1/d?host=" + host);
                    CloseableHttpClient client = HttpClients.createDefault();
                    CloseableHttpResponse response = client.execute(request);
                    String body = EntityUtils.toString(response.getEntity());
                    client.close();

                    // 解析返回的 IP 列表
                    String[] ips = body.split(";");
                    InetAddress[] addresses = new InetAddress[ips.length];
                    for (int i = 0; i < ips.length; i++) {
                        addresses[i] = InetAddress.getByName(ips[i].trim());
                    }
                    return addresses;

                } catch (Exception e) {
                    // 降级到系统 DNS
                    return InetAddress.getAllByName(host);
                }
            }
        };
    }
}
```

## 网络层内核参数优化

除了应用层，还需要优化操作系统层面的网络参数。

```bash
# /etc/sysctl.conf 网络优化

# 1. TIME_WAIT 复用
net.ipv4.tcp_tw_reuse = 1           # 允许重用 TIME_WAIT 连接
net.ipv4.tcp_fin_timeout = 15        # FIN 超时时间

# 2. 连接队列
net.core.somaxconn = 65535           # 服务器端 socket 队列最大长度
net.ipv4.tcp_max_syn_backlog = 65535 # SYN 队列长度

# 3. 文件描述符
fs.file-max = 2097152                 # 系统最大文件描述符数

# 4. TCP 缓冲区
net.core.rmem_default = 262144
net.core.rmem_max = 16777216
net.core.wmem_default = 262144
net.core.wmem_max = 16777216
net.ipv4.tcp_rmem = 4096 87380 16777216
net.ipv4.tcp_wmem = 4096 65536 16777216

# 5. 开启 BBR（拥塞控制算法）
net.core.default_qdisc = fq
net.ipv4.tcp_congestion_control = bbr

# 生效配置
sysctl -p
```

## 网络监控体系

网络问题往往来得突然，监控是最好的预警手段。

```yaml
# Prometheus 网络监控指标
groups:
  - name: network_health
    rules:
      # TCP 连接数
      - alert: TcpConnectionHigh
        expr: node_netstat_Tcp_CurrEstab > 10000
        annotations:
          summary: "TCP Established 连接数超过 10000"

      # TIME_WAIT 连接堆积
      - alert: TcpTimeWaitHigh
        expr: node_netstat_Tcp_TimeWait > 50000
        annotations:
          summary: "TIME_WAIT 连接数超过 50000，可能需要调优"

      # 丢包率
      - alert: PacketLossHigh
        expr: rate(node_network_receive_errors[5m]) > 0.01
        annotations:
          summary: "网卡丢包率异常"

      # Feign 调用延迟
      - alert: FeignLatencyHigh
        expr: histogram_quantile(0.99,
            rate(feign_request_duration_seconds_bucket[5m])) > 1
        annotations:
          summary: "Feign 调用 p99 延迟超过 1 秒"
```

## 总结

这次网络性能优化的过程，有几个关键认知：

**第一，网络问题的表象往往指向错误的方向**。「商品服务慢」的表象，导致团队花了很多时间查商品服务的代码和数据库。真正的问题是订单服务的连接池——不是被调用方慢，而是调用方的等待时间长。

**第二，连接池是分布式系统的隐形瓶颈**。在低并发下，连接池够用；在高并发下，连接池大小决定了系统的吞吐量上限。连接池配置必须根据实际 QPS 计算，而不是拍脑袋设置。

**第三，超时配置是容错的第一道防线**。合理的超时配置 + 重试机制，可以在依赖方故障时保护调用方不被拖垮。超时太长会积累请求，超时太短会误判正常请求为失败。

**第四，网络层优化需要应用层和系统层协同**。应用层的连接池配置、熔断策略、超时策略，加上系统层的内核参数优化，才能构建完整的网络容错体系。
