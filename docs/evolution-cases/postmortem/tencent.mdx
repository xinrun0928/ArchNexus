# 腾讯服务雪崩事件

## 事故概述

2020 年，腾讯内部某服务经历了一次典型的**级联雪崩（Cascading Failure）** 事故。事件起因是一次常规的系统压测，测试环境的流量意外地被路由到了生产环境。生产服务在毫无防备的情况下承受了数倍的预期流量，导致核心接口响应时间急剧增加。雪球越滚越大，最终整个服务集群几乎完全不可用，恢复时间超过 4 小时。

这类事故在大型互联网公司中并不罕见。其共性特征是：初始问题（流量突增）→ 部分服务响应变慢 → 调用方超时重试 → 下游压力加倍 → 资源耗尽 → 全面崩溃。雪崩的发生往往只需要几分钟，但排查和恢复可能需要数小时。

## 影响范围

事故影响呈级联扩散态势：

- **核心 API 服务不可用**：用户请求大量超时，核心业务功能完全失效
- **依赖服务被拖垮**：下游的数据库服务、缓存服务因连接数激增而告警
- **前端页面降级**：移动端 App 出现「网络异常」「请求超时」等错误提示
- **客服工单激增**：用户反馈集中涌入，客户服务团队压力骤增

事故持续约 4.5 小时，期间经历了多次恢复尝试失败。团队在尝试扩容时发现，新启动的实例很快也被涌入的请求淹没，根本原因在于上游的流量没有有效控制。

## 根因分析

**超时设置不当**是此次雪崩的起点。当下游服务响应变慢时，如果上游服务的超时时间设置过长，请求会在内存中堆积等待，消耗宝贵的线程/协程资源。当所有可用资源都被耗尽后，服务彻底失去响应能力。很多团队为了「避免误报」，将超时时间设置得很长（如 30 秒甚至更长），这个做法在正常情况下可能没问题，但一旦下游出现问题，长超时反而会成为压垮系统的最后一根稻草。

**缺乏熔断机制**加剧了故障扩散。当某个依赖服务不可用或响应异常时，调用方应该立即「熔断」对这个服务的调用，快速失败并返回降级响应，而不是持续等待或重试。事故中的服务对所有下游调用都采用了相同的重试策略，导致对已经异常的下游服务的重试请求进一步加重了系统负担。

**容量预估缺失**使系统在面对突发流量时毫无防备。压测流量意外进入生产环境，说明隔离机制失效。同时，系统在设计时可能没有充分考虑流量峰值的上限，导致实际流量超过设计容量时没有任何保护措施。

**线程池隔离不足**：不同优先级的请求共用同一个线程池，当低优先级请求（如预热请求）阻塞时，高优先级的核心请求也被影响。应该按照服务调用关系划分独立的线程池，防止故障在服务间扩散。

## 改进措施

**合理设置超时并启用熔断**：为每个下游调用设置合理的超时时间（建议根据 P99 延迟的 2-3 倍设定），并启用熔断器（Circuit Breaker）。当失败率超过阈值时，熔断器打开，后续请求直接返回降级响应而非实际调用。经过一段冷却时间后，熔断器进入半开状态，允许一个试探性请求通过，如果成功则恢复正常，否则继续熔断。

**实现多级限流机制**：

- **入口限流**：在 API Gateway 层面限制单个 IP 或用户的请求频率
- **服务间限流**：使用令牌桶或滑动窗口算法限制每个服务的 QPS
- **数据库连接池限流**：确保不会因为突发流量耗尽数据库连接

**服务隔离与降级**：将核心业务和非核心业务隔离到不同的线程池或服务实例中。当非核心服务出现问题时，只影响自身，不影响核心业务。同时为每个服务定义降级方案——当依赖服务不可用时，返回预设的缓存数据或友好提示。

**建立流量隔离机制**：严格区分测试环境和生产环境的网络配置，确保压测流量不会进入生产环境。对于跨环境的流量，应该在边界处设置硬阻断。

**定期进行混沌工程演练**：主动在测试环境中模拟服务超时、实例宕机、网络分区等故障，验证系统的容错能力是否如预期工作。很多公司正是通过混沌工程发现了熔断机制失效、限流配置不合理等问题。
