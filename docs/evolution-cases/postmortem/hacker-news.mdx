# Hacker News 数据库宕机事件

## 事故概述

Hacker News（简称 HN）是 Y Combinator 旗下的科技新闻聚合网站，基于 PostgreSQL 构建。HN 曾多次因数据库负载过高而出现服务降级或不可用。其中最严重的一次发生在 2016 年 3 月，**网站服务完全中断超过 24 小时**，用户无法访问任何内容。

HN 使用的数据库是标准的 PostgreSQL，托管在单台服务器上。网站流量虽然在互联网产品中算不上特别大，但在热门新闻发布期间，访问量会在短时间内急剧攀升，加上 HN 特有的投票和评论机制对数据库的读写压力，数据库很容易成为瓶颈。

## 影响范围

事故期间：

- **网站完全不可用**：用户访问 hn.algolia.com 或 news.ycombinator.com 均返回错误
- **API 服务中断**：依赖 HN API 的第三方应用和工具无法获取数据
- **持续时间长**：服务中断超过 24 小时，期间多次尝试恢复均告失败
- **社区讨论热烈**：HN 的用户在社交媒体上热烈讨论此次宕机，甚至 HN 自身的评论区也因为宕机无法使用

## 根因分析

**Web 型数据库架构的局限性**是 HN 频繁宕机的根本原因。HN 的数据库架构长期停留在「单数据库」模式，没有采用读写分离、主从复制等常见的数据库扩展方案。当流量增加时，数据库的 CPU 和 IO 成为瓶颈，无法通过简单地增加硬件来解决问题。

**索引维护与查询优化缺失**：HN 的核心查询包括「获取最新 N 条新闻」「获取某条新闻的评论树」「按分数排序的热门新闻」等。这些查询在数据量较小时执行很快，但随着数据增长，缺少优化的查询会导致全表扫描。以评论查询为例，HN 使用递归 CTE（Common Table Expression）来获取评论树，当评论层级较深时，这个查询的执行时间会急剧增加。

**缺乏有效的限流和熔断机制**：当数据库接近过载时， HN 没有在应用层或数据库代理层实现请求限流。所有请求都被无差别地发送到数据库，导致数据库雪崩式地拒绝请求。

**容量规划不足**：HN 团队对流量峰值缺乏准确的预估。当某条新闻突然爆火时，短时间内大量用户涌入，数据库毫无防备地承受冲击。好的做法是建立流量预测模型，提前扩容或开启缓存保护。

## 改进措施

**读写分离架构**：将读请求（新闻列表、评论展示）和写请求（投票、评论）分离。写操作发往主库，读操作可以分散到多个从库。HN 在事故后确实进行了架构升级，采用了类似读写分离的方案。

**引入缓存层**：对热门新闻列表、用户信息等高频访问的数据进行缓存（Redis 或 Memcached），减少数据库查询压力。HN 的很多页面实际上可以完全通过缓存响应，数据库只负责处理写操作和缓存刷新。

**数据库查询优化**：为高频查询创建合适的索引，必要时使用物化视图（Materialized View）预计算热门列表。定期审查慢查询日志，针对性地进行优化。

**应用层限流保护**：在数据库代理层（如 PgBouncer）或应用层实现限流，当数据库负载过高时主动拒绝部分请求而非让数据库在过载状态下崩溃。这是一种有损保护，但能避免完全不可用的最坏情况。

**流量监控与告警**：建立数据库负载监控，当 CPU、连接数、查询队列长度等指标超过阈值时提前告警，在用户感知到问题之前采取措施。
