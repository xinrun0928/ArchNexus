# GitHub 数据库迁移降级事故

## 事故概述

2018 年 10 月，GitHub 经历了一次重大服务降级事故，持续时间约 24 小时。事故的起因是一次看似常规的**数据库 Schema 变更（DDL 操作）**：为了给 `表A` 添加一个新索引，运维团队在主库上执行了一条 `ALTER TABLE` 语句。就是这个操作导致了 GitHub 整个平台的服务降级。

## 影响范围

事故期间，GitHub 的多项核心功能受到影响：

- **Git 操作降级**：代码推送（push）、拉取（pull）出现长时间等待或失败
- **Web 界面响应缓慢**：页面加载时间显著增加，部分请求超时
- **API 服务异常**：第三方集成和 API 调用失败率上升
- **CI/CD 服务中断**：GitHub Actions 和 Travis CI 等依赖 GitHub Webhook 的服务受到影响

受影响的用户覆盖全球开发者和企业客户，包括使用 GitHub 作为主要代码托管平台的众多科技公司。事故在社交媒体上引发了大量讨论，GitHub 的状态页面在整个事件期间持续更新。

## 根因分析

**DDL 操作阻塞写事务**是此次事故的直接原因。在 MySQL 的默认配置下，`ALTER TABLE` 操作会阻塞该表的所有写操作（INSERT/UPDATE/DELETE）。当在包含数亿行数据的主表上执行 `ALTER TABLE` 时，阻塞时间可能持续数分钟到数小时，期间所有依赖该表的写操作都会排队等待。

**主从复制延迟加剧问题**：GitHub 的数据库架构采用主从复制。主库上的 DDL 操作需要同步到多个从库，而从库在重放 DDL 时同样会出现短暂的不可用窗口。当多个从库同时处理 DDL 时，复制延迟会显著增加，进一步扩大了影响范围。

**变更窗口评估不足**：团队在评估变更影响时，可能低估了在大表上添加索引所需的时间。`ALGORITHM=INPLACE` 或 `ALGORITHM=COPY` 的选择、`LOCK` 参数的配置都会显著影响 DDL 的执行行为和阻塞时间。

**缺乏灰度验证机制**：变更直接应用到生产主库，缺少预发布环境的真实流量验证。如果在预发布环境先执行同样操作，可以提前发现潜在问题并制定应对方案。

## 改进措施

**使用 pt-online-schema-change 或 gh-ost**：这类工具通过创建影子表、复制数据、原子切换的方式实现大表 DDL，不直接锁表。gh-ost 由 GitHub 自行开发，正是在此次事故后开源并被广泛采用。

**采用蓝绿部署策略**：准备一组完全同步的新数据库实例，在新实例上完成所有 Schema 变更，验证通过后切换流量。这种方式将变更窗口从「在线变更」转变为「离线准备+瞬时切换」。

**强制变更评审流程**：所有涉及大表的 DDL 必须经过数据库管理员（DBA）评审，评估影响范围、执行时间、所需资源，制定回滚方案后方可执行。

**建立数据库变更灰度验证机制**：在生产环境执行变更前，先在隔离的验证环境使用真实流量进行测试，确保变更不会对应用层产生负面影响。

**完善监控告警**：对数据库的 DDL 执行时间、写操作队列长度、复制延迟等关键指标设置告警，当指标异常时及时人工介入。
