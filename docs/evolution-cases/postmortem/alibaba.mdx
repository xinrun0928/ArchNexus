# 阿里云光缆故障事件

## 事故概述

2019 年 3 月 3 日凌晨，阿里云位于华北 2 可用区 B 的部分云服务出现访问异常，故障持续约 3 小时。经调查确认，事故原因是**合作运营商的施工意外导致一条骨干光缆被挖断**。这条光缆承载着华北 2 可用区与其他可用区及互联网骨干网之间的主要通信链路。

故障发生后，阿里云立即启动应急预案，尝试通过备用链路恢复网络连接，但由于主要链路的流量无法完全迁移到备用链路，核心服务的可用性受到了严重影响。

## 影响范围

此次故障的影响范围包括：

- **云产品中断**：ECS 云服务器、RDS 数据库、OSS 对象存储、SLB 负载均衡等核心产品出现不同程度的访问异常
- **服务不可用**：大量部署在该可用区的在线业务出现页面无法访问、API 调用超时等问题
- **地域差异**：华北地区用户受影响最为严重，其他地域通过骨干网访问阿里云华北节点的服务也出现延迟增加或超时

据阿里云官方公告，故障期间华北 2 可用区的云服务可用性下降至约 70%，受影响业务数量达到数万家。故障持续约 3 小时后才完全恢复，期间阿里云通过微博、官网公告等方式持续更新故障进展。

## 根因分析

**单链路依赖**是此次故障的根本原因。尽管阿里云在可用区层面实现了基础设施的冗余设计，但可用区之间的互联链路以及与互联网骨干网的连接，**在某些路由方向上存在单点依赖**。当主要光缆被挖断后，备用链路无法承载全部流量，导致网络拥塞和服务降级。

**容灾演练不足**加剧了故障的持续时间。备用链路的设计容量是否经过真实故障场景的验证存疑。如果平时没有进行过「主要链路中断」的切换演练，实际故障时才发现备用链路容量不足，将导致宝贵的恢复时间被浪费。

**告警响应延迟**也值得关注。从光缆被挖断到故障被用户感知，中间存在一定时间差。这说明网络层的监控告警可能没有覆盖到所有关键链路的可用性指标。

## 改进措施

**多链路冗余与自动切换**：在主要链路上增加备用链路的数量，并实现网络流量的自动切换。当主链路中断时，流量应能在秒级自动切换到备用链路，而非依赖人工干预。

**定期容灾演练**：建立网络层面的故障演练机制，定期模拟主要链路中断场景，验证备用链路的切换能力和容量配置。演练应该覆盖不同场景：单链路中断、多链路中断、跨地域切换等。

**网络监控覆盖度提升**：将监控覆盖到所有关键网络节点和链路，增加主动探测机制，及时发现网络质量下降而非等到完全中断才告警。

**多可用区架构的必要性**：对于核心业务，应该充分利用云厂商提供的多可用区部署能力，将服务分散部署在不同可用区，避免单可用区故障导致全量服务不可用。应用程序应具备自动切换可用区的能力，而非假设单个可用区始终可用。
