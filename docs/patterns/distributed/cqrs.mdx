# CQRS 模式

你的电商系统，随着业务增长，读写压力都在增加。产品详情页 QPS 10 万，订单写入 QPS 只有 1000。但数据库只有一套，读写互相影响。

你可能想到读写分离：主库写入，从库读取。但这还不够——**读和写的需求差异越来越大**：写需要事务保证，读需要复杂查询；写只需要几个字段，读需要几十个字段的聚合。

CQRS 模式正是为了解决这个问题：**把命令（写）和查询（读）完全分开，使用不同的模型**。

## 问题背景

### 传统 CRUD 的问题

```java
// 传统做法：一个模型服务所有场景
public class Order {
    private Long id;
    private String orderNo;
    private Long userId;
    private List<OrderItem> items;
    private BigDecimal totalAmount;
    private OrderStatus status;
    // ... 50+ 字段
}

public class OrderService {
    // 写入
    public Order createOrder(CreateOrderRequest request) {
        Order order = new Order();
        // 映射、校验、保存
        return orderRepository.save(order);
    }

    // 查询 1：订单列表（分页）
    public Page<Order> listOrders(Long userId, int page, int size) {
        return orderRepository.findByUserId(userId, PageRequest.of(page, size));
    }

    // 查询 2：订单详情（包含商品信息）
    public OrderDetailDTO getOrderDetail(Long orderId) {
        // 需要 join 多张表
        return orderRepository.findDetailById(orderId);
    }

    // 查询 3：订单统计（销售额、订单数）
    public OrderStatistics getStatistics(Long userId, Date startDate, Date endDate) {
        // 复杂的聚合查询
        return orderRepository.getStatistics(userId, startDate, endDate);
    }
}
```

问题分析：

- 一个 `Order` 模型承担了所有职责
- 读和写的需求差异大，互相牵制
- 读需要 join、聚合，写需要事务保证
- 无法针对读写分别优化

## CQRS 模式的核心

```
核心思想：
┌──────────────────────────────────────────────────────────────────┐
│                         CQRS 模式                               │
├──────────────────────────────────────────────────────────────────┤
│                                                                  │
│   写路径（Command）              读路径（Query）                  │
│   ┌────────────────┐           ┌────────────────┐              │
│   │   Command       │           │   Query        │              │
│   │   Handler       │           │   Handler      │              │
│   └───────┬────────┘           └───────┬────────┘              │
│           │                              │                      │
│           ▼                              ▼                      │
│   ┌────────────────┐           ┌────────────────┐              │
│   │   Domain       │           │   Read Model   │              │
│   │   Model        │ ────────▶ │   (视图)       │              │
│   │   (领域模型)   │  同步     │                │              │
│   └────────────────┘           └────────────────┘              │
│                                                                  │
│   写入优化                 读取优化                              │
│   - 事务                    - 索引                               │
│   - 校验                    - 聚合                               │
│   - 业务规则                - 反范式化                            │
│                                                                  │
└──────────────────────────────────────────────────────────────────┘
```

**关键特点**：

1. **分离读写模型**：写用领域模型，读用物化视图
2. **不同存储**：写用 OLTP 数据库，读用 OLAP 数据库或专门结构
3. **不同接口**：命令接口和查询接口分离
4. **最终一致**：写后读可能短暂不一致

## 代码实现

### 最简版本：内存同步

```java
// 写模型（领域模型）
public class Order {
    private Long id;
    private String orderNo;
    private Long userId;
    private List<OrderItem> items;
    private BigDecimal totalAmount;
    private OrderStatus status;
    private Date createdAt;

    // 业务方法
    public void addItem(Product product, int quantity) {
        // 业务校验
        if (quantity <= 0) {
            throw new IllegalArgumentException("数量必须大于 0");
        }
        // 添加商品
        this.items.add(new OrderItem(product.getId(), product.getPrice(), quantity));
        recalculateTotal();
    }

    public void pay() {
        if (this.status != OrderStatus.PENDING) {
            throw new IllegalStateException("订单状态不正确");
        }
        this.status = OrderStatus.PAID;
    }

    private void recalculateTotal() {
        this.totalAmount = items.stream()
            .map(item -> item.getPrice().multiply(BigDecimal.valueOf(item.getQuantity())))
            .reduce(BigDecimal.ZERO, BigDecimal::add);
    }
}

// 读模型（物化视图）
public class OrderReadModel {
    private Long id;
    private String orderNo;
    private String userName;        // 反范式化
    private int itemCount;          // 聚合
    private BigDecimal totalAmount;
    private String status;
    private String statusText;      // 冗余状态文本
    private String createdAt;       // 格式化的时间
}

// 命令处理器
public class OrderCommandHandler {
    @Autowired
    private OrderRepository orderRepository;
    @Autowired
    private EventPublisher eventPublisher;

    // 处理创建订单命令
    public Order handle(CreateOrderCommand command) {
        Order order = new Order();
        order.setUserId(command.getUserId());
        order.setOrderNo(generateOrderNo());

        for (OrderItemDTO item : command.getItems()) {
            Product product = productService.get(item.getProductId());
            order.addItem(product, item.getQuantity());
        }

        Order saved = orderRepository.save(order);

        // 发布领域事件
        eventPublisher.publish(new OrderCreatedEvent(saved));

        return saved;
    }

    // 处理支付命令
    public void handle(PayOrderCommand command) {
        Order order = orderRepository.findById(command.getOrderId())
            .orElseThrow(() -> new OrderNotFoundException(command.getOrderId()));

        order.pay();
        orderRepository.save(order);

        eventPublisher.publish(new OrderPaidEvent(order));
    }
}

// 查询处理器
public class OrderQueryHandler {
    @Autowired
    private OrderReadRepository readRepository;

    // 查询订单列表
    public Page<OrderReadModel> listOrders(Long userId, int page, int size) {
        return readRepository.findByUserId(userId, PageRequest.of(page, size));
    }

    // 查询订单详情
    public OrderDetailReadModel getOrderDetail(Long orderId) {
        return readRepository.findDetailById(orderId)
            .orElseThrow(() -> new OrderNotFoundException(orderId));
    }
}
```

### 生产版本：事件驱动同步

```java
// 领域事件
public abstract class DomainEvent {
    private final String eventId;
    private final long occurredAt;

    protected DomainEvent() {
        this.eventId = UUID.randomUUID().toString();
        this.occurredAt = System.currentTimeMillis();
    }
}

public class OrderCreatedEvent extends DomainEvent {
    private final Order order;

    public OrderCreatedEvent(Order order) {
        this.order = order;
    }
}

public class OrderItemAddedEvent extends DomainEvent {
    private final Long orderId;
    private final OrderItem item;

    public OrderItemAddedEvent(Long orderId, OrderItem item) {
        this.orderId = orderId;
        this.item = item;
    }
}

// 事件发布器
public interface EventPublisher {
    void publish(DomainEvent event);
}

// 事件订阅器（更新读模型）
public class OrderReadModelUpdater {
    @Autowired
    private OrderReadRepository readRepository;

    @EventListener
    public void onOrderCreated(OrderCreatedEvent event) {
        Order order = event.getOrder();

        OrderReadModel readModel = new OrderReadModel();
        readModel.setId(order.getId());
        readModel.setOrderNo(order.getOrderNo());
        readModel.setUserName(userService.getUserName(order.getUserId()));
        readModel.setItemCount(order.getItems().size());
        readModel.setTotalAmount(order.getTotalAmount());
        readModel.setStatus(order.getStatus().name());
        readModel.setStatusText(order.getStatus().getDisplayName());
        readModel.setCreatedAt(formatDate(order.getCreatedAt()));

        readRepository.save(readModel);
    }

    @EventListener
    public void onOrderItemAdded(OrderItemAddedEvent event) {
        readRepository.updateItemCount(event.getOrderId(),
            readRepository.findById(event.getOrderId()).map(OrderReadModel::getItemCount).orElse(0) + 1);
    }
}
```

### 使用 Kafka 实现事件驱动

```java
// 事件发布到 Kafka
@Service
public class KafkaEventPublisher implements EventPublisher {
    @Autowired
    private KafkaTemplate<String, DomainEvent> kafkaTemplate;

    @Override
    public void publish(DomainEvent event) {
        String topic = getTopicForEvent(event);
        kafkaTemplate.send(topic, event.getAggregateId(), event);
    }
}

// 事件消费者（读模型更新服务）
@Service
public class ReadModelConsumer {
    @Autowired
    private OrderReadRepository readRepository;

    @KafkaListener(topics = "order-events", groupId = "read-model-updater")
    public void consume(ConsumerRecord<String, DomainEvent> record) {
        DomainEvent event = record.value();

        if (event instanceof OrderCreatedEvent) {
            handleOrderCreated((OrderCreatedEvent) event);
        } else if (event instanceof OrderPaidEvent) {
            handleOrderPaid((OrderPaidEvent) event);
        } else if (event instanceof OrderCancelledEvent) {
            handleOrderCancelled((OrderCancelledEvent) event);
        }
    }

    private void handleOrderCreated(OrderCreatedEvent event) {
        // 构建读模型
        OrderReadModel model = buildReadModel(event.getOrder());
        readRepository.save(model);
    }
}
```

## CQRS 的优势与挑战

### 优势

| 优势 | 说明 |
|---|---|
| 读写解耦 | 读写可以独立优化 |
| 性能提升 | 读侧可以针对查询优化 |
| 可扩展性 | 读写可以独立扩展 |
| 灵活性 | 读模型可以随时重塑 |
| 复杂度隔离 | 业务逻辑和查询逻辑分离 |

### 挑战

| 挑战 | 说明 |
|---|---|
| 最终一致 | 写后读可能短暂不一致 |
| 复杂度增加 | 需要管理多个模型 |
| 同步成本 | 需要处理读写同步 |
| 错误处理 | 同步失败需要补偿 |

## 何时使用 CQRS

**适合使用 CQRS 的场景**：

- 读写压力差异明显的系统
- 需要针对读写分别优化的系统
- 复杂业务逻辑，需要清晰分离
- 需要支持多种查询视图
- 高并发系统，需要独立扩展

**不适合使用 CQRS 的场景**：

- 简单的 CRUD 系统
- 读写压力均衡的系统
- 强一致性要求高的系统
- 团队不熟悉事件驱动

## 总结

CQRS 模式通过分离命令和查询职责，让读写可以独立优化，是解决复杂业务系统读写性能差异的利器。

**核心要点**：

- 命令处理写入领域模型，查询处理读取物化视图
- 写侧和读侧使用不同的模型和存储
- 通过事件驱动实现读写同步
- 接受最终一致，换取性能和灵活性

**注意事项**：

- 权衡一致性和性能
- 设计好事件格式和版本
- 考虑同步失败的处理
- 监控读写延迟
