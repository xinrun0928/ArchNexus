# Saga 模式

你正在设计一个电商下单系统，涉及订单服务、库存服务、支付服务、物流服务。一次下单需要这四个服务全部成功。但问题来了：库存服务成功了，支付服务失败了——已经扣掉的库存怎么办？

分布式事务的两阶段提交（2PC）可以解决这个问题，但它有个致命问题：**协调者宕机时，所有参与者都傻等了**。

Saga 模式正是为了解决这个问题：**不追求强一致，而是用「补偿」的方式，实现最终一致**。

## 问题背景

### 分布式事务的困境

```java
// 电商下单场景：涉及多个服务
public class OrderService {
    public void createOrder(Order order) {
        // 1. 创建订单
        orderService.create(order);

        // 2. 扣减库存（调用库存服务）
        inventoryService.deduct(order.getItems());  // 成功

        // 3. 扣款（调用支付服务）
        paymentService.charge(order.getUserId(), order.getAmount());  // 失败！
    }
}
```

问题：库存已经扣了，但支付失败了。怎么办？

### 2PC 的问题

两阶段提交（2PC）可以实现强一致，但问题明显：

| 问题 | 说明 |
|---|---|
| 协调者单点 | 协调者宕机，所有参与者阻塞 |
| 锁定资源 | 提交阶段前，所有资源被锁定 |
| 阻塞 | 参与者等待协调者通知，阻塞时间长 |
| 实现复杂 | 需要处理各种异常场景 |

## Saga 模式的核心

```
核心思想：
┌─────────────────────────────────────────────────────────────┐
│                      Saga 模式                              │
├─────────────────────────────────────────────────────────────┤
│  将长事务分解为多个本地事务 + 补偿事务                       │
│                                                             │
│  操作序列：                                                 │
│  ┌──────────┐   ┌──────────┐   ┌──────────┐   ┌──────────┐│
│  │ Step 1   │ → │ Step 2   │ → │ Step 3   │ → │ Step 4   ││
│  │ 创建订单 │   │ 扣减库存 │   │ 扣款     │ → │ 创建物流 ││
│  └──────────┘   └──────────┘   └──────────┘   └──────────┘│
│       ↓              ↓              ↓              ↓         │
│  ┌──────────┐   ┌──────────┐   ┌──────────┐   ┌──────────┐│
│  │ 取消订单 │ ← │ 恢复库存 │ ← │ 退款     │ ← │ 取消物流 ││
│  │ (补偿1)  │   │ (补偿2)  │   │ (补偿3)  │   │ (补偿4)  ││
│  └──────────┘   └──────────┘   └──────────┘   └──────────┘│
└─────────────────────────────────────────────────────────────┘
```

**关键特点**：

1. **无锁定**：每个步骤提交后立即释放资源
2. **补偿机制**：失败时通过补偿事务回滚
3. **最终一致**：短时间内可能不一致，但最终会达到一致状态
4. **异步执行**：各步骤可以异步进行

## 两种 Saga 编排方式

### 编排式 Saga（Choreography）

```
事件驱动：每个参与者监听事件并执行自己的操作

  ┌──────────┐    [订单创建事件]    ┌──────────┐
  │ Order    │ ─────────────────▶ │ Inventory│
  │ Service  │                    │ Service  │
  └──────────┘                    └────┬─────┘
       ▲                                 │
       │       [库存已扣事件]              │
       └─────────────────────────────────┘
                    │
       ┌────────────┴────────────┐
       ▼                          ▼
  ┌──────────┐             ┌──────────┐
  │ Payment  │             │ Payment  │
  │ Service  │             │ Service  │
  └──────────┘             └────┬─────┘
                                 │
                    [支付完成事件]
```

```java
// 订单服务
public class OrderService {
    @Autowired
    private ApplicationEventPublisher publisher;

    public void createOrder(Order order) {
        order.setStatus(OrderStatus.CREATED);
        orderRepository.save(order);

        // 发布订单创建事件
        publisher.publishEvent(new OrderCreatedEvent(order));  // [!code highlight]
    }

    @EventListener
    public void handlePaymentFailed(PaymentFailedEvent event) {
        // 补偿：取消订单
        Order order = orderRepository.findById(event.getOrderId());
        order.setStatus(OrderStatus.CANCELLED);
        orderRepository.save(order);

        // 发布补偿事件
        publisher.publishEvent(new OrderCancelledEvent(order));
    }
}

// 库存服务
public class InventoryService {
    @EventListener
    public void handleOrderCreated(OrderCreatedEvent event) {
        // 扣减库存
        inventoryService.deduct(event.getOrder().getItems());
        // 发布库存已扣事件
        publisher.publishEvent(new InventoryDeductedEvent(event.getOrder()));
    }

    @EventListener
    public void handleOrderCancelled(OrderCancelledEvent event) {
        // 补偿：恢复库存
        inventoryService.restore(event.getOrder().getItems());
    }
}

// 支付服务
public class PaymentService {
    @EventListener
    public void handleInventoryDeducted(InventoryDeductedEvent event) {
        // 扣款
        try {
            paymentService.charge(event.getOrder());
            publisher.publishEvent(new PaymentCompletedEvent(event.getOrder()));
        } catch (PaymentException e) {
            // 发布支付失败事件
            publisher.publishEvent(new PaymentFailedEvent(event.getOrder(), e.getMessage()));
        }
    }
}
```

### 编排式 Saga 的问题

- 服务间耦合度高
- 难以追踪完整事务
- 事件循环依赖风险

### 编排式 Saga（Orchestration）

```
中心协调者：专门的服务控制整个流程

  ┌─────────────┐
  │  Saga       │
  │  Orchestrator│
  └──────┬──────┘
         │
    ┌────┴────┐
    │         │
    ▼         ▼
┌────────┐ ┌────────┐
│ Order  │ │Inventory│
│ Service│ │ Service │
└────┬───┘ └───┬────┘
     │         │
     ▼         ▼
┌────────┐ ┌────────┐
│Payment │ │ Shipping│
│ Service│ │ Service │
└────────┘ └────────┘
```

```java
// Saga 协调器
public class OrderSagaOrchestrator {
    private final OrderService orderService;
    private final InventoryService inventoryService;
    private final PaymentService paymentService;
    private final ShippingService shippingService;

    public void execute(Order order) {
        SagaState state = new SagaState(order);

        try {
            // Step 1: 创建订单
            orderService.create(order);
            state.markStepCompleted(SagaStep.CREATE_ORDER);

            // Step 2: 扣减库存
            inventoryService.deduct(order.getItems());
            state.markStepCompleted(SagaStep.DEDUCT_INVENTORY);

            // Step 3: 扣款
            paymentService.charge(order.getUserId(), order.getAmount());
            state.markStepCompleted(SagaStep.CHARGE);

            // Step 4: 创建物流
            shippingService.create(order);
            state.markStepCompleted(SagaStep.CREATE_SHIPPING);

        } catch (Exception e) {
            // 补偿已完成的步骤
            compensate(state, e);
        }
    }

    private void compensate(SagaState state, Exception originalError) {
        // 补偿要按相反顺序执行
        for (SagaStep step : state.getCompletedStepsReversed()) {
            try {
                switch (step) {
                    case CREATE_SHIPPING:
                        shippingService.cancel(state.getOrder());
                        break;
                    case CHARGE:
                        paymentService.refund(state.getOrder());
                        break;
                    case DEDUCT_INVENTORY:
                        inventoryService.restore(state.getOrder().getItems());
                        break;
                    case CREATE_ORDER:
                        orderService.cancel(state.getOrder());
                        break;
                }
            } catch (Exception compensateError) {
                // 记录补偿失败，但继续补偿其他步骤
                log.error("补偿失败: step={}, order={}", step, state.getOrder().getId(), compensateError);
            }
        }
        throw new SagaExecutionException("Saga 执行失败", originalError);
    }
}

// Saga 状态
public class SagaState {
    private final Order order;
    private final List<SagaStep> completedSteps = new ArrayList<>();

    public SagaState(Order order) {
        this.order = order;
    }

    public void markStepCompleted(SagaStep step) {
        completedSteps.add(step);
    }

    public List<SagaStep> getCompletedStepsReversed() {
        List<SagaStep> reversed = new ArrayList<>(completedSteps);
        Collections.reverse(reversed);
        return reversed;
    }

    public Order getOrder() {
        return order;
    }
}
```

## 实际应用：使用 Eventuate Tram

```java
// 定义 Saga
@Saga
public class OrderSaga {
    @Autowired
    private SagaInstanceFactory sagaInstanceFactory;

    public void startSaga(Order order) {
        sagaInstanceFactory.create(this, new OrderSagaData(order));
    }

    @Autowired
    private OrderService orderService;
    @Autowired
    private InventoryService inventoryService;
    @Autowired
    private PaymentService paymentService;

    @SagaStart
    public void createOrder(OrderSagaData data) {
        orderService.create(data.getOrder());
    }

    @SagaStep(sagaType = "DEDUCT_INVENTORY")
    public void deductInventory(OrderSagaData data) {
        inventoryService.deduct(data.getOrder().getItems());
    }

    @SagaStep(sagaType = "CHARGE")
    public void chargePayment(OrderSagaData data) {
        paymentService.charge(data.getOrder());
    }

    @Compensation(sagaType = "DEDUCT_INVENTORY")
    public void compensateInventory(OrderSagaData data) {
        inventoryService.restore(data.getOrder().getItems());
    }

    @Compensation(sagaType = "CHARGE")
    public void compensatePayment(OrderSagaData data) {
        paymentService.refund(data.getOrder());
    }
}
```

## Saga vs 2PC

| 维度 | Saga | 2PC |
|---|---|---|
| 一致性 | 最终一致 | 强一致 |
| 资源锁定 | 不锁定 | 锁定 |
| 性能 | 高 | 低 |
| 复杂度 | 高（补偿逻辑） | 中（协调者复杂） |
| 宕机恢复 | 自动补偿 | 依赖日志 |
| 适用场景 | 长事务、分布式 | 短事务 |

## Saga 的补偿策略

### 策略一：正向恢复（Retry）

```
失败时重试正向操作，直到成功
适用于：临时性故障（网络抖动、服务暂时不可用）

T1: 扣款 ──▶ 失败 ──▶ 重试 ──▶ 成功
```

### 策略二：反向补偿（Compensate）

```
失败时按相反顺序执行补偿操作
适用于：业务性故障（余额不足、库存不足）

T1: 扣款成功 ──▶ T2: 库存不足 ──▶ 补偿: 退款
```

### 策略三：幂等重试

```
每个操作都是幂等的，可以安全重试
适用于：所有场景

扣款: amount, idempotencyId=xxx
```

## 常见问题

### 问题一：补偿逻辑的复杂度

```java
// 错误：补偿逻辑不完整
public void deductInventory(List<Item> items) {
    for (Item item : items) {
        inventory.decrement(item.getSku(), item.getQuantity());
    }
}

public void restoreInventory(List<Item> items) {
    // 忘记恢复某个商品
    for (int i = 0; i < items.size() - 1; i++) {  // [!code error]
        inventory.increment(items.get(i).getSku(), items.get(i).getQuantity());
    }
}

// 正确：补偿逻辑必须完整
public void restoreInventory(List<Item> items) {
    for (Item item : items) {
        inventory.increment(item.getSku(), item.getQuantity());
    }
}
```

```java
// 错误：补偿失败就直接抛异常
```

### 问题三：并发修改

```java
        throw new InsufficientBalanceException();

    int updated = accountMapper.debitWithOptimisticLock(
        order.getUserId(),
        order.getAmount(),
        account.getVersion()  // [!code highlight]
    );
    if (updated == 0) {
        throw new ConcurrentModificationException();
    }
}
```

## Saga 的适用场景

**适合 Saga 的场景**：

- 跨服务的业务流程（订单、支付、物流）
- 长事务（业务流程时间长）
- 允许最终一致的业务

**不适合 Saga 的场景**：

- 需要强一致的事务（银行转账）
- 补偿逻辑难以实现的业务
- 业务规则复杂，补偿风险高

## 总结

Saga 模式通过将长事务分解为多个本地事务，并配合补偿机制，实现了分布式系统的最终一致性。

**核心要点**：

- 用补偿代替回滚
- 每个操作必须幂等
- 补偿要按相反顺序执行
- 记录补偿失败的任务用于重试

**编排方式**：

- 事件驱动（Choreography）：松耦合，但难以追踪
- 中心协调（Orchestration）：清晰可控，但有单点

**注意事项**：

- 补偿逻辑的完整性和正确性
- 补偿失败的恢复机制
- 并发控制的考虑
