# 分布式设计模式

订单服务扣了库存，支付服务收了钱，但物流服务宕机了。这条订单该算什么状态？库存扣了，钱收了，发货失败——是重试？还是直接退款？钱原路退回去之后，库存要不要加回来？

这就是分布式系统的本质问题：**单机的 ACID 事务不再有效，你需要用分布式的方式来保证数据一致性。**

但一致性只是冰山一角。当系统拆成几十上百个服务之后，服务间如何通信？日志如何统一收集？熔断、限流、重试谁来管？监控、链路追踪怎么做？这些问题不是加个框架就能解决的，你需要一套经过验证的**分布式设计模式**。

## 为什么分布式系统需要设计模式

单体系统的问题我们都清楚：代码耦合、部署不灵活、一个模块拖垮整个系统。拆成微服务之后，这些问题解决了，但同时又引入了一系列新问题：

**数据一致性**：原本一个数据库搞定的事，现在分散到多个服务，每个服务有自己的数据库。如何保证跨服务的数据一致性？

**服务协作**：服务之间如何通信？同步调用还是异步消息？超时了怎么办？某个服务挂了怎么办？

**横切关注点**：日志、监控、熔断、重试、限流……每个服务都要重复实现一遍吗？

**运维复杂度**：服务拆开了，如何追踪一个请求经过哪些服务？日志如何关联？配置如何管理？

分布式设计模式，就是针对这些问题的通用解决方案。它们不是某个框架的专利，而是经过大量生产环境验证的架构经验。

## 核心模式一览

本章节涵盖五种分布式架构中的核心模式：

### 事务协调模式

| 模式 | 解决的问题 | 核心思想 |
|---|---|---|
| **Saga 模式** | 跨服务的分布式事务 | 将长事务拆成多个本地事务，配合补偿机制 |
| **CQRS 模式** | 读写压力不均、数据异构 | 命令与查询分离，读写使用不同模型 |
| **事件溯源模式** | 状态变更审计、可回溯 | 用事件流代替状态快照，完整记录所有变更 |

### 基础设施模式

| 模式 | 解决的问题 | 核心思想 |
|---|---|---|
| **Sidecar 模式** | 横切关注点耦合主服务 | 将通用功能抽离为独立进程，与应用同生命周期部署 |
| **Ambassador 模式** | 客户端与外部服务解耦 | 在客户端部署代理，处理重试、限流、协议转换等 |

## 模式之间的关系

这五种模式并非孤立存在，而是相互配合解决分布式系统的不同层面问题。

对于**数据一致性**问题，Saga 模式通过补偿机制替代传统分布式事务，适合跨服务的业务流程；CQRS 模式通过读写分离优化性能，同时降低读写冲突；事件溯源模式则用事件流提供完整的审计能力和历史回溯。

对于**服务治理**问题，Sidecar 模式将日志、监控、安全等横切功能与应用解耦，每个服务无需重复实现；Ambassador 模式则在客户端侧统一处理与外部服务的交互细节。

## 技术选型建议

引入任何一种模式都是有代价的。在做技术选型之前，需要先想清楚：

**Saga 模式**：适合业务流程稳定、步骤可控的场景。如果业务流程经常变化，补偿逻辑会变得非常复杂。如果系统对一致性要求极高（比如金融转账），Saga 的最终一致性可能不够。

**CQRS 模式**：适合读写压力差异明显、数据结构差异大的场景。但如果团队不熟悉事件驱动架构，引入 CQRS 会增加系统复杂度。

**事件溯源模式**：适合需要完整审计日志、支持时间回溯的场景。如果业务不需要这些能力，事件溯源会带来额外的存储和处理开销。

**Sidecar 模式**：适合微服务架构下需要统一治理能力的场景。但需要配合服务网格（如 Istio、Linkerd）才能发挥最大价值。

**Ambassador 模式**：适合需要大量外部服务调用、需要统一预处理逻辑的场景。如果外部服务数量不多，直接在应用内处理可能更简单。
