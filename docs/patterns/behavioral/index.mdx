# 行为型设计模式

当代码从「数据结构 + 函数」演化为「对象之间相互协作」时，一个问题随之浮现：对象之间应该如何通信？调用方需要知道多少细节？谁来承担「发起请求」和「处理请求」的责任？

行为型设计模式解决的，正是这个对象间协作的问题。它不是告诉你要做什么，而是告诉你「对象之间如何对话」。

## 模式分类

行为型模式可以按其核心关注点分为三类：

| 类别 | 解决的问题 | 涉及模式 |
|---|---|---|
| 对象间通信 | 如何让对象在不紧耦合的情况下相互通知 | 观察者、中介者 |
| 行为封装 | 如何把可变的行为封装为可替换的对象 | 策略、状态、命令 |
| 结构组合 | 如何把多个对象组合成可遍历的结构 | 迭代器、访问者 |
| 流程解耦 | 如何把处理流程拆分为独立可复用的步骤 | 责任链、模板方法 |
| 状态保存 | 如何在不破坏封装的前提下保存和恢复状态 | 备忘录 |
| 语法抽象 | 如何用对象表示一门语言或表达式 | 解释器 |

## 核心特征

行为型模式有三个显著特征：

**对象间通信**。对象不应该通过直接调用彼此的方法来协作，而应该通过中间层来解耦。例如观察者模式让 Subject 和 Observer 通过注册/通知机制通信，双方都不知道对方的存在。

**算法封装**。把变化的算法从对象中剥离出来，封装成独立对象。策略模式允许在运行时切换算法；命令模式把请求封装为对象，支持撤销、重做、日志记录。

**责任解耦**。把请求的发送者和接收者解耦。责任链模式让多个对象都有机会处理请求，调用者不需要知道是谁处理的；中介者模式让对象通过中介者来通信，而不是直接互相调用。

## 与其他类型的区别

相比创建型模式（关注对象如何创建）和结构型模式（关注对象如何组合），行为型模式的关注点在于**对象之间的行为协作**。当你发现代码中出现大量对象之间的直接调用、行为难以扩展或替换、请求发送者和处理者紧耦合时，往往就需要考虑行为型模式。

## 章节内容

本章节将逐一探讨各类行为型设计模式：

**通信模式**：观察者模式让对象自动响应状态变化；中介者模式通过中心协调��减少对象间的直接依赖。

**封装模式**：策略模式把算法封装为可互换的对象；命令模式把请求封装为可记录、可撤销的操作；状态模式让对象根据内部状态改变自己的行为。

**结构模式**：迭代器模式提供统一的方式遍历集合；访问者模式把对数据的操作从数据结构中分离出来。

**流程模式**：责任链模式让多个处理器依次处理请求；模板方法模式用继承定义算法的骨架，具体步骤由子类实现。

**辅助模式**：备忘录模式在不破坏封装的前提下保存对象状态；解释器模式为特定语言定义表达式和解释器。

掌握这些模式，你将能够设计出更灵活、更易扩展的对象协作系统。
