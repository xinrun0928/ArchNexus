# 架构型设计模式

代码写多了，你可能会遇到这样的困惑：明明两个人实现同样的功能，代码结构却天差地别。一个改一处全局生效，一个模块独立互不影响；一个加需求越叠越乱，一个扩展新功能像搭积木。

这种差异，根源不在编码能力，而在**架构设计**。

架构型设计模式解决的，不是「某个类该怎么写」的问题，而是「系统整体该怎么组织」的问题。它定义了模块之间的职责边界、交互方式和演进方向。选对了架构，系统可以优雅地生长；选错了架构，技术债会像滚雪球一样越滚越大。

## 架构与设计模式的区别

很多人容易把架构模式和 GoF 设计模式混为一谈。简单区分：

| 维度 | GoF 设计模式 | 架构型设计模式 |
|---|---|---|
| 粒度 | 代码级别，关注类和对象之间的关系 | 系统级别，关注模块或服务之间的组织 |
| 抽象层次 | 解决特定场景下的代码结构问题 | 解决整个系统的结构性问题 |
| 关注点 | 如何优雅地组织代码 | 如何让系统易于理解、扩展和维护 |
| 典型示例 | 单例、工厂、观察者 | 分层、六边形、CQRS |

两者并非对立，而是互补的关系。好的架构为代码提供舞台，好的设计模式让架构落地。

## 为什么要学习架构模式

架构模式不是「银弹」，但它是经验的沉淀。学习架构模式的价值在于：

**站在前人的肩膀上**。很多问题已经被反复遇到并解决过了，比如「如何让业务逻辑与基础设施解耦」「如何让系统支持高并发写入」——架构模式把这些解决方案提炼成可复用的思路。

**做出更理性的选型**。没有银弹意味着没有万能方案。学了架构模式，你才能理解每种方案的 trade-off，而不是盲目追新或凭感觉选型。

**让团队沟通更高效**。当团队对「分层架构」「CQRS」有共同认知后，讨论架构设计时不需要从零解释概念，直接在共识基础上推进。

## 章节内容

本章节将系统讲解六种核心架构模式：

- **分层架构**：最经典的分层思想，理解 MVC 三层架构的演进与局限
- **六边形架构**：以业务为核心，端口与适配器彻底解耦基础设施
- **微内核架构**：核心系统 + 插件化设计，适用于功能可扩展性要求高的场景
- **事件驱动架构**：通过事件总线解耦组件，构建高响应式系统
- **CQRS 架构**：读写分离的极致演进，解决复杂业务场景下的读写冲突
- **Serverless 架构**：无服务器化趋势下，重新思考架构与运维的边界

每一章都会从「解决了什么问题」出发，探讨「为什么不用更简单的方案」，以及「它的代价是什么、什么场景下不应该用」。学完这一章，你面对一个新的业务需求时，应该能回答出：**我为什么选择这种架构，而不是其他方案**。
