# CQRS 架构

你的电商系统，数据模型越做越复杂：订单要展示商品详情、用户信息、库存状态、物流进度……一个查询要 join 五六张表。

写操作呢？只是简单的插入和更新。

问题出在哪？**读写需求差异巨大，却用同一个模型服务所有场景。**

CQRS 架构正是为了解决这个问题：**把命令（写）和查询（读）完全分开，用不同的模型满足不同的需求**。

## 问题背景

### CRUD 架构的困境

```java
// 传统 Order 模型服务所有场景
public class Order {
    private Long id;
    private String orderNo;
    private Long userId;
    private String userName;       // 反范式
    private String userEmail;
    private List<OrderItem> items;
    private BigDecimal totalAmount;
    private String status;

    // 1. 订单列表页：需要 orderNo, userName, totalAmount, status, createdAt
    // 2. 订单详情页：需要所有字段 + items + logistics
    // 3. 订单统计：需要 sum(totalAmount), count(*), group by date
    // 4. 用户订单：需要分页、按时间排序
}

// 查询方法越来越复杂
public Page<OrderDTO> listOrders(Long userId, int page, int size) {
    // 这个查询要 join 多张表
    return orderRepository.findByUserIdWithDetails(userId, page, size);
}

public OrderDetailDTO getOrderDetail(Long orderId) {
    // 这个查询更复杂，要获取商品详情、物流信息、支付信息
    return orderRepository.findDetailById(orderId);
}

public OrderStatistics getStatistics(Long userId) {
    // 这个是聚合查询
    return orderRepository.calculateStatistics(userId);
}
```

问题分析：

- **读写比例失衡**：读多写少，但用同一套模型
- **查询复杂**：一个模型要满足多种查询需求
- **性能瓶颈**：join 太多影响性能
- **难以优化**：读和写互相牵制

## CQRS 架构的核心

```
核心思想：
┌─────────────────────────────────────────────────────────────────┐
│                      CQRS 架构                                  │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  写路径 (Command Side)                                         │
│  ┌─────────────┐    ┌─────────────┐    ┌─────────────┐          │
│  │   Command   │───▶│  Domain     │───▶│   Event     │          │
│  │             │    │   Model     │    │   Store     │          │
│  └─────────────┘    └─────────────┘    └──────┬──────┘          │
│                                               │                  │
│  ┌───────────────────────────────────────────┘                  │
│                                                                  │
│  读路径 (Query Side)                                            │
│  ┌─────────────┐    ┌─────────────┐    ┌─────────────┐          │
│  │   Query     │◀───│   Read      │◀───│   Material- │          │
│  │   Handler   │    │   Model     │    │   ized View │          │
│  └─────────────┘    └─────────────┘    └─────────────┘          │
│                                                                  │
│  ┌─────────────────────────────────────────────────────────────┐│
│  │                     不同的存储                               ││
│  │  写侧：OLTP 数据库（事务优先）                               ││
│  │  读侧：OLAP 数据库 / Elasticsearch / Redis / Data Warehouse ││
│  └─────────────────────────────────────────────────────────────┘│
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

**关键概念**：

1. **Command（命令）**：改变系统状态的请求
2. **Query（查询）**：不改变状态的请求
3. **Read Model（读模型）**：针对查询优化的数据模型
4. **Write Model（写模型）**：包含完整业务逻辑的领域模型

### 最简版本：内存同步

```java
// ===== Command Side =====

// 订单命令
public sealed interface OrderCommand {
    record CreateOrderCommand(
        CustomerId customerId,
        List<OrderItemRequest> items
    ) implements OrderCommand {}

    record ConfirmOrderCommand(OrderId orderId) implements OrderCommand {}

    record CancelOrderCommand(OrderId orderId, String reason) implements OrderCommand {}
}

// 订单聚合根
public class Order {
    private OrderId id;
    private CustomerId customerId;
    private List<OrderItem> items;
    private Money totalAmount;
    private OrderStatus status;
    private List<DomainEvent> domainEvents = new ArrayList<>();

    // 工厂方法
    public static Order create(CustomerId customerId, List<OrderItem> items) {
        Order order = new Order();
        order.id = OrderId.generate();
        order.customerId = customerId;
        order.items = new ArrayList<>(items);
        order.totalAmount = calculateTotal(items);
        order.status = OrderStatus.PENDING;

        order.addDomainEvent(new OrderCreatedEvent(order));
        return order;
    }

    // 领域行为
    public void confirm() {
        if (this.status != OrderStatus.PENDING) {
            throw new IllegalStateException("只有待确认订单可以确认");
        }
        this.status = OrderStatus.CONFIRMED;
        addDomainEvent(new OrderConfirmedEvent(this));
    }

    public void cancel(String reason) {
        if (this.status == OrderStatus.SHIPPED || this.status == OrderStatus.DELIVERED) {
            throw new IllegalStateException("已发货不能取消");
        }
        this.status = OrderStatus.CANCELLED;
        addDomainEvent(new OrderCancelledEvent(this, reason));
    }

    private void addDomainEvent(DomainEvent event) {
        domainEvents.add(event);
    }

    public List<DomainEvent> getDomainEvents() {
        List<DomainEvent> events = new ArrayList<>(domainEvents);
        domainEvents.clear();
        return events;
    }
}

// 命令处理器
@Service
public class OrderCommandHandler {

    @Autowired
    private OrderRepository orderRepository;

    public Order handle(CreateOrderCommand command) {
        List<OrderItem> items = command.items().stream()
            .map(this::toOrderItem)
            .collect(Collectors.toList());

        Order order = Order.create(command.customerId(), items);
        orderRepository.save(order);

        // 发布领域事件
        for (DomainEvent event : order.getDomainEvents()) {
            eventPublisher.publish(event);
        }

        return order;
    }

    public void handle(ConfirmOrderCommand command) {
        Order order = orderRepository.findById(command.orderId());
        order.confirm();
        orderRepository.save(order);

        for (DomainEvent event : order.getDomainEvents()) {
            eventPublisher.publish(event);
        }
    }
}
```

```java
public class OrderListItem {
    private Long id;

    private BigDecimal totalAmount;

    private Long id;
    private String orderNo;

    // 订单基本信息
    private CustomerInfo customer;
    private List<OrderItemDetail> items;
    private PaymentInfo payment;
    private LogisticsInfo logistics;

    // 适合详情页的字段
}

public class OrderStatistics {
    private Long customerId;
    private int totalOrders;
    private BigDecimal totalSpent;
    private int thisMonthOrders;

    // 适合统计的字段
}

// 读模型构建器
@Service
public class OrderReadModelBuilder {
    @Autowired
    private JdbcTemplate jdbcTemplate;

    @KafkaListener(topics = "order-events")
    public void handleOrderEvent(DomainEvent event) {
        if (event instanceof OrderCreatedEvent e) {
            buildListItem(e.getOrder());
        } else if (event instanceof OrderConfirmedEvent e) {
            updateStatus(e.getOrderId(), "CONFIRMED");
        } else if (event instanceof OrderCancelledEvent e) {
            updateStatus(e.getOrderId(), "CANCELLED");
        }
    }

    private void buildListItem(Order order) {
        jdbcTemplate.update("""
            INSERT INTO order_read_models (id, order_no, customer_name, item_count, total_amount, status, created_at)
            VALUES (?, ?, ?, ?, ?, ?, ?)
            ON CONFLICT (id) DO UPDATE SET ...
            """,
            order.getId().getValue(),
            order.getOrderNo(),
            order.getCustomerName(),
            order.getItems().size(),
            order.getTotalAmount().getValue(),
            order.getStatus().name(),
            order.getCreatedAt()
        );
    }
}

// 查询处理器
@Service
public class OrderQueryHandler {

    @Autowired
    private JdbcTemplate jdbcTemplate;

    public Page<OrderListItem> listOrders(Long customerId, int page, int size) {
        String sql = """
            SELECT * FROM order_read_models
            WHERE customer_id = ?
            ORDER BY created_at DESC
            LIMIT ? OFFSET ?
            """;

        List<OrderListItem> items = jdbcTemplate.query(sql,
            (rs, rowNum) -> mapToListItem(rs),
            customerId, size, page * size
        );

        long total = jdbcTemplate.queryForObject(
            "SELECT COUNT(*) FROM order_read_models WHERE customer_id = ?",
            Long.class, customerId
        );

        return new Page<>(items, total, page, size);
    }

    public OrderDetail getOrderDetail(Long orderId) {
        // 从多个读模型中组装详情
        OrderListItem basic = jdbcTemplate.queryForObject(
            "SELECT * FROM order_read_models WHERE id = ?",
            (rs, rowNum) -> mapToListItem(rs),
            orderId
        );

        List<OrderItemDetail> items = getItemDetails(orderId);
        PaymentInfo payment = getPaymentInfo(orderId);
        LogisticsInfo logistics = getLogisticsInfo(orderId);

        return new OrderDetail(basic, items, payment, logistics);
    }

    public OrderStatistics getStatistics(Long customerId) {
        String sql = """
            SELECT
                customer_id,
                COUNT(*) as total_orders,
                SUM(total_amount) as total_spent,
                SUM(CASE WHEN created_at >= ? THEN 1 ELSE 0 END) as this_month_orders
            FROM order_read_models
            WHERE customer_id = ?
            GROUP BY customer_id
            """;

        return jdbcTemplate.queryForObject(sql,
            (rs, rowNum) -> new OrderStatistics(
                customerId,
                rs.getInt("total_orders"),
                rs.getBigDecimal("total_spent"),
                rs.getInt("this_month_orders")
            ),
            LocalDate.now().withDayOfMonth(1), customerId
        );
    }
}
```

### 生产版本：事件驱动同步

```java
// ===== 事件同步 =====

@Service
public class OrderReadModelSyncService {

    @Autowired
    private OrderEsRepository elasticsearchRepository;
    @Autowired
    private OrderRedisRepository redisRepository;

    @KafkaListener(topics = "order-events", groupId = "read-model-sync")
    public void handleOrderEvent(ConsumerRecord<String, DomainEvent> record) {
        DomainEvent event = record.value();

        try {
            switch (event) {
                case OrderCreatedEvent e -> handleCreated(e);
                case OrderConfirmedEvent e -> handleConfirmed(e);
                case OrderCancelledEvent e -> handleCancelled(e);
                case OrderShippedEvent e -> handleShipped(e);
                case OrderDeliveredEvent e -> handleDelivered(e);
                default -> log.warn("未知事件类型: {}", event.getClass());
            }
        } catch (Exception e) {
            log.error("同步读模型失败: {}", event.getEventId(), e);
            sendToDeadLetterQueue(event);
        }
    }

    private void handleCreated(OrderCreatedEvent event) {
        Order order = event.getOrder();

        // 更新 MySQL 读模型
        orderReadRepository.save(toReadModel(order));

        // 更新 Elasticsearch（用于搜索）
        OrderSearchDocument doc = toSearchDocument(order);
        elasticsearchRepository.save(doc);

        // 更新 Redis（用于缓存）
        redisRepository.cacheOrder(order.getId(), order);
    }

    private void handleConfirmed(OrderConfirmedEvent event) {
        updateReadModel(event.getOrderId(), "CONFIRMED");
        invalidateCache(event.getOrderId());
    }

    private void handleShipped(OrderShippedEvent event) {
        // 更新物流信息
        OrderReadModel model = orderReadRepository.findById(event.getOrderId());
        model.setLogisticsId(event.getTrackingNumber());
        model.setStatus("SHIPPED");
        model.setShippedAt(event.getShippedAt());
        orderReadRepository.save(model);
    }
}

// 异步更新 + 最终一致
public class AsyncReadModelSyncService {

    @Autowired
    private KafkaTemplate<String, Object> kafkaTemplate;

    private void handleCreated(OrderCreatedEvent event) {
        // 先发送到 Kafka，由专门的消费者处理
        kafkaTemplate.send("read-model-sync", event.getOrderId().getValue(), event);

        // 这样可以：
        // 1. 削峰填谷
        // 2. 失败重试
        // 3. 批量处理优化
    }
}
```

## CQRS vs 传统 CRUD

| 维度 | CQRS | CRUD |
|---|---|---|
| 模型 | 分离读写模型 | 统一模型 |
| 存储 | 可用不同存储 | 单一存储 |
| 查询 | 针对查询优化 | 简单查询 |
| 一致性 | 最终一致 | 强一致 |
| 复杂度 | 高 | 低 |
| 性能 | 可分别优化 | 互相牵制 |

## CQRS 的优势与挑战

### 优势

| 优势 | 说明 |
|---|---|
| 读写解耦 | 可以针对读写分别优化 |
| 性能提升 | 读模型针对查询优化 |
| 可扩展性 | 读写可以独立扩展 |
| 灵活性 | 支持多种读模型视图 |
| 复杂度隔离 | 命令和查询逻辑分离 |

### 挑战

| 挑战 | 说明 |
|---|---|
| 最终一致 | 写后读可能短暂不一致 |
| 复杂度增加 | 需要管理多个模型 |
| 同步成本 | 需要处理读写同步 |
| 错误处理 | 同步失败需要补偿 |

```java
@Service
public class OrderSearchService {

    @Autowired
    private ElasticsearchOperations elasticsearchTemplate;

    public Page<OrderSearchResult> search(OrderSearchCriteria criteria) {
        BoolQueryBuilder query = new BoolQueryBuilder();

        if (criteria.getKeyword() != null) {
            query.must(m -> m.multiMatch(
                criteria.getKeyword(),
                "orderNo", "customerName", "productNames"
            ));
        }

        if (criteria.getStatus() != null) {
            query.filter(f -> f.term("status", criteria.getStatus()));
        }

        if (criteria.getDateRange() != null) {
            query.filter(f -> f.range(r -> r
                .field("createdAt")
                .gte(criteria.getDateRange().getStart())
                .lte(criteria.getDateRange().getEnd())
            ));
        }

        return elasticsearchTemplate.search(query, OrderSearchResult.class);
    }
}
```

## 常见问题

### 问题一：数据一致性问题

```java
// 问题：写入成功但读模型未更新
orderCommandHandler.createOrder(command);  // 成功
// 此时读模型可能还是旧的

// 解决方案 1：接受最终一致
// 用户会短暂看到旧数据，通常可以接受

// 解决方案 2：返回同步结果
public OrderResponse createOrder(CreateOrderCommand command) {
    Order order = orderCommandHandler.handle(command);

    // 同步等待读模型更新
    waitForReadModelUpdate(order.getId());

    // 返回最新的读模型
    return orderQueryHandler.getOrderDetail(order.getId());
}

// 解决方案 3：乐观 UI
// 前端先显示写入结果，后续更新时再刷新
```

## 总结

CQRS 架构通过分离命令和查询职责，让读写可以独立优化，是解决复杂业务系统读写性能差异的利器。

**核心要点**：

- 命令处理写入领域模型，查询处理读取物化视图
- 写侧和读侧使用不同的模型和存储
- 通过事件驱动实现读写同步
- 接受最终一致，换取性能和灵活性

**优势**：

- 读写解耦，可以分别优化
- 支持多种读模型视图
- 可扩展性强
- 复杂度隔离

**挑战**：

- 最终一致性问题
- 复杂度增加
- 需要管理多个模型

**适用场景**：

- 读写压力差异明显的系统
- 需要多种查询视图的系统
- 复杂业务逻辑需要清晰分离
- 高并发系统需要独立扩展
