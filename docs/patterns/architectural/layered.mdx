# 分层架构

从 DOS 到移动互联网，软件开发已经发展了几十年。但有一个架构模式，从诞生到现在，一直是大多数应用系统的默认选择——分层架构。

不是因为它最先进，而是因为它最符合人类认知：**每一层只关注自己的事**。

## 问题背景

### 单体架构的混乱

```java
// 错误示例：一个类处理所有事情
public class UserService {
    public void register(String username, String password, String email) {
        // 1. 参数校验（Presentation）
        if (username == null || username.length() < 3) {
            throw new IllegalArgumentException("用户名必须至少3个字符");
        }

        // 2. 业务逻辑（Business）
        User existing = userRepository.findByUsername(username);
        if (existing != null) {
            throw new BusinessException("用户名已存在");
        }

        // 3. 数据持久化（Data Access）
        User user = new User(username, password, email);
        userRepository.save(user);

        // 4. 发送邮件（Infrastructure）
        emailService.send(email, "欢迎注册");

        // 5. 日志记录
        logger.info("新用户注册: {}", username);

        // ... 越写越长，越来越乱
    }
}
```

问题分析：

- **职责不清**：校验、业务、持久化、邮件混在一起
- **难以测试**：所有逻辑耦合，无法单独测试
- **难以复用**：注册逻辑无法在其他地方复用
- **难以变更**：换邮件服务要改核心代码

## 分层架构的核心

```
核心思想：
┌─────────────────────────────────────────────────────────────────┐
│                        分层架构                                  │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  ┌──────────────────────────────────────────────────────────┐   │
│  │            Presentation Layer（展示层）                   │   │
│  │  职责：处理用户界面、请求响应、参数校验                    │   │
│  │  技术：Controller、DTO、View Model                        │   │
│  └──────────────────────────────────────────────────────────┘   │
│                              │                                   │
│                              ▼                                   │
│  ┌──────────────────────────────────────────────────────────┐   │
│  │            Application Layer（应用层）                    │   │
│  │  职责：编排业务流程、事务管理、消息分发                    │   │
│  │  技术：Service、Command、Query                            │   │
│  └──────────────────────────────────────────────────────────┘   │
│                              │                                   │
│                              ▼                                   │
│  ┌──────────────────────────────────────────────────────────┐   │
│  │              Domain Layer（领域层）                      │   │
│  │  职责：业务规则、领域模型、实体、值对象                    │   │
│  │  技术：Entity、Value Object、Domain Service              │   │
│  └──────────────────────────────────────────────────────────┘   │
│                              │                                   │
│                              ▼                                   │
│  ┌──────────────────────────────────────────────────────────┐   │
│  │          Infrastructure Layer（基础设施层）                │   │
│  │  职责：数据持久化、消息发送、第三方服务调用                │   │
│  │  技术：Repository、Message Producer、External Client      │   │
│  └──────────────────────────────────────────────────────────┘   │
│                                                                  │
│  依赖规则：上层依赖下层，下层不依赖上层                          │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

**关键特点**：

1. **职责分离**：每一层只关注自己的职责
2. **单向依赖**：上层依赖下层，下层不依赖上层
3. **可测试性**：每层可以独立测试
4. **可替换性**：下层实现可以替换，不影响上层

## 代码实现

### 经典三层架构

```java
// ===== Presentation Layer =====

// DTO：数据传输对象
public class CreateUserRequest {
    @NotBlank(message = "用户名不能为空")
    @Size(min = 3, max = 20, message = "用户名长度3-20个字符")
    private String username;

    @NotBlank(message = "密码不能为空")
    @Size(min = 6, message = "密码至少6位")
    private String password;

    @NotBlank(message = "邮箱不能为空")
    @Email(message = "邮箱格式不正确")
    private String email;

    // getters and setters
}

// Controller
@RestController
@RequestMapping("/api/users")
public class UserController {

    private final UserApplicationService userService;

    @PostMapping
    public ResponseEntity<UserResponse> create(@Valid @RequestBody CreateUserRequest request) {
        UserResponse response = userService.createUser(request);
        return ResponseEntity.status(HttpStatus.CREATED).body(response);
    }

    @GetMapping("/{id}")
    public ResponseEntity<UserResponse> get(@PathVariable Long id) {
        UserResponse response = userService.getUser(id);
        return ResponseEntity.ok(response);
    }
}
```

```java
// ===== Application Layer =====

// 应用服务
@Service
@Transactional
public class UserApplicationService {

    private final UserDomainService userDomainService;
    private final UserRepository userRepository;
    private final EventPublisher eventPublisher;

    public UserApplicationService(
            UserDomainService userDomainService,
            UserRepository userRepository,
            EventPublisher eventPublisher) {
        this.userDomainService = userDomainService;
        this.userRepository = userRepository;
        this.eventPublisher = eventPublisher;
    }

    public UserResponse createUser(CreateUserRequest request) {
        // 1. 创建领域对象
        User user = User.create(request.getUsername(), request.getPassword(), request.getEmail());

        // 2. 执行业务逻辑
        userDomainService.validateUniqueUsername(user, userRepository);

        // 3. 持久化
        User saved = userRepository.save(user);

        // 4. 发布领域事件
        eventPublisher.publish(new UserCreatedEvent(saved));

        // 5. 返回响应
        return UserResponse.from(saved);
    }
}
```

```java
// ===== Domain Layer =====

// 实体
public class User {
    private Long id;
    private String username;
    private String password;
    private String email;
    private UserStatus status;
    private Date createdAt;

    // 私有构造函数，通过工厂方法创建
    private User() {}

    // 工厂方法：创建新用户
    public static User create(String username, String password, String email) {
        User user = new User();
        user.username = username;
        user.password = hashPassword(password);
        user.email = email;
        user.status = UserStatus.ACTIVE;
        user.createdAt = new Date();
        return user;
    }

    // 领域方法：更改邮箱
    public void changeEmail(String newEmail) {
        if (newEmail == null || !newEmail.contains("@")) {
            throw new IllegalArgumentException("邮箱格式不正确");
        }
        this.email = newEmail;
    }

    // 领域方法：激活用户
    public void activate() {
        if (this.status != UserStatus.PENDING) {
            throw new IllegalStateException("只有待激活状态的用户才能激活");
        }
        this.status = UserStatus.ACTIVE;
    }

    private static String hashPassword(String password) {
        // 密码哈希
        return BCrypt.hashpw(password, BCrypt.gensalt());
    }
}

// 领域服务：处理跨实体业务逻辑
public class UserDomainService {

    public void validateUniqueUsername(User user, UserRepository repository) {
        User existing = repository.findByUsername(user.getUsername());
        if (existing != null && !existing.getId().equals(user.getId())) {
            throw new BusinessException("用户名已被占用");
        }
    }
}
```

```java
// ===== Infrastructure Layer =====

// 仓储接口（定义在 Domain 层）
public interface UserRepository {
    User save(User user);
    User findById(Long id);
    User findByUsername(String username);
    void delete(Long id);
}

// JPA 实现
@Repository
public class JpaUserRepository implements UserRepository {

    @Autowired
    private UserJpaRepository jpaRepository;

    @Override
    @Transactional
    public User save(User user) {
        UserEntity entity = toEntity(user);
        UserEntity saved = jpaRepository.save(entity);
        return toDomain(saved);
    }

    @Override
    public User findById(Long id) {
        return jpaRepository.findById(id)
            .map(this::toDomain)
            .orElse(null);
    }

    @Override
    public User findByUsername(String username) {
        return jpaRepository.findByUsername(username)
            .map(this::toDomain)
            .orElse(null);
    }

    private UserEntity toEntity(User user) {
        return UserEntity.builder()
            .id(user.getId())
            .username(user.getUsername())
            .password(user.getPassword())
            .email(user.getEmail())
            .status(user.getStatus().name())
            .createdAt(user.getCreatedAt())
            .build();
    }

    private User toDomain(UserEntity entity) {
        User user = User.create(entity.getUsername(), entity.getPassword(), entity.getEmail());
        // 使用反射或 Builder 设置 ID
        return user;
    }
}

// 事件发布器
@Component
public class DomainEventPublisher implements EventPublisher {

    @Autowired
    private ApplicationEventPublisher springPublisher;

    @Override
    public void publish(DomainEvent event) {
        springPublisher.publishEvent(event);
    }
}
```

```
└─────────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────────┐
│                    Application Layer（应用层）                   │
│   职责：用例编排、事务边界、事件发布                              │
└─────────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────────┐
│                     Domain Layer（领域层）                       │
│   职责：业务规则、领域模型、领域服务                              │
│   注意：这一层没有任何外部依赖                                   │
└─────────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────────┐
│                  Infrastructure Layer（基础设施层）               │
```


## 依赖倒置原则（DIP）

分层架构的核心原则：**依赖倒置**。

```
传统依赖：
┌──────────────┐
│ Presentation │ ────────▶ Application
└──────────────┘                │
      ▲                         │
      │                         ▼
      │                ┌──────────────┐
      │                │   Domain     │
      │                └──────────────┘
      │                         ▲
      │                         │
      │                ┌──────────────┐
      └────────────────│ Infrastructure│
                       └──────────────┘

依赖倒置后：
┌──────────────┐
│ Presentation │
└──────────────┘
      │
      ▼
┌──────────────┐
│ Application  │
└──────────────┘
      │
      ▼
┌──────────────┐     ┌────────────────────┐
│    Domain    │◀────│  Infrastructure    │
│              │     │  (实现 Domain 接口) │
└──────────────┘     └────────────────────┘
```

```java
// Domain 层定义接口
public interface UserRepository {
    User save(User user);
    User findById(Long id);
}

// Infrastructure 层实现接口
public class JpaUserRepository implements UserRepository {
    // 实现细节
}

// Application 层依赖接口，不依赖实现
public class UserService {
    private final UserRepository userRepository;

    public UserService(UserRepository userRepository) {
        this.userRepository = userRepository;  // 依赖注入
    }
}
```

## 分层架构的优势与挑战

### 优势

| 优势 | 说明 |
|---|---|
| 清晰的结构 | 每层职责明确 |
| 可维护性 | 变更影响范围小 |
| 可测试性 | 每层可独立测试 |
| 可扩展性 | 可以单独扩展某层 |
| 团队协作 | 不同团队负责不同层 |

### 挑战

| 挑战 | 说明 |
|---|---|
| 过度分层 | 层数过多反而降低效率 |
| 贫血模型 | Domain 层只有 getter/setter |
| 领域逻辑泄漏 | 业务逻辑写在 Application 层 |
| 循环依赖 | 依赖关系混乱 |

## 常见问题

### 问题一：领域逻辑泄漏

```java
// 错误：业务逻辑写在 Application 层
public class OrderService {
    public void createOrder(OrderRequest request) {
        // 业务逻辑不应该在这里
        if (request.getItems().isEmpty()) {
            throw new BusinessException("订单不能为空");  // [!code warning]
        }

        BigDecimal total = BigDecimal.ZERO;
        for (OrderItem item : request.getItems()) {
            if (item.getPrice().compareTo(BigDecimal.ZERO) < 0) {
                throw new BusinessException("价格不能为负");  // [!code warning]
            }
            total = total.add(item.getPrice().multiply(BigDecimal.valueOf(item.getQuantity())));
        }

        // 折扣逻辑
        if (total.compareTo(new BigDecimal("1000")) >= 0) {
            total = total.multiply(new BigDecimal("0.9"));  // [!code warning]
        }

        // ...
    }
}

// 正确：业务逻辑在 Domain 层
public class Order {
    private List<OrderItem> items;
    private BigDecimal totalAmount;

    public void addItem(Product product, int quantity) {
        if (quantity <= 0) {
            throw new IllegalArgumentException("数量必须大于0");
        }
        // 领域逻辑
    }

    public void applyDiscount(BigDecimal threshold, BigDecimal rate) {
        if (this.totalAmount.compareTo(threshold) >= 0) {
            this.totalAmount = this.totalAmount.multiply(rate);
        }
    }
}
```

### 问题二：贫血模型

```java
// 错误：贫血模型
public class User {
    private Long id;
    private String username;
    private String email;

    // 只有 getter/setter，没有行为
    public Long getId() { return id; }
    public void setId(Long id) { this.id = id; }
    public String getUsername() { return username; }
    public void setUsername(String username) { this.username = username; }
}

// 正确：充血模型
public class User {
    private Long id;
    private String username;
    private String email;
    private UserStatus status;

    // 工厂方法
    public static User register(String username, String email, String password) {
        User user = new User();
        user.username = username;
        user.email = email;
        user.status = UserStatus.PENDING;
        return user;
    }

    // 领域行为
    public void activate() {
        if (this.status != UserStatus.PENDING) {
            throw new IllegalStateException("只有待激活状态才能激活");
        }
        this.status = UserStatus.ACTIVE;
    }

    public void changeEmail(String newEmail) {
        if (!newEmail.contains("@")) {
            throw new IllegalArgumentException("邮箱格式不正确");
        }
        this.email = newEmail;
    }
}
```

```java
// 错误：过度分层
Controller
  └─> ApplicationService
        └─> DomainService
              └─> Repository
                    └─> DAO

                    └─> DataSource
```

## 何时使用分层架构

**适合分层架构的场景**：

- 中大型企业应用
- 业务逻辑复杂的系统
- 需要长期维护的系统
- 团队分工明确的组织

**不适合分层架构的场景**：

- 简单的 CRUD 应用
- 需要快速迭代的创业项目
- 微服务架构（需要更细粒度的拆分）

## 总结

分层架构是软件工程中最经典、最成熟的架构模式。它通过将系统划分为不同的层，实现了职责分离、依赖倒置、可测试性等目标。

**核心要点**：

- 每层只关注自己的职责
- 上层依赖下层，下层通过接口被依赖
- 领域层是核心，不依赖任何外部
- 基础设施层实现领域定义的接口

**注意事项**：

- 避免领域逻辑泄漏到上层
- 避免贫血模型
- 适度分层，不过度设计
- 遵循依赖倒置原则
