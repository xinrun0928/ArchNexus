# 六边形架构

你开发了一个订单系统，开始用的是 MySQL 数据库。过了一年，业务要求换 PostgreSQL。再过一年，又要支持 MongoDB。每次换存储，代码改动都很大。

问题出在哪？**业务代码和基础设施代码耦合在一起了。**

六边形架构正是为了解决这个问题：**把业务逻辑放在核心，外围是适配器——无论是数据库、Web、消息队列，都只是「插上去」的适配器**。

## 问题背景

### 传统分层架构的问题

```java
// 错误示例：业务逻辑和数据访问混在一起
public class OrderService {
    @Autowired
    private OrderRepository orderRepository;  // 直接依赖具体实现

    public Order createOrder(OrderRequest request) {
        // 业务逻辑

        // 直接调用 JPA
        OrderEntity entity = new OrderEntity();
        entity.setUserId(request.getUserId());
        entity.setItems(request.getItems());
        orderRepository.save(entity);  // 耦合了 JPA

        return entity.toOrder();
    }
}
```

问题分析：

- 业务逻辑直接依赖数据库实现
- 换数据库要改核心业务代码
- 难以测试：测试需要真实数据库
- 难以扩展：增加新的数据源很困难

**关键概念**：

1. **Core（核心）**：包含所有业务逻辑，不依赖任何外部
2. **Port（端口）**：核心定义的接口，是外部世界访问核心的入口
3. **Adapter（适配器）**：实现端口的具体技术实现

## 代码实现

### 核心层定义

```java
// ===== Core: Domain Model =====

// 订单实体
public class Order {
    private OrderId id;
    private CustomerId customerId;
    private List<OrderItem> items;
    private Money totalAmount;
    private OrderStatus status;

    // 工厂方法
    public static Order create(CustomerId customerId, List<OrderItem> items) {
        Order order = new Order();
        order.id = OrderId.generate();
        order.customerId = customerId;
        order.items = new ArrayList<>(items);
        order.totalAmount = calculateTotal(items);
        order.status = OrderStatus.PENDING;
        return order;
    }

    // 领域行为
    public void confirm() {
        if (this.status != OrderStatus.PENDING) {
            throw new IllegalStateException("只有待确认订单可以确认");
        }
        this.status = OrderStatus.CONFIRMED;
    }

    public void cancel() {
        if (this.status == OrderStatus.SHIPPED || this.status == OrderStatus.DELIVERED) {
            throw new IllegalStateException("已发货或已送达的订单不能取消");
        }
        this.status = OrderStatus.CANCELLED;
    }

    private static Money calculateTotal(List<OrderItem> items) {
        return items.stream()
            .map(item -> item.getPrice().multiply(item.getQuantity()))
            .reduce(Money.ZERO, Money::add);
    }
}
```

```java
// ===== Core: Ports (Interfaces) =====

// 入站端口：处理外部请求
public interface OrderServicePort {
    OrderDTO createOrder(CreateOrderCommand command);
    OrderDTO confirmOrder(OrderId orderId);
    OrderDTO cancelOrder(OrderId orderId);
    OrderDTO getOrder(OrderId orderId);
    List<OrderDTO> getOrdersByCustomer(CustomerId customerId);
}

// 出站端口：访问外部资源
public interface OrderRepository {
    Order save(Order order);
    Order findById(OrderId id);
    List<Order> findByCustomerId(CustomerId customerId);
    void delete(OrderId id);
}

public interface NotificationPort {
    void sendOrderConfirmation(Order order);
    void sendOrderCancellation(Order order);
}

public interface PaymentGatewayPort {
    PaymentResult charge(CustomerId customerId, Money amount);
    RefundResult refund(PaymentTransactionId transactionId, Money amount);
}
```

```java
// ===== Core: Application Service =====

// 应用服务：编排业务流程
public class OrderApplicationService implements OrderServicePort {

    private final OrderRepository orderRepository;
    private final NotificationPort notificationPort;
    private final PaymentGatewayPort paymentGateway;

    public OrderApplicationService(
            OrderRepository orderRepository,
            NotificationPort notificationPort,
            PaymentGatewayPort paymentGateway) {
        this.orderRepository = orderRepository;
        this.notificationPort = notificationPort;
        this.paymentGateway = paymentGateway;
    }

    @Override
    public OrderDTO createOrder(CreateOrderCommand command) {
        // 1. 构建订单
        List<OrderItem> items = command.getItems().stream()
            .map(this::toOrderItem)
            .collect(Collectors.toList());

        Order order = Order.create(command.getCustomerId(), items);

        // 2. 扣款
        PaymentResult paymentResult = paymentGateway.charge(
            command.getCustomerId(),
            order.getTotalAmount()
        );

        if (!paymentResult.isSuccess()) {
            throw new PaymentFailedException(paymentResult.getMessage());
        }

        // 3. 保存订单
        Order saved = orderRepository.save(order);

        // 4. 发送通知
        notificationPort.sendOrderConfirmation(saved);

        return OrderDTO.from(saved);
    }

    @Override
    public OrderDTO confirmOrder(OrderId orderId) {
        Order order = orderRepository.findById(orderId);
        order.confirm();
        Order saved = orderRepository.save(order);
        return OrderDTO.from(saved);
    }
}
```

```java
// REST 适配器
@RestController
@RequestMapping("/api/orders")
public class OrderRestAdapter implements OrderServicePort {

    private final OrderServicePort orderService;

    @PostMapping
    public ResponseEntity<OrderDTO> create(@RequestBody @Valid CreateOrderRequest request) {
        CreateOrderCommand command = toCommand(request);
        OrderDTO result = orderService.createOrder(command);
        return ResponseEntity.status(HttpStatus.CREATED).body(result);
    }

    @PostMapping("/{id}/confirm")
    public ResponseEntity<OrderDTO> confirm(@PathVariable String id) {
        OrderDTO result = orderService.confirmOrder(OrderId.of(id));
        return ResponseEntity.ok(result);
    }

    @PostMapping("/{id}/cancel")
    public ResponseEntity<Void> cancel(@PathVariable String id) {
        orderService.cancelOrder(OrderId.of(id));
        return ResponseEntity.noContent();
    }

    private CreateOrderCommand toCommand(CreateOrderRequest request) {
        // 转换逻辑
    }
}

// 消息适配器（消费端）
@Component
public class OrderMessageAdapter {
    private final OrderServicePort orderService;

    @Autowired
    private ApplicationEventPublisher eventPublisher;

    @KafkaListener(topics = "order-commands", groupId = "order-service")
    public void handleMessage(OrderCommandMessage message) {
        switch (message.getType()) {
            case "CREATE":
                orderService.createOrder(message.getCommand());
                break;
            case "CONFIRM":
                orderService.confirmOrder(message.getOrderId());
                break;
        }
    }
}
```

```java
// ===== Adapters: Secondary Adapters (Driven) =====

// JPA 适配器
@Repository
public class JpaOrderRepository implements OrderRepository {

    @Autowired
    private OrderJpaRepository jpaRepository;

    @Override
    @Transactional
    public Order save(Order order) {
        OrderEntity entity = toEntity(order);
        OrderEntity saved = jpaRepository.save(entity);
        return toDomain(saved);
    }

    @Override
    public Order findById(OrderId id) {
        return jpaRepository.findById(id.getValue())
            .map(this::toDomain)
            .orElseThrow(() -> new OrderNotFoundException(id));
    }

    private OrderEntity toEntity(Order order) {
        return OrderEntity.builder()
            .id(order.getId().getValue())
            .customerId(order.getCustomerId().getValue())
            .totalAmount(order.getTotalAmount().getValue())
            .status(order.getStatus().name())
            .build();
    }

    private Order toDomain(OrderEntity entity) {
        // 从实体转换为领域对象
        return Order.reconstitute(...);
    }
}

// 消息通知适配器
@Component
public class EmailNotificationAdapter implements NotificationPort {

    @Autowired
    private JavaMailSender mailSender;

    @Override
    public void sendOrderConfirmation(Order order) {
        SimpleMailMessage message = new SimpleMailMessage();
        message.setTo(order.getCustomerEmail());
        message.setSubject("订单确认 - " + order.getId());
        message.setText("您的订单已确认...");

}

// 第三方支付适配器
@Component
public class StripePaymentAdapter implements PaymentGatewayPort {

    private final RestTemplate restTemplate;

    @Override
    public PaymentResult charge(CustomerId customerId, Money amount) {
        HttpHeaders headers = new HttpHeaders();
        headers.set("Authorization", "Bearer " + apiKey);

        Map<String, Object> body = Map.of(
            "amount", amount.getCents(),
            "currency", amount.getCurrency(),
            "customer", customerId.getValue()
        );

        try {
            ResponseEntity<StripeChargeResponse> response = restTemplate.exchange(
                "https://api.stripe.com/v1/charges",
                HttpMethod.POST,
                new HttpEntity<>(body, headers),
                StripeChargeResponse.class
            );

            return PaymentResult.success(response.getBody().getId());
        } catch (Exception e) {
            return PaymentResult.failed(e.getMessage());
        }
    }
}
```

## 六边形架构的变体

### 经典六边形 vs 洋葱架构

```
经典六边形：
┌─────────────────────────────────────┐
│           Adapters                  │
│  ┌───────────────────────────────┐  │
│  │         Ports                  │  │
│  │  ┌─────────────────────────┐  │  │
│  │  │        Core             │  │  │
│  │  │   Domain + Application  │  │  │
│  │  └─────────────────────────┘  │  │
│  └───────────────────────────────┘  │
└─────────────────────────────────────┘

洋葱架构：
┌─────────────────────────────────────┐
│           Infrastructure           │
│  ┌───────────────────────────────┐  │
│  │         Application          │  │
│  │  ┌─────────────────────────┐  │  │
│  │  │         Domain         │  │  │
│  │  │    (最核心，零依赖)     │  │  │
│  │  └─────────────────────────┘  │  │
│  └───────────────────────────────┘  │
└─────────────────────────────────────┘

两种架构本质上是一样的，只是视觉表现不同：
- 六边形强调「适配器可插拔」
- 洋葱强调「依赖向内」
```

### 清洁架构（Clean Architecture）

```
┌─────────────────────────────────────────────────────────────────┐
│                    Clean Architecture Layers                     │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  Entities（实体）：最内层，包含企业级业务规则                      │
│                                                                  │
│  Use Cases（用例）：应用层业务规则                                │
│                                                                  │
│  Interface Adapters（接口适配器）：控制器、网关                   │
│                                                                  │
│  Frameworks & Drivers（框架和驱动）：数据库、Web、UI              │
│                                                                  │
│  依赖方向从外向内，内部不知道外部的存在                            │
└─────────────────────────────────────────────────────────────────┘
```

## 六边形架构的优势

| 优势 | 说明 |
|---|---|
| 可测试性 | 核心完全独立，可以轻松 mock 所有依赖 |
| 可替换性 | 任何适配器都可以替换，不影响核心 |
| 可维护性 | 变更影响范围小 |
| 清晰的结构 | 依赖方向清晰 |
| 并行开发 | 前端和后端可以并行开发 |

## 六边形架构 vs 分层架构

| 维度 | 六边形架构 | 分层架构 |
|---|---|---|
| 核心 | 业务逻辑 | 领域层 |
| 依赖方向 | 向内 | 向下 |
| 外部访问 | 通过端口 | 直接依赖 |
| 可测试性 | 高 | 中 |
| 复杂度 | 较高 | 较低 |
| 适用场景 | 需要高可替换性 | 一般业务系统 |

## 常见问题

### 问题一：过度设计

```java
// 错误：为每个方法都创建端口
public interface UserRepository {
    User findById(Long id);
    User findByUsername(String username);
    User findByEmail(String email);
    User findByPhone(String phone);
    // ... 一个接口 50 个方法
}

// 正确：按功能划分，或者使用泛型仓储
public interface ReadRepository<T, ID> {
    T findById(ID id);
    List<T> findAll();
}

public interface UserRepository extends ReadRepository<User, Long> {
    User findByUsername(String username);
    User findByEmail(String email);
}
```

### 问题二：贫血模型

六边形架构不要求充血模型，但如果 Domain 层全是贫血模型，就失去了意义。

```java
// 错误：Domain 完全是贫血
public class Order {
    private Long id;
    private String status;
    // 只有 getter/setter

    public Long getId() { return id; }
    public void setId(Long id) { this.id = id; }
    // ...
}

// 正确：Domain 有业务行为
public class Order {
    private OrderId id;
    private OrderStatus status;

    public void confirm() {
        if (this.status != OrderStatus.PENDING) {
            throw new IllegalStateException("只有待确认订单可以确认");
        }
        this.status = OrderStatus.CONFIRMED;
    }
}
```

## 六边形架构的实际应用

### Spring Boot 集成

```java
// 配置类：注入适配器
@Configuration
public class HexagonConfiguration {

    // 出站适配器
    @Bean
    public OrderRepository orderRepository(JpaOrderRepository jpaRepository) {
        return new CachingOrderRepository(jpaRepository);  // 包装一层缓存
    }

    // 应用服务
    @Bean
    public OrderServicePort orderServicePort(
            OrderRepository orderRepository,
            NotificationPort notificationPort,
            PaymentGatewayPort paymentGateway) {
        return new OrderApplicationService(orderRepository, notificationPort, paymentGateway);
    }

    // 入站适配器
    @Bean
    public OrderController orderController(OrderServicePort orderService) {
        return new OrderRestAdapter(orderService);
    }
}
```

## 总结

六边形架构通过将核心业务逻辑与外部依赖完全解耦，实现了高度的可测试性和可替换性。

**核心要点**：

- Core 完全独立，不依赖任何外部
- Port 是核心定义的接口，定义了与外部的交互方式
- Adapter 实现 Port，是具体的技术实现
- 依赖方向永远从外向内

**优势**：

- 核心完全可测试
- 适配器可随时替换
- 支持多数据源
- 支持并行开发

**挑战**：

- 概念理解需要时间
- 过度设计风险
- 编码工作量增加

**适用场景**：

- 需要高可测试性的系统
- 需要支持多数据源的系统
- 微服务架构中的服务设计
- DDD 战术设计
