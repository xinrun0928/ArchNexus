# 生产者-消费者模式

一个订单系统，每秒要处理 1000 个订单。你第一反应是多线程并发处理——但如果订单处理涉及数据库写入、库存扣减、短信通知这些 IO 操作，直接开一堆线程去抢资源，数据库连接池会瞬间被打满，系统反而会更慢。

问题出在哪？**生产者和消费者没有解耦。**

生产者-消费者模式正是为了解决这个问题：**通过一个共享队列，把「生产」和「消费」两个环节解耦，让它们各自按照自己的节奏运行**。

## 问题背景

```java
// 错误示例：同步处理，生产和消费强耦合
public class OrderProcessor {
    public void processOrder(Order order) {
        // 数据库写入
        orderRepository.save(order);

        // 库存扣减
        inventoryService.deduct(order.getItems());

        // 发送通知
        notificationService.send(order);

        // 如果任何一个步骤慢，整个流程都被拖住
    }
}

// 调用方
for (Order order : orders) {
    orderProcessor.processOrder(order);  // 一个接一个，同步阻塞
}
```

问题分析：

- 如果 `orderRepository.save()` 变慢，后续的 `inventoryService` 和 `notificationService` 都要等待
- 1000 个订单同时来，会创建 1000 个线程去访问数据库，连接池瞬间耗尽
- 生产和消费强耦合，无法独立扩缩容

## 生产者-消费者模式的核心

```
核心思想：
┌──────────────┐         ┌──────────────┐         ┌──────────────┐
│  Producer    │ ──────▶ │   Queue      │ ──────▶ │  Consumer    │
│  (生产者)    │         │   (队列)     │         │  (消费者)   │
└──────────────┘         └──────────────┘         └──────────────┘
     快速放入                缓冲带                按自己的节奏取
```

**关键特点**：

1. **解耦**：生产者只管往队列放数据，不关心谁在消费
2. **缓冲**：队列作为缓冲带，削峰填谷
3. **异步**：生产者无需等待消费者完成
4. **并发控制**：消费者数量可以独立控制

## 代码实现

### 最简版本：手动实现

```java
// 订单任务
public class OrderTask {
    private final Order order;

    public OrderTask(Order order) {
        this.order = order;
    }

    public void execute() {
        // 处理订单
    }
}

// 线程安全的订单队列
public class OrderQueue {
    private final Queue<OrderTask> queue = new LinkedList<>();
    private final int capacity;

    public OrderQueue(int capacity) {
        this.capacity = capacity;
    }

    // 生产者调用：放入任务
    public synchronized void put(OrderTask task) throws InterruptedException {
        while (queue.size() >= capacity) {
            wait();  // 队列满了，等待
        }
        queue.offer(task);
        notifyAll();  // 通知消费者有新任务
    }

    // 消费者调用：取出任务
    public synchronized OrderTask take() throws InterruptedException {
        while (queue.isEmpty()) {
            wait();  // 队列空了，等待
        }
        notifyAll();  // 通知生产者队列有空间了
        return queue.poll();
    }
}

// 消费者线程
public class Consumer implements Runnable {
    private final OrderQueue queue;
    private final AtomicInteger processedCount;

    public Consumer(OrderQueue queue, AtomicInteger processedCount) {
        this.queue = queue;
        this.processedCount = processedCount;
    }

    @Override
    public void run() {
        try {
            while (true) {
                OrderTask task = queue.take();
                task.execute();
                processedCount.incrementAndGet();
            }
        } catch (InterruptedException e) {
            Thread.currentThread().interrupt();
        }
    }
}

// 使用示例
public class OrderProcessor {
    public void start(int consumerCount) {
        OrderQueue queue = new OrderQueue(1000);  // 队列容量 1000
        AtomicInteger processedCount = new AtomicInteger(0);

        // 启动消费者线程
        ExecutorService consumers = Executors.newFixedThreadPool(consumerCount);
        for (int i = 0; i < consumerCount; i++) {
            consumers.submit(new Consumer(queue, processedCount));
        }

        // 生产者不断往队列放订单
        for (Order order : orderSource) {
            queue.put(new OrderTask(order));
        }
    }
}
```

### 生产版本：使用 BlockingQueue

```java
public class OrderProcessor {
    private final BlockingQueue<OrderTask> queue;
    private final ExecutorService consumerPool;
    private final AtomicLong processedCount = new AtomicLong(0);
    private final AtomicLong rejectedCount = new AtomicLong(0);

    // 拒绝策略：队列满时记录日志并拒绝
    private final RejectedExecutionHandler rejectionHandler = (r, executor) -> {
        OrderTask task = (OrderTask) r;
        log.warn("订单处理队列已满，拒绝订单: {}", task.getOrder().getId());
        rejectedCount.incrementAndGet();
    };

    public OrderProcessor(int queueCapacity, int consumerCount) {
        // 使用有界队列，队列满时拒绝
        this.queue = new LinkedBlockingQueue<>(queueCapacity);

        this.consumerPool = Executors.newFixedThreadPool(consumerCount, r -> {
            Thread t = new Thread(r);
            t.setName("order-consumer-" + t.getId());
            return t;
        });

        // 启动消费者
        for (int i = 0; i < consumerCount; i++) {
            consumerPool.submit(new OrderConsumer());
        }
    }

    // 生产方法：提交订单
    public boolean submitOrder(Order order) {
        OrderTask task = new OrderTask(order);
        boolean accepted = queue.offer(task, 5, TimeUnit.SECONDS);
        if (!accepted) {
            rejectedCount.incrementAndGet();
        }
        return accepted;
    }

    // 消费者
    private class OrderConsumer implements Runnable {
        @Override
        public void run() {
            try {
                while (!Thread.currentThread().isInterrupted()) {
                    OrderTask task = queue.poll(1, TimeUnit.SECONDS);
                    if (task != null) {
                        processTask(task);
                    }
                }
            } catch (InterruptedException e) {
                Thread.currentThread().interrupt();
            }
        }

        private void processTask(OrderTask task) {
            try {
                // 处理订单的各个步骤
                processDatabase(task.getOrder());
                processInventory(task.getOrder());
                processNotification(task.getOrder());
                processedCount.incrementAndGet();
            } catch (Exception e) {
                log.error("处理订单失败: {}", task.getOrder().getId(), e);
                // 可以重新放回队列重试，或者发送到死信队列
                handleFailure(task);
            }
        }
    }

    // 关闭时优雅退出
    public void shutdown() {
        consumerPool.shutdown();
        try {
            if (!consumerPool.awaitTermination(60, TimeUnit.SECONDS)) {
                consumerPool.shutdownNow();
            }
        } catch (InterruptedException e) {
            consumerPool.shutdownNow();
        }
    }
}
```

### 使用线程池 + 队列的现代实现

```java
public class AsyncOrderService {
    private final ThreadPoolExecutor executor;
    private final BlockingQueue<Runnable> workQueue;

    public AsyncOrderService() {
        // 核心线程 4 个，最大线程 16 个，队列容量 1000
        this.workQueue = new LinkedBlockingQueue<>(1000);

        this.executor = new ThreadPoolExecutor(
            4,  // 核心线程数
            16,  // 最大线程数
            60L, TimeUnit.SECONDS,  // 空闲线程存活时间
            workQueue,
            // 自定义线程工厂，带业务标识
            new ThreadFactoryBuilder()
                .setNameFormat("order-processor-%d")
                .build(),
            // 拒绝策略
            new ThreadPoolExecutor.CallerRunsPolicy()  // 由调用方执行
        );
    }

    public CompletableFuture<OrderResult> processOrderAsync(Order order) {
        return CompletableFuture.supplyAsync(() -> {
            // 模拟数据库操作
            orderRepository.save(order);
            // 模拟库存扣减
            inventoryService.deduct(order.getItems());
            // 模拟发送通知
            notificationService.send(order);
            return new OrderResult(order.getId(), true);
        }, executor);
    }
}
```

## 队列类型的选择

| 队列类型 | 适用场景 | 特点 |
|---|---|---|
| `LinkedBlockingQueue` | 大多数场景 | 无界或有界，吞吐量高 |
| `ArrayBlockingQueue` | 需要精确控制容量 | 有界，基于数组，有界后吞吐量稍低 |
| `PriorityBlockingQueue` | 任务需要优先级 | 按优先级排序 |
| `DelayQueue` | 延迟任务 | 元素需要实现 `Delayed` 接口 |
| `SynchronousQueue` | 直接交接 | 不存储元素，生产者必须等消费者 |

## 消费者数量的计算

消费者线程数不是越多越好，需要根据任务类型来定：

| 任务类型 | 瓶颈 | 建议线程数 |
|---|---|---|
| CPU 密集型 | CPU | `CPU 核心数 + 1` |
| IO 密集型 | 网络/磁盘 | `CPU 核心数 × (1 + IO 等待时间/CPU 时间)` |
| 混合型 | 综合 | 通过压测确定 |

```
经验公式：
线程数 = CPU 核心数 × (1 + 等待时间/计算时间)

例如：CPU 核心数 8，IO 等待时间 100ms，计算时间 20ms
线程数 = 8 × (1 + 100/20) = 48
```

### 日志处理

```java
public class LogCollector {
    private final BlockingQueue<LogEvent> queue = new LinkedBlockingQueue<>(10000);

    // 生产者：收集日志
    public void collect(LogLevel level, String message) {
        queue.offer(new LogEvent(level, message, System.currentTimeMillis()));
    }

    // 消费者：批量写入磁盘
    public void startBatchConsumer() {
        Executors.newSingleThreadScheduledExecutor().scheduleAtFixedRate(() -> {
            List<LogEvent> batch = new ArrayList<>();
            queue.drainTo(batch, 1000);  // 批量取出最多 1000 条

            if (!batch.isEmpty()) {
                logWriter.writeBatch(batch);  // 批量写入
            }
        }, 0, 1, TimeUnit.SECONDS);
    }
}
```

## 常见问题

### 问题一：队列容量设置

队列太大 → 内存占用高，延迟增加
队列太小 → 容易拒绝，无法削峰

**建议**：根据 `QPS × 平均处理时间` 估算正常情况下的队列深度，再乘以峰值系数（2~3 倍）。

### 问题二：消费者异常处理

```java
// 错误：异常没处理，直接跳过
while (true) {
    Task task = queue.take();
    task.execute();  // 如果抛出异常，线程就退出了
}

// 正确：捕获异常，任务放回队列重试
while (true) {
    try {
        Task task = queue.take();
        task.execute();
    } catch (Exception e) {
        // 重新放回队列
        queue.put(task);
        log.warn("任务处理失败，重新入队", e);
    }
}
```

### 问题三：优雅关闭

```java
public void shutdown() {
    // 1. 停止接收新任务
    executor.shutdown();

    // 2. 等待队列中的任务完成
    if (!executor.awaitTermination(30, TimeUnit.SECONDS)) {
        // 3. 超时后强制中断
        executor.shutdownNow();

        // 4. 记录未完成的任务
        queue.forEach(task -> log.warn("未完成的任务: {}", task));
    }
}
```

## 总结

生产者-消费者模式通过队列解耦了生产与消费两个环节，让它们可以独立扩缩容、独立处理异常、独立调优。

**核心要点**：

- 使用 `BlockingQueue` 作为共享队列
- 生产者和消费者线程数可以独立配置
- 队列容量需要根据业务特点估算
- 需要处理好消费者异常和优雅关闭

**适用场景**：

- 订单处理、消息推送等异步任务
- 日志收集、指标上报等批量处理
- 任何「生产」和「消费」速度不匹配的场景

**不适用场景**：

- 需要实时响应的场景（队列本身就有延迟）
- 任务之间有强依赖关系的场景
