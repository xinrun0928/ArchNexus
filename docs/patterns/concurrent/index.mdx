# 并发设计模式

想象一个场景：订单系统高峰期，每秒涌入上万笔订单。负责接收订单的线程忙得不可开交，负责库存校验的线程却大部分时间在空等，负责发送通知的线程更是被晾在一边。线程之间的协作乱成一锅粥，CPU 利用率却低得可怜。

这就是多线程协作的真实困境——不是线程不够用，而是线程之间的配合没有章法。

## 线程间协作的本质问题

线程间协作面临三个核心挑战：

**速度不匹配**。生产者速度和消费者速度往往不一致。如果生产者一股脑往队列里塞数据，消费者来不及处理，数据就会堆积；如果消费者速度比生产者快，大量时间浪费在空等上。

**资源竞争**。当多个线程同时访问共享资源时，读操作可能被写操作打断，写操作可能被并发修改。没有有效的协调机制，数据一致性问题随之而来。

**依赖等待**。一个线程的结果可能是另一个线程的输入。如何让等待的线程在条件满足时及时被唤醒，而不是反复轮询浪费 CPU？

这些问题的解决方案，不应该是「加锁就能解决」或者「用队列就行」这种简单思路。每一种协作场景都有其特点，需要针对性的设计。

## 并发模式的价值

并发设计模式是经过无数实践验证的解决方案。它们不只是代码模板，更是对特定协作场景的深刻抽象：

| 场景 | 模式 | 解决的问题 |
|---|---|---|
| 生产者与消费者速度不匹配 | Producer-Consumer | 缓冲、削峰、解耦 |
| 读多写少场景 | Read-Write Lock | 减少锁竞争，提升并发读性能 |
| 异步调用等待结果 | Future | 非阻塞获取计算结果 |
| 条件不满足时等待 | Guarded Suspension | 线程安全地等待条件成立 |
| 状态驱动的不必要操作 | Balking | 避免无效的重复执行 |

理解这些模式的本质，能够帮助你在遇到具体问题时快速找到合适的方案，而不是从零开始摸索。

## 章节内容

本章节将系统讲解五种核心的并发设计模式：

- **生产者-消费者模式**：通过缓冲队列解耦生产者和消费者，让它们各自以自己的节奏运行
- **读写锁模式**：区分读和写的访问模式，在保证数据一致性的同时最大化并发读性能
- **Future 模式**：将异步执行与结果获取分离，避免同步阻塞
- **Guarded Suspension 模式**：在条件不满足时优雅地挂起线程，条件满足时精准唤醒
- **Balking 模式**：当对象状态不符合操作要求时，直接放弃而不是排队等待

每种模式都会从「问题背景」「解决方案」「实现要点」「适用场景」四个维度展开，结合 Java 代码示例，帮助你建立对并发协作模式的完整认知。
