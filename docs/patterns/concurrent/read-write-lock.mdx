# 读写锁模式

你接手了一个缓存系统，读操作占了 99%，写操作只有 1%。为了保证线程安全，你给所有读写操作都加上了 `synchronized`——结果系统吞吐量反而下降了，因为读请求也在互相等待。

问题出在哪？**读和写的并发度没有区分。** 读写锁模式正是为了解决这个问题：**读操作可以并发，写操作需要独占**。

## 问题背景

```java
// 错误示例：所有操作都加锁，读读也要互斥
public class CacheService {
    private Map<String, Object> cache = new HashMap<>();
    private final Object lock = new Object();

    // 读操作
    public Object get(String key) {
        synchronized (lock) {
            return cache.get(key);
        }
    }

    // 写操作
    public void put(String key, Object value) {
        synchronized (lock) {
            cache.put(key, value);
        }
    }
}
```

问题分析：

- 1000 个读请求同时到来，都要排队等锁
- 实际上读操作之间不冲突，可以同时进行
- 99% 的读请求被 1% 的写请求拖累

## 读写锁模式的核心

```
核心思想：
┌─────────────────────────────────────────────┐
│              ReadWriteLock                  │
├─────────────────────────────────────────────┤
│  读锁：多个线程可以同时持有，互不阻塞        │
│  写锁：独占锁，一次只能有一个线程持有        │
│                                             │
│  读 ──▶ [读] [读] [读]  // 可以并发          │
│  写 ──▶ [写]           // 独占              │
└─────────────────────────────────────────────┘
```

**关键特点**：

1. **读读不互斥**：多个线程可以同时读取
2. **读写互斥**：读的时候不能写，写的时候不能读
3. **写写互斥**：同一时间只能有一个写操作

## 代码实现

### 基本实现

```java
import java.util.concurrent.locks.ReadWriteLock;
import java.util.concurrent.locks.ReentrantReadWriteLock;

public class CacheService {
    private Map<String, Object> cache = new HashMap<>();
    private final ReadWriteLock rwLock = new ReentrantReadWriteLock();

    // 读操作：获取读锁
    public Object get(String key) {
        rwLock.readLock().lock();
        try {
            return cache.get(key);
        } finally {
            rwLock.readLock().unlock();
        }
    }

    // 写操作：获取写锁
    public void put(String key, Object value) {
        rwLock.writeLock().lock();
        try {
            cache.put(key, value);
        } finally {
            rwLock.writeLock().unlock();
        }
    }

    // 批量写：写锁保证原子性
    public void putAll(Map<String, Object> updates) {
        rwLock.writeLock().lock();
        try {
            cache.putAll(updates);
        } finally {
            rwLock.writeLock().unlock();
        }
    }

    // 清空缓存
    public void clear() {
        rwLock.writeLock().lock();
        try {
            cache.clear();
        } finally {
            rwLock.writeLock().unlock();
        }
    }
}
```

### 缓存 + 过期机制

```java
public class ExpirableCache<K, V> {
    private final Map<K, CacheEntry<V>> cache = new HashMap<>();
    private final ReadWriteLock rwLock = new ReentrantReadWriteLock();
    private final ScheduledExecutorService cleaner;

    private static class CacheEntry<V> {
        final V value;
        final long expireTime;

        CacheEntry(V value, long ttlMillis) {
            this.value = value;
            this.expireTime = System.currentTimeMillis() + ttlMillis;
        }

        boolean isExpired() {
            return System.currentTimeMillis() > expireTime;
        }
    }

    public ExpirableCache() {
        // 每分钟清理过期缓存
        this.cleaner = Executors.newSingleThreadScheduledExecutor(r -> {
            Thread t = new Thread(r, "cache-cleaner");
            t.setDaemon(true);
            return t;
        });
        cleaner.scheduleAtFixedRate(this::cleanExpired, 1, 1, TimeUnit.MINUTES);
    }

    public V get(K key) {
        rwLock.readLock().lock();
        try {
            CacheEntry<V> entry = cache.get(key);
            if (entry == null || entry.isExpired()) {
                return null;
            }
            return entry.value;
        } finally {
            rwLock.readLock().unlock();
        }
    }

    public void put(K key, V value, long ttlMillis) {
        rwLock.writeLock().lock();
        try {
            cache.put(key, new CacheEntry<>(value, ttlMillis));
        } finally {
            rwLock.writeLock().unlock();
        }
    }

    private void cleanExpired() {
        rwLock.writeLock().lock();
        try {
            cache.entrySet().removeIf(entry -> entry.getValue().isExpired());
        } finally {
            rwLock.writeLock().unlock();
        }
    }

    public void shutdown() {
        cleaner.shutdown();
    }
}
```

### 缓存加载（Cache-Aside）

```java
public class CacheAside<K, V> {
    private final Map<K, V> cache = new ConcurrentHashMap<>();
    private final ReadWriteLock rwLock = new ReentrantReadWriteLock();
    private final Function<K, V> loader;  // 缓存未命中时加载

    public CacheAside(Function<K, V> loader) {
        this.loader = loader;
    }

    public V get(K key) {
        // 先从缓存读
        V value = cache.get(key);
        if (value != null) {
            return value;
        }

        // 缓存未命中，加载数据
        rwLock.writeLock().lock();
        try {
            // 双重检查：可能其他线程已经加载了
            value = cache.get(key);
            if (value != null) {
                return value;
            }

            // 加载并放入缓存
            value = loader.apply(key);
            cache.put(key, value);
            return value;
        } finally {
            rwLock.writeLock().unlock();
        }
    }

    // 批量加载，避免缓存雪崩
    public Map<K, V> getAll(List<K> keys) {
        Map<K, V> result = new HashMap<>();
        List<K> missingKeys = new ArrayList<>();

        // 先批量从缓存读
        for (K key : keys) {
            V value = cache.get(key);
            if (value != null) {
                result.put(key, value);
            } else {
                missingKeys.add(key);
            }
        }

        // 批量加载缺失的
        if (!missingKeys.isEmpty()) {
            rwLock.writeLock().lock();
            try {
                for (K key : missingKeys) {
                    V value = cache.get(key);
                    if (value == null) {
                        value = loader.apply(key);
                        cache.put(key, value);
                    }
                    result.put(key, value);
                }
            } finally {
                rwLock.writeLock().unlock();
            }
        }

        return result;
    }
}
```

## 读写锁 vs 普通锁

| 场景 | 普通 `synchronized` | `ReentrantReadWriteLock` |
|---|---|---|
| 100 读 1 写 | 100 + 1 = 101 次加锁 | 1 次写锁，100 次读锁（并发） |
| 读并发度 | 0（串行） | 高 |
| 写并发度 | 0（串行） | 0（仍然独占） |
| 适用场景 | 读写均衡 | **读多写少** |

```
性能对比（100 读 + 1 写，并发线程 100）：

synchronized：所有操作串行执行
  Thread-1: [读]
  Thread-2: [读]          // 等待
  Thread-3: [读]          // 等待
  ...
  Thread-100: [读]         // 等待
  Thread-99:  [写]        // 等待
  总耗时: 101 × T

ReadWriteLock：读操作并发
  Thread-1~100: [读] [读] [读] ... [读]  // 并发执行
  Thread-99:    [写]                    // 等待读完成
  总耗时: T + T = 2T
```

## 读写锁的锁降级

读写锁支持锁降级：**从写锁降级为读锁**。但不支持锁升级。

```java
public Object getAndUpdate(String key, Function<Object, Object> updater) {
    rwLock.writeLock().lock();
    try {
        Object value = cache.get(key);
        Object newValue = updater.apply(value);
        cache.put(key, newValue);
        // 锁降级：释放写锁，获取读锁
        rwLock.readLock().lock();  // [!code highlight]
    } finally {
        rwLock.writeLock().unlock();  // 释放写锁，但还持有读锁
    }
    try {
        return cache.get(key);
    } finally {
        rwLock.readLock().unlock();
    }
}
```

:::warning
**不要锁升级**：从读锁升级为写锁会导致死锁。

```java
// 错误：锁升级会导致死锁
rwLock.readLock().lock();
try {
    // ...
    rwLock.writeLock().lock();  // 永远等待，因为其他读线程持有读锁
} finally {
    rwLock.readLock().unlock();
    rwLock.writeLock().unlock();
}
```
:::

## 读写锁的实际应用

### 配置中心

```java
public class ConfigCenter {
    private final Map<String, String> configs = new HashMap<>();
    private final ReadWriteLock rwLock = new ReentrantReadWriteLock();

    // 读取配置：高频操作
    public String get(String key, String defaultValue) {
        rwLock.readLock().lock();
        try {
            return configs.getOrDefault(key, defaultValue);
        } finally {
            rwLock.readLock().unlock();
        }
    }

    // 读取所有配置
    public Map<String, String> getAll() {
        rwLock.readLock().lock();
        try {
            return new HashMap<>(configs);
        } finally {
            rwLock.readLock().unlock();
        }
    }

    // 更新配置：低频操作
    public void update(String key, String value) {
        rwLock.writeLock().lock();
        try {
            configs.put(key, value);
        } finally {
            rwLock.writeLock().unlock();
        }
    }

    // 批量更新
    public void updateAll(Map<String, String> updates) {
        rwLock.writeLock().lock();
        try {
            configs.putAll(updates);
        } finally {
            rwLock.writeLock().unlock();
        }
    }
}
```

### 计数器

```java
public class Statistics {
    private final Map<String, Long> counters = new ConcurrentHashMap<>();
    private final ReadWriteLock rwLock = new ReentrantReadWriteLock();

    // 计数 +1
    public void increment(String metric) {
        counters.compute(metric, (k, v) -> v == null ? 1L : v + 1);
    }

    // 获取统计值
    public long get(String metric) {
        rwLock.readLock().lock();
        try {
            return counters.getOrDefault(metric, 0L);
        } finally {
            rwLock.readLock().unlock();
        }
    }

    // 获取所有统计（需要快照）
    public Map<String, Long> getSnapshot() {
        rwLock.readLock().lock();
        try {
            return new HashMap<>(counters);
        } finally {
            rwLock.readLock().unlock();
        }
    }

    // 重置统计
    public void reset() {
        rwLock.writeLock().lock();
        try {
            counters.clear();
        } finally {
            rwLock.writeLock().unlock();
        }
    }
}
```

## 读写锁的公平性

`ReentrantReadWriteLock` 支持公平和非公平模式：

```java
// 非公平模式（默认）：插队
// 写锁释放后，等待的读锁和新来的读锁竞争
ReadWriteLock unfairLock = new ReentrantReadWriteLock();

// 公平模式：先来先得
// 等待时间最长的线程优先获得锁
ReadWriteLock fairLock = new ReentrantReadWriteLock(true);
```

| 模式 | 优点 | 缺点 |
|---|---|---|
| 非公平 | 吞吐量高 | 可能造成写线程饥饿 |
| 公平 | 无饥饿 | 吞吐量低 |

**建议**：大多数场景用非公平模式，除非有严格的公平性要求。

## 常见问题

### 问题一：读锁下的数据一致性问题

```java
// 错误：在读锁下进行耗时操作
rwLock.readLock().lock();
try {
    Object value = cache.get(key);
    // 耗时操作在锁内，导致其他线程长时间等待
    process(value);  // [!code warning]
} finally {
    rwLock.readLock().unlock();
}
```

**正确做法**：把耗时操作放到锁外

```java
// 正确：先获取数据，释放锁后再处理
Object value;
rwLock.readLock().lock();
try {
    value = cache.get(key);
} finally {
    rwLock.readLock().unlock();
}

if (value != null) {
    process(value);  // 耗时操作在锁外
}
```

### 问题二：StampedLock 替代方案

Java 8 引入了 `StampedLock`，它比 `ReadWriteLock` 性能更好：

```java
public class CacheServiceWithStampedLock {
    private Map<String, Object> cache = new HashMap<>();
    private final StampedLock stampedLock = new StampedLock();

    // 读操作
    public Object get(String key) {
        long stamp = stampedLock.readLock();
        try {
            return cache.get(key);
        } finally {
            stampedLock.unlockRead(stamp);
        }
    }

    // 写操作
    public void put(String key, Object value) {
        long stamp = stampedLock.writeLock();
        try {
            cache.put(key, value);
        } finally {
            stampedLock.unlockWrite(stamp);
        }
    }

    // 乐观读：不需要真正获取锁
    public Object getOptimistic(String key) {
        long stamp = stampedLock.tryOptimisticRead();
        Object value = cache.get(key);
        // 检查读取期间是否有写操作
        if (!stampedLock.validate(stamp)) {
            // 有写操作，降级为悲观读
            stamp = stampedLock.readLock();
            try {
                value = cache.get(key);
            } finally {
                stampedLock.unlockRead(stamp);
            }
        }
        return value;
    }
}
```

## 总结

读写锁模式通过区分读锁和写锁，显著提升了「读多写少」场景的并发性能。

**核心要点**：

- 读锁可以并发获取，写锁独占
- 读写锁不支持锁升级，但支持锁降级
- 注意读锁下的数据一致性
- `StampedLock` 是更现代的选择

**适用场景**：

- 缓存系统（读多写少）
- 配置中心（读多写少）
- 统计计数器（读多写少）
- 文档编辑器（读多写少）

**不适用场景**：

- 读写均衡或写多读少（此时普通锁性能更好）
- 读操作本身就很耗时（锁占用时间过长）
- 需要严格公平性的场景
