# 适配器模式

你的系统需要对接一个第三方支付接口，但对方的 SDK 接口和你的系统不兼容：你的系统用 `PayResult`，对方用 `ThirdPartyResponse`。你不想改现有代码，也不想改第三方 SDK，怎么办？

适配器模式正是为了解决这个问题：**把一个类的接口转换成另一个接口，让原本不兼容的类可以合作**。

## 问题背景

```java
// 你的系统期望的接口
public interface PaymentGateway {
    PayResult pay(String orderId, BigDecimal amount);
}

// 第三方 SDK 的接口
public class ThirdPartySDK {
    public ThirdPartyResponse processPayment(String transactionId, int cents) {
        // ...
    }
}
```

问题：第三方接口的参数和返回值都和你的不兼容，不能直接用。

## 适配器模式的核心

适配器模式将一个类的接口转换成客户端期望的另一个接口，让原本接口不兼容的类可以一起工作。

```
核心思想：
┌──────────────┐          ┌──────────────┐
│   Client     │────────▶│   Target     │
│   (客户端)   │          │   (目标接口) │
└──────────────┘          └──────┬───────┘
                                 │ 使用
                          ┌──────▼───────┐
                          │   Adapter    │
                          │   (适配器)    │
                          └──────┬───────┘
                                 │ 调用
                          ┌──────▼───────┐
                          │   Adaptee    │
                          │  (被适配者)  │
                          └──────────────┘
```

## 代码实现

### 类适配器（通过继承）

```java
// 目标接口
public interface PaymentGateway {
    PayResult pay(String orderId, BigDecimal amount);
}

// 被适配者（第三方 SDK）
public class ThirdPartySDK {
    public ThirdPartyResponse processPayment(String transactionId, int cents) {
        // 模拟调用
        return new ThirdPartyResponse("SUCCESS", transactionId);
    }
}

// 适配器：通过继承实现
public class PaymentAdapter extends ThirdPartySDK implements PaymentGateway {

    @Override
    public PayResult pay(String orderId, BigDecimal amount) {
        // 1. 参数转换：元 → 分
        int cents = amount.multiply(new BigDecimal("100")).intValue();

        // 2. 调用第三方 SDK
        ThirdPartyResponse response = processPayment(orderId, cents);

        // 3. 结果转换：第三方响应 → 系统响应
        return new PayResult(
            response.isSuccess(),
            orderId,
            response.getTransactionId(),
            amount
        );
    }
}
```

### 对象适配器（通过组合）

```java
// 适配器：通过组合实现
public class PaymentAdapter implements PaymentGateway {
    // 持有被适配者的引用
    private ThirdPartySDK thirdPartySDK;

    public PaymentAdapter(ThirdPartySDK thirdPartySDK) {
        this.thirdPartySDK = thirdPartySDK;
    }

    @Override
    public PayResult pay(String orderId, BigDecimal amount) {
        int cents = amount.multiply(new BigDecimal("100")).intValue();
        ThirdPartyResponse response = thirdPartySDK.processPayment(orderId, cents);

        return new PayResult(
            response.isSuccess(),
            orderId,
            response.getTransactionId(),
            amount
        );
    }
}
```

### 使用

```java
// 客户端代码：无感知地使用适配器
public class OrderService {
    private PaymentGateway paymentGateway;

    // 通过依赖注入，屏蔽具体实现
    public OrderService(PaymentGateway paymentGateway) {
        this.paymentGateway = paymentGateway;
    }

    public void payOrder(String orderId, BigDecimal amount) {
        PayResult result = paymentGateway.pay(orderId, amount);
        if (result.isSuccess()) {
            // 更新订单状态
        }
    }
}

// 使用
public class Main {
    public static void main(String[] args) {
        ThirdPartySDK sdk = new ThirdPartySDK();
        PaymentGateway gateway = new PaymentAdapter(sdk);

        OrderService orderService = new OrderService(gateway);
        orderService.payOrder("ORD-123", new BigDecimal("99.99"));
    }
}
```

## 类适配器 vs 对象适配器

| 维度 | 类适配器 | 对象适配器 |
|---|---|---|
| 实现方式 | 继承 | 组合 |
| 灵活性 | 低（受限于继承） | 高（可以适配被适配者的子类） |
| 覆盖行为 | 可以覆盖被适配者的方法 | 需要委托 |
| 多重继承 | Java 不支持 | 可以工作 |

**推荐**：大多数情况下使用对象适配器。

## 适配器模式的实际应用

### 日志框架适配

```java
// 你的系统定义的日志接口
public interface Logger {
    void info(String message);
    void error(String message);
}

// 第三方日志库
public class Slf4jLogger {
    public void log(String level, String msg) { }
}

// 适配器
public class Slf4jLoggerAdapter implements Logger {
    private Slf4jLogger logger;

    public Slf4jLoggerAdapter(Slf4jLogger logger) {
        this.logger = logger;
    }

    @Override
    public void info(String message) {
        logger.log("INFO", message);
    }

    @Override
    public void error(String message) {
        logger.log("ERROR", message);
    }
}
```

### 数据格式适配

```java
// 你的系统期望的格式
public interface DataParser {
    User parseUser(String data);
}

// XML 格式的旧系统
public class XmlDataParser {
    public Document parse(String xml) {
        // 解析 XML
        return new Document(xml);
    }
}

// 适配器：XML → User
public class XmlUserParserAdapter implements DataParser {
    private XmlDataParser xmlParser;

    @Override
    public User parseUser(String xmlData) {
        Document doc = xmlParser.parse(xmlData);

        return User.builder()
            .firstName(doc.getValue("//user/firstName"))
            .lastName(doc.getValue("//user/lastName"))
            .email(doc.getValue("//user/email"))
            .build();
    }
}
```

### API 版本适配

```java
// 最新的 API 接口
public interface UserService {
    User getUser(String userId);
    void updateUser(User user);
}

// 旧版 API 实现
public class LegacyUserService {
    public Map<String, Object> fetchUser(String id) {
        // 旧版 API 返回 Map
        return Map.of("name", "John", "email", "john@example.com");
    }
}

// 适配器
public class UserServiceAdapter implements UserService {
    private LegacyUserService legacyService;

    @Override
    public User getUser(String userId) {
        Map<String, Object> legacyData = legacyService.fetchUser(userId);

        return User.builder()
            .firstName((String) legacyData.get("name"))
            .email((String) legacyData.get("email"))
            .build();
    }

    @Override
    public void updateUser(User user) {
        // 转换为旧版格式，调用旧版 API
    }
}
```

## 适配器 vs 装饰器 vs 代理

很多人分不清这三个模式：

| 模式 | 目的 | 改变什么 |
|---|---|---|
| 适配器 | 让不兼容的接口合作 | 接口 |
| 装饰器 | 动态添加功能 | 行为 |
| 代理 | 控制对对象的访问 | 访问方式 |

```
适配器：让「方形的钉子」能装进「圆形的孔」
装饰器：给对象「穿上外套」
代理：「替身」代替对象工作
```

## 适配器模式的特点

### 优点

1. **解耦**：客户端代码与具体实现解耦
2. **复用**：复用现有类，不需要修改
3. **符合开闭原则**：添加新适配器不影响现有代码

### 缺点

1. **增加复杂度**：引入适配器层会增加代码复杂度
2. **性能开销**：适配器有额外的调用开销
3. **隐藏问题**：可能掩盖接口设计问题

## 总结

适配器模式通过转换接口，让不兼容的类可以一起工作。

**核心要点**：

- 将一个类的接口转换成另一个接口
- 让原本不兼容的类可以合作
- 客户端无感知地使用适配器

**使用场景**：

- 集成第三方库或 SDK
- 接口版本兼容
- 数据格式转换
- 系统迁移时的兼容层

**类适配器 vs 对象适配器**：大多数情况下用对象适配器。
