# 结构型设计模式

面向对象编程的核心优势是什么？很多人会说是「封装」「继承」「多态」——这没错，但如果只能选一个答案，我选「组合」。

为什么？继承是静态的，一旦类被定义，它的结构就固定了。你无法在运行时动态地给一个类添加行为，也无法在不修改源码的情况下改变对象之间的协作关系。而组合是动态的——你可以在运行时组装对象，决定它们如何协作，可以随时替换某个部件而不用改动整体。

这才是结构型设计模式要解决的问题：如何通过对象组合，以灵活、可复用的方式构建复杂的软件结构。

## 为什么组合优于继承

继承的弱点在实践中暴露得很彻底。当你发现一个父类的方法不适合某个子类时，通常只有两条路：要么在子类里 Override，要么改父类——但改父类可能影响其他所有子类。这就是所谓的「类爆炸」问题：层级越来越深，耦合越来越紧，改一处影响一片。

组合把这个���系倒过来。你不再问「这个类应该继承谁」，而是问「这个对象需要什么能力」。

```java
// 继承方式：星巴克咖啡
class CoffeeMachine {
    void makeCoffee() { ... }
}

class StarbucksCoffeeMachine extends CoffeeMachine {
    @Override
    void makeCoffee() {
        // 加入星巴克特有的配方
    }
}
```

一旦需要加「加糖」「加奶」等功能，继承就开始失控——你需要组合出 `StarbucksWithSugar`、`StarbucksWithMilk`、`StarbucksWithSugarAndMilk`……每增加一种配料，类的数量就是指数级增长。

而用组合：

```java
// 组合方式
interface Condiment {
    Coffee make(Coffee coffee);
}

class Sugar implements Condiment {
    public Coffee make(Coffee coffee) {
        return new Coffee(coffee.getBase() + ", sugar");
    }
}

class Milk implements Condiment {
    public Coffee make(Coffee coffee) {
        return new Coffee(coffee.getBase() + ", milk");
    }
}

// 使用时：想加什么就加什么，数量不受限制
Coffee myCoffee = new Coffee("espresso");
myCoffee = new Sugar().make(myCoffee);
myCoffee = new Milk().make(myCoffee);
```

组合让扩展从「改类」变成「加类」。新增一种配料只需要写一个新的 `Condiment` 实现，不用动任何现有代码。这就是所谓的「开闭原则」——对扩展开放，对修改封闭。

## 七种结构型模式概览

结构型模式一共有七种，它们解决的问题都可以归结为「如何组织对象之间的关系」：

| 模式 | 核心问题 | 典型场景 |
|---|---|---|
| **适配器模式**（Adapter） | 接口不兼容 | 将旧系统接入新接口 |
| **装饰器模式**（Decorator） | 动态添加职责 | 给对象添加日志、缓存、校验 |
| **代理模式**（Proxy） | 访问控制 | 延迟加载、权限控制、远程调用 |
| **外观模式**（Facade） | 接口过多 | 简化子系统交互入口 |
| **桥接模式**（Bridge） | 多维度变化 | 平台与功能正交扩展 |
| **组合模式**（Composite） | 整体与部分一致 | 树形结构组织 |
| **享元模式**（Flyweight） | 对象太多太重 | 大量细粒度对象的共享 |

这里有一个重要的区分：**装饰器、代理、适配器**看起来都是在「包裹」一个对象，但它们的意图完全不同：

- **适配器**：让不兼容的接口变得兼容，侧重「翻译」
- **装饰器**：动态地给对象添加职责，侧重「增强」
- **代理**：控制对对象的访问，侧重「保护或延迟」

理解意图比记忆代码更重要。同样是包裹��意图不同，设计的方向就完全不同。

## 章节内容

本章节将逐一分析这七种结构型模式：

- **适配器模式**：如何让「语言不通」的接口顺利对话
- **装饰器模式**：如何在运行时给对象叠加能力
- **代理模式**：如何在访问层面做文章
- **外观模式**：如何给复杂的子系统一个简单的门面
- **桥接模式**：如何让两个维度的变化独立扩展
- **组合模式**：如何让「整体」和「部分」用同一套接口操作
- **享元模式**：如何通过共享减少对象数量

每种模式都会围绕三个核心问题展开：**它解决了什么**、**核心实现思路**、**什么场景不该用**。学设计模式不是为了背代码，而是为了理解每种设计背后的权衡——知道什么时候用，比知道怎么用更重要。
