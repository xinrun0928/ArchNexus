# 抽象工厂模式

工厂方法模式解决了"一个产品线"的问题。但如果你的系统需要支持**多个产品线**（比如同时支持 Windows 风格和 Mac 风格的各种组件），工厂方法就力不从心了。

抽象工厂模式正是为了解决"产品族"问题而生的。

## 问题背景

假设你开发一个 UI 框架，需要同时支持 Windows 和 Mac 两种风格：

```java
// Windows 风格
public class WindowsButton { }
public class WindowsMenu { }
public class WindowsDialog { }

// Mac 风格
public class MacButton { }
public class MacMenu { }
public class MacDialog { }
```

如果用工厂方法，你需要为每种风格定义一套工厂：

```java
// Windows 工厂
public class WindowsFactory {
    public Button createButton() { return new WindowsButton(); }
    public Menu createMenu() { return new WindowsMenu(); }
    public Dialog createDialog() { return new WindowsDialog(); }
}

// Mac 工厂
public class MacFactory {
    public Button createButton() { return new MacButton(); }
    public Menu createMenu() { return new MacMenu(); }
    public Dialog createDialog() { return new MacDialog(); }
}
```

这就是抽象工厂模式。

## 抽象工厂模式的核心

抽象工厂模式提供一个创建**产品族**的接口，而子类决定这个族里具体有什么产品。

```
核心思想：
┌─────────────┐
│ AbstractFactory │
│ (抽象工厂)     │
└──────┬──────┘
       │ 创建
       ▼
┌─────────────┐    ┌─────────────┐
│   Button    │    │    Menu     │
└─────────────┘    └─────────────┘

ConcreteFactoryA        ConcreteFactoryB
(Windows 工厂)           (Mac 工厂)
    │                       │
    ▼                       ▼
WindowsButton          MacButton
WindowsMenu            MacMenu
```

## 代码实现

### 产品接口层

```java
// 产品族接口
public interface Button {
    void paint();
}

public interface Menu {
    void paint();
}

public interface Dialog {
    void paint();
}
```

### 具体产品

```java
// Windows 风格
public class WindowsButton implements Button {
    @Override
    public void paint() {
        System.out.println("Windows Button");
    }
}

public class WindowsMenu implements Menu {
    @Override
    public void paint() {
        System.out.println("Windows Menu");
    }
}

public class WindowsDialog implements Dialog {
    @Override
    public void paint() {
        System.out.println("Windows Dialog");
    }
}

// Mac 风格
public class MacButton implements Button {
    @Override
    public void paint() {
        System.out.println("Mac Button");
    }
}

public class MacMenu implements Menu {
    @Override
    public void paint() {
        System.out.println("Mac Menu");
    }
}

public class MacDialog implements Dialog {
    @Override
    public void paint() {
        System.out.println("Mac Dialog");
    }
}
```

### 抽象工厂

```java
// 抽象工厂接口
public interface GUIFactory {
    Button createButton();
    Menu createMenu();
    Dialog createDialog();
}

// 具体工厂
public class WindowsFactory implements GUIFactory {
    @Override
    public Button createButton() {
        return new WindowsButton();
    }

    @Override
    public Menu createMenu() {
        return new WindowsMenu();
    }

    @Override
    public Dialog createDialog() {
        return new WindowsDialog();
    }
}

// 具体工厂
public class MacFactory implements GUIFactory {
    @Override
    public Button createButton() {
        return new MacButton();
    }

    @Override
    public Menu createMenu() {
        return new MacMenu();
    }

    @Override
    public Dialog createDialog() {
        return new MacDialog();
    }
}
```

### 使用

```java
public class Application {
    private GUIFactory factory;
    private Button button;
    private Menu menu;

    // 通过构造函数注入工厂
    public Application(GUIFactory factory) {
        this.factory = factory;
    }

    public void createUI() {
        button = factory.createButton();
        menu = factory.createMenu();
    }

    public void paint() {
        button.paint();
        menu.paint();
    }
}

// 客户端代码
public class Main {
    public static void main(String[] args) {
        // 根据配置创建工厂
        String os = System.getProperty("os.name");
        GUIFactory factory;

        if (os.startsWith("Windows")) {
            factory = new WindowsFactory();
        } else {
            factory = new MacFactory();
        }

        Application app = new Application(factory);
        app.createUI();
        app.paint();
    }
}
```

## 抽象工厂 vs 工厂方法

| 维度 | 工厂方法 | 抽象工厂 |
|---|---|---|
| 产品 | 一个产品（多个子类） | 多个相关产品（产品族） |
| 创建方式 | 子类决定创建哪个 | 子类决定创建哪些 |
| 工厂数量 | 一个工厂类 | 每个产品族一个工厂 |
| 扩展方式 | 加新具体产品 | 加新产品族（要改接口） |

```
工厂方法：产品线扩展容易，工厂类不变
抽象工厂：产品族扩展容易，产品类不变

工厂方法：添加新具体产品 → 加一个子类
抽象工厂：添加新产品族 → 加一整套子类
```

## 工厂方法的实际应用

### 数据库访问层

```java
// DAO 接口（产品）
public interface UserDao {
    User findById(Long id);
    void save(User user);
}

public interface OrderDao {
    Order findById(Long id);
    void save(Order order);
}

// 抽象工厂
public interface DaoFactory {
    UserDao createUserDao();
    OrderDao createOrderDao();
}

// MySQL 工厂
public class MySqlDaoFactory implements DaoFactory {
    @Override
    public UserDao createUserDao() {
        return new MySqlUserDao();
    }

    @Override
    public OrderDao createOrderDao() {
        return new MySqlOrderDao();
    }
}

// PostgreSQL 工厂
public class PostgresDaoFactory implements DaoFactory {
    @Override
    public UserDao createUserDao() {
        return new PostgresUserDao();
    }

    @Override
    public OrderDao createOrderDao() {
        return new PostgresOrderDao();
    }
}
```

### 序列化工厂

```java
// 序列化器接口
public interface Serializer {
    String serialize(Object obj);
    Object deserialize(String data);
}

public interface Deserializer {
    Object deserialize(String data);
}

// 抽象工厂
public interface SerializationFactory {
    Serializer createSerializer();
    Deserializer createDeserializer();
}

// JSON 工厂
public class JsonFactory implements SerializationFactory {
    @Override
    public Serializer createSerializer() {
        return new JsonSerializer();
    }

    @Override
    public Deserializer createDeserializer() {
        return new JsonDeserializer();
    }
}

// Protocol Buffers 工厂
public class ProtobufFactory implements SerializationFactory {
    @Override
    public Serializer createSerializer() {
        return new ProtobufSerializer();
    }

    @Override
    public Deserializer createDeserializer() {
        return new ProtobufDeserializer();
    }
}
```

## 抽象工厂模式的特点

### 优点

1. **产品族一致性**：保证同一族的产品一起使用
2. **解耦**：客户端代码不需要知道具体实现
3. **符合开闭原则**：产品族扩展容易

### 缺点

1. **产品族扩展困难**：添加新产品族要改接口
2. **类数量增加**：每个产品族都要一套工厂
3. **增加新产品的困难**：如果产品结构不稳定，抽象工厂需要频繁修改

## 适用场景

**适合使用抽象工厂的场景**：

- 系统需要支持多个产品族
- 同一个族的產品应该一起使用
- 希望提供一个产品类库，只暴露接口，不暴露实现

**不适合使用抽象工厂的场景**：

- 产品族不固定，经常变化
- 需要经常添加新产品
- 只关心单个产品的创建

## 总结

抽象工厂模式通过"产品族"的概念，解決了相关产品需要一起创建的问题。

**核心要点**：

- 提供创建产品族的接口
- 保证同一族的产品一起使用
- 符合开闭原则的"部分扩展"

**与工厂方法的区别**：

- 工厂方法：产品线扩展容易
- 抽象工厂：产品族扩展容易

选择哪种模式，取决于你的产品结构是"一条线变化"还是"一整套变化"。
