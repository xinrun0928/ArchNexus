# 单例模式

在很多场景中，我们需要一个全局唯一的对象：配置管理器、线程池、缓存、日志系统。这个对象从程序启动到结束，应该只有**一个实例**，任何地方访问都是同一个对象。

这就是单例模式要解决的问题。

## 问题背景

假设我们需要一个配置管理器：

```java
// 错误示例：每次 new 都是新对象
public class ConfigManager {
    private Map<String, String> configs = new HashMap<>();

    public void set(String key, String value) {
        configs.put(key, value);
    }

    public String get(String key) {
        return configs.get(key);
    }
}

// 使用
ConfigManager cm1 = new ConfigManager();
cm1.set("db_host", "localhost");

ConfigManager cm2 = new ConfigManager();
System.out.println(cm2.get("db_host")); // null！不是同一个对象
```

问题很明显：每次 `new` 都创建新对象，`cm1` 设置的值 `cm2` 看不到。

## 单例模式的核心

单例模式确保一个类**只有一个实例**，并提供一个全局访问点。

### 核心要点

1. **私有构造函数**：禁止外部通过 `new` 创建实例
2. **私有静态实例**：类内部维护唯一实例
3. **公有静态方法**：提供全局访问点

## 实现方式

### 饿汉式

```java
public class Singleton {
    // 类加载时就创建实例（饿汉）
    private static final Singleton INSTANCE = new Singleton();

    // 私有构造函数
    private Singleton() {
    }

    // 公有访问方法
    public static Singleton getInstance() {
        return INSTANCE;
    }
}
```

**优点**：线程安全，JVM 保证
**缺点**：类加载时就创建，可能浪费资源（如果从未使用）

### 懒汉式（线程不安全）

```java
public class Singleton {
    private static Singleton INSTANCE;

    private Singleton() {
    }

    public static Singleton getInstance() {
        if (INSTANCE == null) {
            INSTANCE = new Singleton();
        }
        return INSTANCE;
    }
}
```

**优点**：延迟加载，需要时才创建
**缺点**：线程不安全，多线程下可能创建多个实例

### 懒汉式（线程安全）

```java
public class Singleton {
    private static volatile Singleton INSTANCE;

    private Singleton() {
    }

    public static synchronized Singleton getInstance() {
        if (INSTANCE == null) {
            INSTANCE = new Singleton();
        }
        return INSTANCE;
    }
}
```

**优点**：延迟加载 + 线程安全
**缺点**：每次调用都要加锁，性能开销

### 双重检查锁定

```java
public class Singleton {
    // volatile 防止指令重排
    private static volatile Singleton INSTANCE;

    private Singleton() {
    }

    public static Singleton getInstance() {
        // 第一次检查：避免不必要的加锁
        if (INSTANCE == null) {
            synchronized (Singleton.class) {
                // 第二次检查：防止多线程重复创建
                if (INSTANCE == null) {
                    INSTANCE = new Singleton();
                }
            }
        }
        return INSTANCE;
    }
}
```

**优点**：延迟加载 + 高性能 + 线程安全
**缺点**：实现稍复杂

### 静态内部类

```java
public class Singleton {
    private Singleton() {
    }

    // 静态内部类，只有调用 getInstance 时才会加载
    private static class Holder {
        private static final Singleton INSTANCE = new Singleton();
    }

    public static Singleton getInstance() {
        return Holder.INSTANCE;
    }
}
```

**优点**：延迟加载 + 高性能 + 线程安全（JVM 保证）
**缺点**：无法传参初始化

### 枚举单例

```java
public enum Singleton {
    INSTANCE;

    public void doSomething() {
    }
}
```

**优点**：线程安全 + 防反射 + 防反序列化
**缺点**：无法延迟初始化

## 单例模式的实际应用

### 配置管理器

```java
public class ConfigManager {
    private static volatile ConfigManager INSTANCE;
    private Properties properties = new Properties();

    private ConfigManager() {
        // 从配置文件加载
    }

    public static ConfigManager getInstance() {
        if (INSTANCE == null) {
            synchronized (ConfigManager.class) {
                if (INSTANCE == null) {
                    INSTANCE = new ConfigManager();
                }
            }
        }
        return INSTANCE;
    }

    public String get(String key) {
        return properties.getProperty(key);
    }
}
```

### 线程池

```java
public class ThreadPoolManager {
    private static volatile ThreadPoolManager INSTANCE;
    private ExecutorService executor;

    private ThreadPoolManager() {
        executor = Executors.newFixedThreadPool(10);
    }

    public static ThreadPoolManager getInstance() {
        if (INSTANCE == null) {
            synchronized (ThreadPoolManager.class) {
                if (INSTANCE == null) {
                    INSTANCE = new ThreadPoolManager();
                }
            }
        }
        return INSTANCE;
    }

    public void execute(Runnable task) {
        executor.execute(task);
    }
}
```

## 单例模式的问题

### 问题一：隐藏的依赖

```java
// 错误：直接依赖具体类
public class OrderService {
    private ConfigManager config = ConfigManager.getInstance();
}
```

单例模式让类之间的依赖变得隐式。更好的方式是依赖注入：

```java
// 正确：通过构造函数注入
public class OrderService {
    private ConfigManager config;

    public OrderService(ConfigManager config) {
        this.config = config;
    }
}
```

### 问题二：难以测试

单例难以 mock，难以在测试中替换实现。使用依赖注入可以解决这个问题。

### 问题三：违背单一职责

单例同时负责"管理实例"和"执行业务"，违背了单一职责原则。

## 单例模式 vs 静态方法

很多人认为可以用静态方法代替单例：

```java
// 静态方法方式
public class ConfigManager {
    private static Map<String, String> configs = new HashMap<>();

    public static void set(String key, String value) {
        configs.put(key, value);
    }

    public static String get(String key) {
        return configs.get(key);
    }
}
```

**单例的优势**：

- 单例是对象，可以实现接口，可以继承
- 单例可以被 mock，更容易测试
- 单例可以延迟初始化
- 单例可以使用多态，不同环境返回不同实现

**静态方法的优势**：

- 更简单，不需要考虑线程安全
- 性能更好（无方法调度开销）

## 总结

单例模式确保类只有一个实例，并提供全局访问点。

**适用场景**：

- 全局配置管理器
- 线程池管理器
- 日志管理器
- 缓存管理器

**推荐实现**：

- 简单场景：饿汉式
- 需要延迟加载：静态内部类或双重检查锁定
- 需要防反射/反序列化：枚举单例
