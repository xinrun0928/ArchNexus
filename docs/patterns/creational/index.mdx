# 创建型设计模式

一个支付系统需要同时支持支付宝、微信支付和银联通道。业务代码里散落着 `new AlipayClient()`、`new WechatPayClient()`、`new UnionPayClient()`——每当新增一种支付方式，修改点遍布数十个文件。这是创建型模式要解决的第一个问题：**对象「怎么创建」决定了系统能不能灵活应对变化**。

对象的创建远不是一句 `new` 能概括的。创建过程可能涉及复杂的构造逻辑、隐藏的依赖关系、重复创建的性能开销，以及对象整个生命周期内的一致性要求。创建型设计模式正是针对这些场景的系统化解决方案。

## 为什么需要创建型模式

直接 `new` 一个对象看似简单，但在复杂系统中会迅速暴露问题：

**紧耦合**。当代码直接依赖具体类型时，换一个实现需要改源码：

```java
// 每次换实现都要改这里
PaymentService service = new AlipayClient();
service.pay(amount);
```

**构造复杂度**。有些对象需要十几二十个参数，分步构造，或者在不同场景下有不同的默认值。直接 `new` 让构造逻辑和业务代码混在一起。

**重复创建**。每次请求都 new 一个新对象，在高频场景下会产生大量 GC 压力，而有些对象本该是单份的。

**隐藏依赖**。一个对象在创建时可能依赖其他多个对象，这些依赖链往往对调用方是透明的。调用方不知道创建过程中发生了什么，也不知道会不会创建失败。

创建型模式用各自的设计思路分别回应了这些问题。

## 五种创建型模式

| 模式 | 核心思想 | 解决什么问题 |
|---|---|---|
| 单例模式（Singleton） | 一个类只有一个实例 | 全局唯一对象，避免重复创建 |
| 工厂方法模式（Factory Method） | 子类决定创建哪个对象 | 将创建逻辑延迟到子类，避免调用方依赖具体类型 |
| 抽象工厂模式（Abstract Factory） | 一族相关对象统一创建 | 产品族切换，保证产品间一致性 |
| 建造者模式（Builder） | 分步构造复杂对象 | 解决多参数构造的可读性和可维护性问题 |
| 原型模式（Prototype） | 复制已有对象 | 避免重复初始化的开销，支持动态克隆 |

## 章节内容

- **单例模式**：全局唯一实例的多种实现方式与线程安全问题
- **工厂方法模式**：通过子类延迟决定「创建什么」，解耦产品与创建逻辑
- **抽象工厂模式**：围绕产品族设计，确保一族产品间的创建一致性
- **建造者模式**：用链式调用构造复杂对象，分离对象表示与构建逻辑
- **原型模式**：通过克隆而非构造创建对象，适用于高频创建与状态复制场景

## 适用场景

不要为了用模式而用模式。创建型模式适用的大多数场景有一个共同特征：**对象创建过程本身包含了业务逻辑或复杂决策**，或者**系统需要灵活应对对象的类型变化**。

如果对象创建逻辑极其简单，且未来几乎不会变化，直接 `new` 就是最好的选择——这是最小化设计原则的核心。
