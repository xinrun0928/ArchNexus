# 工厂方法模式

假设你开发一个物流系统，初期只支持公路运输。你写了这样的代码：

```java
public class TransportService {
    public void deliver(String packageId) {
        Truck truck = new Truck();
        truck.deliver(packageId);
    }
}
```

后来业务扩展，需要支持海运、空运、铁路运输。你发现每加一种运输方式，都要修改 `TransportService`，还要到处 `if-else`。

工厂方法模式正是为了解决这个问题：**把对象创建封装起来，让子类决定创建什么对象**。

## 问题背景

```java
// 初始版本：只支持卡车
public class TransportService {
    public void deliver(String packageId) {
        Truck truck = new Truck();
        truck.deliver(packageId);
    }
}
```

扩展后的代码变成了这样：

```java
// 扩展后：大量 if-else
public class TransportService {
    public void deliver(String packageId, String transportType) {
        if ("TRUCK".equals(transportType)) {
            Truck truck = new Truck();
            truck.deliver(packageId);
        } else if ("SHIP".equals(transportType)) {
            Ship ship = new Ship();
            ship.deliver(packageId);
        } else if ("PLANE".equals(transportType)) {
            Plane plane = new Plane();
            plane.deliver(packageId);
        }
        // 每次加新类型都要改这里
    }
}
```

问题：违反开闭原则，每次加新运输方式都要改现有代码。

## 工厂方法模式的核心

工厂方法模式定义一个创建对象的接口，但让**子类决定实例化哪个类**。

```
核心思想：
Product（产品）    ──由──  Factory（工厂）
                         │
                         │ 决定创建
                         ▼
                      ConcreteFactory
```

### 结构

- **Product（产品）**：定义产品的接口
- **ConcreteProduct（具体产品）**：实现产品接口
- **Factory（工厂）**：定义创建产品的抽象方法
- **ConcreteFactory（具体工厂）**：实现工厂方法，创建具体产品

## 代码实现

### 产品层

```java
// 产品接口
public interface Transport {
    void deliver(String packageId);
}

// 具体产品
public class Truck implements Transport {
    @Override
    public void deliver(String packageId) {
        System.out.println("Truck delivering: " + packageId);
    }
}

// 具体产品
public class Ship implements Transport {
    @Override
    public void deliver(String packageId) {
        System.out.println("Ship delivering: " + packageId);
    }
}

// 具体产品
public class Plane implements Transport {
    @Override
    public void deliver(String packageId) {
        System.out.println("Plane delivering: " + packageId);
    }
}
```

### 工厂层

```java
// 抽象工厂
public abstract class TransportFactory {
    // 工厂方法：子类决定创建什么
    public abstract Transport createTransport();

    // 使用产品的通用逻辑
    public void deliver(String packageId) {
        Transport transport = createTransport();
        transport.deliver(packageId);
    }
}

// 具体工厂：创建卡车
public class TruckFactory extends TransportFactory {
    @Override
    public Transport createTransport() {
        return new Truck();
    }
}

// 具体工厂：创建轮船
public class ShipFactory extends TransportFactory {
    @Override
    public Transport createTransport() {
        return new Ship();
    }
}

// 具体工厂：创建飞机
public class PlaneFactory extends TransportFactory {
    @Override
    public Transport createTransport() {
        return new Plane();
    }
}
```

### 使用

```java
// 客户端代码
public class LogisticsApp {
    public void processOrder(String packageId, String transportType) {
        TransportFactory factory;

        switch (transportType) {
            case "TRUCK":
                factory = new TruckFactory();
                break;
            case "SHIP":
                factory = new ShipFactory();
                break;
            case "PLANE":
                factory = new PlaneFactory();
                break;
            default:
                throw new IllegalArgumentException("Unknown transport type");
        }

        factory.deliver(packageId);
    }
}
```

## 工厂方法 vs 简单工厂

很多人分不清工厂方法和简单工厂：

| 维度 | 简单工厂 | 工厂方法 |
|---|---|---|
| 工厂类 | 一个工厂类，包含所有创建逻辑 | 抽象工厂 + 具体工厂 |
| 扩展方式 | 修改工厂类 | 添加新具体工厂 |
| 符合 OCP | 否 | 是 |
| 复杂度 | 低 | 高 |

```java
// 简单工厂：不符合 OCP
public class SimpleTransportFactory {
    public static Transport create(String type) {
        if ("TRUCK".equals(type)) {
            return new Truck();
        } else if ("SHIP".equals(type)) {
            return new Ship();
        }
        // 每次加新产品都要改这里
        return null;
    }
}
```

## 工厂方法的实际应用

### JDBC 驱动加载

```java
// 抽象工厂方法
public abstract class DaoFactory {
    public abstract UserDao createUserDao();

    // 通用操作
    public void processUser(Long userId) {
        UserDao dao = createDao();
        User user = dao.findById(userId);
        // 处理用户
    }
}

// MySQL 工厂
public class MySqlDaoFactory extends DaoFactory {
    @Override
    public UserDao createUserDao() {
        return new MySqlUserDao();
    }
}

// PostgreSQL 工厂
public class PostgresDaoFactory extends DaoFactory {
    @Override
    public UserDao createUserDao() {
        return new PostgresUserDao();
    }
}
```

### 日志框架

```java
// 日志工厂
public abstract class LoggerFactory {
    public abstract Logger createLogger();

    public void log(String message) {
        Logger logger = createLogger();

    }
}

// 文件日志工厂
public class FileLoggerFactory extends LoggerFactory {
    @Override

    }

// 控制台日志工厂
public class ConsoleLoggerFactory extends LoggerFactory {

    public Logger createLogger() {

    }
}
```

## 工厂方法模式的特点

### 优点

1. **符合开闭原则**：添加新产品只需添加新工厂
2. **单一职责**：产品创建逻辑在子类
3. **解耦**：客户端代码不需要知道具体产品类名

### 缺点

1. **类数量增加**：每个产品都要对应一个工厂
2. **结构变复杂**：增加了抽象层

## 工厂方法 vs 其他创建型模式

| 模式 | 适用场景 |
|---|---|
| 工厂方法 | 产品种类稳定，但具体产品类型经常变化 |
| 抽象工厂 | 产品族（多个相关产品）需要一起创建 |
| 建造者 | 产品由多个部分组成，创建过程固定 |
| 原型 | 复制已有对象 |

## 总结

工厂方法模式通过子类决定创建什么对象，实现了对象的创建与使用分离。

**核心要点**：

- 定义创建对象的接口，让子类决定
- 符合开闭原则，扩展时不修改现有代码
- 客户端依赖抽象（接口），不依赖具体

**使用场景**：

- 一个类无法预知需要创建什么类的对象
- 希望由子类决定创建什么对象
- 产品类相对稳定，具体产品类经常变化
