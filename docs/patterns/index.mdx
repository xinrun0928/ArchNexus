# 设计模式

代码写多了，你会发现一个问题：同样的问题被反复解决，每次都要重新想方案。有时候，你隐约感觉"这个问题我之前遇到过"，但就是想不起当时是怎么处理的。

设计模式就是这些"反复出现的问题"的通用解决方案。它不是具体的代码，而是一种经过验证的**思维框架**——告诉你"遇到这类问题，应该这样想"。

## 本章节内容

本章节涵盖以下设计模式：

### GoF 设计模式

| 类别 | 模式 |
|---|---|
| 创建型模式 | 单例、工厂方法、抽象工厂、建造者、原型 |
| 结构型模式 | 适配器、装饰器、代理、外观、桥接、组合、享元 |
| 行为型模式 | 策略、观察者、责任链、命令、模板方法、迭代器、状态、访问者、中介者、备忘录、解释器 |

### 扩展模式

| 类别 | 模式 |
|---|---|
| 并发设计模式 | 生产者-消费者、读写锁、Future、Guarded Suspension、Balking |
| 分布式设计模式 | Saga、CQRS、事件溯源、Sidecar、Ambassador |
| 架构模式 | 分层、六边形、微内核、事件驱动、CQRS、Serverless |

## 学习设计模式的目的

学习设计模式不是为了"炫耀我知道 23 种模式"，而是为了：

1. **复用经验**：站在前人的肩膀上，避免重复踩坑
2. **沟通语言**：和其他开发者用共同的概念讨论问题
3. **代码质量**：写出更易维护、更易扩展的代码
4. **识别代码**：看懂开源项目、框架中的设计决策

## 学习设计模式的误区

### 误区一：过度设计

不是所有场景都需要设计模式。有时候，简单的代码比复杂的设计更好。

```
反例：为了展示"我懂设计模式"，在一个只有两三个类的项目中使用工厂模式、策略模式、装饰器模式层层嵌套。

正例：代码简单到一眼就能看懂，不需要任何设计模式。
```

### 误区二：教条主义

设计模式是指导原则，不是硬性规定。遇到问题时，模式只是给你一个思路，最终还是要根据实际情况调整。

```
反例：必须用单例，不能用静态方法。

正例：单例和静态方法各有优劣，根据场景选择。
```

### 误区三：模式至上

设计模式是手段，不是目的。代码的可读性、可维护性才是目的。

```
反例：为了用设计模式而用设计模式，导致代码复杂度上升。

正例：代码简单清晰，必要时引入模式解决问题。
```

## 设计模式与 SOLID 原则

设计模式背后是 SOLID 原则：

| 原则 | 含义 |
|---|---|
| **S**ingle Responsibility（单一职责） | 一个类只做一件事 |
| **O**pen/Closed（开闭原则） | 对扩展开放，对修改关闭 |
| **L**iskov Substitution（里氏替换） | 子类可以替换父类 |
| **I**nterface Segregation（接口隔离） | 接口要小而专 |
| **D**ependency Inversion（依赖倒置） | 依赖抽象，不依赖具体 |

好的设计模式应用，一定是符合 SOLID 原则的。

## 前置知识

在学习设计模式之前，建议你掌握：

- 面向对象基础：类、对象、继承、多态
- 接口与抽象类
- UML 类图基础
- Java/C++/Python 等面向对象语言基础
