# 乐观并发控制

冲突很少发生，何必每次都加锁？

乐观并发控制（Optimistic Concurrency Control，OCC）是一种"假设冲突很少，冲突了再处理"的策略。

## 核心思想

```
┌─────────────────────────────────────────────────────────────────┐
│                    乐观并发控制的三阶段                                │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │                                                         │   │
│  │  阶段一：读取（Read Phase）                           │   │
│  │  - 事务正常读取数据，不加锁                          │   │
│  │  - 将读取的值保存在本地变量中                       │   │
│  │                                                         │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                  │
│                         ↓                                        │
│                                                                  │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │                                                         │   │
│  │  阶段二：验证（Validate Phase）                       │   │
│  │  - 检查其他事务是否修改了本事务读过的数据         │   │
│  │  - 检测读写冲突或写写冲突                         │   │
│  │  - 如果有冲突，事务回滚                          │   │
│  │                                                         │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                  │
│                         ↓                                        │
│                                                                  │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │                                                         │   │
│  │  阶段三：写入（Write Phase）                          │   │
│  │  - 验证通过后，写入修改                          │   │
│  │  - 提交事务                                        │   │
│  │                                                         │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

## 乐观锁的实现

### 版本号机制

```sql
-- 数据库表设计：添加版本号字段
CREATE TABLE accounts (
    id BIGINT PRIMARY KEY,
    balance DECIMAL(10, 2),
    version INT DEFAULT 0  -- 乐观锁版本号
);

-- 读取数据时获取版本号
SELECT id, balance, version FROM accounts WHERE id = 1;
-- 结果：id=1, balance=1000.00, version=3

-- 更新时检查版本号
UPDATE accounts
SET balance = balance - 100, version = version + 1
WHERE id = 1 AND version = 3;

-- 如果 version 不匹配，返回 0 行，表示更新失败
-- 业务层捕获后重试
```

### Java 实现

```java
// 乐观锁服务
@Service
public class OptimisticLockService {
    @Autowired private AccountRepository accountRepository;

    @Transactional
    public void transfer(String fromAccountId, String toAccountId,
                       BigDecimal amount, Long expectedVersion) {
        // 1. 查询当前版本
        Account account = accountRepository.findById(fromAccountId)
            .orElseThrow(() -> new AccountNotFoundException(fromAccountId));

        // 2. 验证版本号
        if (!account.getVersion().equals(expectedVersion)) {
            throw new OptimisticLockException(
                "版本号不匹配，数据已被其他事务修改");
        }

        // 3. 检查余额
        if (account.getBalance().compareTo(amount) < 0) {
            throw new InsufficientBalanceException("余额不足");
        }

        // 4. 更新（数据库层会自动检查版本）
        account.setBalance(account.getBalance().subtract(amount));
        accountRepository.save(account);
    }
}

// Spring Data JPA 版本号自动检查
@Entity
public class Account {
    @Id
    private String id;

    private BigDecimal balance;

    @Version  // JPA 乐观锁注解
    private Long version;
}

// 更新失败时抛出 org.springframework.orm.ObjectOptimisticLockingFailureException
```

### Redis 实现

```java
// Redis 乐观锁：使用 WATCH 监视键
@Service
public class RedisOptimisticLockService {
    @Autowired private StringRedisTemplate redisTemplate;

    public boolean transfer(String fromKey, String toKey, int amount) {
        // 1. 监视键
        redisTemplate.execute(new SessionCallback<Object>() {
            @Override
            public Object execute(RedisOperations operations) throws Exception {
                operations.watch(fromKey, toKey);

                // 2. 读取当前值
                Integer fromBalance = Integer.parseInt(
                    operations.opsForValue().get(fromKey));
                Integer toBalance = Integer.parseInt(
                    operations.opsForValue().get(toKey));

                // 3. 检查余额
                if (fromBalance < amount) {
                    operations.unwatch();
                    return false;
                }

                // 4. 开启事务
                operations.multi();
                operations.opsForValue().set(fromKey,
                    String.valueOf(fromBalance - amount));
                operations.opsForValue().set(toKey,
                    String.valueOf(toBalance + amount));

                // 5. 执行，如果键被修改则事务失败
                List<Object> result = operations.exec();
                return result != null && !result.isEmpty();
            }
        });
    }

    // 使用 Lua 脚本（更推荐）
    public boolean transferLua(String fromKey, String toKey, int amount) {
        String script = """
            if redis.call('GET', KEYS[1]) >= ARGV[1] then
                redis.call('DECRBY', KEYS[1], ARGV[1])
                redis.call('INCRBY', KEYS[2], ARGV[1])
                return 1
            else
                return 0
            end
            """;

        RedisScript<Long> redisScript = new RedisScript<>(script, Long.class);
        Long result = redisTemplate.execute(redisScript,
            List.of(fromKey, toKey),
            String.valueOf(amount));

        return result != null && result == 1;
    }
}
```

## 冲突检测

```java
// 乐观锁冲突检测器
public class OptimisticConflictDetector {
    // 事务读集：事务读取的所有数据项
    private final Set<DataItem> readSet = new HashSet<>();

    // 事务写集：事务写入的所有数据项
    private final Set<DataItem> writeSet = new HashSet<>();

    public void addToReadSet(String itemId, Object value, long version) {
        readSet.add(new DataItem(itemId, value, version));
    }

    public void addToWriteSet(String itemId, Object value) {
        writeSet.add(new DataItem(itemId, value, -1));
    }

    // 验证本事务与其他事务的冲突
    public boolean validate(Map<String, DataItem> committedItems,
                          Map<String, DataItem> activeItems) {
        // 检查本事务的读是否与已提交事务的写冲突
        for (DataItem readItem : readSet) {
            DataItem committedItem = committedItems.get(readItem.getItemId());

            // 已提交的数据项被本事务读取后又被修改
            if (committedItem != null &&
                readItem.getVersion() < committedItem.getVersion()) {
                // 读到了过期数据，冲突
                return false;
            }
        }

        // 检查本事务的写是否与其他活跃事务的读冲突
        for (DataItem writeItem : writeSet) {
            for (DataItem activeRead : activeItems) {
                if (writeItem.getItemId().equals(activeRead.getItemId())) {
                    // 其他活跃事务读了本事务要写的数据，冲突
                    return false;
                }
            }
        }

        return true;
    }
}
```

## 重试机制

```java
// 乐观锁重试策略
@Service
public class RetryableTransferService {
    @Autowired private AccountRepository accountRepository;
    private static final int MAX_RETRIES = 3;

    public void transferWithRetry(String fromId, String toId,
                                 BigDecimal amount) {
        int attempt = 0;
        while (attempt < MAX_RETRIES) {
            try {
                attempt++;
                transfer(fromId, toId, amount);
                return;  // 成功，退出
            } catch (ObjectOptimisticLockingFailureException e) {
                if (attempt >= MAX_RETRIES) {
                    throw new TransferException(
                        "转账失败，已重试 " + MAX_RETRIES + " 次", e);
                }
                log.warn("乐观锁冲突，第 {} 次重试", attempt);
                sleep(calculateBackoff(attempt));
            }
        }
    }

    @Transactional
    public void transfer(String fromId, String toId, BigDecimal amount) {
        Account from = accountRepository.findById(fromId)
            .orElseThrow(() -> new AccountNotFoundException(fromId));
        Account to = accountRepository.findById(toId)
            .orElseThrow(() -> new AccountNotFoundException(toId));

        from.setBalance(from.getBalance().subtract(amount));
        to.setBalance(to.getBalance().add(amount));

        accountRepository.save(from);
        accountRepository.save(to);
    }

    // 指数退避
    private long calculateBackoff(int attempt) {
        return (long) Math.pow(2, attempt) * 100;  // 200, 400, 800ms
    }

    private void sleep(long millis) {
        try {
            Thread.sleep(millis);
        } catch (InterruptedException e) {
            Thread.currentThread().interrupt();
        }
    }
}
```

## 适用场景

```
┌─────────────────────────────────────────────────────────────────┐
│                    乐观锁适用场景                                    │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │ 适合使用乐观锁的场景                                  │   │
│  ├─────────────────────────────────────────────────────────┤   │
│  │                                                         │   │
│  │  - 冲突较少的场景：库存充足、大部分人只读不写       │   │
│  │  - 读多写少的场景：更新操作不频繁                   │   │
│  │  - 短事务：事务执行时间短                         │   │
│  │  - 注重响应时间：无锁等待                         │   │
│  │                                                         │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                  │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │ 不适合使用乐观锁的场景                                │   │
│  ├─────────────────────────────────────────────────────────┤   │
│  │                                                         │   │
│  │  - 冲突频繁的场景：抢票、秒杀                       │   │
│  │  - 写多的场景：大量并发更新同一条记录              │   │
│  │  - 长事务：事务执行时间长                         │   │
│  │  - 注重吞吐量的场景：重试开销不可接受              │   │
│  │                                                         │   │
│  │  → 这些场景应该使用悲观锁                          │   │
│  │                                                         │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

## 总结

乐观并发控制适合冲突较少的场景。

**核心机制**：

- 读取阶段：不加锁，正常读取
- 验证阶段：检测是否发生冲突
- 写入阶段：验证通过则提交

**实现方式**：

- 版本号：最常用
- CAS：Compare-And-Swap
- 时间戳：基于时间戳排序

**适用条件**：

- 冲突概率低
- 重试成本可接受
- 追求高并发度

**核心认知**：乐观锁的本质是用重试换并发。在冲突很少时，这种策略非常高效；但冲突多时，不断重试反而会浪费资源。
