# 并发控制理论

多个事务同时运行时，如果没有适当的控制，就会出现数据不一致。

并发控制是数据库和分布式系统中的核心问题。它的目标是在保证数据一致性的前提下，最大化系统的并发度。

## 串行调度 vs 并发调度

```sql
-- 场景：两个事务同时转账
-- 账户 A：1000 元
-- 账户 B：1000 元

-- 事务 T1：A 转给 B 100 元
-- 事务 T2：B 转给 A 50 元

-- 串行调度（正确）
T1: BEGIN
T1: SELECT balance FROM accounts WHERE id = 'A';  -- 1000
T1: UPDATE accounts SET balance = balance - 100 WHERE id = 'A';  -- 900
T1: UPDATE accounts SET balance = balance + 100 WHERE id = 'B';  -- 1100
T1: COMMIT

T2: BEGIN
T2: SELECT balance FROM accounts WHERE id = 'B';  -- 1100
T2: UPDATE accounts SET balance = balance - 50 WHERE id = 'B';  -- 1050
T2: UPDATE accounts SET balance = balance + 50 WHERE id = 'A';  -- 950
T2: COMMIT

-- 最终结果：A = 950，B = 1050，总金额 = 2000（正确）

-- 并发调度（错误）
T1: BEGIN
T1: SELECT balance FROM accounts WHERE id = 'A';  -- 1000
T2: BEGIN
T2: SELECT balance FROM accounts WHERE id = 'B';  -- 1000
T1: UPDATE accounts SET balance = 900 WHERE id = 'A';
T2: UPDATE accounts SET balance = 950 WHERE id = 'B';  -- [!code error] 基于旧值
T2: UPDATE accounts SET balance = 1050 WHERE id = 'A';  -- [!code error] 基于旧值
T1: UPDATE accounts SET balance = 1100 WHERE id = 'B';  -- [!code error] 基于旧值
-- 最终结果：A = 1050，B = 1100，总金额 = 2150（错误！凭空多了 150）
```

### 冲突类型

```java
// 冲突操作分析
/**
 * 两个操作冲突，当且仅当：
 * 1. 它们属于不同的事务
 * 2. 它们访问同一个数据项
 * 3. 至少有一个是写操作
 *
 * 冲突类型：
 * - Read-Write 冲突（读-写）：一个读，一个写
 * - Write-Read 冲突（写-读）：一个写，一个读
 * - Write-Write 冲突（写-写）：两个都写
 *
 * Read-Read 不冲突：两个都读，不会造成数据不一致
 */

// 冲突检测示例
public class ConflictDetector {
    public boolean hasConflict(Operation op1, Operation op2) {
        // 不同事务
        if (op1.getTransactionId() == op2.getTransactionId()) {
            return false;
        }

        // 不同数据项
        if (!op1.getDataItem().equals(op2.getDataItem())) {
            return false;
        }

        // 都是读操作
        if (op1.isRead() && op2.isRead()) {
            return false;
        }

        return true;  // 存在冲突
    }
}
```

## 并发控制的两大流派

```
┌─────────────────────────────────────────────────────────────────┐
│                    并发控制策略分类                                   │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │ 乐观并发控制（Optimistic CC）                           │   │
│  ├─────────────────────────────────────────────────────────┤   │
│  │                                                         │   │
│  │  假设冲突很少发生，事务执行时不加锁                   │   │
│  │  提交前检测是否发生冲突                              │   │
│  │                                                         │   │
│  │  适用场景：                                          │   │
│  │  - 冲突较少的场景                                    │   │
│  │  - 读多写少的场景                                    │   │
│  │                                                         │   │
│  │  优点：并发度高，无死锁                              │   │
│  │  缺点：冲突检测失败的回滚开销                        │   │
│  │                                                         │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                  │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │ 悲观并发控制（Pessimistic CC）                         │   │
│  ├─────────────────────────────────────────────────────────┤   │
│  │                                                         │   │
│  │  假设冲突很可能发生，事务开始时就加锁                 │   │
│  │  整个事务期间持有锁                                   │   │
│  │                                                         │   │
│  │  适用场景：                                          │   │
│  │  - 冲突频繁的场景                                    │   │
│  │  - 写多的场景                                        │   │
│  │                                                         │   │
│  │  优点：避免冲突失败                                  │   │
│  │  缺点：并发度低，可能死锁                           │   │
│  │                                                         │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

## 时间戳排序

```java
// 基于时间戳的并发控制
public class TimestampOrdering {
    private final AtomicLong globalTimestamp = new AtomicLong(0);

    // 每个数据项维护两个时间戳
    private static class DataItem {
        long readTimestamp;  // 最大读时间戳
        long writeTimestamp;  // 最大写时间戳
        Object value;
    }

    private final Map<String, DataItem> dataItems = new ConcurrentHashMap<>();

    public ReadResult read(long transactionId, String itemId) {
        DataItem item = dataItems.get(itemId);

        // 读时间戳检查
        if (transactionId < item.writeTimestamp) {
            // 事务在数据项最后写入之前开始，应该读取旧值
            // 这里简化处理，实际实现更复杂
        }

        // 更新读时间戳
        item.readTimestamp = Math.max(item.readTimestamp, transactionId);

        return new ReadResult(item.value);
    }

    public boolean write(long transactionId, String itemId, Object value) {
        DataItem item = dataItems.get(itemId);

        // 写时间戳检查
        if (transactionId < item.readTimestamp) {
            // 有更晚的读操作，必须回滚
            return false;
        }

        // 执行写
        item.value = value;
        item.writeTimestamp = transactionId;

        return true;
    }
}
```

## 多版本并发控制

```java
// MVCC 核心思想
/**
 * MVCC = Multi-Version Concurrency Control
 *
 * 核心思想：
 * - 每次写操作创建新版本，不覆盖旧版本
 * - 读操作读取合适的历史版本
 * - 读写互不阻塞
 *
 * 版本链：
 * [Version 3] → [Version 2] → [Version 1]
 *   (最新)          ↑             ↑
 *               当前读           历史
 */
public class MVCCStorage {
    // 版本节点
    private static class Version {
        Object value;
        long createTimestamp;
        long deleteTimestamp;  // 软删除标记
        Version previous;
    }

    // 读取：找到合适版本的记录
    public Object read(String key, long transactionId) {
        Version version = findVersion(key, transactionId);
        if (version != null && version.deleteTimestamp == 0) {
            return version.value;
        }
        return null;
    }

    // 写入：创建新版本
    public void write(String key, Object value, long transactionId) {
        Version oldVersion = findVersion(key, transactionId);
        Version newVersion = new Version();
        newVersion.value = value;
        newVersion.createTimestamp = transactionId;
        newVersion.previous = oldVersion;

        // 原子替换
        versions.put(key, newVersion);
    }

    // 删除：标记删除版本
    public void delete(String key, long transactionId) {
        Version version = findVersion(key, transactionId);
        version.deleteTimestamp = transactionId;
    }
}
```

## 总结

并发控制的目标是在一致性和并发度之间找到平衡。

**核心概念**：

- 串行调度 vs 并发调度
- 可串行化调度
- 冲突检测（读-写、写-读、写-写）

**两大策略**：

- 乐观并发控制：冲突少时性能好
- 悲观并发控制：冲突多时更稳定

**常见实现**：

- 两阶段锁（悲观）
- 时间戳排序（乐观）
- MVCC（读写分离）
