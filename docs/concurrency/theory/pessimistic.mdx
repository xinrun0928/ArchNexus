# 悲观并发控制

冲突很可能发生，不如一开始就锁住。

悲观并发控制（Pessimistic Concurrency Control，PCC）是一种"假设冲突很可能发生，从一开始就加锁"的策略。

### 两阶段锁的变体

```java
/**
 * 严格两阶段锁（Strict 2PL）
 * - 扩展阶段：获取锁
 * - 收缩阶段：事务提交后才释放所有锁
 * - 优点：避免级联回滚
 *
 * 强严格两阶段锁（Strong Strict 2PL）
 * - 收缩阶段：事务提交或回滚后才释放写锁
 * - 读锁可以提前释放
 * - 常用模式
 */

// 严格两阶段锁示例
@Service
public class StrictTwoPhaseLocking {
    @Transactional
    public void transfer(Account from, Account to, BigDecimal amount) {
        // 扩展阶段：获取锁
        synchronized (from) {
            synchronized (to) {
                // 检查余额
                if (from.getBalance().compareTo(amount) < 0) {
                    throw new InsufficientBalanceException();
                }

                // 执行转账
                from.setBalance(from.getBalance().subtract(amount));
                to.setBalance(to.getBalance().add(amount));
            }
            // 收缩阶段：事务提交后释放锁
        }
    }
}
```

## 悲观锁的 SQL 实现

```sql
-- MySQL 悲观锁：SELECT ... FOR UPDATE
START TRANSACTION;

-- 获取行级写锁
SELECT * FROM accounts WHERE id = 1 FOR UPDATE;

-- 账户被锁定，其他事务无法获取这行的锁
UPDATE accounts SET balance = balance - 100 WHERE id = 1;
UPDATE accounts SET balance = balance + 100 WHERE id = 2;

COMMIT;  -- 释放锁

-- SELECT ... LOCK IN SHARE MODE（读锁）
START TRANSACTION;
SELECT * FROM accounts WHERE id = 1 LOCK IN SHARE MODE;
-- 其他事务可以读，但无法修改（写锁）
COMMIT;
```

```java
// Spring Data JPA 悲观锁
@Repository
public interface AccountRepository extends JpaRepository<Account, String> {

    // 使用悲观锁查询
    @Lock(LockModeType.PESSIMISTIC_WRITE)
    @Query("SELECT a FROM Account a WHERE a.id = :id")
    Optional<Account> findByIdWithLock(@Param("id") String id);

    // 使用读锁查询
    @Lock(LockModeType.PESSIMISTIC_READ)
    @Query("SELECT a FROM Account a WHERE a.id = :id")
    Optional<Account> findByIdWithReadLock(@Param("id") String id);
}

// 服务层使用
@Service
public class PessimisticTransferService {
    @Autowired private AccountRepository accountRepository;

    @Transactional
    public void transfer(String fromId, String toId, BigDecimal amount) {
        // 按固定顺序获取锁，避免死锁
        String firstId = fromId.compareTo(toId) < 0 ? fromId : toId;
        String secondId = fromId.compareTo(toId) < 0 ? toId : fromId;

        Account first = accountRepository.findByIdWithLock(firstId)
            .orElseThrow(() -> new AccountNotFoundException(firstId));
        Account second = accountRepository.findByIdWithLock(secondId)
            .orElseThrow(() -> new AccountNotFoundException(secondId));

        Account from = fromId.equals(firstId) ? first : second;
        Account to = fromId.equals(firstId) ? second : first;

        if (from.getBalance().compareTo(amount) < 0) {
            throw new InsufficientBalanceException();
        }

        from.setBalance(from.getBalance().subtract(amount));
        to.setBalance(to.getBalance().add(amount));

        accountRepository.save(from);
        accountRepository.save(to);
    }
}
```

## 锁粒度控制

```java
// 锁粒度控制示例
/**
 * 锁粒度越大，并发度越低，但死锁概率越低
 * 锁粒度越小，并发度越高，但死锁概率越高
 */

// 粗粒度锁：一个锁保护所有账户
@Service
public class CoarseGrainedLock {
    private final Object globalLock = new Object();

    public void transfer(String fromId, String toId, BigDecimal amount) {
        synchronized (globalLock) {
            // 所有转账操作串行化
            doTransfer(fromId, toId, amount);
        }
    }
}

// 细粒度锁：每个账户一个锁
@Service
public class FineGrainedLock {
    private final ConcurrentHashMap<String, Object> accountLocks =
        new ConcurrentHashMap<>();

    public void transfer(String fromId, String toId, BigDecimal amount) {
        // 按 ID 排序获取锁，避免死锁
        Object firstLock = accountLocks.computeIfAbsent(fromId, k -> new Object());
        Object secondLock = accountLocks.computeIfAbsent(toId, k -> new Object());

        Object lock1 = firstId.compareTo(secondId) < 0 ? firstLock : secondLock;
        Object lock2 = firstId.compareTo(secondId) < 0 ? secondLock : firstLock;

        synchronized (lock1) {
            synchronized (lock2) {
                doTransfer(fromId, toId, amount);
            }
        }
    }
}

// 数据库行锁 vs 表锁
// 行锁：并发度高，但锁管理复杂
SELECT * FROM accounts WHERE id = 1 FOR UPDATE;  -- 行锁

// 表锁：并发度低，但简单
LOCK TABLE accounts WRITE;  -- 表锁
-- 所有行都被锁定
```

## 与乐观锁的对比

| 维度 | 乐观锁 | 悲观锁 |
|---|---|---|
| 假设 | 冲突很少 | 冲突很多 |
| 加锁时机 | 提交时 | 开始时 |
| 冲突处理 | 回滚重试 | 阻塞等待 |
| 并发度 | 高 | 低 |
| 吞吐量 | 高 | 低（锁竞争） |
| 适用场景 | 读多写少 | 写多读少 |

## 总结

悲观并发控制适合冲突频繁的场景。

**核心机制**：

- 事务开始即加锁
- 持有锁直到事务结束
- 冲突时阻塞等待

**两阶段锁**：

- 扩展阶段：只获取锁
- 收缩阶段：只释放锁

**关键设计**：

- 按固定顺序获取锁，避免死锁
- 合理选择锁粒度
- 控制持有锁时间

**核心认知**：悲观锁用等待换确定性。在冲突频繁时，这种策略比乐观锁的重试更高效。
