# 并发控制策略选择

乐观锁还是悲观锁？这不是一个技术问题，而是一个业务问题。

并发控制策略的选择需要综合考虑业务特点、数据特征和系统要求。

## 典型场景选择

### 场景一：用户资料更新

```java
/**
 * 场景：用户修改自己的个人资料
 * 特点：每个用户只操作自己的数据，冲突概率极低
 * 选择：乐观锁（版本号）
 */

@Service
public class UserProfileService {
    @Autowired private UserRepository userRepository;

    @Transactional
    public void updateProfile(String userId, UserProfile profile, Long version) {
        User user = userRepository.findById(userId)
            .orElseThrow(() -> new UserNotFoundException(userId));

        // 版本号检查
        if (!user.getVersion().equals(version)) {
            throw new OptimisticLockException("数据已被修改");
        }

        user.setName(profile.getName());
        user.setEmail(profile.getEmail());
        user.setPhone(profile.getPhone());

        userRepository.save(user);
    }
}
```

### 场景二：库存扣减

```java
/**
 * 场景：电商秒杀扣减库存
 * 特点：大量并发抢同一商品，冲突概率极高
 * 选择：悲观锁 + Redis Lua
 */

@Service
public class InventoryService {
    @Autowired private RedisTemplate<String, String> redisTemplate;

    private static final String STOCK_KEY = "inventory:stock:";

    // Redis Lua 脚本：原子扣减
    public boolean deductStock(Long productId, int quantity) {
        String key = STOCK_KEY + productId;

        String script = """
            local stock = tonumber(redis.call('GET', KEYS[1]) or 0)
            if stock >= tonumber(ARGV[1]) then
                redis.call('DECRBY', KEYS[1], ARGV[1])
                return 1
            else
                return 0
            end
            """;

        RedisScript<Long> redisScript = new RedisScript<>(script, Long.class);
        Long result = redisTemplate.execute(redisScript,
            List.of(key), String.valueOf(quantity));

        return result != null && result == 1;
    }

    // 扣减成功后，异步更新数据库
    @Transactional
    public void syncInventory(Long productId, int quantity) {
        // 使用悲观锁更新数据库
        Optional<Product> product = productRepository.findByIdWithLock(productId);
        if (product.isPresent()) {
            Product p = product.get();
            p.setStock(p.getStock() - quantity);
            productRepository.save(p);
        }
    }
}
```

### 场景三：账户余额更新

```java
/**
 * 场景：银行转账
 * 特点：金额必须精确，不允许丢失更新
 * 选择：悲观锁（按固定顺序加锁）
 */

@Service
public class BankTransferService {
    @Transactional
    public void transfer(String fromAccountId, String toAccountId,
                        BigDecimal amount) {
        // 按 ID 排序获取锁，避免死锁
        String firstId = fromAccountId.compareTo(toAccountId) < 0
            ? fromAccountId : toAccountId;
        String secondId = fromAccountId.compareTo(toAccountId) < 0
            ? toAccountId : fromAccountId;

        Account first = accountRepository.findByIdWithLock(firstId)
            .orElseThrow(() -> new AccountNotFoundException(firstId));
        Account second = accountRepository.findByIdWithLock(secondId)
            .orElseThrow(() -> new AccountNotFoundException(secondId));

        Account from = fromAccountId.equals(firstId) ? first : second;
        Account to = fromAccountId.equals(firstId) ? second : first;

        if (from.getBalance().compareTo(amount) < 0) {
            throw new InsufficientBalanceException();
        }

        from.setBalance(from.getBalance().subtract(amount));
        to.setBalance(to.getBalance().add(amount));

        accountRepository.save(from);
        accountRepository.save(to);
    }
}
```

### 场景四：乐观锁 + 降级策略

```java
/**
 * 场景：订单状态更新
 * 特点：冲突概率中等，希望在高并发时保持可用
 * 选择：乐观锁 + 降级到悲观锁
 */

@Service
public class OrderStatusService {
    private static final int OPTIMISTIC_MAX_RETRIES = 3;

    @Transactional
    public void updateOrderStatus(String orderId, OrderStatus newStatus,
                                 Long expectedVersion) {
        Order order = orderRepository.findById(orderId)
            .orElseThrow(() -> new OrderNotFoundException(orderId));

        // 尝试乐观锁更新
        if (order.getVersion().equals(expectedVersion)) {
            order.setStatus(newStatus);
            orderRepository.save(order);
            return;
        }

        // 乐观锁失败，降级为悲观锁
        pessimisticUpdate(orderId, newStatus);
    }

    @Transactional
    public void pessimisticUpdate(String orderId, OrderStatus newStatus) {
        Order order = orderRepository.findByIdWithLock(orderId)
            .orElseThrow(() -> new OrderNotFoundException(orderId));

        order.setStatus(newStatus);
        orderRepository.save(order);
    }
}
```

## 总结

并发控制策略的选择取决于业务特点。

**选择原则**：

- 冲突少 + 重试成本低：乐观锁
- 冲突多 + 不允许失败：悲观锁
- 业务多样性：混合策略

**关键评估维度**：

- 冲突概率
- 重试成本
- 一致性要求

**实践建议**：

- 先用乐观锁，性能不够再加悲观锁
- 悲观锁要按固定顺序加锁
- 核心数据用悲观锁，辅助数据用乐观锁
