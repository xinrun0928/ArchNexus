# 死锁预防策略

预防死锁的核心是破坏 Coffman 条件中的至少一个条件。

死锁预防是在系统设计层面消除死锁的可能性，比死锁检测和恢复更高效。

## 策略一：固定锁顺序

### Java 示例

```java
// 按固定顺序获取锁：永远不会死锁
public class FixedOrderLockExample {
    private final ConcurrentHashMap<String, Account> accounts =
        new ConcurrentHashMap<>();

    public void transfer(String fromId, String toId, BigDecimal amount) {
        // 按 ID 排序，确保所有线程按相同顺序获取锁
        String firstId = fromId.compareTo(toId) < 0 ? fromId : toId;
        String secondId = fromId.compareTo(toId) < 0 ? toId : fromId;

        Account first = accounts.get(firstId);
        Account second = accounts.get(secondId);

        // 固定顺序：先锁 first，再锁 second
        synchronized (first) {
            synchronized (second) {
                if (first.getBalance().compareTo(amount) < 0) {
                    throw new InsufficientBalanceException();
                }

                first.setBalance(first.getBalance().subtract(amount));
                second.setBalance(second.getBalance().add(amount));
            }
        }
    }
}
```

### 数据库示例

```sql
-- 按 ID 排序获取锁
START TRANSACTION;

-- T1: UPDATE accounts SET ... WHERE id IN ('A', 'B') ORDER BY id;
-- T2: UPDATE accounts SET ... WHERE id IN ('B', 'A') ORDER BY id;
-- 两个事务都按相同顺序锁定 A 和 B，不会死锁

-- 错误做法（可能导致死锁）
-- T1: UPDATE accounts SET ... WHERE id = 'A';
-- T2: UPDATE accounts SET ... WHERE id = 'B';
-- T1: UPDATE accounts SET ... WHERE id = 'B';  -- 等待 T2
-- T2: UPDATE accounts SET ... WHERE id = 'A';  -- 等待 T1
```

## 策略二：一次性获取所有锁

```java
// 一次性获取所有需要的锁
public class AllAtOnceLock {
    private final RedissonClient redisson;

    public void transferWithAllLocks(String fromId, String toId,
                                     BigDecimal amount) {
        // 获取所有锁的列表（按排序后的顺序）
        List<String> lockIds = Arrays.asList(fromId, toId)
            .stream()
            .sorted()
            .map(id -> "account:lock:" + id)
            .collect(Collectors.toList());

        // 尝试获取所有锁
        List<RLock> locks = new ArrayList<>();
        boolean allAcquired = true;

        for (String lockId : lockIds) {
            RLock lock = redisson.getLock(lockId);
            boolean acquired = lock.tryLock(10, 30, TimeUnit.SECONDS);
            if (!acquired) {
                allAcquired = false;
                break;
            }
            locks.add(lock);
        }

        // 如果没有全部获取，释放已获取的锁
        if (!allAcquired) {
            for (RLock lock : locks) {
                if (lock.isHeldByCurrentThread()) {
                    lock.unlock();
                }
            }
            throw new LockAcquisitionException("无法获取所有锁");
        }

        try {
            // 执行转账
            doTransfer(fromId, toId, amount);
        } finally {
            // 统一释放所有锁
            for (RLock lock : locks) {
                if (lock.isHeldByCurrentThread()) {
                    lock.unlock();
                }
            }
        }
    }
}
```

## 策略三：使用超时锁

```java
// 使用带超时的锁
public class TimeoutLock {
    private final ReentrantLock lock = new ReentrantLock();

    public void doWithTimeout(long timeout, TimeUnit unit,
                              Runnable action) {
        try {
            // 尝试获取锁，最多等待指定时间
            if (lock.tryLock(timeout, unit)) {
                try {
                    action.run();
                } finally {
                    lock.unlock();
                }
            } else {
                // 获取锁超时
                throw new LockTimeoutException(
                    "获取锁超时: " + timeout + " " + unit);
            }
        } catch (InterruptedException e) {
            Thread.currentThread().interrupt();
            throw new LockInterruptedException(
                "获取锁被中断");
        }
    }

    // 示例：重试机制
    public void doWithRetry(String resourceId, Runnable action) {
        int maxRetries = 3;
        long baseTimeout = 100;

        for (int i = 0; i < maxRetries; i++) {
            try {
                if (lock.tryLock(baseTimeout * (i + 1), TimeUnit.MILLISECONDS)) {
                    try {
                        action.run();
                        return;
                    } finally {
                        lock.unlock();
                    }
                }
            } catch (InterruptedException e) {
                Thread.currentThread().interrupt();
                throw new LockInterruptedException("重试被中断");
            }

            // 指数退避
            try {
                Thread.sleep(baseTimeout * (i + 1));
            } catch (InterruptedException e) {
                Thread.currentThread().interrupt();
                break;
            }
        }

        throw new LockAcquisitionException("重试次数耗尽");
    }
}
```

## 策略四：资源排序（数据库）

```sql
-- 数据库死锁预防：所有操作按相同顺序访问资源

-- 方案一：使用全局排序的 ID
-- 确保所有事务按 ID 顺序访问行

-- 转账操作：总是先访问 ID 小的账户
START TRANSACTION;

SELECT * FROM accounts
WHERE id IN ('A', 'B')
ORDER BY id
FOR UPDATE;

UPDATE accounts SET balance = balance - 100 WHERE id = 'A';
UPDATE accounts SET balance = balance + 100 WHERE id = 'B';

COMMIT;

-- 方案二：使用单一锁对象
-- 通过一个全局锁来序列化所有敏感操作
SELECT * FROM global_lock WHERE id = 1 FOR UPDATE;
-- 执行所有需要保护的逻辑
UPDATE global_lock SET version = version + 1 WHERE id = 1;
COMMIT;
```

## 总结

死锁预防的关键是破坏 Coffman 条件。

**最实用的策略**：

- 固定锁顺序：最简单有效
- 超时锁：兜底策略
- 一次性获取所有锁：适合复杂场景

**选择建议**：

- 简单场景：固定锁顺序
- 复杂场景：按顺序一次性获取
- 需要容错：使用超时锁

**核心认知**：死锁预防比分死锁检测和恢复更高效。设计系统时就应该考虑死锁预防。
