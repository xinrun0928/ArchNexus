# 乐观锁与悲观锁的选择

没有最好的策略，只有最适合的策略。

乐观锁和悲观锁各有优劣，选择时需要综合考虑业务特点、数据特征和系统要求。

## 典型场景选择

### 场景一：用户资料更新

**特点**：每个用户只操作自己的数据，冲突概率极低

```java
@Service
public class UserProfileService {
    // 选择：乐观锁（版本号）
    @Transactional
    public void updateProfile(String userId, UserProfile profile) {
        User user = userRepository.findById(userId)
            .orElseThrow(() -> new UserNotFoundException(userId));

        if (!user.getVersion().equals(profile.getExpectedVersion())) {
            throw new OptimisticLockException("数据已被修改");
        }

        user.setName(profile.getName());
        user.setEmail(profile.getEmail());
        user.setPhone(profile.getPhone());

        userRepository.save(user);
    }
}
```

### 场景二：库存扣减

**特点**：大量并发抢同一商品，冲突概率极高

```java
@Service
public class InventoryService {
    // 选择：悲观锁（Redis Lua + 数据库行锁）
    private static final String STOCK_KEY = "inventory:stock:";

    public boolean deductStock(Long productId, int quantity) {
        String key = STOCK_KEY + productId;

        // Redis Lua 原子操作
        String script = """
            if tonumber(redis.call('GET', KEYS[1]) or 0) >= tonumber(ARGV[1]) then
                redis.call('DECRBY', KEYS[1], ARGV[1])
                return 1
            else
                return 0
            end
            """;

        RedisScript<Long> redisScript = new RedisScript<>(script, Long.class);
        return Boolean.TRUE.equals(
            redisTemplate.execute(redisScript, List.of(key), String.valueOf(quantity)) == 1
        );
    }
}
```

### 场景三：银行转账

**特点**：金额必须精确，不允许任何丢失更新

```java
@Service
public class BankTransferService {
    // 选择：悲观锁（按固定顺序加锁）
    @Transactional
    public void transfer(String fromId, String toId, BigDecimal amount) {
        String firstId = fromId.compareTo(toId) < 0 ? fromId : toId;
        String secondId = fromId.compareTo(toId) < 0 ? toId : fromId;

        Account first = accountRepository.findByIdWithLock(firstId).orElseThrow();
        Account second = accountRepository.findByIdWithLock(secondId).orElseThrow();

        Account from = fromId.equals(firstId) ? first : second;
        Account to = fromId.equals(firstId) ? second : first;

        if (from.getBalance().compareTo(amount) < 0) {
            throw new InsufficientBalanceException();
        }

        from.setBalance(from.getBalance().subtract(amount));
        to.setBalance(to.getBalance().add(amount));

        accountRepository.save(from);
        accountRepository.save(to);
    }
}
```

### 场景四：文档协作

**特点**：冲突中等，希望最终一致

```java
@Service
public class DocumentCollaborationService {
    // 选择：乐观锁 + CRDT
    @Transactional
    public void updateDocument(String docId, String userId,
                             String content, Long expectedVersion) {
        Document doc = documentRepository.findById(docId)
            .orElseThrow(() -> new DocumentNotFoundException(docId));

        // 使用乐观锁检测冲突
        if (!doc.getVersion().equals(expectedVersion)) {
            // 冲突：使用 CRDT 合并
            String mergedContent = crdtMerge(doc.getContent(), content);
            doc.setContent(mergedContent);
            doc.setVersion(doc.getVersion() + 1);
        } else {
            doc.setContent(content);
            doc.setVersion(doc.getVersion() + 1);
        }

        documentRepository.save(doc);
    }
}
```

## 总结

策略选择的关键是理解业务对冲突概率、重试成本和一致性要求的权衡。

**选择原则**：

- 冲突少 + 重试成本低：乐观锁
- 冲突多 + 不允许失败：悲观锁
- 复杂业务：混合策略

**决策建议**：

- 先用乐观锁，性能瓶颈时再加悲观锁
- 悲观锁要按固定顺序加锁
- 不同数据层使用不同策略
