# 乐观锁与悲观锁

并发环境下，多个线程同时修改同一份数据，如何保证数据一致性？这个问题没有标准答案，因为答案取决于业务场景对冲突概率、重试成本和一致性要求的权衡。

乐观锁和悲观锁代表了两种截然不同的思路：前者相信冲突是少数，先干再说，冲突了再补救；后者相信冲突是常态，先加锁再操作，用锁等待换取确定性。这两种策略各有适用场景，选对了能大幅提升系统吞吐量，选错了轻则性能暴跌，重则死锁频发。

## 核心对比

两种锁策略的本质区别在于「何时检测冲突」：

| 维度 | 乐观锁 | 悲观锁 |
|---|---|---|
| 假设 | 冲突是少数 | 冲突是常态 |
| 加锁时机 | 提交时检查 | 操作前加锁 |
| 冲突处理 | 自旋重试 | 阻塞等待 |
| 适用场景 | 读多写少 | 写多读少 |
| 吞吐量 | 高（无锁等待） | 低（锁竞争） |
| 一致性 | 最终一致 | 强一致 |

乐观锁的核心是「检查-更新」原子化，通过版本号或 CAS 保证更新时数据未被修改。悲观锁的核心是「先锁后操作」，通过数据库行锁或分布式锁确保同一时刻只有一个事务能操作数据。

## 学习路径

本章节将系统讲解乐观锁与悲观锁的实现原理和选型策略：

- **乐观锁的实现**：版本号与 CAS 两种方案的具体实现，JPA `@Version`、Java `Atomic*` 类的用法，以及重试机制的实现
- **悲观锁的实现**：数据库行锁（`SELECT ... FOR UPDATE`）、Spring Data JPA 悲观锁注解、分布式悲观锁（Redisson）
- **CAS 原理与 ABA 问题**：CAS 的硬件级原子性，ABA 问题的成因，以及 `AtomicStampedReference`、`StampedLock` 等解决方案
- **乐观锁与悲观锁的选择**：根据冲突概率、事务长度、重试成本等维度，如何在具体业务场景中做出决策

## 章节内容

本章节深入探讨并发控制的核心策略：

- **乐观锁**：通过版本号或 CAS 实现，先操作后检查，冲突时重试，适合读多写少、冲突概率低的场景
- **悲观锁**：通过锁机制实现，先加锁后操作，冲突时阻塞等待，适合写多读少、冲突概率高的场景
- **CAS 原理**：硬件级原子指令，无锁编程的基础，理解 ABA 问题是深入并发编程的关键
- **选型策略**：没有银弹，根据业务特点选择合适的锁策略，混合使用也是常见做法
