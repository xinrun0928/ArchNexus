# 乐观锁的实现

乐观锁的核心假设是「冲突是少数」，先操作后检查，冲突时重试。

本节重点讲解数据库乐观锁的两种主流实现方式和 Java 代码示例。

## 核心思想

乐观锁的工作流程分为三步：

```
┌─────────────────────────────────────────────────────────────────┐
│                    乐观锁工作流程                                   │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │                                                         │   │
│  │  1. 读取：读取当前版本号 / 待修改值                      │   │
│  │                                                         │   │
│  │  2. 修改：执行业务逻辑，计算新值                         │   │
│  │                                                         │   │
│  │  3. 提交：带原版本号写入，数据库判断是否冲突              │   │
│  │                                                         │   │
│  │     冲突  → 回滚业务操作，重试（重新读取 → 修改 → 提交） │   │
│  │     无冲突 → 提交成功                                    │   │
│  │                                                         │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

悲观锁是「先加锁再操作」，乐观锁是「先操作，提交时检查冲突」。前者适合写多的场景，后者适合读多写少的场景。

## 方案一：版本号字段

### 数据库表设计

```sql
CREATE TABLE orders (
    id          VARCHAR(36)  PRIMARY KEY,
    status      VARCHAR(20)  NOT NULL DEFAULT 'pending',
    amount      DECIMAL(10,2) NOT NULL,
    version     INT          NOT NULL DEFAULT 0,  -- 版本号字段
    created_at  TIMESTAMP    NOT NULL DEFAULT CURRENT_TIMESTAMP,
    updated_at  TIMESTAMP    NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP
);
```

`version` 字段初始为 `0`，每次更新时自增。更新条件必须同时满足「主键匹配」和「版本号匹配」，否则影响行数为 0，表示有并发冲突。

### 纯 SQL 实现

```sql
-- 普通更新（无乐观锁）—— 并发不安全
UPDATE orders SET status = 'paid' WHERE id = '1001';

-- 乐观锁更新：WHERE 条件包含 version
UPDATE orders
SET status = 'paid',
    version = version + 1
WHERE id = '1001'
  AND version = 5;  -- 提交时检查版本号

-- 影响行数为 0？说明有并发冲突
-- 影响行数为 1？说明更新成功
```

### Spring Data JPA 实现

```java
@Entity
@Table(name = "orders")
public class Order {
    @Id
    private String id;

    private String status;

    private BigDecimal amount;

    @Version  // JPA 自动维护版本号
    private Long version;
}
```

JPA 的 `@Version` 注解会在每次持久化时自动递增版本号，并自动在 UPDATE 语句的 WHERE 条件中加入版本检查，冲突时抛出 `OptimisticLockException`。

```java
@Service
public class OptimisticOrderService {
    @Autowired private OrderRepository orderRepository;

    // JPA 自动处理版本检查，冲突时抛 OptimisticLockException
    @Transactional
    public void updateOrderStatus(String orderId, String newStatus) {
        Order order = orderRepository.findById(orderId)
            .orElseThrow(() -> new OrderNotFoundException(orderId));

        order.setStatus(newStatus);
        orderRepository.save(order);
        // JPA 自动追加 WHERE version = ? 条件
    }
}
```

手动版本检查的写法：

```java
@Service
public class ManualVersionOrderService {
    @Autowired private JdbcTemplate jdbcTemplate;

    public int updateStatus(String orderId, String newStatus, Long expectedVersion) {
        // UPDATE ... WHERE id = ? AND version = ?
        // 影响行数为 0 时说明版本不匹配（被其他事务修改过）
        String sql = """
            UPDATE orders
            SET status = ?, version = version + 1
            WHERE id = ? AND version = ?
            """;

        return jdbcTemplate.update(sql, newStatus, orderId, expectedVersion);
    }
}
```

### 重试机制

乐观锁冲突后需要重试，以下是典型的重试实现：

```java
@Service
public class RetryOrderService {
    @Autowired private OrderRepository orderRepository;

    private static final int MAX_RETRIES = 3;

    public void payOrder(String orderId, BigDecimal payAmount) {
        int attempts = 0;

        while (true) {
            try {
                executePayment(orderId, payAmount);
                return;
            } catch (OptimisticLockException e) {
                attempts++;
                if (attempts >= MAX_RETRIES) {
                    throw new PaymentFailedException("支付失败，重试次数超限");
                }
                // 重试前短暂退避，避免连续冲突
                try {
                    Thread.sleep(50L * attempts);
                } catch (InterruptedException ie) {
                    Thread.currentThread().interrupt();
                    throw new PaymentFailedException("支付被中断");
                }
            }
        }
    }

    @Transactional
    public void executePayment(String orderId, BigDecimal payAmount) {
        Order order = orderRepository.findById(orderId)
            .orElseThrow(() -> new OrderNotFoundException(orderId));

        if (order.getStatus().equals("paid")) {
            return;
        }

        BigDecimal newAmount = order.getAmount().subtract(payAmount);
        if (newAmount.compareTo(BigDecimal.ZERO) < 0) {
            throw new InsufficientBalanceException();
        }

        order.setAmount(newAmount);
        order.setStatus("paid");
        orderRepository.save(order);
        // JPA 自动检查版本号，冲突时抛 OptimisticLockException
    }
}
```

## 方案二：CAS（Compare-And-Swap）

版本号方案依赖数据库的 WHERE 条件检查，CAS 方案则是在应用层通过原子操作实现相同逻辑。相比版本号，CAS 方案的检查和更新在同一条原子指令中完成，不需要事务支持，适合纯内存场景。

### Java Atomic 系列

Java 的 `java.util.concurrent.atomic` 包提供了基于 CAS 的原子类，无需加锁即可保证并发安全：

```java
// AtomicInteger：计数器
AtomicInteger stock = new AtomicInteger(100);
stock.incrementAndGet();   // 原子递增
stock.decrementAndGet();   // 原子递减
stock.compareAndSet(100, 101);  // CAS：等于 100 才更新为 101

// AtomicLong：处理超过 int 范围的值
AtomicLong viewCount = new AtomicLong(0);
long current = viewCount.get();
viewCount.compareAndSet(current, current + 1);

// AtomicReference：引用类型的 CAS
AtomicReference<Order> currentOrder = new AtomicReference<>();
currentOrder.compareAndSet(oldOrder, newOrder);
```

### 乐观锁计数器

```java
@Service
public class OptimisticCounterService {
    private final AtomicInteger successCount = new AtomicInteger(0);
    private final AtomicInteger failCount = new AtomicInteger(0);

    // 原子递增成功计数
    public void recordSuccess() {
        successCount.incrementAndGet();
    }

    // 乐观重试：CAS + 自旋
    public boolean decrementStock(AtomicInteger stock, int quantity) {
        while (true) {
            int current = stock.get();
            if (current < quantity) {
                return false;  // 库存不足
            }
            if (stock.compareAndSet(current, current - quantity)) {
                return true;  // CAS 成功
            }
            // CAS 失败，说明有并发修改，自旋重试
        }
    }

    public int getSuccessCount() {
        return successCount.get();
    }
}
```

### 带条件检查的 CAS

业务中很少有纯粹的递增递减，通常需要在 CAS 前做条件判断。以下是一种常见模式：

```java
@Service
public class InventoryCasService {
    private final ConcurrentHashMap<Long, AtomicInteger> stockMap = new ConcurrentHashMap<>();

    public void initStock(Long productId, int quantity) {
        stockMap.put(productId, new AtomicInteger(quantity));
    }

    // 乐观扣减：CAS + 条件检查
    public boolean deductStock(Long productId, int quantity) {
        AtomicInteger stock = stockMap.get(productId);
        if (stock == null) {
            return false;
        }

        while (true) {
            int current = stock.get();
            if (current < quantity) {
                return false;  // 库存不足，直接返回
            }
            if (stock.compareAndSet(current, current - quantity)) {
                return true;
            }
            // CAS 失败（说明并发线程也在改），自旋重试
        }
    }
}
```

### 批量原子操作

单条 CAS 适合简单场景，多条数据的一致更新需要更复杂的设计：

```java
@Service
public class AccountTransferService {
    private final ConcurrentHashMap<String, AtomicReference<BigDecimal>> accounts =
        new ConcurrentHashMap<>();

    // 转账：先扣后加，任意一步失败都需要回滚
    public boolean transfer(String fromId, String toId, BigDecimal amount) {
        AtomicReference<BigDecimal> from = accounts.get(fromId);
        AtomicReference<BigDecimal> to = accounts.get(toId);
        if (from == null || to == null) {
            return false;
        }

        while (true) {
            BigDecimal fromBalance = from.get();
            BigDecimal toBalance = to.get();

            if (fromBalance.compareTo(amount) < 0) {
                return false;  // 余额不足
            }

            BigDecimal newFrom = fromBalance.subtract(amount);
            BigDecimal newTo = toBalance.add(amount);

            // 两条 CAS 必须同时成功
            boolean fromCas = from.compareAndSet(fromBalance, newFrom);
            boolean toCas = to.compareAndSet(toBalance, newTo);

            if (fromCas && toCas) {
                return true;
            }
            // 任意一条失败，说明有并发修改，重试
        }
    }
}
```

上述转账代码存在一个潜在问题：两条 CAS 中间有间隙，如果线程 A 和线程 B 同时进行方向相反的转账，可能出现互相覆盖的情况。这是 CAS 在多变量场景下的固有限制，解决方案是用更上层的锁（ReentrantLock）保护，或引入分布式事务。

## 两种方案对比

| 维度 | 版本号（数据库） | CAS（内存） |
|---|---|---|
| 适用场景 | 持久化数据、多进程共享 | 单进程内存数据 |
| 冲突处理 | 数据库自动检测，应用层抛异常 | CAS 失败返回 false，应用层自旋 |
| 事务依赖 | 需要事务支持 | 不需要事务 |
| 范围限制 | 全局有效 | 单 JVM 内有效 |
| 重试代价 | 较高（重新查询、重新计算） | 较低（纯内存操作） |

## 总结

乐观锁通过「读-改-写」三步和版本检查，将并发控制从「锁等待」转化为「重试」，适合读多写少、冲突概率低的场景。

**版本号方案**适合数据库持久化数据，JPA 的 `@Version` 注解可以零成本接入。

**CAS 方案**适合纯内存计数、状态切换等轻量场景，Java 的 `Atomic*` 系列类已经封装好了所有细节。

**需要注意**：

- 乐观锁不是银弹。冲突率超过 20% 时，乐观锁的重试开销会远超悲观锁。
- 重试要有上限，防止无限自旋耗尽 CPU。
- CAS 在多变量场景下有局限性，涉及多条数据的强一致性时，应考虑悲观锁或分布式事务。
