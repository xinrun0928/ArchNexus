# 悲观锁的实现

悲观锁通过在事务开始时就获取锁来保证一致性。

本节重点讲解数据库悲观锁的 SQL 实现和 Java 代码示例。

## 数据库行锁

```sql
-- MySQL SELECT ... FOR UPDATE
START TRANSACTION;

-- 获取行级写锁
SELECT * FROM orders WHERE id = '1001' FOR UPDATE;

-- 锁住订单后，其他事务无法修改这条记录
UPDATE orders SET status = 'paid' WHERE id = '1001';

COMMIT;  -- 提交事务，释放锁

-- SELECT ... LOCK IN SHARE MODE（读锁）
START TRANSACTION;

SELECT * FROM orders WHERE id = '1001' LOCK IN SHARE MODE;

-- 其他事务可以读取，但无法修改（写锁）
COMMIT;
```

## 批量锁定

```sql
-- 锁定多个行（按 ID 排序）
START TRANSACTION;

-- 按固定顺序锁定，避免死锁
SELECT * FROM orders
WHERE id IN ('1001', '1002', '1003')
ORDER BY id
FOR UPDATE;

-- 执行操作
UPDATE orders SET status = 'shipped' WHERE id = '1001';
UPDATE orders SET status = 'shipped' WHERE id = '1002';
UPDATE orders SET status = 'shipped' WHERE id = '1003';

COMMIT;
```

## Spring Data JPA 悲观锁

```java
@Repository
public interface OrderRepository extends JpaRepository<Order, String> {

    // PESSIMISTIC_WRITE：获取排他锁
    @Lock(LockModeType.PESSIMISTIC_WRITE)
    @Query("SELECT o FROM Order o WHERE o.id = :id")
    Optional<Order> findByIdWithPessimisticWriteLock(@Param("id") String id);

    // PESSIMISTIC_READ：获取共享锁
    @Lock(LockModeType.PESSIMISTIC_READ)
    @Query("SELECT o FROM Order o WHERE o.id = :id")
    Optional<Order> findByIdWithPessimisticReadLock(@Param("id") String id);

    // 带超时的锁
    @Lock(LockModeType.PESSIMISTIC_WRITE)
    @QueryHints({
        @QueryHint(name = "jakarta.persistence.lock.timeout", value = "3000")
    })
    @Query("SELECT o FROM Order o WHERE o.id = :id")
    Optional<Order> findByIdWithLockTimeout(@Param("id") String id);
}
```

## 服务层实现

```java
@Service
public class PessimisticOrderService {
    @Autowired private OrderRepository orderRepository;

    @Transactional
    public void processOrder(String orderId, OrderAction action) {
        // 获取悲观锁
        Order order = orderRepository.findByIdWithPessimisticWriteLock(orderId)
            .orElseThrow(() -> new OrderNotFoundException(orderId));

        // 在锁保护下进行操作
        switch (action) {
            case PAY:
                if (order.getStatus() != OrderStatus.PENDING) {
                    throw new InvalidOrderStateException("订单状态不允许支付");
                }
                order.setStatus(OrderStatus.PAID);
                break;
            case SHIP:
                if (order.getStatus() != OrderStatus.PAID) {
                    throw new InvalidOrderStateException("订单状态不允许发货");
                }
                order.setStatus(OrderStatus.SHIPPED);
                break;
            case CANCEL:
                if (order.getStatus() == OrderStatus.COMPLETED) {
                    throw new InvalidOrderStateException("已完成订单不能取消");
                }
                order.setStatus(OrderStatus.CANCELLED);
                break;
        }

        orderRepository.save(order);
    }

    // 转账：按固定顺序加锁避免死锁
    @Transactional
    public void transfer(String fromId, String toId, BigDecimal amount) {
        // 确保按 ID 排序获取锁
        String firstId = fromId.compareTo(toId) < 0 ? fromId : toId;
        String secondId = fromId.compareTo(toId) < 0 ? toId : fromId;

        Account first = accountRepository.findByIdWithPessimisticWriteLock(firstId)
            .orElseThrow(() -> new AccountNotFoundException(firstId));
        Account second = accountRepository.findByIdWithPessimisticWriteLock(secondId)
            .orElseThrow(() -> new AccountNotFoundException(secondId));

        Account from = fromId.equals(firstId) ? first : second;
        Account to = fromId.equals(firstId) ? second : first;

        if (from.getBalance().compareTo(amount) < 0) {
            throw new InsufficientBalanceException();
        }

        from.setBalance(from.getBalance().subtract(amount));
        to.setBalance(to.getBalance().add(amount));

        accountRepository.save(from);
        accountRepository.save(to);
    }
}
```

## 分布式悲观锁

```java
@Service
public class DistributedPessimisticLock {
    @Autowired private RedissonClient redisson;

    public <T> T executeWithLock(String resourceId, Supplier<T> action) {
        RLock lock = redisson.getLock("pessimistic:" + resourceId);
        lock.lock();  // 默认无限等待
        try {
            return action.get();
        } finally {
            lock.unlock();
        }
    }

    // 带超时
    public <T> T executeWithLockTimeout(String resourceId, long waitTime,
                                      TimeUnit unit, Supplier<T> action) {
        RLock lock = redisson.getLock("pessimistic:" + resourceId);
        try {
            if (lock.tryLock(waitTime, unit)) {
                return action.get();
            }
            throw new LockAcquisitionException("获取锁超时: " + resourceId);
        } catch (InterruptedException e) {
            Thread.currentThread().interrupt();
            throw new LockAcquisitionException("获取锁被中断: " + resourceId);
        } finally {
            if (lock.isHeldByCurrentThread()) {
                lock.unlock();
            }
        }
    }
}
```

## 总结

悲观锁通过提前加锁来保证一致性。

**实现方式**：

- SQL：`SELECT ... FOR UPDATE`
- JPA：`@Lock(LockModeType.PESSIMISTIC_WRITE)`
- 分布式：Redisson `RLock`

**关键要点**：

- 按固定顺序加锁避免死锁
- 控制锁粒度，减少锁冲突
- 设置合理的锁超时时间
