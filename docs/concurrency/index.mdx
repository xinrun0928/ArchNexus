# 并发

你以为代码写对了，测试也通过了，但上线后，系统开始出现奇怪的问题：明明余额没变，账户却少钱了；两个用户同时下单，库存却超卖了；服务跑着跑着，突然所有接口都不响应了。这些问题，往往不在业务逻辑里，而在**并发**里。

并发是现代后端开发无法回避的核心议题。当多个执行单元（线程、进程）同时访问共享资源时，如何保证数据的一致性和系统的正确性，是每个后端工程师必须掌握的能力。

## 核心问题

### 为什么并发这么难？

并发问题之所以难以发现和调试，有三个主要原因：

**问题隐藏深**：并发 bug 往往不是每次都出现，只有在特定时序条件下才会触发。测试环境可能跑一万次都不出问题，生产环境流量一上来就开始暴露。

**复现困难**：由于时序依赖，同样的输入在不同运行中可能产生不同结果。开发机器上复现不了，线上却频繁出现。

**根因隐蔽**：一个看似简单的「余额不对」问题，可能根源在数据库隔离级别；一个「死循环」卡死，可能根因是锁顺序不一致。

### 并发控制的两大方向

解决并发问题，核心思路有两种：

**悲观策略**：假设冲突必然发生，从一开始就加锁保护。适合写多读多、冲突频繁的场景。代价是并发度低，可能产生死锁。

**乐观策略**：假设冲突很少发生，不加锁，冲突时再处理。适合读多写少、冲突不多的场景。代价是冲突时需要重试，可能浪费计算资源。

没有绝对更好的策略，只有更适合的场景。关键在于正确评估业务特点。

### 死锁的四个必要条件

死锁的发生必须同时满足四个条件：

- **互斥**：资源只能被一个线程持有
- **持有并等待**：线程持有资源的同时等待其他资源
- **不可抢占**：资源不能被强制释放，只能主动释放
- **循环等待**：存在循环依赖链

打破其中任何一个条件，就可以避免死锁。这也是死锁处理的核心思路。

## 章节内容

### 并发控制

从理论层面理解并发问题的本质：为什么需要并发控制？什么是串行等价？乐观策略和悲观策略各有什么特点？如何根据业务特点选择合适的控制策略？

### 锁机制

深入讲解各种锁的实现原理和使用场景：synchronized 的锁升级机制、ReentrantLock 的可重入特性、读写锁的适用场景、Condition 的条件等待。理解锁的背后原理，才能用对、用好锁。

### 乐观与悲观锁

CAS（Compare-And-Swap）的底层原理、ABA 问题的产生与解决、数据库中的乐观锁（版本号）和悲观锁（SELECT FOR UPDATE）的实现方式。理解两种策略的 trade-off，才能做出正确的技术选型。

### 事务隔离

数据库事务隔离级别的定义（READ UNCOMMITTED、READ COMMITTED、REPEATABLE READ、SERIALIZABLE）、各种并发问题的成因（脏读、不可重复读、幻读）、MVCC 的实现原理。理解隔离级别，才能正确配置数据库。

### 死锁处理

死锁的检测方法（wait-dog graph）、死锁的恢复策略、死锁的预防原则（固定锁顺序、资源一次性申请）、活锁与饥饿问题。死锁是并发系统中最危险的问题之一，需要掌握排查和预防能力。

### 并发数据结构

Java 并发包（juc）中的核心数据结构：ConcurrentHashMap 的分段锁和 CAS 优化、CopyOnWriteArrayList 的适用场景、BlockingQueue 的生产者-消费者模式、ConcurrentLinkedQueue 的无锁实现。善用现成的并发数据结构，避免重复造轮子。
