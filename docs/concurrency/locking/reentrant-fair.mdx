# 可重入锁与公平锁

一个线程获取了锁，还能再次获取吗？等待的线程，谁先获取锁？

可重入锁和公平锁是两种重要的锁特性，它们解决了不同的问题。

### 可重入 vs 不可重入

```java
// 不可重入锁：会导致死锁
public class NonReentrantLock {
    private boolean locked = false;

    public synchronized void lock() throws InterruptedException {
        while (locked) {
            wait();  // 已持有锁的线程再次调用会阻塞
        }
        locked = true;
    }

    public synchronized void unlock() {
        locked = false;
        notify();
    }

    public void doA() throws InterruptedException {
        lock();
        try {
            doB();  // 这里会死锁！
        } finally {
            unlock();
        }
    }

    public void doB() throws InterruptedException {
        lock();  // 尝试再次获取锁，已持有的线程会阻塞
        try {
            // ...
        } finally {
            unlock();
        }
    }
}

// 可重入锁：计数器实现
public class ReentrantLock {
    private boolean locked = false;
    private Thread ownerThread = null;
    private int count = 0;

    public synchronized void lock() throws InterruptedException {
        Thread current = Thread.currentThread();

        if (ownerThread == current) {
            // 同一线程重入，计数器增加
            count++;
            return;
        }

        while (locked) {
            wait();
        }

        locked = true;
        ownerThread = current;
        count = 1;
    }

    public synchronized void unlock() {
        if (ownerThread != Thread.currentThread()) {
            throw new IllegalMonitorStateException();
        }

        count--;
        if (count == 0) {
            locked = false;
            ownerThread = null;
            notify();
        }
    }
}
```

## Java 中的可重入锁

```java
// synchronized 是可重入锁
public class SynchronizedReentrant {
    public synchronized void methodA() {
        System.out.println("A start");
        methodB();  // 重入
        System.out.println("A end");
    }

    public synchronized void methodB() {
        System.out.println("B start");
        // ...
        System.out.println("B end");
    }
}

// ReentrantLock 也是可重入锁
public class ReentrantLockReentrant {
    private final ReentrantLock lock = new ReentrantLock();

    public void methodA() {
        lock.lock();
        try {
            System.out.println("A start");
            methodB();  // 重入
            System.out.println("A end");
        } finally {
            lock.unlock();
        }
    }

    public void methodB() {
        lock.lock();  // 重入
        try {
            System.out.println("B start");
            // ...
            System.out.println("B end");
        } finally {
            lock.unlock();  // 注意：重入计数为 2，需要 unlock 两次
        }
    }
}
```

## 公平锁

```
┌─────────────────────────────────────────────────────────────────┐
│                    公平锁 vs 非公平锁                                │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │ 公平锁（Fair）                                         │   │
│  ├─────────────────────────────────────────────────────────┤   │
│  │                                                         │   │
│  │  等待时间最长的线程先获取锁。                     │   │
│  │  FIFO 顺序，绝对公平。                               │   │
│  │                                                         │   │
│  │  Thread-A ──────────→ [获取锁]                      │   │
│  │  Thread-B ──┐                                         │   │
│  │  Thread-C ──┴─→ [等待 A] → [等待 B] → [获取锁]    │   │
│  │                                                         │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                  │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │ 非公平锁（Nonfair）                                    │   │
│  ├─────────────────────────────────────────────────────────┤   │
│  │                                                         │   │
│  │  允许插队，可能比等待中的线程先获取锁。           │   │
│  │  性能更高，但可能饥饿。                              │   │
│  │                                                         │   │
│  │  Thread-A ──────────→ [释放锁]                       │   │
│  │  Thread-B ──┐                                         │   │
│  │  Thread-C ──┴─→ [尝试获取] → [可能插队] → [获取锁]│   │
│  │                                                         │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

### Java 实现

```java
// ReentrantLock 的公平性
public class FairLockExample {
    // 公平锁：等待时间最长的线程先获取
    private final ReentrantLock fairLock = new ReentrantLock(true);

    // 非公平锁（默认）：可能插队
    private final ReentrantLock unfairLock = new ReentrantLock(false);

    // 公平锁的等待队列
    // 严格 FIFO，不允许插队
    public void fairAccess() {
        fairLock.lock();
        try {
            // 临界区
        } finally {
            fairLock.unlock();
        }
    }

    // 非公平锁的竞争
    // 在锁释放和新线程获取之间有时间窗口
    // 新线程可能在这个窗口内获取锁
    public void unfairAccess() {
        unfairLock.lock();
        try {
            // 临界区
        } finally {
            unfairLock.unlock();
        }
    }
}
```

│  │                                                         │   │
│  │  缺点：                                              │   │
│  │  - 可能导致线程饥饿                                │   │
│  │  - 获取锁的顺序不确定                              │   │
│  │                                                         │   │

### 实践建议

```java
// 典型场景分析

// 场景一：高频计数器
public class CounterExample {
    private final ReentrantLock lock = new ReentrantLock();  // 默认非公平

    private int count = 0;

    // 锁持有时间极短（count++）
    // 非公平锁性能更好
    public void increment() {
        lock.lock();
        try {
            count++;
        } finally {
            lock.unlock();
        }
    }
}

// 场景二：任务调度器
public class TaskScheduler {
    // 需要公平性，保证任务按顺序执行
    private final ReentrantLock lock = new ReentrantLock(true);

    // 锁持有时间较长（执行任务）
    // 公平锁避免饥饿
    public void execute(Task task) {
        lock.lock();
        try {
            task.run();  // 可能耗时较长
        } finally {
            lock.unlock();
        }
    }
}

// 场景三：连接池
public class ConnectionPool {
    // 非公平锁更好
    // 线程获取连接是高频操作
    // 非公平锁吞吐量更高
    private final ReentrantLock lock = new ReentrantLock(false);

    public Connection getConnection() {
        lock.lock();
        try {
            return acquireFromPool();
        } finally {
            lock.unlock();
        }
    }
}
```

## 总结

可重入锁和公平锁是两种不同的锁特性。

**可重入锁**：

- 同一线程可以多次获取同一把锁
- 每次获取计数 +1，每次释放计数 -1
- synchronized 和 ReentrantLock 都是可重入的

**公平锁**：

- 等待时间最长的线程先获取锁
- FIFO 顺序，绝对公平
- 吞吐量低于非公平锁

**选择建议**：

- 绝大多数场景：非公平可重入锁
- 需要严格公平性：公平可重入锁
- 同一线程内调用：必须可重入锁

**核心认知**：Java 的 ReentrantLock 默认是非公平可重入锁，这是性能最优的配置。只有在明确需要公平性时才使用公平锁。
