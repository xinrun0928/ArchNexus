# 事务隔离

两个并发的事务同时执行：A 转账给 B 100 元，A 扣款成功，但 B 还没收到钱时，B 去查余额——看到的还是旧数据。你觉得这是 A 的问题？B 的问题？还是数据库的问题？

都不是。这是**事务隔离**的问题。

## 为什么需要事务隔离

数据库要同时服务大量并发请求。如果每个事务都能「看到」其他事务的中间状态，就会产生各种数据异常：读到未提交的修改、同一行数据两次读结果不一致、查询结果集突然多了或少了记录。这些听起来是小问题，但在金融系统、库存扣减、订单处理等场景下，一个不一致的数据可能导致严重的业务损失。

事务隔离级别，就是用来控制「能看到多少」的工具。隔离得越严，一致性越好，但性能越差——这是数据库领域最经典的权衡。

## 核心问题：隔离级别与并发异常

SQL 标准定义了四级隔离级别，每一级都旨在消除特定的并发异常：

| 隔离级别 | 脏读 | 不可重复读 | 幻读 |
|---|---|---|---|
| READ UNCOMMITTED | 可能 | 可能 | 可能 |
| READ COMMITTED | 不可能 | 可能 | 可能 |
| REPEATABLE READ | 不可能 | 不可能 | 可能 |
| SERIALIZABLE | 不可能 | 不可能 | 不可能 |

问题来了：MySQL 的 REPEATABLE READ 真的能防止幻读吗？PostgreSQL 的 MVCC 和 MySQL 的 MVCC 在实现上有什么区别？为什么大多数业务系统用 READ COMMITTED 就够了？

这些问题会贯穿整个章节。

## 章节内容

### 事务隔离级别

理解四种标准隔离级别是基础。每种级别对并发异常的防护能力不同，性能损耗也不同。你需要知道：MySQL 默认是 REPEATABLE READ，而 PostgreSQL 默认是 READ COMMITTED——这不是随意的选择，背后有各自的考量。

### 脏读、不可重复读、幻读

三种异常是理解隔离级别的钥匙。先搞懂「会发生什么」，才能理解「隔离级别在防什么」。这一节会通过具体场景演示每种异常的成因和危害。

### MVCC 原理

MVCC（Multi-Version Concurrency Control）是现代数据库实现高并发读写的核心机制。它让读写不再互斥，显著提升了并发性能。但 MVCC 不是银弹——它有自己的适用场景和边界条件。理解 ReadView 的生成时机，是掌握 MVCC 的关键。

### 隔离级别选择

没有最好的隔离级别，只有最合适的。选择隔离级别需要考虑业务一致性需求、性能要求、以及数据库的默认行为。这一节会给出不同场景下的选择依据。

## 学习路径

```
levels → anomalies → mvcc → selection
  ↑         ↑         ↑        ↑
 基础     理解问题   理解原理   应用决策
```

建议按顺序学习：先掌握隔离级别定义，再理解三种异常，然后深入 MVCC 原理，最后学会根据业务场景做决策。如果你已经熟悉前两部分，可以直接跳到 MVCC 原理，但隔离级别选择建议最后阅读——因为它需要综合前面所有知识。

## 核心问题

学完这章，你应该能回答：

- 为什么 SERIALIZABLE 能防止所有异常，但大多数系统不用它？
- MVCC 中 ReadView 是什么时候生成的？不同的生成时机对隔离级别有什么影响？
- MySQL 和 PostgreSQL 在 MVCC 实现上的核心差异是什么？
- 你的业务场景，应该选哪个隔离级别？为什么？
