# 四层 vs 七层负载均衡

你的系统流量越来越大，一台负载均衡器扛不住了。产品经理问：「是加机器还是升级配置？」

但这个问题背后还有一个更根本的问题：**你需要的是四层负载均衡还是七层负载均衡？**

## 网络模型回顾

```
┌─────────────────────────────────────────────────────────────────┐
│                      OSI 七层模型                                 │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  第 7 层：应用层    HTTP, FTP, SMTP, DNS, SSH                   │
│  第 6 层：表示层    SSL, TLS, JPEG, ASCII                       │
│  第 5 层：会话层    NetBIOS, RPC                               │
│  ────────────────────────────────────────────────────────       │
│  第 4 层：传输层    TCP, UDP                                    │
│  第 3 层：网络层    IP, ICMP, ARP                              │
│  第 2 层：数据链路层  Ethernet, PPP, Frame Relay              │
│  第 1 层：物理层    光纤, 电缆, 集线器                         │
│                                                                  │
│  四层负载均衡：工作在第 4 层（传输层）                          │
│  七层负载均衡：工作在第 7 层（应用层）                          │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

## 四层负载均衡

### 工作原理

```
┌─────────────────────────────────────────────────────────────────┐
│                    四层负载均衡原理                              │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  客户端 ──▶ [四层 LB] ──▶ 后端服务器                          │
│                                                                  │
│  工作在传输层（TCP/UDP）                                        │
│  只看 IP + 端口，不解析内容                                     │
│                                                                  │
│  ┌─────────┐     ┌─────────┐     ┌─────────┐                 │
│  │ Client   │     │  L4 LB  │     │ Server A │                 │
│  │ :1000   │ ──▶ │         │ ──▶ │         │                 │
│  │ → :80   │     │ 转发    │     │         │                 │
│  └─────────┘     └─────────┘     └─────────┘                 │
│                                                                  │
│  LB 做的事情：                                                 │
│  - 解析 IP 头和 TCP 头                                         │
│  - 根据算法选择后端                                             │
│  - 修改目标 IP + 端口                                          │
│  - NAT 转发                                                    │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

### 优势

```
四层负载均衡的优势：

1. 性能高
   - 只解析到第四层，不解析应用层
   - 吞吐量大（L4 VS L7 差 3-5 倍）
   - 延迟低（微秒级）

2. 简单稳定
   - 不解析应用层协议，不受协议限制
   - 适合 TCP/UDP 任意协议
   - 成熟稳定

3. 抗攻击
   - 不解析应用层内容，不受应用层攻击影响
   - DDoS 攻击在四层被拦截
```

### 劣势

```
四层负载均衡的劣势：

1. 无法感知应用层信息
   - 不能基于 URL 路由
   - 不能做 Cookies、Header 改写
   - 不能做 SSL 终结

2. 会话保持困难
   - 只能基于源 IP，精度低
   - NAT 环境下不准确
```

### 常见实现

```nginx
# Nginx 四层负载均衡
stream {
    upstream backend {
        server 127.0.0.1:8001;
        server 127.0.0.1:8002;
        server 127.0.0.1:8003;
    }

    server {
        listen 80;
        proxy_pass backend;
        proxy_connect_timeout 1s;
    }
}
```

```yaml
# Envoy 四层负载均衡
static_resources:
  listeners:
    - name: "tcp_proxy"
      address:
        socket_address:
          address: "0.0.0.0"
          port_value: 80

      filter_chains:
        - filters:
            - name: "envoy.filters.network.tcp_proxy"
              typed_config:
                "@type": "type.googleapis.com/envoy.extensions.filters.network.tcp_proxy.v3.TcpProxy"
                stat_prefix: "tcp"
                cluster: "backend_cluster"

  clusters:
    - name: "backend_cluster"
      connect_timeout: 5s
      type: STRICT_DNS
      lb_policy: ROUND_ROBIN
      hosts:
        - socket_address:
            address: "127.0.0.1"
            port_value: 8001
```

## 七层负载均衡

### 工作原理

```
┌─────────────────────────────────────────────────────────────────┐
│                    七层负载均衡原理                              │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  客户端 ──▶ [七层 LB] ──▶ 后端服务器                          │
│                                                                  │
│  工作在应用层（HTTP/HTTPS）                                     │
│  解析完整请求，感知 URL、Header、Cookie                          │
│                                                                  │
│  ┌─────────┐     ┌─────────┐     ┌─────────┐                 │
│  │ Client   │     │  L7 LB  │     │ Server A │                 │
│  │         │ ──▶ │ URL: /api│ ──▶ │ /api    │                 │
│  │         │     │ Header   │     │         │                 │
│  └─────────┘     └─────────┘     └─────────┘                 │
│                                                                  │
│  LB 做的事情：                                                 │
│  - 解析 HTTP 请求（Method, URL, Header）                       │
│  - 基于 URL 路径、Header、Cookie 路由                          │
│  - SSL 终结                                                     │
│  - 请求改写                                                     │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

### 优势

```
七层负载均衡的优势：

1. 智能路由
   - 基于 URL 路径路由
   - 基于 Header 路由（如 X-User-ID）
   - 基于 Cookie 路由
   - 基于域名路由

2. 安全功能
   - SSL 终结（卸载）
   - WAF（Web 应用防火墙）
   - Bot 检测
   - 请求限流

3. 内容改写
   - Header 添加/删除/改写
   - Cookie 处理
   - 重定向
```

### 劣势

```
七层负载均衡的劣势：

1. 性能开销
   - 需要解析完整 HTTP 请求
   - 吞吐量和延迟都不如四层

2. 复杂度高
   - 需要理解应用层协议
   - 配置复杂
   - 维护成本高
```

### 常见实现

```nginx
# Nginx 七层负载均衡
http {
    upstream backend {
        server 127.0.0.1:8001;
        server 127.0.0.1:8002;
    }

    server {
        listen 80;

        location /api/ {
            proxy_pass http://backend;
            proxy_set_header Host $host;
            proxy_set_header X-Real-IP $remote_addr;
            proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        }

        location /admin/ {
            # 管理员走独立后端
            proxy_pass http://admin_backend;
        }
    }
}
```

```yaml
# Envoy 七层负载均衡
static_resources:
  listeners:
    - name: "http"
      address:
        socket_address:
          address: "0.0.0.0"
          port_value: 80

      filter_chains:
        - filters:
            - name: "envoy.filters.network.http_connection_manager"
              typed_config:
                "@type": "type.googleapis.com/envoy.extensions.filters.network.http_connection_manager.v3.HttpConnectionManager"
                route_config:
                  virtual_hosts:
                    - name: "backend"
                      domains: ["*"]
                      routes:
                        - match: { prefix: "/api/" }
                          route: { cluster: "api_cluster" }
                        - match: { prefix: "/admin/" }
                          route: { cluster: "admin_cluster" }
                        - match: { prefix: "/" }
                          route: { cluster: "default_cluster" }
```

## 四层 vs 七层对比

| 维度 | 四层负载均衡 | 七层负载均衡 |
|---|---|---|
| 工作层 | 传输层（TCP/UDP） | 应用层（HTTP/HTTPS） |
| 转发依据 | IP + 端口 | URL、Header、Cookie |
| 性能 | 高（微秒级延迟） | 中（毫秒级延迟） |
| 吞吐量 | 高 | 中 |
| SSL 终结 | 不支持 | 支持 |
| 智能路由 | 不支持 | 支持 |
| 安全功能 | 有限 | 丰富（WAF、BOT） |
| 适用场景 | 数据库、缓存、TCP 服务 | Web 服务、API 网关 |

## 组合使用

```
┌─────────────────────────────────────────────────────────────────┐
│                    四层 + 七层组合                                │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  ┌─────────┐                                                    │
│  │  客户端  │                                                   │
│  └────┬────┘                                                    │
│       │                                                          │
│       ▼                                                          │
│  ┌─────────┐                                                    │
│  │  L4 LB  │  ← 入口，高性能                                  │
│  │ (TCP)   │    四层转发到 L7                                 │
│  └────┬────┘                                                    │
│       │                                                          │
│       ▼                                                          │
│  ┌─────────┐                                                    │
│  │  L7 LB  │  ← 应用层，智能路由                              │
│  │ (HTTP)  │    七层转发到具体服务                             │
│  └────┬────┘                                                    │
│       │                                                          │
│       ▼                                                          │
│  ┌─────────┐  ┌─────────┐  ┌─────────┐                      │
│  │ Service A│  │Service B│  │Service C│                      │
│  └─────────┘  └─────────┘  └─────────┘                      │
│                                                                  │
│  架构优势：                                                     │
│  - L4 处理大量并发连接，性能高                                  │
│  - L7 做精细化路由，功能丰富                                   │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

## 选型建议

```
选择四层的场景：
- TCP/UDP 服务（MySQL、Redis、MQ）
- 需要高性能的 HTTP 服务
- 不需要感知 URL 或 Header

选择七层的场景：
- HTTP/HTTPS 服务
- 需要基于 URL 路由
- 需要 SSL 终结
- 需要 WAF、BOT 防护
- 需要请求改写

组合使用的场景：
- 大型互联网应用
- 高并发 + 智能路由
- 安全要求高
```

## 实际案例

### Nginx + Keepalived（L4）

```bash
# keepalived.conf
vrrp_instance VI_1 {
    state MASTER
    interface eth0
    virtual_router_id 51
    priority 100
    virtual_ipaddress {
        192.168.1.100
    }
}
```

### Envoy（L7）

```yaml
# 完整的 Envoy 配置示例
static_resources:
  listeners:
    - name: "https"
      address:
        socket_address:
          address: "0.0.0.0"
          port_value: 443

      filter_chains:
        - filters:
            - name: "envoy.filters.network.http_connection_manager"
              typed_config:
                "@type": "type.googleapis.com/envoy.extensions.filters.network.http_connection_manager.v3.HttpConnectionManager"
                codec_type: AUTO
                route_config:
                  virtual_hosts:
                    - name: "backend"
                      domains: ["*"]
                      routes:
                        - match: { prefix: "/" }
                          route: { cluster: "backend" }
                http_filters:
                  - name: "envoy.filters.http.router"
```

## 总结

四层和七层负载均衡各有优势，需要根据实际场景选择。

**核心要点**：

- 四层：高性能，简单，适合基础设施
- 七层：功能丰富，智能，适合应用层
- 组合使用：各取所长

**选择建议**：

- 基础设施层：四层
- 应用层：七层
- 大型系统：四层 + 七层组合
