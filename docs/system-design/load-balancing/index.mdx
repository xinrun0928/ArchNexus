# 负载均衡

想象一下这样的场景：双十一零点，流量洪峰瞬间涌入，单台服务器在 30 秒内被打满，所有接口开始超时。如果此时只有一台服务器，它会直接崩溃；如果有多台服务器但没有负载均衡，请求还是会全部涌向同一台。

负载均衡（Load Balancing）正是为解决这类问题而生。它的核心职责是：将大量并发请求合理地分配到多台服务器上，避免单点过载，同时具备故障检测与自动摘除能力，保障系统整体可用性。

## 为什么需要负载均衡

单机处理的瓶颈是显而易见的：CPU、内存、磁盘 I/O、网络带宽都是有限的资源。当请求量超过单台服务器的处理能力时，响应时间会急剧上升，甚至导致服务完全不可用。

水平扩展是解决单机瓶颈的标准思路——增加服务器数量，让每台机器只处理一部分请求。但这需要一个「调度者」来决定：哪个请求该去哪台服务器？这个调度者就是负载均衡器。

除了流量分发，负载均衡还需要解决三个核心问题：

**单点故障**：如果负载均衡器本身挂了怎么办？现实中通常采用主备部署或集群模式，让多台负载均衡器协同工作。

**健康检测**：当某台后端服务器出现故障（比如 JVM OOM、数据库连接耗尽），负载均衡器必须能及时发现，并将流量切换到健康的节点。

**分配策略**：如何决定将请求发送到哪台服务器？是轮询、按权重分配、还是根据实时负载动态调整？

## 章节内容

本章节将系统讲解负载均衡的各个方面：

- **负载均衡算法**：从轮询、加权到最少连接、源地址哈希，不同算法适用于不同场景
- **四层 vs 七层负载均衡**：了解 OSI 模型不同层的负载均衡机制与各自优劣
- **健康检查机制**：掌握主动检测与被动检测策略，确保故障节点及时下线
- **会话保持策略**：解决分布式会话一致性问题，Cookie 插入、Session 复制与粘性会话
