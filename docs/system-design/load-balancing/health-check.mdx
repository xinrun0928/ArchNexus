# 健康检查机制

你的系统中有一台服务器宕机了，但负载均衡器还在往这台服务器分流量。结果：大量请求超时，用户怨声载道。

问题出在哪？**健康检查没有做好**。

## 健康检查的本质

```
┌─────────────────────────────────────────────────────────────────┐
│                    健康检查的核心问题                            │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  负载均衡器面临的核心问题：                                       │
│  后端服务器还活着吗？                                           │
│                                                                  │
│  不知道 → 继续分发 → 请求失败                                   │
│  知道 → 停止分发 → 没问题                                       │
│                                                                  │
│  健康检查 = 让负载均衡器知道「服务器还活着」                     │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

## 健康检查类型

### 被动健康检查

```
原理：基于请求结果判断服务器健康状态
流程：
1. 转发请求到后端
2. 如果失败次数超过阈值，标记为不健康
3. 如果成功次数超过阈值，标记为健康

优点：无需额外探测流量
缺点：用户会请求到故障节点
```

```java
public class PassiveHealthChecker {
    private final Map<Server, FailureCounter> failureCounters = new ConcurrentHashMap<>();
    private final int failureThreshold = 5;
    private final int successThreshold = 3;

    public void recordSuccess(Server server) {
        FailureCounter counter = failureCounters.computeIfAbsent(server, k -> new FailureCounter());
        counter.reset();
        counter.successCount++;

        if (counter.successCount >= successThreshold && !server.isHealthy()) {
            server.setHealthy(true);
            log.info("服务器 {} 恢复健康", server);
        }
    }

    public void recordFailure(Server server) {
        FailureCounter counter = failureCounters.computeIfAbsent(server, k -> new FailureCounter());
        counter.failureCount++;

        if (counter.failureCount >= failureThreshold && server.isHealthy()) {
            server.setHealthy(false);
            log.warn("服务器 {} 标记为不健康", server);
        }
    }
}
```

### 主动健康检查

```
原理：定期发送探测请求，判断服务器健康状态
流程：
1. 定时发送探测请求到后端
2. 如果探测成功，标记为健康
3. 如果探测失败，标记为不健康

优点：提前发现故障，不影响用户请求
缺点：需要额外的探测流量
```

```java
public class ActiveHealthChecker {
    private final ScheduledExecutorService scheduler = Executors.newScheduledThreadPool(2);
    private final Map<Server, HealthStatus> healthStatus = new ConcurrentHashMap<>();

    public void start(List<Server> servers, int intervalSeconds, int timeoutMs) {
        scheduler.scheduleAtFixedRate(() -> {
            for (Server server : servers) {
                checkServer(server, timeoutMs);
            }
        }, 0, intervalSeconds, TimeUnit.SECONDS);
    }

    private void checkServer(Server server, int timeoutMs) {
        try {
            HttpResponse response = httpClient.get(server.getUrl() + "/health", timeoutMs);
            if (response.isSuccess()) {
                updateHealth(server, HealthStatus.HEALTHY);
            } else {
                updateHealth(server, HealthStatus.UNHEALTHY);
            }
        } catch (Exception e) {
            updateHealth(server, HealthStatus.UNHEALTHY);
        }
    }

    private void updateHealth(Server server, HealthStatus newStatus) {
        HealthStatus oldStatus = healthStatus.put(server, newStatus);
        if (oldStatus != newStatus) {
            if (newStatus == HealthStatus.UNHEALTHY) {
                loadBalancer.remove(server);
                log.warn("移除不健康服务器: {}", server);
            } else if (newStatus == HealthStatus.HEALTHY && oldStatus == HealthStatus.UNHEALTHY) {
                loadBalancer.add(server);
                log.info("添加健康服务器: {}", server);
            }
        }
    }
}
```

## 健康检查方式

### TCP 端口检查

```
原理：尝试建立 TCP 连接
示例：telnet 127.0.0.1 8001

优点：简单，不需要后端配合
缺点：端口通不代表服务正常
```

```nginx
# Nginx TCP 健康检查
upstream backend {
    server 127.0.0.1:8001;
    server 127.0.0.1:8002;
}

# Nginx 不内置主动健康检查，需要第三方模块
```

### HTTP/HTTPS 检查

```
原理：发送 HTTP 请求，检查响应状态码
优点：可以自定义检查逻辑
缺点：需要后端提供健康检查接口
```

```nginx
# Nginx 七层健康检查（需要 nginx_upstream_check_module）
upstream backend {
    server 127.0.0.1:8001;
    server 127.0.0.1:8002;
    check interval=3000 rise=2 fall=3 timeout=1000 type=http;
    check_http_send "GET /health HTTP/1.0\r\n\r\n";
    check_http_expect_alive http_2xx;
}
```

```yaml
# Kubernetes 健康检查
apiVersion: v1
kind: Pod
spec:
  containers:
    - name: app
      livenessProbe:
        httpGet:
          path: /health
          port: 8080
        initialDelaySeconds: 10
        periodSeconds: 15
        timeoutSeconds: 5
        failureThreshold: 3
      readinessProbe:
        httpGet:
          path: /ready
          port: 8080
        initialDelaySeconds: 5
        periodSeconds: 10
        failureThreshold: 3
```

### 业务层健康检查

```java
// 业务层健康检查
@RestController
public class HealthController {

    private final DatabaseHealthChecker databaseHealth;
    private final CacheHealthChecker cacheHealth;
    private final QueueHealthChecker queueHealth;

    @GetMapping("/health")
    public ResponseEntity<Health> health() {
        Health health = new Health();

        // 检查数据库
        try {
            databaseHealth.check();
            health.put("database", "UP");
        } catch (Exception e) {
            health.put("database", "DOWN: " + e.getMessage());
            health.setStatus("UNHEALTHY");
        }

        // 检查缓存
        try {
            cacheHealth.check();
            health.put("cache", "UP");
        } catch (Exception e) {
            health.put("cache", "DEGRADED: " + e.getMessage());
        }

        // 检查消息队列
        try {
            queueHealth.check();
            health.put("queue", "UP");
        } catch (Exception e) {
            health.put("queue", "DEGRADED: " + e.getMessage());
        }

        // 综合判断
        if ("UNHEALTHY".equals(health.getStatus())) {
            return ResponseEntity.status(503).body(health);
        }

        return ResponseEntity.ok(health);
    }

    // 就绪检查：是否准备好接收流量
    @GetMapping("/ready")
    public ResponseEntity<Map<String, Object>> ready() {
        Map<String, Object> status = new HashMap<>();

        // 检查依赖是否就绪
        boolean ready = checkDependencies(status);

        if (ready) {
            return ResponseEntity.ok(status);
        } else {
            return ResponseEntity.status(503).body(status);
        }
    }

    private boolean checkDependencies(Map<String, Object> status) {
        // 检查数据库连接池
        boolean dbReady = databaseHealth.isConnectionPoolReady();
        status.put("database", dbReady ? "ready" : "not_ready");

        // 检查缓存连接
        boolean cacheReady = cacheHealth.isConnected();
        status.put("cache", cacheReady ? "ready" : "not_ready");

        return dbReady && cacheReady;
    }
}
```

## 健康检查的常见问题

### 问题一：检查间隔设置

```
检查间隔太短 → 探测流量太大
检查间隔太长 → 发现故障太慢

建议：
- 核心服务：5-10 秒
- 非核心服务：30-60 秒
```

### 问题二：阈值设置

```java
// 不健康阈值过高
if (failureThreshold > 10) {  // 太高
    // 故障节点可能持续分发大量请求
}

// 不健康阈值过低
if (failureThreshold < 2) {  // 太低
    // 网络抖动导致误判
}

// 建议
int failureThreshold = 3;    // 连续 3 次失败
int successThreshold = 2;     // 连续 2 次成功
```

### 问题三：优雅下线

```java
// 应用关闭前，先从负载均衡器摘除
@PreDestroy
public void deregisterFromLoadBalancer() {
    // 1. 通知负载均衡器摘除
    loadBalancer.deregister(this);

    // 2. 等待现有请求处理完成
    try {
        Thread.sleep(30 * 1000);  // 等待 30 秒
    } catch (InterruptedException e) {
        Thread.currentThread().interrupt();
    }

    // 3. 关闭应用
    shutdown();
}

// Spring Boot
@Component
public class GracefulShutdown implements SmartLifecycle {
    private boolean running = false;

    @Override
    public void start() {
        running = true;
    }

    @Override
    public void stop() {
        // 1. 从负载均衡器摘除
        loadBalancer.deregister();

        // 2. 等待请求处理完成
        waitForRequestsComplete();

        running = false;
    }

    @Override
    public boolean isRunning() {
        return running;
    }
}
```

### 问题四：金丝雀发布

```java
// 金丝雀发布：先分发少量流量
public class CanaryDeployment {
```

## 总结

健康检查是保证系统高可用的关键机制。

**核心要点**：

- 被动检查：基于请求结果，适合快速发现故障
- 主动检查：定期探测，提前发现故障
- 业务层检查：检查依赖，不只是端口

**配置建议**：

- 检查间隔：5-30 秒
- 不健康阈值：3 次
- 健康阈值：2 次
- 超时时间：5 秒

**注意事项**：

- 检查依赖，不只是端口
- 优雅下线，先摘除再关闭
- 金丝雀发布时单独检查
