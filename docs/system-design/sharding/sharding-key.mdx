# 分片键选择

分库分表最大的坑是什么？不是技术实现，而是**分片键选错了**。

选错了分片键，轻则查询受限，重则数据严重倾斜，还可能无法扩容。

## 分片键的重要性

```
┌─────────────────────────────────────────────────────────────────┐
│                    分片键的重要性                                │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  分片键 = 数据路由的依据                                        │
│                                                                  │
│  选对了：                                                       │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │ • 查询高效（直接定位分片）                                │   │
│  │ • 数据均匀（负载分散）                                   │   │
│  │ • 扩缩容简单                                            │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                  │
│  选错了：                                                       │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │ • 大量跨分片查询（性能灾难）                            │   │
│  │ • 数据倾斜（热点分片压力过大）                          │   │
│  │ • 无法扩容（选错键导致路由冲突）                       │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

## 分片键选择原则

### 原则一：高频查询字段

```
分片键应该是查询条件中经常出现的字段

正确做法：
┌─────────────────────────────────────────────────────────┐
│ 查询：SELECT * FROM orders WHERE user_id = ?               │
│ 分片键：user_id                                          │
│ 结果：直接定位到某个分片，查询高效                        │
└─────────────────────────────────────────────────────────┘

错误做法：
┌─────────────────────────────────────────────────────────┐
│ 查询：SELECT * FROM orders WHERE status = ?              │
│ 分片键：user_id                                         │
│ 结果：需要扫描所有分片（全分片扫描）                    │
└─────────────────────────────────────────────────────────┘
```

### 原则二：数据分布均匀

```
分片键的值应该分布均匀

均匀字段：
✓ user_id（哈希后均匀）
✓ order_id（雪花 ID 均匀）

不均匀字段：
✗ status（90% 是待支付）
✗ sex（男女各 50%，但不够细）
✗ province（用户集中在北上广）
```

### 原则三：避免热点

```
分片键不应是时间字段（近期的数据会集中在最后一个分片）

错误示例：
┌─────────────────────────────────────────────────────────┐
│ 按月分片：t_order_202401                                │
│ 问题：2 月的数据全在一个分片，热点严重                    │
└─────────────────────────────────────────────────────────┘

正确示例：
┌─────────────────────────────────────────────────────────┐
│ 按 user_id 哈希分片：t_order_{user_id % 4}              │
│ 优点：用户均匀分布，负载均衡                            │
└─────────────────────────────────────────────────────────┘
```

## 常见场景的分片键选择

### 场景一：用户中心

```
业务场景：
- 核心查询：查询某用户的所有订单
- 核心查询：查询某用户的收货地址

分片键选择：
┌─────────────────────────────────────────────────────────┐
│ 表：orders                                            │
│ 分片键：user_id                                      │
│ 理由：核心查询都基于 user_id                        │
│                                                      │
│ 表：addresses                                        │
│ 分片键：user_id                                      │
│ 理由：按用户查询地址                                 │
└─────────────────────────────────────────────────────────┘
```

### 场景二：订单中心

```
业务场景：
- 核心查询：用户查询自己的订单（WHERE user_id = ?）
- 核心查询：商家查询自己的订单（WHERE merchant_id = ?）
- 核心查询：后台按状态查询（WHERE status = ?）

分片键选择分析：
┌─────────────────────────────────────────────────────────┐
│ 选项 1：user_id                                       │
│ 优点：用户查询高效                                    │
│ 缺点：商家查询全分片扫描                              │
│                                                      │
│ 选项 2：merchant_id                                  │
│ 优点：商家查询高效                                    │
│ 缺点：用户查询全分片扫描                              │
│                                                      │
│ 选项 3：user_id + merchant_id 双重分片                │
│ 优点：支持两种查询                                    │
│ 缺点：存储翻倍，维护复杂                              │
└─────────────────────────────────────────────────────────┘

建议：根据核心业务选择
- C2C 电商（用户查询多）→ user_id
- B2C 电商（商家查询多）→ merchant_id
```

### 场景三：日志系统

```
业务场景：
- 核心查询：按时间范围查询
- 核心查询：按用户 ID 查询

分片键选择：
┌─────────────────────────────────────────────────────────┐
│ 表：access_logs                                       │
│ 分片键：created_date（按天分表）                      │
│ 理由：核心查询都带时间条件                           │
│                                                      │
│ 组合索引表：log_by_user                              │
│ 分片键：user_id                                      │
│ 理由：冗余一份按用户维度的数据                        │
└─────────────────────────────────────────────────────────┘
```

## 常见问题

### 问题一：查询不带分片键

```
问题：前端查询不带 user_id，只有手机号

解决方案：
┌─────────────────────────────────────────────────────────┐
│ 1. 缓存映射：将 user_id → phone 缓存起来            │
│ 2. 异构索引表：创建 phone → user_id 的映射表        │
│ 3. ES 搜索引擎：通过 phone 查到 user_id，再查分片    │
└─────────────────────────────────────────────────────────┘
```

```java
// 方案：异构索引
public class PhoneToUserIndex {
    // phone → user_id 映射
    public Long getUserIdByPhone(String phone) {
        return phoneIndex.get(phone);
    }
}

// 使用
public List<Order> getOrdersByPhone(String phone, int page, int pageSize) {
    // 1. 通过 phone 查到 user_id
    Long userId = phoneToUserIndex.getUserIdByPhone(phone);
    if (userId == null) {
        return Collections.emptyList();
    }

    // 2. 通过 user_id 路由到具体分片
    return orderRepository.findByUserId(userId, page, pageSize);
}
```

### 问题二：多字段查询

```
问题：需要按多个字段组合查询

示例：查询某商家在某个时间段的订单

解决方案：
┌─────────────────────────────────────────────────────────┐
│ 1. 冗余字段：merchant_id + created_date 组合           │
│    存储：merchant_id_created                         │
│    分片键：(merchant_id, created_date)              │
│                                                      │
│ 2. ES 搜索：所有查询先到 ES，再路由分片             │
│                                                      │
│ 3. 业务层聚合：先按一个键查，再过滤                 │
└─────────────────────────────────────────────────────────┘
```

### 问题三：分片键变更

```
问题：业务发展后发现分片键选错了

代价：
┌─────────────────────────────────────────────────────────┐
│ • 需要迁移全量数据                                      │
│ • 迁移期间双写                                         │
│ • 迁移后需要灰度切换                                   │
│ • 可能需要停服                                          │
└─────────────────────────────────────────────────────────┘

建议：
• 分片键选择时慎重考虑
• 预留扩展空间
• 使用代理层（ShardingSphere）减少绑定
```

## 总结

分片键选择是分库分表最关键的决策。

**核心要点**：

- 选择高频查询字段作为分片键
- 确保数据分布均匀
- 避免热点字段
- 必要时使用异构索引

**决策建议**：

- 列出所有查询场景
- 评估查询频率和数据分布
- 考虑扩缩容影响
- 必要时冗余存储

**常见选择**：

| 业务表 | 推荐分片键 | 原因 |
|---|---|---|
| 订单表 | user_id 或 merchant_id | 按用户/商家查询为主 |
| 用户表 | user_id | 按用户查询为主 |
| 日志表 | created_date | 按时间范围查询 |
| 消息表 | conversation_id | 按会话查询 |
