# 缓存设计

一个接口响应时间 500ms，加了缓存后变成 5ms。看起来立竿见影。但两周后的某个凌晨，运维突然发现缓存命中率跌到 1%，所有请求都打到数据库上，系统濒临崩溃。问题出在哪？

缓存是性能优化的第一把利器，但用不好反而会引入更难排查的一致性问题。它是架构中最容易上手、也最容易挖坑的设计之一。

## 缓存的核心价值

缓存的本质是「用空间换时间」。将热点数据存储在访问速度更快的介质中，减少对底层存储的访问次数。

缓存的核心价值体现在三个方面：

**降低延迟**。内存访问延迟通常是纳秒级，而机械硬盘是毫秒级，差距可达数万倍。一次缓存命中，可以将接口响应时间从几百毫秒降低到个位数毫秒。

**减轻后端压力**。数据库连接数是稀缺资源，当 QPS 达到一定量级后，数据库会成为系统的瓶颈。缓存可以拦截大量读请求，让数据库只处理必要的访问。

**提升系统吞吐量**。在高并发场景下，缓存可以将系统的整体吞吐量提升一到两个数量级，让有限的数据库资源服务更多用户。

## 缓存的常见陷阱

引入缓存不只是「加一层 if-else」。以下几个问题，是线上事故的高发区：

**一致性问题**。缓存和数据库是两份数据，任何一份的更新都可能产生不一致。当业务对数据实时性要求极高时，缓存反而会成为负担。

**缓存穿透**。大量请求查询不存在的数据，绕过缓存直接打到数据库。在遭受恶意攻击时，可能导致数据库过载。

**缓存击穿**。热点 key 突然过期，瞬间大量请求同时穿透到数据库，造成数据库压力激增。

**缓存雪崩**。大量缓存同时失效，或者缓存服务本身不可用，导致请求全部打到数据库，引发连锁反应。

**容量规划失控**。缓存数据不断堆积，内存耗尽后触发淘汰策略，可能导致命中率剧烈波动。

## 缓存的设计层次

缓存设计需要从多个层次综合考虑：

**读取路径**。数据从数据库写入缓存后，如何被读取？客户端先查缓存还是先查数据库？缓存未命中后如何回源？

**更新策略**。数据变更时，缓存如何同步更新？先更新数据库还是先删除缓存？更新失败如何保证最终一致？

**淘汰策略**。缓存空间有限，当达到容量上限时，哪些数据应该被淘汰？LRU、LFU、FIFO，各有什么优劣？

**过期策略**。缓存数据应该设置多久的 TTL？固定 TTL 还是动态 TTL？过期数据如何处理？
