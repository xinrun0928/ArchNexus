# 缓存穿透、击穿、雪崩

你的系统在缓存层面有三个致命问题：

- 缓存命中率极低
- 缓存过期时大量请求同时打爆数据库
- 整个缓存层集体失效

这三种情况，分别对应缓存的三大问题：**穿透、击穿、雪崩**。

## 问题概览

## 问题一：缓存穿透

### 问题描述

```
问题：查询不存在的数据

正常情况：
- 查询 id=1 → 缓存未命中 → 查数据库 → 写入缓存 → 返回

穿透情况：
- 查询 id=-1 → 缓存未命中 → 查数据库 → 数据库也没有 → 返回空
- 问题：每次都查数据库

危害：
- 数据库压力增大
- 如果被恶意利用（大量不存在的数据），可能导致数据库宕机
```

### 解决方案

#### 方案一：缓存空值

```java
// 缓存空值
@Service
public class CacheNullValueService {
    @Autowired
    private RedisTemplate<String, Object> redisTemplate;
    @Autowired
    private ProductRepository productRepository;

    private static final Object NULL_VALUE = new Object();
    private static final Duration NULL_TTL = Duration.ofMinutes(5);

    public Product getProduct(Long productId) {
        String cacheKey = "product:" + productId;

        // 1. 查缓存
        Object cached = redisTemplate.opsForValue().get(cacheKey);

        if (cached != null) {
            // 2. 命中空值，返回 null
            if (cached == NULL_VALUE) {
                return null;
            }
            // 3. 命中正常值
            return (Product) cached;
        }

        // 4. 缓存未命中，查数据库
        Product product = productRepository.findById(productId);

        // 5. 写入缓存（即使是空值）
        if (product != null) {
            redisTemplate.opsForValue().set(cacheKey, product, Duration.ofMinutes(30));
        } else {
            // 缓存空值，防止穿透
            redisTemplate.opsForValue().set(cacheKey, NULL_VALUE, NULL_TTL);
        }

        return product;
    }
}
```

#### 方案二：布隆过滤器

```
原理：使用布隆过滤器判断数据是否存在

工作流程：
1. 数据写入时，将数据 key 加入布隆过滤器
2. 查询时，先检查布隆过滤器
3. 布隆过滤器说不存在 → 直接返回，不查数据库
4. 布隆过滤器说存在 → 继续查缓存/数据库

优点：内存占用小，查询速度快
缺点：有一定的误判率（不存在的数据可能说存在）
```

```java
// 布隆过滤器
@Service
public class BloomFilterService {
    private final RedisBloomFilter<String> bloomFilter = RedisBloomFilter.create(
        "product-bloom",
        100_000_000,  // 预期数据量
        0.01          // 误判率
    );

    // 初始化时加载数据
    public void init() {
        productRepository.findAll().forEach(product ->
            bloomFilter.add("product:" + product.getId())
        );
    }

    // 查询
    public Product getProduct(Long productId) {
        String key = "product:" + productId;

        // 1. 检查布隆过滤器
        if (!bloomFilter.mightContain(key)) {
            // 布隆过滤器说不存在，直接返回
            return null;
        }

        // 2. 继续查缓存和数据库
        return getProductFromCacheOrDb(productId);
    }

    // 数据写入时
    public void addProduct(Product product) {
        bloomFilter.add("product:" + product.getId());
    }
}
```

## 问题二：缓存击穿

### 问题描述

```
问题：热点 key 过期，大量请求同时打到数据库

场景：
- 某商品是爆款，缓存中有
- 突然缓存过期了
- 大量请求同时查数据库

┌─────────────────────────────────────────────────────────────────┐
│                    缓存击穿                                       │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  缓存（热点 key）                                               │
│       │                                                        │
│       │ 过期                                                    │
│       ▼                                                        │
│  ┌─────────────────────────────────────────────┐               │
│  │                                             │               │
│  │ T1: Request A ──▶ 查 DB                    │               │
│  │ T2: Request B ──▶ 查 DB                    │               │
│  │ T3: Request C ──▶ 查 DB                    │               │
│  │ ...                                        │               │
│  │                                             │               │
│  └─────────────────────────────────────────────┘               │
│  数据库被打爆                                             │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

### 解决方案

#### 方案一：互斥锁

```java
// 互斥锁
@Service
public class MutexCacheService {
    @Autowired
    private RedisTemplate<String, Object> redisTemplate;
    @Autowired
    private ProductRepository productRepository;

    private static final String LOCK_KEY = "lock:product:";
    private static final Duration LOCK_TIMEOUT = Duration.ofSeconds(10);

    public Product getProduct(Long productId) {
        String cacheKey = "product:" + productId;

        // 1. 查缓存
        Product cached = (Product) redisTemplate.opsForValue().get(cacheKey);
        if (cached != null) {
            return cached;
        }

        // 2. 获取锁
        String lockKey = LOCK_KEY + productId;
        Boolean acquired = redisTemplate.opsForValue()
            .setIfAbsent(lockKey, "1", LOCK_TIMEOUT);

        try {
            // 3. 获取到锁，再查一次缓存
            if (Boolean.TRUE.equals(acquired)) {
                cached = (Product) redisTemplate.opsForValue().get(cacheKey);
                if (cached != null) {
                    return cached;
                }

                // 4. 查数据库
                Product product = productRepository.findById(productId);

                // 5. 写入缓存
                if (product != null) {
                    redisTemplate.opsForValue().set(cacheKey, product, Duration.ofMinutes(30));
                }

                return product;
            } else {
                // 5. 未获取到锁，等待后重试
                Thread.sleep(100);
                return getProduct(productId);  // 重试
            }
        } finally {
            // 6. 释放锁
            if (Boolean.TRUE.equals(acquired)) {
                redisTemplate.delete(lockKey);
            }
        }
    }
}
```

#### 方案二：逻辑过期

```
原理：不设置缓存 TTL，而是设置数据的「逻辑过期时间」。数据永不过期，但业务层判断是否需要刷新。

优点：完全避免击穿
缺点：实现复杂，需要额外字段
```

```java
// 逻辑过期
@Service
public class LogicalExpireService {
    @Autowired
    private RedisTemplate<String, Object> redisTemplate;
    @Autowired
    private ProductRepository productRepository;

    public Product getProduct(Long productId) {
        String cacheKey = "product:" + productId;

        // 1. 查缓存
        CacheData<Product> cached = (CacheData<Product>) redisTemplate.opsForValue().get(cacheKey);

        if (cached == null) {
            // 缓存不存在，查询数据库
            Product product = productRepository.findById(productId);
            if (product != null) {
                // 写入缓存，设置逻辑过期时间
                CacheData<Product> cacheData = new CacheData<>(product, Duration.ofMinutes(30));
                redisTemplate.opsForValue().set(cacheKey, cacheData);
            }
            return product;
        }

        // 2. 检查是否逻辑过期
        if (cached.isLogicalExpired()) {
            // 3. 逻辑过期，异步刷新缓存
            String lockKey = "lock:product:" + productId;
            Boolean acquired = redisTemplate.opsForValue()
                .setIfAbsent(lockKey, "1", Duration.ofSeconds(10));

            if (Boolean.TRUE.equals(acquired)) {
                // 异步刷新
                CompletableFuture.runAsync(() -> refreshCache(productId));
            }

            // 4. 返回旧数据
            return cached.getData();
        }

        // 5. 未过期，返回缓存数据
        return cached.getData();
    }

    private void refreshCache(Long productId) {
        Product product = productRepository.findById(productId);
        if (product != null) {
            CacheData<Product> cacheData = new CacheData<>(product, Duration.ofMinutes(30));
            redisTemplate.opsForValue().set("product:" + productId, cacheData);
        }
    }
}

// 缓存数据结构
@Data
public class CacheData<T> {
    private T data;
    private long expireTime;  // 逻辑过期时间（时间戳）

    public CacheData(T data, Duration logicalExpire) {
        this.data = data;
        this.expireTime = System.currentTimeMillis() + logicalExpire.toMillis();
    }

    public boolean isLogicalExpired() {
        return System.currentTimeMillis() > expireTime;
    }
}
```

## 问题三：缓存雪崩

### 问题描述

```
问题：大量 key 同时过期，或缓存服务宕机

场景 1：大量 key 同时过期
- 缓存设置了相同的过期时间
- 到达过期时间后，大量请求同时穿透

场景 2：缓存服务宕机
- Redis 宕机
- 所有请求打到数据库
- 数据库被打爆

┌─────────────────────────────────────────────────────────────────┐
│                    缓存雪崩                                       │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  缓存层                                                        │
│  ┌─────────┐ ┌─────────┐ ┌─────────┐                        │
│  │  key1   │ │  key2   │ │  key3   │                        │
│  │ expire! │ │ expire! │ │ expire! │                        │
│  └─────────┘ └─────────┘ └─────────┘                        │
│       │           │           │                               │
│       └───────────┴───────────┘                               │
│                    │                                           │
│                    ▼                                           │
│  ┌─────────────────────────────────────────────┐               │
│  │              数据库                           │               │
│  │                                             │               │
│  │   请求A ──▶ 查 DB                          │               │
│  │   请求B ──▶ 查 DB                          │               │
│  │   请求C ──▶ 查 DB                          │               │
│  └─────────────────────────────────────────────┘               │
│  数据库被打爆                                             │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

### 解决方案

#### 方案一：过期时间随机化

```java
// 过期时间随机化
@Service
public class RandomExpireService {
    @Autowired
    private RedisTemplate<String, Object> redisTemplate;

    // 基础过期时间 30 分钟，随机偏移 ±5 分钟
    private static final Duration BASE_EXPIRE = Duration.ofMinutes(30);
    private static final Duration RANDOM_OFFSET = Duration.ofMinutes(5);

    public void cacheProduct(Product product) {
        String cacheKey = "product:" + product.getId();

        // 过期时间 = 30 分钟 + 随机偏移 [-5, +5]
        long offset = (long) (Math.random() * 10 - 5) * 60 * 1000;
        Duration expire = BASE_EXPIRE.plusMillis(offset);

        redisTemplate.opsForValue().set(cacheKey, product, expire);
    }
}
```

## 总结

缓存三大问题是缓存架构中最常见的问题。

**核心对比**：

| 问题 | 原因 | 解决方案 |
|---|---|---|
| 穿透 | 查询不存在的数据 | 缓存空值、布隆过滤器 |
| 击穿 | 热点 key 过期 | 互斥锁、逻辑过期 |
| 雪崩 | 大量 key 同时过期 | 随机过期、多级缓存、高可用 |

**最佳实践**：

- 穿透 + 击穿：布隆过滤器 + 互斥锁
- 雪崩：过期时间随机化 + 多级缓存
- 兜底：限流 + 降级 + 高可用
