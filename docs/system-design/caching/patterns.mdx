# 缓存模式

你的系统每次查询都要访问数据库，数据库压力大得惊人。你决定加缓存，但加完缓存后，问题反而更多了：缓存和数据库不一致、缓存穿透、缓存雪崩……

问题出在哪？**没有选择正确的缓存模式**。

## 模式一：Cache Aside（旁路缓存）

### 原理

```
Cache Aside 是最常用的缓存模式。

读操作：
1. 先查缓存
2. 缓存命中 → 返回缓存数据
3. 缓存未命中 → 查数据库 → 写入缓存 → 返回数据

写操作：
1. 先更新数据库
2. 再删除缓存（或更新缓存）

┌─────────────────────────────────────────────────────────────────┐
│                    Cache Aside 读流程                            │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  Client                                                      │
│     │                                                         │
│     ├─▶ 查缓存 ──▶ 命中 ──▶ 返回                              │
│     │                                                         │
│     └─▶ 查缓存 ──▶ 未命中                                     │
│                    │                                           │
│                    ▼                                           │
│              查数据库                                           │
│                    │                                           │
│                    ▼                                           │
│              写入缓存                                           │
│                    │                                           │
│                    ▼                                           │
│               返回数据                                          │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

### 实现

```java
// Cache Aside 实现
@Service
public class CacheAsideService {
    @Autowired
    private RedisTemplate<String, Object> redisTemplate;
    @Autowired
    private ProductRepository productRepository;

    // 读取
    public Product getProduct(Long productId) {
        // 1. 查缓存
        String cacheKey = "product:" + productId;
        Product cached = (Product) redisTemplate.opsForValue().get(cacheKey);

        if (cached != null) {
            return cached;  // 缓存命中
        }

        // 2. 缓存未命中，查数据库
        Product product = productRepository.findById(productId);

        if (product != null) {
            // 3. 写入缓存
            redisTemplate.opsForValue().set(cacheKey, product, Duration.ofMinutes(30));
        }

        return product;
    }

    // 写入：先更新数据库，再删除缓存
    public void updateProduct(Product product) {
        // 1. 更新数据库
        productRepository.save(product);

        // 2. 删除缓存（不是更新！）
        String cacheKey = "product:" + product.getId();
        redisTemplate.delete(cacheKey);
    }

    // 删除
    public void deleteProduct(Long productId) {
        // 1. 删除数据库
        productRepository.deleteById(productId);

        // 2. 删除缓存
        String cacheKey = "product:" + productId;
        redisTemplate.delete(cacheKey);
    }
}
```

### 优缺点

```
优点：
✓ 实现简单
✓ 数据一致性较好（数据库是主数据源）
✓ 适合读多写少场景

缺点：
✗ 首次访问有缓存穿透（查数据库）
✗ 写入时可能有短暂不一致
✗ 缓存失效时大量请求打到数据库
```

## 模式二：Read Through（读穿透）

### 原理

```
Read Through 特点：应用层只访问缓存，缓存负责从数据库加载数据。

对比 Cache Aside：
- Cache Aside：应用层决定「什么时候查数据库」
- Read Through：缓存决定「什么时候查数据库」

应用层代码更简洁：
product = cache.get(key)  // 一行代码，缓存自动处理
```

### 实现

```java
// Read Through 实现
@Service
public class ReadThroughService {
    @Autowired
    private LoadingCache<Long, Product> productCache;

    // 读取：只需要调用缓存
    public Product getProduct(Long productId) {
        // 缓存自动处理数据库查询
        return productCache.get(productId);
    }
}

// 配置
@Configuration
public class CacheConfig {
    @Bean
    public LoadingCache<Long, Product> productCache(
            ProductRepository productRepository) {
        return Caffeine.newBuilder()
            .maximumSize(10_000)
            .expireAfterWrite(10, TimeUnit.MINUTES)
            .build(productId -> {
                // 缓存未命中时，自动从数据库加载
                return productRepository.findById(productId);
            });
    }
}
```

### 优缺点

```
优点：
✓ 应用层代码简洁
✓ 缓存自动处理加载逻辑
✓ 减少缓存穿透

缺点：
✗ 应用层对缓存有依赖
✗ 首次加载慢（没有预热）
✗ 不适合分布式缓存（如 Redis）
```

## 模式三：Write Through（写穿透）

### 原理

```
Write Through 特点：写入时同时更新缓存和数据库。

操作顺序：
1. 写入缓存
2. 写入数据库
3. 两者都成功才算成功

优点：缓存和数据库强一致
缺点：写入性能差（需要两次写入）
```

### 实现

```java
// Write Through 实现
@Service
public class WriteThroughService {
    @Autowired
    private RedisTemplate<String, Object> redisTemplate;
    @Autowired
    private ProductRepository productRepository;

    // 写入：同时写入缓存和数据库
    @Transactional
    public Product updateProduct(Product product) {
        // 1. 写入缓存
        String cacheKey = "product:" + product.getId();
        redisTemplate.opsForValue().set(cacheKey, product, Duration.ofMinutes(30));

        // 2. 写入数据库
        return productRepository.save(product);
    }
}
```

### 优缺点

```
优点：
✓ 缓存和数据库强一致
✓ 读取始终命中缓存

缺点：
✗ 写入性能差
✗ 实现复杂（需要处理部分失败）
✗ 不适合高并发写入场景
```

## 模式四：Write Behind（异步写入）

### 原理

```
Write Behind 特点：写入时只更新缓存，异步更新数据库。

操作顺序：
1. 写入缓存（立即返回）
2. 异步写入数据库

优点：写入性能极高
缺点：可能丢数据（缓存宕机）
```

### 实现

```java
// Write Behind 实现
@Service
public class WriteBehindService {
    @Autowired
    private RedisTemplate<String, Object> redisTemplate;
    @Autowired
    private KafkaTemplate<String, Object> kafkaTemplate;

    // 写入：只写缓存，异步写数据库
    public void updateProduct(Product product) {
        // 1. 写入缓存（立即返回）
        String cacheKey = "product:" + product.getId();
        redisTemplate.opsForValue().set(cacheKey, product);

        // 2. 发送消息，异步更新数据库
        kafkaTemplate.send("product-updates", new ProductUpdateEvent(product));
    }
}

// 消费者异步处理
@Service
public class ProductUpdateConsumer {
    @Autowired
    private ProductRepository productRepository;

    @KafkaListener(topics = "product-updates")
    public void handleProductUpdate(ProductUpdateEvent event) {
        productRepository.save(event.getProduct());
    }
}
```

### 优缺点

```
优点：
✓ 写入性能极高
✓ 抗写流量高峰

缺点：
✗ 可能丢数据
✗ 实现复杂
✗ 需要处理消息丢失、重复消费
```

## 组合使用

### 读写分离 + 缓存

```
实际项目中，通常组合使用多种模式：

读：Cache Aside
写：Write Behind（高并发）或 Write Through（一致性要求高）
```

```java
// 组合使用示例
@Service
public class HybridCacheService {
    @Autowired
    private RedisTemplate<String, Object> redisTemplate;
    @Autowired
    private ProductRepository productRepository;
    @Autowired
    private KafkaTemplate<String, Object> kafkaTemplate;

    // 读取：Cache Aside
    public Product getProduct(Long productId) {
        String cacheKey = "product:" + productId;
        Product cached = (Product) redisTemplate.opsForValue().get(cacheKey);

        if (cached != null) {
            return cached;
        }

        Product product = productRepository.findById(productId);
        if (product != null) {
            redisTemplate.opsForValue().set(cacheKey, product, Duration.ofMinutes(30));
        }
        return product;
    }

    // 写入：Write Behind
    public void updateProduct(Product product) {
        // 1. 更新缓存
        String cacheKey = "product:" + product.getId();
        redisTemplate.opsForValue().set(cacheKey, product);

        // 2. 异步更新数据库
        kafkaTemplate.send("product-updates", new ProductUpdateEvent(product));
    }
}
```

## 总结

缓存模式是缓存使用的核心问题。

**核心要点**：

| 模式 | 读 | 写 | 一致性 | 性能 |
|---|---|---|---|---|
| Cache Aside | 缓存→未命中→DB | DB→删缓存 | 弱一致 | 中 |
| Read Through | 缓存自动加载 | 缓存自动更新 | 弱一致 | 中 |
| Write Through | 缓存命中 | 缓存+DB | 强一致 | 差 |
| Write Behind | 缓存命中 | 缓存→异步DB | 弱一致 | 好 |

**选择建议**：

- 大多数场景：Cache Aside
- 本地缓存：Read Through
- 一致性要求高：Write Through
- 高并发写入：Write Behind
