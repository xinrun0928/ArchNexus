# 数据模型设计

你设计了一个订单系统，数据库表建好了，代码也写完了。上线后发现订单查询慢得要命，一查才发现：没有索引、没有分区、历史数据堆积……

问题出在哪？**数据模型设计时没考虑未来的查询模式**。

数据模型不是「把字段塞进表里」，而是**用数据结构支持业务场景**。

## Step 2: 确定查询模式

### 查询模式分析

```
电商订单系统查询模式：

高频查询（需要优化）：
1. 用户查看自己的订单列表（分页）
   - SELECT * FROM orders WHERE user_id = ? ORDER BY created_at DESC

2. 订单详情查询（包含商品、支付、物流）
   - SELECT * FROM orders WHERE id = ?
   - SELECT * FROM order_items WHERE order_id = ?

3. 后台查看订单（按状态、日期筛选）
   - SELECT * FROM orders WHERE status = ? AND created_at BETWEEN ? AND ?

4. 统计查询（订单量、销售额）
   - SELECT COUNT(*), SUM(total_amount) FROM orders WHERE ...

低频查询（可接受较慢）：
5. 用户搜索订单（按商品名）
   - SELECT * FROM orders o JOIN order_items oi ON o.id = oi.order_id WHERE ...

6. 商品销量统计
   - SELECT product_id, SUM(quantity) FROM order_items GROUP BY product_id
```

## Step 3: 设计数据模型

### 基础表结构

```sql
-- 用户表
CREATE TABLE users (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    username VARCHAR(50) NOT NULL UNIQUE,
    email VARCHAR(100) NOT NULL UNIQUE,
    phone VARCHAR(20),
    password_hash VARCHAR(255) NOT NULL,
    status TINYINT DEFAULT 1 COMMENT '1-正常 0-禁用',
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,

    INDEX idx_username (username),
    INDEX idx_email (email),
    INDEX idx_created_at (created_at)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;

-- 订单表
CREATE TABLE orders (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    order_no VARCHAR(32) NOT NULL UNIQUE COMMENT '订单号',
    user_id BIGINT NOT NULL,
    status VARCHAR(20) NOT NULL COMMENT 'PENDING/PAID/SHIPPED/DELIVERED/CANCELLED',
    total_amount DECIMAL(12, 2) NOT NULL COMMENT '订单总金额',
    discount_amount DECIMAL(12, 2) DEFAULT 0 COMMENT '优惠金额',
    freight_amount DECIMAL(10, 2) DEFAULT 0 COMMENT '运费',
    receiver_name VARCHAR(50) NOT NULL,
    receiver_phone VARCHAR(20) NOT NULL,
    receiver_address TEXT NOT NULL,
    remark VARCHAR(255),
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    paid_at TIMESTAMP NULL,
    shipped_at TIMESTAMP NULL,
    delivered_at TIMESTAMP NULL,

    INDEX idx_user_id (user_id),
    INDEX idx_status (status),
    INDEX idx_created_at (created_at),
    INDEX idx_user_status_created (user_id, status, created_at),
    INDEX idx_order_no (order_no)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;

-- 订单商品表
CREATE TABLE order_items (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    order_id BIGINT NOT NULL,
    product_id BIGINT NOT NULL,
    product_name VARCHAR(200) NOT NULL COMMENT '商品快照-名称',
    sku_code VARCHAR(50) NOT NULL COMMENT '商品 SKU',
    price DECIMAL(12, 2) NOT NULL COMMENT '单价快照',
    quantity INT NOT NULL,
    subtotal DECIMAL(12, 2) NOT NULL COMMENT '小计',
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,

    INDEX idx_order_id (order_id),
    INDEX idx_product_id (product_id)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;

-- 支付表
CREATE TABLE payments (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    payment_no VARCHAR(32) NOT NULL UNIQUE COMMENT '支付单号',
    order_id BIGINT NOT NULL,
    user_id BIGINT NOT NULL,
    amount DECIMAL(12, 2) NOT NULL,
    channel VARCHAR(20) NOT NULL COMMENT 'ALIPAY/WECHAT/CARD',
    status VARCHAR(20) NOT NULL COMMENT 'PENDING/SUCCESS/FAILED/REFUNDED',
    paid_at TIMESTAMP NULL,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,

    INDEX idx_order_id (order_id),
    INDEX idx_user_id (user_id),
    INDEX idx_status (status)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
```

### 索引设计原则

```java
// 索引设计决策

/*
索引设计原则：

1. 覆盖高频查询
   - 查询需要哪些字段，就建哪些索引
   - 查询条件 + 返回字段 = 索引列

2. 最左前缀匹配
   - 复合索引 (a, b, c) 可以支持 (a), (a,b), (a,b,c)
   - 不能支持 (b), (c), (b,c)

3. 避免过多索引
   - 每个索引都会影响写入性能
   - 索引不是越多越好

4. 考虑数据分布
   - 区分度低的字段不适合建索引（如性别）
   - 考虑字段基数（cardinality）
*/

// 示例：订单查询索引

// 查询 1: 用户订单列表
// SELECT * FROM orders WHERE user_id = ? ORDER BY created_at DESC
// 索引：idx_user_created (user_id, created_at DESC)

// 查询 2: 后台订单筛选
// SELECT * FROM orders WHERE status = ? AND created_at BETWEEN ? AND ?
// 索引：idx_status_created (status, created_at)

// 查询 3: 订单号查询
// SELECT * FROM orders WHERE order_no = ?
// 索引：order_no 唯一索引
```

### 分区设计

```sql
-- 按月分区（适合历史数据查询）

CREATE TABLE orders (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    order_no VARCHAR(32) NOT NULL,
    user_id BIGINT NOT NULL,
    status VARCHAR(20) NOT NULL,
    total_amount DECIMAL(12, 2) NOT NULL,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    -- ... 其他字段

    INDEX idx_user_id (user_id),
    INDEX idx_status (status),
    INDEX idx_created_at (created_at)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4
PARTITION BY RANGE (UNIX_TIMESTAMP(created_at)) (
    PARTITION p202401 VALUES LESS THAN (UNIX_TIMESTAMP('2024-02-01 00:00:00')),
    PARTITION p202402 VALUES LESS THAN (UNIX_TIMESTAMP('2024-03-01 00:00:00')),
    PARTITION p202403 VALUES LESS THAN (UNIX_TIMESTAMP('2024-04-01 00:00:00')),
    PARTITION p202404 VALUES LESS THAN (UNIX_TIMESTAMP('2024-05-01 00:00:00')),
    PARTITION p_future VALUES LESS THAN MAXVALUE
);
```

## Step 4: 验证和优化

### 慢查询分析

```java
// 验证查询性能

/*
慢查询分析步骤：

1. 开启慢查询日志
   slow_query_log = ON
   long_query_time = 1

2. 分析慢查询
   EXPLAIN SELECT * FROM orders WHERE user_id = ? ...

3. 检查执行计划
   - type: 查询类型（ALL 全表扫描，index 索引扫描）
   - key: 使用的索引
   - rows: 扫描行数
   - Extra: 额外信息（Using index, Using filesort 等）
*/

// 示例：EXPLAIN 分析
EXPLAIN SELECT o.*, u.username
FROM orders o
JOIN users u ON o.user_id = u.id
WHERE o.user_id = 12345
AND o.status = 'PAID'
ORDER BY o.created_at DESC
LIMIT 20;
```

### 常见问题及优化

```sql
-- 问题 1：深分页查询慢
-- 错误：
SELECT * FROM orders
WHERE user_id = 12345
ORDER BY created_at DESC
LIMIT 1000000, 20;  -- 扫描 1000020 行，只返回 20 行

-- 优化：使用游标分页
SELECT * FROM orders
WHERE user_id = 12345
AND created_at < '2024-01-01 00:00:00'
ORDER BY created_at DESC
LIMIT 20;


-- 问题 2：count(*) 慢
-- 优化：使用统计表或近似计数
SELECT COUNT(*) FROM orders WHERE user_id = 12345;  -- 慢

-- 优化：维护 count 字段
SELECT order_count FROM user_stats WHERE user_id = 12345;  -- 快


-- 问题 3：JOIN 太多
-- 优化：反范式化，提前冗余
ALTER TABLE orders ADD COLUMN user_username VARCHAR(50);
ALTER TABLE orders ADD COLUMN user_email VARCHAR(100);
-- 读取时不需要 JOIN 用户表
```

## 数据模型演进

### 读优化：增加冗余字段

```sql
-- 读取场景：订单列表需要显示用户名、商品名
-- 每次 JOIN 效率低，可以增加冗余字段

ALTER TABLE orders ADD COLUMN user_username VARCHAR(50) DEFAULT '';
ALTER TABLE order_items ADD COLUMN product_name VARCHAR(200) DEFAULT '';


-- 查询优化后
SELECT
    o.id,
    o.order_no,
    o.user_username,     -- 不需要 JOIN
    o.total_amount,
    o.status,
    GROUP_CONCAT(oi.product_name) AS products  -- 不需要 JOIN
FROM orders o
LEFT JOIN order_items oi ON o.id = oi.order_id
WHERE o.user_id = ?
GROUP BY o.id
ORDER BY o.created_at DESC
LIMIT 20;
```

-- 写入场景：访问日志量大，不需要实时入库

## 总结

数据模型设计是系统设计的核心环节。

**设计流程**：

1. **识别实体**：从业务场景抽象核心实体
2. **确定关系**：明确实体间的关系
3. **分析查询**：明确查询模式和性能要求
4. **设计表结构**：选择合适的数据类型和约束
5. **设计索引**：覆盖高频查询
6. **考虑分区**：处理大数据量

**核心原则**：

- 数据模型服务于查询模式
- 索引不是越多越好
- 适度的反范式化可以提升性能
- 考虑未来的扩展性

**常见错误**：

- ❌ 不分析查询就建表
- ❌ 建太多索引影响写入
- ❌ 深分页查询
- ❌ JOIN 太多表
