# 需求分析与边界确定

面试官问：「如何设计一个搜索建议系统？」

你开始画架构图、写代码，结果面试官打断你：「等等，你是设计给谁用的？是内部系统还是面向用户的？日活多少？」

你发现这些问题你都没想过。

需求分析，是系统设计最重要、也最容易被忽略的一步。

## 需求分析的本质

很多人以为需求分析就是「听清楚题目」。但真正的需求分析，是**在模糊的问题中找到清晰的设计边界**。

```
┌─────────────────────────────────────────────────────────────────┐
│                    需求分析的核心问题                            │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  不是「要做什么功能」                                            │
│                                                                  │
│  而是：                                                          │
│  ┌────────────────────────────────────────────────────────────┐ │
│  │  1. 目标用户是谁？                                          │ │
│  │  2. 使用场景是什么？                                        │ │
│  │  3. 规模有多大？                                           │ │
│  │  4. 性能要求是什么？                                        │ │
│  │  5. 有什么约束条件？                                        │ │
│  └────────────────────────────────────────────────────────────┘ │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

## 功能性需求 vs 非功能性需求

### 功能性需求（Feature Requirements）

功能性需求定义了**系统要做什么**。

```
搜索建议系统的功能需求：
- 用户输入时，实时显示搜索建议
- 支持前缀匹配（输入 "a"，显示 "apple", "amazon" 等）
- 每个建议词显示搜索热度
- 支持中英文混合
- 用户点击建议词后跳转到搜索结果页
```

### 非功能性需求（Non-Functional Requirements）

非功能性需求定义了**系统要做到什么程度**。

```
搜索建议系统的非功能性需求：
- 延迟：P99 < 50ms
- 可用性：99.9%
- 支持 100 万并发用户
- 数据新鲜度：建议词每分钟更新一次
- 数据量：1000 万个建议词
```

:::tip
非功能性需求往往比功能性需求更重要。一个跑得慢、动不动挂掉的系统，功能再多也没人用。
:::

## 需求澄清的框架

### 问题维度分析法

```
                            ┌─────────────┐
                            │   用户      │
                            │  Who?       │
                            └──────┬──────┘
                                   │
        ┌──────────────────────────┼──────────────────────────┐
        │                          │                          │
   ┌────▼────┐              ┌─────▼─────┐              ┌─────▼─────┐
   │   场景   │              │    规模    │              │   性能    │
   │  What?  │              │  How much? │              │   How?   │
   └─────────┘              └───────────┘              └───────────┘
```

### 具体澄清问题

拿到一个问题，按这个框架澄清：

```
1. 用户（Who）
   - 谁会使用这个系统？
   - 是内部用户还是外部用户？
   - 有不同的用户角色吗？

2. 场景（What）
   - 主要使用场景是什么？
   - 有什么特殊需求？
   - 需要支持哪些操作？

3. 规模（How much）
   - 用户规模多大？
   - 数据规模多大？
   - 增长预期是多少？

4. 性能（How）
   - 延迟要求多少？
   - 可用性要求几个 9？
   - 需要什么 SLA？
```

## 案例分析：设计一个 URL 缩短服务

### 初始问题

```
面试官：设计一个 URL 缩短服务（类似 bit.ly）
```

### 澄清问题

```
我应该澄清几个问题：

1. 用户和场景
   - 这是面向消费者的服务（如 bit.ly）还是企业内部使用？
   - 主要用途是什么？（营销链接、分享链接、内部链接）

2. 规模
   - 每天生成多少短链接？
   - 每天访问量多大？
   - 需要存储多久的历史数据？

3. 功能
   - 需要支持自定义短链接吗？
   - 需要统计功能吗？
   - 需要过期机制吗？

4. 性能
   - 访问延迟要求多少？
   - 可用性要求多少？
```

### 假设明确

```
基于澄清后的需求，我们假设：

用户和场景：
- 面向消费者的营销链接分享
- 用户输入长 URL，生成短链接分享到社交媒体

规模：
- 日活用户：100 万
- 每天新增短链接：100 万
- 每天访问量：1 亿
- 峰值 QPS：10 万

功能：
- 生成短链接
- 访问短链接（302 重定向）
- 查看链接统计（可选，优化时考虑）

性能：
- 生成接口 P99 < 100ms
- 访问接口 P99 < 50ms
- 可用性：99.99%
```

## 需求拆解方法

### 功能拆解

```
URL 缩短服务功能拆解：

┌──────────────────────────────────────────────────────────────┐
│                        URL 缩短服务                          │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  ┌─────────────────┐                                        │
│  │   核心功能      │                                        │
│  ├─────────────────┤                                        │
│  │ - 生成短链接    │  用户提供长 URL，返回短 URL            │
│  │ - 访问短链接    │  访问短链接，重定向到长 URL            │
│  └─────────────────┘                                        │
│                                                              │
│  ┌─────────────────┐                                        │
│  │   扩展功能      │                                        │
│  ├─────────────────┤                                        │
│  │ - 自定义别名    │  用户指定短链接后缀                    │
│  │ - 链接过期      │  设置有效期                          │
│  │ - 访问统计      │  记录访问量、来源等                   │
│  │ - 批量生成      │  一次生成多个链接                    │
│  └─────────────────┘                                        │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

### 数据拆解

```
数据需求拆解：

用户数据：
- 短链接元信息：short_code, original_url, created_at, expires_at
- 访问日志：short_code, timestamp, referer, ip, user_agent
- 聚合统计：short_code, click_count, last_accessed_at

数据量估算：
- 每天 100 万新增链接
- 每天 1 亿次访问
- 假设平均访问量每个链接 100 次
- 一年数据量：3.65 亿条访问日志
```

### 接口拆解

```
API 接口拆解：

┌──────────────────────────────────────────────────────────────┐
│                         API 设计                            │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  POST /api/v1/shorten                                       │
│  生成短链接                                                  │
│  Request:  { original_url, custom_alias?, expires_at? }   │
│  Response: { short_url, expires_at }                       │
│                                                              │
│  GET  /{short_code}                                         │
│  访问短链接（302 重定向）                                    │
│  Response: 302 Redirect to original_url                      │
│                                                              │
│  GET  /api/v1/shorten/{short_code}/stats (可选)            │
│  查看统计                                                    │
│  Response: { click_count, top_referrers, ... }              │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

## 边界条件识别

### 系统边界

```
URL 缩短服务系统边界：

┌──────────────────────────────────────────────────────────────┐
│                        系统边界                              │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  在系统内：                                                  │
│  ✓ 生成短链接                                                │
│  ✓ 存储 URL 映射关系                                         │
│  ✓ 访问重定向                                                │
│  ✓ 访问统计（可选）                                          │
│                                                              │
│  在系统外（不负责）：                                         │
│  ✗ URL 内容过滤（是否违规内容）                               │
│  ✗ URL 可访问性（目标服务器是否正常）                          │
│  ✗ 用户认证（可选）                                          │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

### 技术边界

```
技术边界（明确不做的）：

- 不做链接预览图生成（可以让客户端实现）
- 不做垃圾链接检测（可以后续扩展）
- 不做用户系统（当前版本无状态）
- 不做 API 限流（第一版先不做）
```

## 需求优先级

### MVP vs 完整版

```
MVP（最小可行产品）：
┌─────────────────────────────────────────┐
│  ✓ 生成短链接                          │
│  ✓ 访问重定向                          │
│  ○ 访问统计（优化时再做）               │
│  ○ 自定义别名（优化时再做）             │
│  ○ 链接过期（优化时再做）               │
└─────────────────────────────────────────┘

完整版：
┌─────────────────────────────────────────┐
│  ✓ 生成短链接                          │
│  ✓ 访问重定向                          │
│  ✓ 访问统计                            │
│  ✓ 自定义别名                          │
│  ✓ 链接过期                            │
│  ✓ 批量生成                            │
│  ✓ API 限流                            │
└─────────────────────────────────────────┘
```

## 总结

需求分析不是可有可无的前戏，而是整个系统设计的基础。

**核心要点**：

1. **分清功能和性能**：功能是做什么，性能是做多好
2. **澄清约束条件**：规模、延迟、可用性都是约束
3. **明确系统边界**：什么做，什么不做
4. **区分 MVP 和完整版**：先解决核心问题

**澄清问题清单**：

- 用户是谁？场景是什么？
- 规模多大？增长预期呢？
- 延迟要求多少？
- 可用性要求几个 9？
- 有什么约束条件？

**常见误区**：

- ❌ 不问清楚就开始画图
- ❌ 把所有功能都当核心功能
- ❌ 忽视非功能性需求
- ❌ 做了很多不需要的功能
