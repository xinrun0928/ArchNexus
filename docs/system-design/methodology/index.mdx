# 设计方法论

系统设计是一门艺术，但艺术也有方法可循。

设计方法论不是教你「背答案」，而是教你「思考问题的方式」。面试中，面试官想看到的不是你的知识储备，而是你的思考过程。

## 方法论的意义

```
面试 vs 实际工作

面试：
- 时间有限（45 分钟）
- 考察思考过程
- 需要展示沟通能力
- 没有标准答案

实际工作：
- 时间充裕
- 需要产出可维护的方案
- 需要团队评审
- 有标准答案（或接近标准答案）
```

### 方法论能帮你的

1. **不遗漏关键点**：系统地思考每个维度
2. **有效沟通**：有条理地表达设计思路
3. **权衡取舍**：理解方案之间的 trade-off
4. **举一反三**：一个方法解决多个问题

## 系统设计常见维度

### 1. 功能边界

```
核心问题：
- 系统需要支持哪些功能？
- 不需要支持哪些功能？
- MVP 是什么？完整版有哪些扩展？
```

### 2. 数据存储

```
核心问题：
- 需要存储什么数据？
- 数据量和增长预期是多少？
- 查询模式是什么？
- 需要什么一致性级别？
```

### 3. 扩展性

```
核心问题：
- 当前规模是多少？增长预期呢？
- 水平扩展还是垂直扩展？
- 瓶颈在哪？如何解决？
```

### 4. 可用性

```
核心问题：
- SLA 要求是什么？
- 单点故障有哪些？如何消除？
- 故障隔离怎么做？
- 降级策略是什么？
```

### 5. 性能

```
核心问题：
- 延迟要求是多少？
- 吞吐量要求是多少？
- 瓶颈在哪？如何优化？
```

### 6. 成本

```
核心问题：
- 基础设施成本多少？
- 开发维护成本多少？
- ROI 合理吗？
```

## 面试中的方法论

### 开场：需求澄清

```
面试官：设计一个 Twitter

你：好的，我想先澄清几个问题：

1. 功能范围
   - 核心功能是发推、关注、首页时间线吗？
   - 需要支持图片、视频吗？
   - 需要搜索功能吗？

2. 用户规模
   - 日活用户大概多少？
   - 每天发多少推文？
   - 关注数有上限吗？

3. 性能要求
   - 首页加载延迟要求多少？
   - 发推延迟要求多少？

4. 其他
   - 需要展示趋势话题吗？
   - 需要私信功能吗？

面试官：日活 1 亿，每天发推 5 亿条，其他你先按简化版考虑。
```

### 中场：方案设计

```
你：基于这些信息，我来分析一下：

1. 规模估算
   - 1 亿用户，每天 5 亿推文
   - 平均每个用户关注 200 人
   - 首页需要展示最新推文

2. 核心挑战
   - 海量数据存储
   - 高并发写入
   - 低延迟读取

3. 高层设计
   ...

4. 详细设计
   ...
```

### 收尾：权衡分析

```
你：方案总结一下：

优势：
- 写入 QPS 可以达到 10 万+
- 读取延迟 P99 < 100ms
- 存储成本可控

劣势：
- 维护成本高
- 迁移复杂度大
- 有一定延迟（最终一致）

替代方案：
- 可以用 Kafka 替代 fan-out 服务，提高吞吐量
- 可以加一层缓存，降低数据库压力
```

## 常见设计模式

### 读写分离

```
适合场景：
- 读多写少
- 可以接受最终一致

架构：
┌─────────────┐     ┌─────────────┐
│   Write     │────▶│  Primary DB │
└─────────────┘     └──────┬──────┘
                            │ 同步
              ┌─────────────┼─────────────┐
              ▼             ▼             ▼
         ┌────────┐   ┌────────┐   ┌────────┐
         │Replica │   │Replica │   │Replica │
         └────────┘   └────────┘   └────────┘
```

### 缓存优先

```
适合场景：
- 热点数据
- 读远多于写

架构：
┌─────────────┐     ┌─────────────┐
│    Cache    │◀───▶│   Database  │
└──────┬──────┘     └─────────────┘
       │
       ▼
┌─────────────┐
│   Client    │
└─────────────┘
```

### 异步处理

```
适合场景：
- 非实时性操作
- 可拆分的长任务

架构：
┌─────────────┐     ┌─────────────┐     ┌─────────────┐
│   Request   │────▶│    Queue   │────▶│   Worker    │
└─────────────┘     └─────────────┘     └─────────────┘
```

## 总结

方法论是系统设计的底层能力。

**核心要点**：

1. **先澄清再设计**：不要假设，理解真正的需求
2. **先高层再细节**：先画框架，再填细节
3. **权衡优于完美**：没有最优解，只有最适合的解
4. **表达清晰有条理**：好的设计也要好的表达

**面试 Tips**：

- 不要急于画图，先问清楚问题
- 边画边讲，让面试官跟上你的思路
- 主动提出 trade-off，展示权衡思维
- 不确定时说「不确定」，不要瞎猜
