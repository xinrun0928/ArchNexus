# 消息顺序性

你的系统用消息队列处理订单状态变更：创建 → 支付 → 发货 → 签收。

但用户反馈：明明已经支付了，系统却显示「待支付」。

这就是消息顺序性问题：**消息乱序导致的状态不一致**。

## 消息顺序性的三个层次

```
┌─────────────────────────────────────────────────────────────────┐
│                    消息顺序性的三个层次                            │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  ┌─────────────┐                                               │
│  │ 全局顺序    │                                               │
│  │  所有消息    │  全局有序（性能差，一般不用）                  │
│  │  按顺序消费  │                                               │
│  └─────────────┘                                               │
│                                                                  │
│  ┌─────────────┐                                               │
│  │ 分区顺序    │  同分区有序（Kafka 默认）                       │
│  │ 同分区      │                                               │
│  │ 按顺序消费  │                                               │
│  └─────────────┘                                               │
│                                                                  │
│  ┌─────────────┐                                               │
│  │ 局部顺序    │  同一 key 的消息有序                           │
│  │ 同一 key    │                                               │
│  │ 按顺序消费  │                                               │
│  └─────────────┘                                               │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

## 乱序的原因

### 生产端乱序

```
生产端乱序场景：

1. 并发发送：多个 Producer 同时发送
2. 重试导致：消息发送失败重试，后来居上
3. 异步发送：回调顺序不确定

┌─────────────────────────────────────────────────────────────────┐
│                    生产端乱序                                     │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  Producer ──▶ [消息A 发送中] ──▶ Broker A                     │
│  Producer ──▶ [消息B 发送中] ──▶ Broker B                     │
│  Producer ──▶ [消息A 重试] ──▶ Broker B  ←── B 先到！         │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

### Broker 乱序

```
Broker 乱序场景：

1. 分区分配：消息发送到不同分区
2. Leader 切换：Leader 切换导致顺序不确定
3. 多副本同步：Follower 同步顺序不同

┌─────────────────────────────────────────────────────────────────┐
│                    Broker 乱序                                     │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  Producer ──▶ [消息A] ──▶ Partition 0 ──▶ Consumer           │
│  Producer ──▶ [消息B] ──▶ Partition 1 ──▶ Consumer           │
│                                                                  │
│  Consumer 可能先收到 B 再收到 A ──▶ 乱序                     │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

### 消费端乱序

```
消费端乱序场景：

1. 并发消费：多个 Consumer 同时消费
2. 异步处理：消息处理异步化
3. 批量消费：一次拉取多条，处理顺序不确定

┌─────────────────────────────────────────────────────────────────┐
│                    消费端乱序                                     │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  Consumer ──▶ [拉取消息A, B, C]                               │
│  Consumer ──▶ [并发处理] ──▶ A 和 C 先完成                   │
│  Consumer ──▶ [处理结果] ──▶ C 先被消费                      │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

## Kafka 顺序保证

### Kafka 的顺序模型

```
Kafka 的分区顺序模型：

1. 分区内有序：同一分区内，消息按 offset 顺序消费
2. 分区间无序：不同分区的消息，顺序不确定
3. 消费者组内有序：同一分区同一时间只有一个 Consumer 消费

┌─────────────────────────────────────────────────────────────────┐
│                    Kafka 分区模型                                 │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  Topic                                                         │
│  ┌─────────────┐ ┌─────────────┐ ┌─────────────┐           │
│  │ Partition 0 │ │ Partition 1 │ │ Partition 2 │           │
│  │  offset 0   │ │  offset 0   │ │  offset 0   │           │
│  │  offset 1   │ │  offset 1   │ │  offset 1   │           │
│  │  offset 2   │ │  offset 2   │ │  offset 2   │           │
│  └──────┬──────┘ └──────┬──────┘ └──────┬──────┘           │
│         │                 │                 │                    │
│         ▼                 ▼                 ▼                    │
│    Consumer 0      Consumer 1      Consumer 2               │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

### Kafka 全局有序实现

```java
// Kafka 全局有序配置
@Service
public class KafkaOrderedProducer {
    @Autowired
    private KafkaTemplate<String, String> kafkaTemplate;

    // 全局有序：只用 1 个分区
    public void sendOrderedMessage(String message) {
        // 发到同一个分区
        kafkaTemplate.send("order-topic", "global", message);
    }
}

// 消费者配置
@Service
public class KafkaOrderedConsumer {
    @KafkaListener(
        topics = "order-topic",
        groupId = "order-group",
        containerProperties = {
            @Property(name = "max.poll.records", value = "1")  // 每次只拉一条
        }
    )
    public void consume(ConsumerRecord<String, String> record) {
        log.info("消费消息: offset={}, value={}", record.offset(), record.value());
        processMessage(record.value());
    }
}
```

### Kafka 单 Key 有序实现

```java
// Kafka 按 key 有序
@Service
public class KafkaKeyOrderedProducer {
    @Autowired
    private KafkaTemplate<String, String> kafkaTemplate;

    // 按订单 ID 有序
    public void sendOrderMessage(String orderId, OrderEvent event) {
        // 相同 orderId 的消息发送到同一个分区
        kafkaTemplate.send("order-topic", orderId, JSON.toJSONString(event));
    }
}

// 消费者配置
@Service
public class KafkaKeyOrderedConsumer {
    @KafkaListener(
        topics = "order-topic",
        groupId = "order-group",
        containerProperties = {
            @Property(name = "max.poll.records", value = "10"),
            @Property(name = "concurrency", value = "3")  // 并发数 = 分区数
        }
    )
    public void consume(ConsumerRecord<String, String> record) {
        String orderId = record.key();
        String value = record.value();
        log.info("消费消息: orderId={}, offset={}, value={}",
            orderId, record.offset(), value);
        processMessage(orderId, value);
    }
}
```

## RabbitMQ 顺序保证

### RabbitMQ 的顺序模型

```
RabbitMQ 的顺序模型：

1. 单队列有序：同一个队列内的消息按入队顺序消费
2. 多队列无序：不同队列的消息，顺序不确定
3. 消费者顺序：单消费者顺序处理

┌─────────────────────────────────────────────────────────────────┐
│                    RabbitMQ 队列模型                             │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  Exchange                                                      │
│     │                                                          │
│     ├──▶ Queue A ──▶ Consumer 1（顺序消费）                   │
│     │                                                          │
│     ├──▶ Queue B ──▶ Consumer 2（顺序消费）                   │
│     │                                                          │
│     └──▶ Queue C ──▶ Consumer 3（顺序消费）                   │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

### RabbitMQ 单队列有序实现

```java
// RabbitMQ 配置
@Configuration
public class RabbitOrderedConfig {
    @Bean
    public Queue orderQueue() {
        // 单队列，保证顺序
        return QueueBuilder.durable("order.queue")
            .build();
    }

    @Bean
    public DirectExchange orderExchange() {
        return new DirectExchange("order.exchange");
    }

    @Bean
    public Binding orderBinding() {
        return BindingBuilder.bind(orderQueue())
            .to(orderExchange())
            .with("order");  // 固定 routing key
    }
}

// 生产者：按订单 ID 路由
@Service
public class RabbitOrderedProducer {
    @Autowired
    private RabbitTemplate rabbitTemplate;

    public void sendOrderMessage(String orderId, OrderEvent event) {
        // 所有订单消息都发到同一个队列
        rabbitTemplate.convertAndSend("order.exchange", "order", event);
    }
}

// 消费者：单消费者顺序消费
@Service
public class RabbitOrderedConsumer {
    @RabbitListener(queues = "order.queue", concurrency = "1")
    public void consume(Message message) {
        OrderEvent event = JSON.parseObject(message.getBody(), OrderEvent.class);
        log.info("消费消息: orderId={}, type={}",
            event.getOrderId(), event.getType());
        processMessage(event);
    }
}
```

## 消费端顺序处理

### 乱序消费的处理策略

```java
// 策略一：队列排序
@Service
public class OrderedConsumer {
    @Autowired
    private ConcurrentHashMap<String, ConcurrentLinkedQueue<String>> pendingQueues;

    @KafkaListener(topics = "order-topic", groupId = "order-group")
    public void consume(ConsumerRecord<String, String> record) {
        String orderId = record.key();
        String event = record.value();

        // 1. 放入队列
        ConcurrentLinkedQueue<String> queue = pendingQueues.computeIfAbsent(
            orderId, k -> new ConcurrentLinkedQueue<>()
        );
        queue.add(event);

        // 2. 按顺序处理
        processQueueInOrder(orderId, queue);
    }

    private void processQueueInOrder(String orderId, ConcurrentLinkedQueue<String> queue) {
        while (true) {
            String event = queue.peek();
            if (event == null) break;

            // 检查前置条件
            OrderEvent orderEvent = JSON.parseObject(event, OrderEvent.class);
            if (canProcess(orderEvent)) {
                queue.poll();
                processMessage(orderEvent);
            } else {
                // 等待前置条件
                break;
            }
        }
    }
}

// 策略二：时间窗口排序
@Service
public class TimeWindowOrderedConsumer {
    private static final long WINDOW_MS = 1000;

    @KafkaListener(topics = "order-topic", groupId = "order-group")
    public void consume(ConsumerRecord<String, String> record) {
        String orderId = record.key();
        String event = record.value();

        // 放入缓冲区
        buffer(orderId, event);

        // 检查窗口是否过期
        checkAndProcessWindow(orderId);
    }

    private void checkAndProcessWindow(String orderId) {
        List<OrderEvent> events = getBufferedEvents(orderId);
        if (events.isEmpty()) return;

        // 检查最老的事件是否超过窗口
        OrderEvent oldest = events.get(0);
        if (System.currentTimeMillis() - oldest.getTimestamp() > WINDOW_MS) {
            // 窗口过期，按顺序处理
            events.stream()
                .sorted(Comparator.comparing(OrderEvent::getTimestamp))
                .forEach(this::processMessage);
            clearBuffer(orderId);
        }
    }
}
```

### 幂等 + 版本号处理乱序

```java
// 版本号处理乱序
@Data
@Entity
public class Order {
    private Long id;
    private String orderId;
    private Integer version;  // 版本号

    // 更新时检查版本号
    public void applyEvent(OrderEvent event) {
        if (event.getVersion() <= this.version) {
            // 版本号不增加，忽略旧消息
            log.warn("忽略旧消息: eventVersion={}, currentVersion={}",
                event.getVersion(), this.version);
            return;
        }

        // 更新状态
        this.version = event.getVersion();
        this.status = event.getStatus();
    }
}

// 消费者
@Service
public class VersionedConsumer {
    @Autowired
    private OrderRepository orderRepository;

    @KafkaListener(topics = "order-topic", groupId = "order-group")
    public void consume(ConsumerRecord<String, String> record) {
        OrderEvent event = JSON.parseObject(record.value(), OrderEvent.class);

        // 乐观锁处理
        Order order = orderRepository.findByOrderId(event.getOrderId());
        if (order.applyEvent(event)) {
            orderRepository.save(order);
        }
    }
}
```

## 总结

消息顺序性是业务正确性的关键。

**核心要点**：

| 层级 | 实现方式 | 性能 |
|---|---|---|
| 全局有序 | 单分区 + 单消费者 | 差 |
| 分区有序 | 按 key 路由到分区 | 中 |
| 局部有序 | 消费端队列排序 | 好 |

**Kafka 顺序保证**：

- 相同 key → 相同分区 → 有序
- 使用 `max.poll.records=1` + `concurrency=分区数`

**RabbitMQ 顺序保证**：

- 单队列 → 单消费者 → 有序
- 多队列 → 按 key 路由到不同队列

**消费端乱序处理**：

- 版本号机制
- 时间窗口排序
- 幂等处理
