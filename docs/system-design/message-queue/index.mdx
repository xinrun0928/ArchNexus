# 消息队列

电商大促零点秒杀，库存服务需要扣减库存，营销服务需要更新用户积分，物流服务需要创建发货单。如果每个服务都同步调用，库存服务一旦变慢，整个下单链路都会被拖垮。但如果把「更新积分」「创建物流」这些非核心链路异步化，让它们各自按自己的节奏消费，下单接口的响应时间可以从 500ms 降到 50ms。这背后依赖的正是消息队列。

消息队列（Message Queue）是分布式系统中实现异步通信的核心组件。它允许生产者和消费者通过「队列」这个中间层进行解耦：生产者只管发消息，不用关心谁来消费；消费者只管取消息，不用关心谁生产的。这种松耦合的架构，是应对高并发、构建高可用系统的关键。

## 核心价值

消息队列的价值主要体现在三个方面。

**异步处理**。同步调用意味着调用方必须等待被调用方返回才能继续。假设一个业务流程涉及 A、B、C 三个服务，其中 C 是非核心服务（发通知、发短信），等待 C 完成会白白浪费 100ms。但如果把 C 异步化，A 发完消息就可以返回，B 和 C 各自消费，整体响应时间从 300ms 降到 200ms。在高并发场景下，这种异步优势会被放大。

**系统解耦**。没有 MQ 时，如果营销服务挂了，可能导致下单服务也跟着失败——因为下单流程里嵌入了营销逻辑。引入 MQ 后，下单服务只负责发消息，营销服务自己订阅队列。如果营销服务挂了，消息会暂存在队列里，等恢复后继续处理，下单服务完全不感知。

**流量削峰**。瞬时流量高峰是电商、票务系统的常态。如果按峰值容量部署服务器，平时会浪费大量资源。MQ 的作用是把瞬时流量「蓄」起来，让后端服务按自己的节奏处理。比如秒杀活动，10 万QPS 的下单请求先全部落入队列，后端服务按 1000QPS 的速度慢慢消费，既不会打爆数据库，也不会让用户请求超时。

## 常见陷阱

MQ 用不好，轻则丢消息，重则系统故障。下面是三个最容易踩的坑。

**消息丢失**。MQ 的可靠性链路涉及三个环节：生产者发消息到 broker、broker 存储消息、消费者从 broker 拉取消息。其中任何一个环节出问题，都可能导致消息丢失。比如生产者没等 broker 确认就继续发，broker 宕机后消息就没了；消费者还没处理完就 ACK，服务重启后消息就丢了。保障消息可靠性需要：生产者确认机制（Confirm）、broker 持久化（刷盘 + 副本）、消费者手动 ACK。

**消息重复消费**。与消息丢失相反的问题是重复消费。消费者处理完业务后，还没来得及 ACK 就宕机了，MQ 会认为消息没消费成功，重新投递。对于幂等性不强的业务（比如扣库存），重复消费会导致库存被扣多次。这种问题的解法是业务上实现幂等，比如利用数据库唯一索引、Redis 防重Key、消息 ID 记录表。

**消息乱序**。有些业务场景对消息顺序有严格要求，比如订单流程必须是「创建 → 支付 → 发货」。但 MQ 的分区（Partition）和多消费者并发消费，都可能打乱消息顺序。如果使用 Kafka，同一个 Partition 内的消息是有序的，但如果需要全局有序，就需要把相关消息路由到同一个 Partition，这又会牺牲一部分性能。实际开发中，需要根据业务对顺序的敏感程度，选择合适的分区策略。

## 选型要点

主流消息队列有三剑客：Kafka、RabbitMQ、RocketMQ。选型时需要考虑几个维度。

**吞吐量**。Kafka 基于顺序写入优化，单机可达到百万级 TPS，适合日志收集、大数据实时计算等场景。RocketMQ 同样面向高吞吐，稳定性也不错。RabbitMQ 单机吞吐在万级，更适合低延迟的小消息场景。

**延迟**。RabbitMQ 的 Pigeon 微服务通讯延迟可到毫秒级，适合对实时性要求高的场景。Kafka 的延迟相对较高，一般在几毫秒到几十毫秒。

**功能丰富度**。RabbitMQ 支持多种协议（AMQP、STOMP、MQTT），路由策略灵活，适合复杂的业务场景。RocketMQ 支持事务消息、延迟消息，功能更贴近电商场景。Kafka 只有基本的发布订阅，更适合作为数据管道。

**可靠性**。三者都支持持久化和副本机制。Kafka 的ISR 副本同步机制，RocketMQ 的主从同步，RabbitMQ 的镜像队列，都能满足大多数可靠性需求。但事务消息目前只有 RocketMQ 和 Kafka 事务特性（Transactional outbox）原生支持。

没有完美的消息队列，只有适合业务场景的选择。如果团队规模小、业务简单，RabbitMQ 上手快；如果日活千万、流量峰值高，Kafka 更稳；如果做电商交易链路、需要事务消息，RocketMQ 更合适。
