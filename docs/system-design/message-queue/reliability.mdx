# 消息可靠性保证

你的系统用消息队列做异步处理，结果用户反馈：订单创建了，但没发货。

排查后发现：消息发送失败了，但业务代码没有回滚。

消息可靠性是消息队列最核心的问题：**消息不能丢，不能重复处理**。

## 生产端可靠性

### 问题分析

```
生产端可能丢消息的场景：

1. 网络抖动：消息没发送到 Broker
2. Broker 故障：消息没写入磁盘
3. 发送超时：客户端超时放弃

┌─────────────────────────────────────────────────────────────────┐
│                    生产端丢消息场景                               │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  Producer ──▶ [网络抖动] ──▶ Broker                           │
│                                                                  │
│  Producer ──▶ [Broker 宕机] ──▶ 消息丢失                       │
│                                                                  │
│  Producer ──▶ [超时] ──▶ 放弃 ──▶ 消息丢失                   │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

### 解决方案：同步发送 + 重试

```java
// Kafka 可靠发送
@Service
public class KafkaReliableProducer {
    @Autowired
    private KafkaTemplate<String, String> kafkaTemplate;

    public void sendMessage(String topic, String key, String value) {
        // 1. 设置acks=all，等待所有副本确认
        ProducerRecord<String, String> record = new ProducerRecord<>(topic, key, value);

        try {
            // 2. 同步发送
            Future<RecordMetadata> future = kafkaTemplate.send(record);
            RecordMetadata metadata = future.get(10, TimeUnit.SECONDS);

            log.info("发送成功: topic={}, partition={}, offset={}",
                metadata.topic(), metadata.partition(), metadata.offset());
        } catch (Exception e) {
            // 3. 发送失败，记录日志
            log.error("发送失败: topic={}, key={}, value={}", topic, key, value, e);

            // 4. 补偿处理
            compensateSend(topic, key, value);
        }
    }

    private void compensateSend(String topic, String key, String value) {
        // 补偿策略：
        // 1. 记录到本地数据库
        // 2. 定时重试
        // 3. 人工处理
    }
}
```

```java
// RabbitMQ 可靠发送
@Service
public class RabbitReliableProducer {
    @Autowired
    private RabbitTemplate rabbitTemplate;

    public void sendMessage(String exchange, String routingKey, Object message) {
        // 1. 开启事务模式
        rabbitTemplate.setChannelTransacted(true);

        try {
            rabbitTemplate.convertAndSend(exchange, routingKey, message);
            // 事务提交
        } catch (Exception e) {
            // 事务回滚
            log.error("发送失败", e);
            compensateSend(exchange, routingKey, message);
        }
    }

    // Confirm 模式（异步确认）
    public void sendWithConfirm(String exchange, String routingKey, Object message) {
        rabbitTemplate.setConfirmCallback((correlationData, ack, cause) -> {
            if (!ack) {
                log.error("消息发送失败: {}", cause);
                compensateSend(exchange, routingKey, message);
            }
        });
        rabbitTemplate.convertAndSend(exchange, routingKey, message);
    }
}
```

## Broker 可靠性

### 问题分析

```
Broker 可能丢消息的场景：

1. 异步刷盘：消息在内存中，机器宕机丢失
2. 副本同步失败：Leader 挂了，Follower 还没同步
3. 分区分配：消息只在 Leader 上

┌─────────────────────────────────────────────────────────────────┐
│                    Broker 丢消息场景                               │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  Producer ──▶ Leader ──▶ [宕机] ──▶ Follower 未同步           │
│                                                     消息丢失     │
│  Producer ──▶ Leader ──▶ [异步刷盘] ──▶ 内存中               │
│                                               宕机              │
│                                              消息丢失           │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

### 解决方案：持久化 + 多副本

```yaml
# Kafka Broker 配置
server.properties:
  # 持久化配置
  flush.messages: 1           # 每条消息都刷盘
  flush.ms: 1000             # 最多 1 秒刷盘

  # 副本配置
  replication.factor: 3        # 每个分区 3 个副本
  min.insync.replicas: 2      # 至少 2 个副本同步成功

  # Broker 配置
  unclean.leader.election.enable: false  # 不允许从非同步副本选举
  broker.rack: /rack1          # 机架感知
```

```yaml
# RocketMQ Broker 配置
brokerClusterName: DefaultCluster
brokerName: broker-a
brokerId: 0
deleteWhen: 04
fileReservedTime: 48
brokerRole: ASYNC_MASTER
flushDiskType: SYNC_FLUSH  # 同步刷盘
```

## 消费端可靠性

### 问题分析

```
消费端可能丢消息的场景：

1. 自动 ACK：消息还没处理完就确认了
2. 处理异常：消息处理失败，但没有重试
3. 程序崩溃：消息还没处理完

┌─────────────────────────────────────────────────────────────────┐
│                    消费端丢消息场景                               │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  Consumer ──▶ [处理中] ──▶ [ACK] ──▶ 崩溃                     │
│                               │                                  │
│                               └── 消息已 ACK，没处理完───▶ 丢失   │
│                                                                  │
│  Consumer ──▶ [异常] ──▶ [未处理] ──▶ 没有重试                 │
│                               │                                  │
│                               └── 消息未处理 ──▶ 丢失             │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

### 解决方案：手动 ACK + 重试机制

```java
// Kafka 手动提交
@Service
public class KafkaReliableConsumer {
    @Autowired
    private KafkaTemplate<String, String> kafkaTemplate;

    @KafkaListener(topics = "order-topic", groupId = "order-group")
    public void consume(ConsumerRecord<String, String> record, Acknowledgment ack) {
        try {
            String value = record.value();
            log.info("消费消息: key={}, offset={}", record.key(), record.offset());

            // 处理消息
            processOrder(value);

            // 处理成功，手动提交
            ack.acknowledge();
        } catch (Exception e) {
            log.error("处理失败", e);
            // 处理失败，不 ACK，消息会重试
            // 或者记录到死信队列
            handleFailedMessage(record);
        }
    }

    private void handleFailedMessage(ConsumerRecord<String, String> record) {
        // 1. 记录到数据库
        // 2. 发送死信队列
        // 3. 人工处理
    }
}
```

```java
// RabbitMQ 手动 ACK
@Service
public class RabbitReliableConsumer {
    @RabbitListener(queues = "order.queue")
    public void consume(Message message, Channel channel, @Header(AmqpHeaders.DELIVERY_TAG) long tag) {
        try {
            String body = new String(message.getBody());
            log.info("消费消息: {}", body);

            // 处理消息
            processOrder(body);

            // 处理成功，手动 ACK
            channel.basicAck(tag, false);
        } catch (Exception e) {
            log.error("处理失败", e);
            // 处理失败，NACK
            // requeue=false 表示不重试，进入死信队列
            channel.basicNack(tag, false, false);
        }
    }
}
```

## 消息幂等性

### 问题分析

```
消息重复的场景：

1. 生产者重试：发送失败重试，可能重复发送
2. 消费者重试：消费失败重试，消息重复
3. Broker 重发：Broker 重启，重复投递

幂等处理 = 同一个消息处理多次，结果一致
```

### 解决方案

```java
// 方案一：唯一键去重
@Service
public class IdempotentConsumer {
    @Autowired
    private OrderRepository orderRepository;

    public void processOrder(OrderMessage message) {
        // 1. 检查是否已处理
        if (orderRepository.existsByMessageId(message.getMessageId())) {
            log.warn("消息已处理: messageId={}", message.getMessageId());
            return;
        }

        // 2. 处理业务
        Order order = createOrder(message);

        // 3. 记录处理结果
        orderRepository.save(order);

        log.info("消息处理成功: messageId={}, orderId={}",
            message.getMessageId(), order.getId());
    }
}

// 方案二：Redis 去重
@Service
public class RedisIdempotentConsumer {
    @Autowired
    private RedisTemplate<String, String> redisTemplate;

    private static final Duration DEDUP_WINDOW = Duration.ofHours(24);

    public void processOrder(OrderMessage message) {
        String dedupKey = "dedup:order:" + message.getMessageId();

        // 1. Redis SETNX 去重
        Boolean isNew = redisTemplate.opsForValue().setIfAbsent(dedupKey, "1", DEDUP_WINDOW);

        if (!Boolean.TRUE.equals(isNew)) {
            log.warn("消息已处理: messageId={}", message.getMessageId());
            return;
        }

        // 2. 处理业务
        processOrder(message);
    }
}

// 方案三：数据库唯一约束
@Data
@Entity
@Table(name = "orders")
public class Order {
    @Id
    private Long id;

    @Column(unique = true)
    private String messageId;  // 消息 ID 唯一约束
}
```

## 总结

消息可靠性需要从生产端、Broker、消费端三个环节共同保证。

**核心要点**：

| 环节 | 关键措施 |
|---|---|
| 生产端 | acks=all、重试、幂等发送 |
| Broker | 多副本、同步刷盘、ISR |
| 消费端 | 手动 ACK、幂等处理、死信队列 |

**最佳实践**：

- 开启消息确认机制
- 设置合理的重试次数
- 做好幂等处理
- 监控消息处理延迟
