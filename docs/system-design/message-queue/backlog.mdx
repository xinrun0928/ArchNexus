# 消息积压处理

你的消息队列里堆积了 1000 万条未消费的消息，用户反馈订单状态不更新，客服电话被打爆。

消息积压是消息队列最常见的问题：**消息产生速度 > 消费速度**。

## 消息积压的原因

```
┌─────────────────────────────────────────────────────────────────┐
│                    消息积压的原因分析                               │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  生产端：                                                     │
│  ────────                                                     │
│  • 流量突增（秒杀、促销）                                      │
│  • 业务逻辑变化（新增消息类型）                                 │
│  • 外部系统故障（补偿消息大量涌入）                             │
│                                                                  │
│  消费端：                                                     │
│  ────────                                                     │
│  • 消费者宕机                                                │
│  • 消费逻辑慢（数据库慢查询、外部接口超时）                     │
│  • 消费者数量减少（动态缩容）                                  │
│  • 代码 bug（消费失败未重试、死循环）                          │
│                                                                  │
│  消息队列：                                                   │
│  ─────────                                                     │
│  • Broker 故障                                                │
│  • 分区分配不合理                                             │
│  • 网络抖动                                                   │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

### 跳过积压消息

```
谨慎使用：可能导致消息丢失

1. 跳过旧消息，从最新位置消费
2. 记录跳过位置，人工处理
3. 告警通知
```

```java
// 跳过旧消息
@Service
public class SkipOldMessageConsumer {
    @KafkaListener(topics = "order-topic", groupId = "order-group")
    public void consume(ConsumerRecord<String, String> record) {
        // 跳过积压消息（超过 1 小时的）
        if (record.timestamp() < System.currentTimeMillis() - 3600_000) {
            log.warn("跳过旧消息: offset={}, timestamp={}", record.offset(), record.timestamp());
            // 记录跳过位置
            skipLog.record(record.topic(), record.partition(), record.offset());
            return;
        }

        // 正常处理
        processMessage(record.value());
    }
}
```

### 暂停消费，保护系统

```java
// 暂停消费
@Service
public class PauseConsumer {
    @Autowired
    private KafkaListenerEndpointRegistry registry;

    public void pauseConsumer() {
        // 暂停所有消费者
        registry.getListenerContainers().forEach(container -> {
            container.stop();
            log.info("暂停消费者: {}", container);
        });
    }

    public void resumeConsumer() {
        // 恢复消费者
        registry.getListenerContainers().forEach(container -> {
            container.start();
            log.info("恢复消费者: {}", container);
        });
    }
}
```

## 消费端优化

### 优化消费逻辑

```
问题：消费逻辑慢
原因：数据库慢查询、外部接口超时、复杂计算

解决方案：
1. 异步处理
2. 批量消费
3. 并行处理
4. 预处理
```

```java
// 优化一：异步处理
@Service
public class AsyncConsumer {
    @Autowired
    private ExecutorService executor;

    @KafkaListener(topics = "order-topic", groupId = "order-group")
    public void consume(ConsumerRecord<String, String> record) {
        // 快速响应，不阻塞
        executor.submit(() -> processMessage(record.value()));
    }
}

// 优化二：批量消费
@Service
public class BatchConsumer {
    @KafkaListener(
        topics = "order-topic",
        groupId = "order-group",
        containerProperties = {
            @Property(name = "max.poll.records", value = "500"),
            @Property(name = "fetch.min.bytes", value = "102400"),
            @Property(name = "fetch.max.wait.ms", value = "500")
        }
    )
    public void consume(List<ConsumerRecord<String, String>> records) {
        log.info("批量消费: {} 条", records.size());

        // 批量处理
        List<OrderEvent> events = records.stream()
            .map(r -> JSON.parseObject(r.value(), OrderEvent.class))
            .collect(Collectors.toList());

        // 批量入库
        orderService.batchSave(events);
    }
}

// 优化三：并行处理
@Service
public class ParallelConsumer {
    @Autowired
    private ForkJoinPool forkJoinPool;

    @KafkaListener(topics = "order-topic", groupId = "order-group")
    public void consume(ConsumerRecord<String, String> record) {
        forkJoinPool.submit(() -> {
            // 分区并行
            processMessage(record.value());
        });
    }
}
```

### 数据库批量操作

```java
// 批量插入优化
@Service
public class BatchDbConsumer {
    @Autowired
    private JdbcTemplate jdbcTemplate;

    public void batchProcess(List<OrderEvent> events) {
        String sql = "INSERT INTO order_events (order_id, type, payload, created_at) VALUES (?, ?, ?, ?)";

        // 使用批量插入
        jdbcTemplate.batchUpdate(sql, events, events.size(),
            (ps, event) -> {
                ps.setString(1, event.getOrderId());
                ps.setString(2, event.getType());
                ps.setString(3, JSON.toJSONString(event));
                ps.setTimestamp(4, new Timestamp(event.getTimestamp()));
            });

        log.info("批量入库完成: {} 条", events.size());
    }
}
```

## 生产端控制

### 限流

```java
// 生产端限流
@Service
public class RateLimitedProducer {
    @Autowired
    private KafkaTemplate<String, String> kafkaTemplate;
    private final RateLimiter rateLimiter = RateLimiter.create(10000);  // 1万 QPS

    public void sendMessage(String topic, String key, String value) {
        // 限流
        if (!rateLimiter.tryAcquire()) {
            log.warn("限流丢弃消息: key={}", key);
            return;
        }

        kafkaTemplate.send(topic, key, value);
    }
}
```

## 积压监控与告警

### 积压监控指标

```java
// 积压监控
@Service
public class BacklogMonitor {
    @Autowired
    private KafkaTemplate<String, String> kafkaTemplate;

    @Scheduled(fixedRate = 60000)  // 每分钟检查
    public void monitorBacklog() {
        // 1. 获取积压量
        Map<TopicPartition, Long> lag = getConsumerLag();

        for (Map.Entry<TopicPartition, Long> entry : lag.entrySet()) {
            String topic = entry.getKey().topic();
            int partition = entry.getKey().partition();
            long lagValue = entry.getValue();

            // 2. 告警阈值
            if (lagValue > 10000) {
                sendAlert("积压告警",
                    String.format("Topic: %s, Partition: %d, Lag: %d", topic, partition, lagValue));
            }
        }
    }

    private Map<TopicPartition, Long> getConsumerLag() {
        // 获取消费 lag
        // lag = endOffset - currentOffset
        // 使用 AdminClient 查询
        return Collections.emptyMap();
    }
}
```

## 预防措施

### 容量规划

```
预防胜于治疗：

1. 预估峰值流量
   - 历史数据
   - 业务增长预期
   - 促销活动

2. 评估消费能力
   - 单消费者 QPS
   - 消费者数量
   - 分区数量

3. 设置告警
   - 积压量阈值
   - 消费延迟
   - 错误率
```

### 熔断降级

```java
// 消费端熔断
@Service
public class CircuitBreakerConsumer {
    @Autowired
    private CircuitBreaker circuitBreaker;

    @KafkaListener(topics = "order-topic", groupId = "order-group")
    public void consume(ConsumerRecord<String, String> record) {
        try {
            circuitBreaker.execute(() -> processMessage(record.value()));
        } catch (Exception e) {
            // 熔断时，将消息放入重试队列
            retryQueue.add(record.value());
            log.warn("消费熔断，消息进入重试队列");
        }
    }
}
```

## 总结

消息积压是消息队列的常见问题，需要快速止血 + 长期优化。

**快速止血**：

- 扩容消费者
- 跳过积压消息（谨慎）
- 暂停消费，保护系统

**消费优化**：

- 批量消费
- 异步处理
- 数据库批量操作

**预防措施**：

- 容量规划
- 监控告警
- 熔断降级
