# 数据库扩展策略

你的电商系统订单量从 100 万增长到 10 亿，数据库从单表变成了几百 G。查询变慢，连接耗尽，备份都做不完。

这就是数据库扩展的挑战：**数据量和并发增长时，数据库如何扩展？**

## 策略一：读写分离

### 原理

```
┌─────────────────────────────────────────────────────────────────┐
│                    读写分离架构                                    │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  写入 ──▶ Primary DB                                           │
│           │                                                     │
│           ▼                                                     │
│  ┌─────────────┐                                                │
│  │  主库      │                                                │
│  │  接收写入  │                                                │
│  └──────┬─────┘                                                │
│         │ binlog 同步                                            │
│  ┌──────┴─────┐                                                │
│  │            │                                                 │
│  ▼            ▼                                                 │
│ ┌────┐      ┌────┐                                            │
│ │从库│      │从库│                                            │
│ └────┘      └────┘                                            │
│  ◀──────── 读取 ─────────                                       │
│                                                                  │
│  效果：读性能线性提升，写性能不变                               │
└─────────────────────────────────────────────────────────────────┘
```

### 实现

```yaml
# MySQL 主从配置
# 主库 my.cnf
[mysqld]
server-id = 1
log-bin = mysql-bin
binlog_format = ROW
sync_binlog = 1
innodb_flush_log_at_trx_commit = 1

# 从库 my.cnf
[mysqld]
server-id = 2
relay-log = relay-bin
read_only = 1
replicate-do-db = app_db
```

```java
// 应用层读写分离
@Configuration
public class DataSourceConfig {
    @Bean
    public DataSource dataSource() {
        RoutingDataSource routingDataSource = new RoutingDataSource();
        routingDataSource.setTargetDataSources(createDataSourceMap());
        routingDataSource.setDefaultTargetDataSource(masterDataSource);
        return routingDataSource;
    }

    private Map<Object, Object> createDataSourceMap() {
        Map<Object, Object> map = new HashMap<>();
        map.put("master", masterDataSource);
        map.put("slave1", slaveDataSource1);
        map.put("slave2", slaveDataSource2);
        return map;
    }
}

// 注解方式路由
@Target(ElementType.METHOD)
@Retention(RetentionPolicy.RUNTIME)
public @interface ReadOnly {}

@Component
public class ReadWriteInterceptor {
    @Around("@annotation(readOnly)")
    public Object routeToSlave(ProceedingJoinPoint joinPoint, ReadOnly readOnly) {
        if (readOnly != null) {
            RoutingDataSource.setReadOnly();
        }
        try {
            return joinPoint.proceed();
        } finally {
            RoutingDataSource.clear();
        }
    }
}

// 使用
@Service
public class UserService {
    @Transactional(readOnly = true)  // 自动路由到从库
    public User findById(Long id) {
        return userRepository.findById(id);
    }

    @Transactional  // 路由到主库
    public User create(User user) {
        return userRepository.save(user);
    }
}
```

### 局限性

```
读写分离的局限性：

1. 写入瓶颈仍然存在
   - 所有写入都到主库
   - 主库仍然单点

2. 主从延迟
   - 写入后立即读取可能读不到
   - 需要延迟感知

3. 复制风暴
   - 大查询会拖慢从库
   - 影响其他查询
```

## 策略二：垂直分库

### 原理

```
┌─────────────────────────────────────────────────────────────────┐
│                    垂直分库架构                                    │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  按业务拆分到不同数据库实例：                                   │
│                                                                  │
│  ┌─────────────┐  ┌─────────────┐  ┌─────────────┐           │
│  │ 用户库      │  │ 订单库      │  │ 商品库      │           │
│  │ user_db    │  │ order_db   │  │ product_db│           │
│  └─────────────┘  └─────────────┘  └─────────────┘           │
│                                                                  │
│  优势：业务隔离、资源独立                                      │
└─────────────────────────────────────────────────────────────────┘
```

### 实现

```java
// 多数据源配置
@Configuration
public class MultiDataSourceConfig {
    @Bean
    @ConfigurationProperties(prefix = "spring.datasource.user")
    public DataSource userDataSource() {
        return DataSourceBuilder.create().build();
    }

    @Bean
    @ConfigurationProperties(prefix = "spring.datasource.order")
    public DataSource orderDataSource() {
        return DataSourceBuilder.create().build();
    }

    @Bean
    @ConfigurationProperties(prefix = "spring.datasource.product")
    public DataSource productDataSource() {
        return DataSourceBuilder.create().build();
    }
}

// 使用时指定数据源
@Service
public class UserService {
    @Autowired
    @Qualifier("userDataSource")
    private DataSource userDataSource;
}

// 或者使用包扫描隔离
// 每个模块用自己的 DataSource
```

### 优势与挑战

```
垂直分库优势：
- 业务隔离，故障不扩散
- 资源独立，可独立优化
- 数据量减少，查询更快

垂直分库挑战：
- 跨库查询困难
- 事务无法跨库
- 关联查询需要应用层处理
```

## 策略三：水平分表

### 原理

```
┌─────────────────────────────────────────────────────────────────┐
│                    水平分表架构                                    │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  按分片键拆分数据到多个表/库：                                   │
│                                                                  │
│  t_order (按 user_id 分 4 张表)                                 │
│                                                                  │
│  ┌─────────┐  ┌─────────┐  ┌─────────┐  ┌─────────┐           │
│  │t_order_0│  │t_order_1│  │t_order_2│  │t_order_3│           │
│  │user_id%4│  │user_id%4│  │user_id%4│  │user_id%4│           │
│  │= 0      │  │= 1      │  │= 2      │  │= 3      │           │
│  └─────────┘  └─────────┘  └─────────┘  └─────────┘           │
│                                                                  │
│  效果：数据量线性分散                                            │
└─────────────────────────────────────────────────────────────────┘
```

### 分片策略

```java
// 哈希分片：数据均匀分布
public class HashShardingAlgorithm implements ShardingAlgorithm {
    @Override
    public String doSharding(Collection<String> availableTargetNames, ShardingValue shardingValue) {
        long id = Long.parseLong(shardingValue.getValue().toString());
        int index = (int) (id % availableTargetNames.size());
        return availableTargetNames.stream()
            .sorted()
            .skip(index)
            .findFirst()
            .orElse(availableTargetNames.iterator().next());
    }
}

// 范围分片：按时间或 ID 范围
public class RangeShardingAlgorithm implements ShardingAlgorithm {
    @Override
    public String doSharding(Collection<String> availableTargetNames, ShardingValue shardingValue) {
        long id = Long.parseLong(shardingValue.getValue().toString());
        if (id < 1000000) return "ds_0.t_order_0";
        if (id < 5000000) return "ds_1.t_order_1";
        if (id < 10000000) return "ds_2.t_order_2";
        return "ds_3.t_order_3";
    }
}
```

### 分片后的问题

```
分片后面临的挑战：

1. 跨分片查询
   - 全局查询（如统计、排序）需要多节点
   - 解决方案：异构索引、搜索库

2. 分片键选择
   - 查询必须带分片键
   - 变更代价大

3. 数据迁移
   - 扩缩容需要迁移数据
   - 双写 + 灰度切换

4. 分布式事务
   - 跨库事务无法本地处理
   - 解决方案：分布式事务中间件
```

## 总结

数据库扩展是随着业务增长逐步演进的过程。

**核心要点**：

- 读写分离：提升读性能
- 垂直分库：业务隔离
- 水平分表：数据量分散
- NewSQL：分布式原生方案

**演进建议**：

- 先优化单体，再考虑扩展
- 按需演进，不要过度设计
- 分片键选择要慎重
- 预留扩容能力
