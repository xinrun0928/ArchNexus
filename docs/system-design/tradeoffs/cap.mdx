# CAP 权衡

分布式系统领域有一个著名的定理：**CAP 定理**。它告诉你一个分布式系统不可能同时满足以下三个特性：

- **C**onsistency（一致性）
- **A**vailability（可用性）
- **P**artition Tolerance（分区容错）

你可能觉得这很简单：既然三个里只能选两个，那我选 CA 不就行了？

问题是：**在分布式系统中，分区是必然发生的**。网络会抖动、机器会宕机、数据会丢失——这些都是分区。既然分区一定会发生，那么实际上你只能在「一致性」和「可用性」之间选择。

## CAP 定理的定义

```
┌─────────────────────────────────────────────────────────────────┐
│                         CAP 定理                                 │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│                    Consistency (一致性)                          │
│                         ▲                                        │
│                        ╱ ╲                                       │
│                       ╱   ╲                                     │
│                      ╱     ╲                                    │
│                     ╱       ╲                                   │
│                    ╱         ╲                                  │
│                   ╱           ╲                                 │
│                  ╱    P-A     ╲                                │
│                 ╱               ╲                               │
│                ╱                 ╲                              │
│               ╱─────────────────────╲                           │
│              Consistency    Availability                         │
│                                                                  │
│  在发生分区（P）时，必须在一致性（C）和可用性（A）之间选择      │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

### 三选其二

| 组合 | 说明 | 代表系统 |
|---|---|---|
| CA | 不考虑分区，一致性和可用性都保证 | 单机数据库、传统关系型数据库 |
| CP | 分区时放弃可用性，保证一致性 | ZooKeeper、HBase、MongoDB（特定配置） |
| AP | 分区时放弃一致性，保证可用性 | Cassandra、DynamoDB、CouchDB |

:::warning
**关键误解**：CAP 不是「三选二」，而是「分区时二选一」。在系统没有发生分区时，C 和 A 可以同时满足。
:::

## 一致性 vs 可用性的权衡

### CP 系统：强一致优先

```
CP 系统特点：
- 分区发生时，返回错误，拒绝服务
- 保证数据在所有副本间一致
- 适合对数据一致性要求极高的场景

场景举例：
- 金融交易系统
- 库存扣减系统
- 分布式锁服务
```

```java
// CP 系统行为示例
public class ZookeeperLock implements DistributedLock {
    // ZooKeeper 保证强一致，分区时拒绝请求
    public boolean acquire(String lockPath, long timeout) {
        try {
            zooKeeper.create(lockPath, data, ZOO_OPEN_ACL_UNSAFE,
                CreateMode.EPHEMERAL);
            return true;
        } catch (Exception e) {
            // 分区时抛出异常，不返回过期的锁
            throw new LockAcquisitionException(e);
        }
    }
}
```

### AP 系统：高可用优先

```
AP 系统特点：
- 分区发生时，继续提供服务
- 各分区独立写入，修复后达成一致
- 适合对可用性要求极高的场景

场景举例：
- 社交媒体内容发布
- 新闻资讯阅读
- 电商商品展示
```

```java
// AP 系统行为示例
public class CassandraWrite {
    // Cassandra 保证高可用，写入本地节点即成功
    public void write(String key, String value) {
        // 写入本地节点，立即返回
        // 其他副本异步同步
        session.execute(
            QueryBuilder.insertInto("users")
                .value("key", key)
                .value("value", value)
                .using(ConsistencyLevel.ONE)  // 写入本地即可
        );
    }
}
```

## BASE 理论：实际系统的妥协

CAP 定理是理论，实际系统用的是 **BASE** 理论：

- **B**asically **A**vailable（基本可用）
- **S**oft state（软状态）
- **E**ventually consistent（最终一致）

```
BASE vs ACID

ACID（传统数据库）：
- Atomicity（原子性）
- Consistency（一致性）
- Isolation（隔离性）
- Durability（持久性）
- 要求强一致

BASE（分布式系统）：
- Basic Available（基本可用）
- Soft state（软状态）
- Eventually consistent（最终一致）
- 接受最终一致
```

### 最终一致性的类型

| 类型 | 说明 | 适用场景 |
|---|---|---|
| 因果一致性 | 有因果关系的操作顺序一致 | 社交评论 |
| 读己之所写 | 自己写入的立即可见 | 用户自己查看数据 |
| 会话一致性 | 单会话内顺序一致 | 电商购物车 |
| 单调读一致 | 不会读到更旧的数据 | 任何场景 |
| 单调写一致 | 写入顺序被保留 | 日志、消息 |

## 实际系统中的选择

### 电商库存系统：CP

```
需求分析：
- 库存不能超卖（一致性）
- 但也不能完全不可用（可用性）

解决方案：牺牲部分可用性，保证强一致

实现方式：
- 使用分布式锁或乐观锁
- 库存服务独立部署，多副本保证高可用
- 降级时返回「库存不足」，而不是返回错误的库存数
```

### 社交媒体点赞：AP

```
需求分析：
- 点赞必须高可用（用户随时可能点赞）
- 短暂的不一致可以接受（稍后同步）

解决方案：保证高可用，接受最终一致

实现方式：
- 点赞写入本地缓存和消息队列
- 异步聚合到数据库
- 前端显示「点赞数 +1」，后端异步更新
```

## 常见误解

### 误解一：CAP 是三选二

```
错误理解：
CAP 意味着系统只能选择三个特性中的两个。

正确理解：
CAP 描述的是系统在发生分区时的行为。在没有分区时，C 和 A 可以同时满足。
系统必须容忍网络分区（P），所以在分区时必须在 C 和 A 之间选择。
```

### 误解二：CP 系统完全不可用

```
错误理解：
CP 系统发生分区时完全不可用。

正确理解：
CP 系统在分区时，只影响涉及分区节点的操作。
正常分区的节点仍然可以正常服务。
```

### 误解三：一致性和性能冲突

```
错误理解：
要一致就要牺牲性能。

正确理解：
一致性和性能不冲突。
可以通过：
- 读写分离（读一致性要求低）
- 本地缓存（读性能高）
- 异步复制（写入性能高）
来平衡一致性和性能。
```

## 实践建议

### 如何选择

```
选择 CP 的场景：
- 数据一致性是核心诉求
- 业务可以接受短暂的不可用
- 强一致是监管或合规要求

选择 AP 的场景：
- 可用性是核心诉求
- 短暂的不一致可以接受
- 用户体验优先于数据精确性
```

### 降级策略

```
在 CP 和 AP 之间灵活切换：

正常情况：CP（强一致）
├── 检测到分区
├── 切换到 AP 模式（高可用）
├── 分区恢复后
└── 同步数据，切换回 CP 模式

示例：
- 金融交易：正常 CP，降级时拒绝交易
- 社交媒体：正常 AP，降级时显示缓存数据
```

## 总结

CAP 定理是分布式系统的理论基础，它告诉我们：**在分布式系统中，一致性和可用性不可兼得**。

**核心要点**：

- 分区必然发生，必须在 C 和 A 之间选择
- CP 系统保证强一致，牺牲可用性
- AP 系统保证高可用，接受最终一致
- BASE 理论是实际系统的妥协方案

**选择建议**：

- 金融、库存、订单：选 CP
- 社交、媒体、内容：选 AP
- 核心系统：CP + 降级方案
- 边缘系统：AP + 补偿机制
