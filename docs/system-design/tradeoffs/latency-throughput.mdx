# 延迟 vs 吞吐量

你的系统延迟 50ms，支持 1 万 QPS。产品经理说：「能不能把延迟降到 10ms？」

你优化了 GC、减少了锁、缓存命中率提到 99%——延迟确实降到了 10ms。但 QPS 变成了 5000。

这就是延迟和吞吐量的矛盾：**不是总能找到两全其美的方案**。

## 基本概念

### 延迟（Latency）

延迟是**完成一个操作所需的时间**。

```
延迟示例：
- 数据库查询：5ms
- 缓存查询：0.5ms
- HTTP 请求：50ms
- 磁盘读取：10ms

特点：
- 单位是时间（毫秒、秒）
- 关注单个请求的响应时间
- 影响用户体验
```

### 吞吐量（Throughput）

吞吐量是**单位时间内完成的操作数**。

```
吞吐量示例：
- Web 服务器：10 万 QPS
- 数据库写入：5 万 TPS
- 消息队列：100 万消息/秒

特点：
- 单位是操作数/时间（QPS、TPS）
- 关注系统的处理能力
- 影响系统容量规划
```

### 关系

```
┌─────────────────────────────────────────────────────────────────┐
│                    延迟 vs 吞吐量关系                           │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  吞吐量 = 1 / 延迟 × 并发数                                     │
│                                                                  │
│  场景 1：单线程                                                  │
│  延迟 10ms，吞吐量 = 100 QPS                                     │
│                                                                  │
│  场景 2：100 并发                                                │
│  延迟 10ms，吞吐量 = 10,000 QPS                                   │
│                                                                  │
│  场景 3：延迟变成 100ms                                          │
│  延迟 100ms，吞吐量 = 1,000 QPS（相同并发下）                    │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

## 延迟的类型

### 平均延迟 vs 百分位延迟

```java
// 延迟统计
/*
Latency Distribution:
  平均延迟：50ms
  P50：30ms    （50% 请求在 30ms 内完成）
  P90：80ms    （90% 请求在 80ms 内完成）
  P99：200ms   （99% 请求在 200ms 内完成）
  P999：500ms  （99.9% 请求在 500ms 内完成）

用户感知：
- P50：普通用户的体验
- P99：大多数用户的体验
- P999：少数用户的体验，但可能遇到极端情况
*/

// 延迟监控示例
public class LatencyMonitor {
    private final AtomicLong count = new AtomicLong(0);
    private final AtomicLong totalLatency = new AtomicLong(0);
    private final LongAdder[] latencyBuckets = createBuckets();

    public void record(long latencyMs) {
        count.incrementAndGet();
        totalLatency.addAndGet(latencyMs);
        latencyBuckets[(int) Math.min(latencyMs, 999)].increment();
    }

    public LatencyMetrics getMetrics() {
        long count = this.count.get();
        return new LatencyMetrics(
            totalLatency.get() / count,  // 平均延迟
            getPercentile(50),            // P50
            getPercentile(90),            // P90
            getPercentile(99),            // P99
            getPercentile(99.9)           // P999
        );
    }
}
```

### 串行延迟 vs 并行延迟

```
串行操作：
  A ──────▶ B ──────▶ C ──────▶
  10ms      20ms      30ms
  总延迟 = 10 + 20 + 30 = 60ms

并行操作：
  A ──────▶
               │
  B ──────▶────┤
               │
  C ──────▶────┘
  总延迟 = max(10, 20, 30) = 30ms
```

## 优化延迟

### 瓶颈识别

```java
// 延迟瓶颈分析
/*
分析步骤：
1. 追踪请求全链路
2. 识别耗时最长的步骤
3. 针对瓶颈优化

常见瓶颈：
- 数据库查询（没有索引、全表扫描）
- 远程调用（同步阻塞）
- 锁等待（热点竞争）
- GC 停顿（对象分配过多）
*/

// 使用链路追踪识别瓶颈
public class RequestTracer {
    public void trace(String requestId, String operation) {
        Span span = tracer.startSpan(operation);
        try {
            // 执行操作
            doOperation();
        } finally {
            span.finish();
        }
    }
}
```

### 常见优化手段

```java
// 1. 缓存优化：减少数据库访问
public class CacheExample {
    private final Cache<String, User> cache = Caffeine.newBuilder()
        .maximumSize(10000)
        .expireAfterWrite(10, TimeUnit.MINUTES)
        .build();

    public User getUser(String userId) {
        return cache.get(userId, k -> database.findUser(k));
    }
}

// 2. 异步化：减少等待
public class AsyncExample {
    public CompletableFuture<User> getUserAsync(String userId) {
        return CompletableFuture
            .supplyAsync(() -> database.findUser(userId));
    }

    // 并行获取多个资源
    public CompletableFuture<UserDetail> getUserDetail(String userId) {
        CompletableFuture<User> userFuture = getUserAsync(userId);
        CompletableFuture<Order> orderFuture = getOrderAsync(userId);
        CompletableFuture<Payment> paymentFuture = getPaymentAsync(userId);

        return CompletableFuture.allOf(userFuture, orderFuture, paymentFuture)
            .thenApply(v -> new UserDetail(
                userFuture.join(),
                orderFuture.join(),
                paymentFuture.join()
            ));
    }
}

// 3. 索引优化：减少查询时间
public class IndexExample {
    // 创建合适的索引
    @Query("CREATE INDEX idx_user_status_created ON orders(user_id, status, created_at)")
    public void createIndex() { }

    // 避免全表扫描
    public List<Order> findUserOrders(Long userId, String status) {
        // 使用索引：user_id, status, created_at
        return jdbcTemplate.query(
            "SELECT * FROM orders WHERE user_id = ? AND status = ? ORDER BY created_at DESC",
            userId, status
        );
    }
}
```

## 优化吞吐量

### 瓶颈识别

```java
// 吞吐量瓶颈分析
/*
吞吐量瓶颈通常在：
1. CPU 密集：计算量大
   - 解决：增加 CPU、算法优化
2. IO 密集：等待时间长
   - 解决：异步 IO、增加并发
3. 内存不足：频繁 GC
   - 解决：增加内存、优化对象分配
4. 连接池耗尽：等待连接
   - 解决：增加连接池、调优参数

计算公式：
最大吞吐量 = CPU 核心数 × 有效工作时间占比 / 单请求处理时间
*/

// 吞吐量压测
public class ThroughputTest {
    public void runLoadTest(int concurrency, int durationSec) {
        ExecutorService executor = Executors.newFixedThreadPool(concurrency);
        AtomicLong requestCount = new AtomicLong(0);
        long start = System.currentTimeMillis();

        // 启动压测线程
        for (int i = 0; i < concurrency; i++) {
            executor.submit(() -> {
                while (System.currentTimeMillis() - start < durationSec * 1000) {
                    processRequest();
                    requestCount.incrementAndGet();
                }
            });
        }

        executor.shutdown();
        executor.awaitTermination(durationSec + 10, TimeUnit.SECONDS);

        long elapsed = System.currentTimeMillis() - start;
        double qps = requestCount.get() / (elapsed / 1000.0);
        System.out.println("QPS: " + qps);
    }
}
```

### 常见优化手段

```java
// 1. 增加并发：充分利用 CPU
public class ConcurrencyExample {
    private final ExecutorService executor = Executors.newFixedThreadPool(
        Runtime.getRuntime().availableProcessors() * 2
    );

    public <T> CompletableFuture<T> submit(Callable<T> task) {
        return CompletableFuture.supplyAsync(() -> {
            try {
                return task.call();
            } catch (Exception e) {
                throw new RuntimeException(e);
            }
        }, executor);
    }
}

// 2. 批量处理：减少 IO 次数
public class BatchExample {
    public void batchInsert(List<User> users) {
        // 批量插入，减少数据库交互次数
        jdbcTemplate.batchUpdate(
            "INSERT INTO users (id, name) VALUES (?, ?)",
            users,
            100,  // 每批 100 条
            (ps, user) -> {
                ps.setLong(1, user.getId());
                ps.setString(2, user.getName());
            }
        );
    }
}

// 3. 连接池优化：充分利用连接
public class ConnectionPoolExample {
    private final HikariConfig config = new HikariConfig();
    config.setMaximumPoolSize(50);        // 最大连接数
    config.setMinimumIdle(10);            // 最小空闲
    config.setConnectionTimeout(30000);   // 获取连接超时
    config.setIdleTimeout(600000);        // 空闲超时
    config.setMaxLifetime(1800000);       // 连接最大生命周期
}
```

## 延迟 vs 吞吐量的权衡

### 场景一：低延迟优先

```
场景：实时交易系统
要求：每笔交易延迟 < 100ms
优化策略：
- 减少不必要的等待
- 同步执行，优先返回
- 牺牲吞吐量换取延迟
```

```java
// 低延迟优先的实现
public class LowLatencyService {
    public Response handleRequest(Request request) {
        // 不排队，立即处理
        // 使用高性能队列
        // 减少 GC 压力
        return processImmediately(request);
    }
}
```

### 场景二：高吞吐优先

```
场景：日志收集系统
要求：每秒处理 100 万条日志
优化策略：
- 批量处理
- 异步写入
- 牺牲延迟换取吞吐
```

```java
// 高吞吐优先的实现
public class HighThroughputService {
    private final BlockingQueue<LogEvent> queue = new LinkedBlockingQueue<>(100000);

    public void handleRequest(LogEvent event) {
        queue.offer(event);  // 快速入队，不阻塞
    }

    @Scheduled(fixedDelay = 100)
    public void batchProcess() {
        List<LogEvent> batch = new ArrayList<>();
        queue.drainTo(batch, 1000);  // 批量取出

        if (!batch.isEmpty()) {
            batchWriter.write(batch);  // 批量写入
        }
    }
}
```

### 场景三：平衡策略

```
场景：Web 服务
要求：延迟合理（P99 < 200ms），吞吐够用
优化策略：
- 使用限流保护
- 优雅降级
- 动态调整
```

## 实际案例

### 案例一：接口优化

```
优化前：
- 串行调用 5 个服务
- 每个服务延迟 50ms
- 总延迟 = 250ms
- 吞吐量 = 400 QPS

优化后：
- 并行调用 5 个服务
- 总延迟 = 50ms
- 吞吐量 = 2000 QPS
- 提升 5 倍
```

```java
// 优化：串行改并行
public Response optimizedHandle(Request request) {
    // 串行（慢）
    // User user = userService.getUser(request.getUserId());
    // Order order = orderService.getOrder(request.getOrderId());
    // Payment payment = paymentService.getPayment(request.getPaymentId());

    // 并行（快）
    CompletableFuture<User> userFuture = userService.getUserAsync(request.getUserId());
    CompletableFuture<Order> orderFuture = orderService.getOrderAsync(request.getOrderId());
    CompletableFuture<Payment> paymentFuture = paymentService.getPaymentAsync(request.getPaymentId());

    CompletableFuture.allOf(userFuture, orderFuture, paymentFuture).join();

    return new Response(
        userFuture.join(),
        orderFuture.join(),
        paymentFuture.join()
    );
}
```

## 总结

延迟和吞吐量是性能优化的两个核心维度，需要根据业务场景权衡。

**核心要点**：

- 延迟：单个请求的响应时间
- 吞吐量：单位时间的处理能力
- 两者通常存在权衡关系

**优化策略**：

- 低延迟优先：减少等待，优先返回
- 高吞吐优先：批量处理，增加并发
- 平衡策略：动态调整，优雅降级

**常见优化**：

- 延迟：缓存、异步、索引
- 吞吐量：并发、批量、连接池
