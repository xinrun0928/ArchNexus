# 一致性 vs 可用性

你的系统用户 100 万，日活 10 万。一切正常，直到某天：

- 数据库主库宕机，切换到从库
- 从库数据有 5 分钟延迟
- 这 5 分钟内，用户看到的是旧数据
- 有些用户下完单，发现订单状态还是「待支付」

这就是一致性 vs 可用性的真实代价：**要么接受旧数据，要么拒绝请求**。

## 一致性模型

### 强一致性

```
定义：
- 写入后，所有读取都返回最新值
- 不存在中间状态
- 数据在所有副本间完全一致

实现代价：
- 写入需要同步到所有副本
- 延迟高
- 可用性降低
```

```java
// 强一致性写入
public void writeStrong(String key, String value) {
    // 写入所有副本
    for (Replica replica : replicas) {
        replica.write(key, value);
    }

    // 等待所有副本确认
    // 如果任何副本失败，抛出异常
    if (!allReplicasConfirmed()) {
        throw new ConsistencyException("写入失败：副本同步失败");
    }
}
```

### 弱一致性

```
定义：
- 写入后，不保证后续读取返回最新值
- 数据最终会一致，但不保证时间
- 延迟低，可用性高

特点：
- 写入即返回
- 副本异步同步
- 可能读到旧数据
```

```java
// 弱一致性写入
public void writeWeak(String key, String value) {
    // 写入本地副本，立即返回
    localReplica.write(key, value);

    // 异步同步到其他副本
    asyncExecutor.submit(() -> {
        for (Replica replica : otherReplicas) {
            replica.write(key, value);
        }
    });
}
```

### 最终一致性

```
定义：
- 写入后，如果没有新写入，最终会达到一致状态
- 有一段时间窗口数据不一致
- 大多数分布式系统的默认选择

特点：
- 实现简单
- 性能高
- 需要处理冲突
```

```java
// 最终一致性：带冲突处理的写入
public void writeEventually(String key, String value) {
    // 写入本地
    localReplica.write(key, value);

    // 生成版本号
    long version = generateVersion();

    // 发送到消息队列
    kafka.send("replication-topic", new ReplicationMessage(key, value, version));
}

// 后台同步
@KafkaListener(topics = "replication-topic")
public void syncReplicate(ReplicationMessage message) {
    // 使用向量时钟或时间戳解决冲突
    if (resolveConflict(message.key, message.version)) {
        replica.write(message.key, message.value);
    }
}
```

## 一致性级别

### SQL 标准定义的一致性级别

| 级别 | 说明 | 隔离级别 |
|---|---|---|
| READ UNCOMMITTED | 读取未提交的数据 | 最低 |
| READ COMMITTED | 只能读取已提交的数据 | Oracle 默认 |
| REPEATABLE READ | 同一事务内多次读取一致 | MySQL InnoDB 默认 |
| SERIALIZABLE | 完全串行执行 | 最高，性能最差 |

### 可调的一致性

```
┌─────────────────────────────────────────────────────────────────┐
│                    可调一致性（Tunable Consistency）            │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  R + W > N：强一致性                                            │
│                                                                  │
│  R = 读取副本数                                                │
│  W = 写入副本数                                                │
│  N = 总副本数                                                  │
│                                                                  │
│  示例：N = 3                                                    │
│  R = 2, W = 2 → R + W = 4 > 3 → 强一致                       │
│  R = 1, W = 3 → 写入全部，读取任意 → 强一致                    │
│  R = 1, W = 1 → R + W = 2 < 3 → 最终一致                     │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

```java
// Cassandra 可调一致性配置
public class CassandraConsistency {
    // 强一致性：读取和写入都使用 quorum
    @Query("SELECT * FROM users CONSISTENCY QUORUM")
    public List<User> readStrong();

    @Query("INSERT INTO users ... CONSISTENCY QUORUM")
    public void writeStrong();

    // 高可用：写入使用 ONE，读取使用 ONE
    @Query("SELECT * FROM users CONSISTENCY ONE")
    public List<User> readAvailable();

    @Query("INSERT INTO users ... CONSISTENCY ONE")
    public void writeAvailable();

    // 极端性能：写入使用 ANY（写入协调节点即可）
    @Query("INSERT INTO users ... CONSISTENCY ANY")
    public void writeFast();
}
```

## 可用性模型

### 可用性的度量

```
可用性 SLA：
- 99.9%（三个9）：每年宕机 8.7 小时
- 99.99%（四个9）：每年宕机 52 分钟
- 99.999%（五个9）：每年宕机 5 分钟

业务影响：
- 99.9% → 每月宕机 44 分钟
- 99.99% → 每月宕机 4.4 分钟
- 99.999% → 每月宕机 26 秒
```

### 高可用架构

```java
// 高可用设计：消除单点
public class HighAvailabilityService {
    // 1. 服务冗余：多实例部署
    private final List<ServiceInstance> instances;

    // 2. 健康检查：自动摘除故障实例
    public boolean isHealthy(ServiceInstance instance) {
        try {
            Response response = httpClient.get(instance.getUrl() + "/health");
            return response.isSuccess();
        } catch (Exception e) {
            return false;
        }
    }

    // 3. 负载均衡：自动路由到健康实例
    public Response route(String path) {
        ServiceInstance instance = loadBalancer.choose(instances);
        if (!isHealthy(instance)) {
            // 摘除故障实例，重新选择
            instances.remove(instance);
            instance = loadBalancer.choose(instances);
            instances.add(instance);  // 稍后重新检查
        }
        return httpClient.get(instance.getUrl() + path);
    }
}
```

## 一致性和可用性的权衡

### 一致性优先（CP）

```
场景：金融交易
要求：钱不能算错

策略：
- 写入需要多数节点确认
- 分区时拒绝写入
- 牺牲可用性，保证一致性

实现：
- ZooKeeper
- etcd
- HBase
```

```java
// CP 模式：ZooKeeper 分布式锁
public class DistributedLock {
    private final ZooKeeper zk;
    private final String lockPath;

    public boolean acquire(long timeoutMs) throws InterruptedException {
        // 创建临时顺序节点
        String nodePath = zk.create(lockPath + "/lock-",
            null,
            ZooDefs.Ids.OPEN_ACL_UNSAFE,
            CreateMode.EPHEMERAL_SEQUENTIAL);

        // 获取所有锁节点
        List<String> nodes = zk.getChildren(lockPath, false);

        // 按序号排序，判断是否最小
        Collections.sort(nodes);
        if (nodePath.endsWith(nodes.get(0))) {
            return true;  // 获取到锁
        }

        // 等待前一个节点删除
        String prevNode = nodePath.substring(lockPath.length() + 1);
        int prevIndex = nodes.indexOf(prevNode);
        if (prevIndex > 0) {
            CountDownLatch latch = new CountDownLatch(1);
            zk.exists(lockPath + "/" + nodes.get(prevIndex - 1),
                event -> latch.countDown());
            latch.await(timeoutMs, TimeUnit.MILLISECONDS);
        }

        return true;
    }
}
```

### 可用性优先（AP）

```
场景：社交媒体
要求：随时能发状态

策略：
- 各分区独立工作
- 分区时继续写入
- 牺牲一致性，保证可用性

实现：
- Cassandra
- DynamoDB
- CouchDB
```

```java
// AP 模式：DynamoDB 写入
public class DynamoWrite {
    private final AmazonDynamoDB dynamoDB;

    public void write(Item item) {
        // 写入本地节点，立即返回
        // 其他节点异步同步
        PutItemRequest request = new PutItemRequest()
            .withTableName("items")
            .withItem(toAttributeMap(item))
            .withReturnValues(ReturnValue.ALL_OLD);

        // 使用最终一致性写入
        dynamoDB.putItem(request);

        // 如果需要强一致，可以换用 CONSISTENT_READ
    }
}

// 读取时处理冲突
public class ConflictResolution {
    public String resolve(Item local, Item remote) {
        // 使用最后写入胜出（LWW）
        if (local.getVersion() < remote.getVersion()) {
            return remote.getValue();
        } else {
            return local.getValue();
        }
    }
}
```

## 混合策略

### 核心数据 vs 非核心数据

```
不同数据使用不同策略：

核心数据（账户、订单、库存）：
- 使用强一致性
- 可用性稍低但可接受
- 必要时人工干预

非核心数据（点赞、评论数、推荐）：
- 使用最终一致性
- 保证高可用
- 允许短暂不一致
```

```java
// 混合策略实现
public class HybridDataService {
    // 核心数据：强一致
    public void transfer(Account from, Account to, BigDecimal amount) {
        // 强一致事务
        transactionTemplate.execute(status -> {
            from.debit(amount);
            to.credit(amount);
            accountRepository.save(from);
            accountRepository.save(to);
            return null;
        });
    }

    // 非核心数据：最终一致
    public void likePost(String postId) {
        // 发布到消息队列
        kafka.send("like-events", new LikeEvent(postId));

        // 异步更新点赞数
        // 使用最终一致性
    }
}
```

### 读写分离 + 一致性哈希

```
不同读使用不同节点：

强一致读：
- 读取主库
- 延迟高，但数据最新

弱一致读：
- 读取从库
- 延迟低，但可能读到旧数据
```

```java
// 读写分离
public class ReadWriteSplitting {
    @Autowired
    private DataSource master;  // 主库

    @Autowired
    private DataSource slave;  // 从库

    // 写操作：主库
    @Transactional
    public void write(DataObject obj) {
        jdbcTemplate.setDataSource(master);
        jdbcTemplate.update("INSERT ...", obj.toParams());
    }

    // 强一致读：主库
    public DataObject readStrong(Long id) {
        jdbcTemplate.setDataSource(master);
        return jdbcTemplate.queryForObject(
            "SELECT * FROM data WHERE id = ?", id);
    }

    // 弱一致读：从库
    public DataObject readWeak(Long id) {
        jdbcTemplate.setDataSource(slave);
        return jdbcTemplate.queryForObject(
            "SELECT * FROM data WHERE id = ?", id);
    }
}
```

## 实际案例分析

### 案例一：库存扣减

```
问题：
- 库存不能超卖（一致性）
- 但也要高可用（不能停服）

解决方案：分布式锁 + 消息队列

流程：
1. 扣减请求先获取分布式锁
2. 扣减成功后发送到消息队列
3. 返回用户「扣减成功」
4. 后台异步处理库存更新

如果锁获取失败：
- 拒绝请求，返回「系统繁忙」
- 不超卖，但不可用
```

```java
public class InventoryService {
    private final RedisTemplate<String, String> redis;
    private final KafkaTemplate<String, Object> kafka;

    public boolean deduct(String sku, int quantity) {
        // 获取分布式锁
        String lockKey = "inventory:lock:" + sku;
        Boolean acquired = redis.opsForValue()
            .setIfAbsent(lockKey, "1", Duration.ofSeconds(5));

        if (!acquired) {
            throw new ServiceUnavailableException("系统繁忙，请稍后重试");
        }

        try {
            // 扣减库存
            String stockStr = redis.opsForValue().get("inventory:" + sku);
            int stock = Integer.parseInt(stockStr);

            if (stock < quantity) {
                throw new InsufficientStockException("库存不足");
            }

            redis.opsForValue().decrement("inventory:" + sku, quantity);

            // 发送事件，异步更新数据库
            kafka.send("inventory-events",
                new InventoryDeductEvent(sku, quantity));

            return true;
        } finally {
            redis.delete(lockKey);
        }
    }
}
```

### 案例二：点赞系统

```
问题：
- 点赞要快速响应（可用性）
- 允许短暂不一致（一致性）

解决方案：本地缓存 + 异步聚合

流程：
1. 点赞写入本地缓存，立即返回
2. 异步发送到消息队列
3. 后台聚合更新数据库
4. 数据库记录真实点赞数

可能出现的不一致：
- 点赞后立即查看，显示数字可能稍旧
- 但用户操作本身是成功的
```

## 总结

一致性和可用性的权衡是分布式系统设计的核心问题。

**核心要点**：

- 强一致：写入慢，但数据准
- 最终一致：写入快，可能读到旧
- 不同数据可以采用不同策略

**选择建议**：

| 场景 | 一致性策略 | 可用性策略 |
|---|---|---|
| 金融交易 | 强一致 | 允许短暂不可用 |
| 库存扣减 | 强一致 | 分布式锁 |
| 社交媒体 | 最终一致 | 高可用 |
| 推荐系统 | 最终一致 | 高可用 |

**实践建议**：

- 核心数据强一致，非核心数据弱一致
- 读写分离，区分场景
- 监控数据不一致时长
- 设计补偿机制
