# 从单体到微服务

2018 年，某中型电商公司决定启动从单体架构到微服务的迁移。当时的系统是一套 Java Spring MVC 的单体应用，代码量 50 万行，团队 30 人，部署方式是单台服务器加负载均衡。

CTO 后来回忆说：「那时候每次发布都提心吊胆。30 个工程师在同一个代码库里改代码，代码冲突频繁，集成测试要跑 2 个小时，发布一次要占用整个周末。」

更让他下决心的是，一次代码改动引发了全站故障——一个工程师修改了商品模块的查询逻辑，以为不会影响订单模块，结果两个模块共用了一个缓存 key，修改导致订单模块读取到了错误的缓存数据，产生了大量虚假订单。

这次故障成为迁移的导火索。

这一节用这个真实案例，完整还原一次长达三年的微服务拆分过程。

## 背景：为什么要拆

### 当时的痛点

```
代码管理：30 人共享一个代码库，代码冲突频繁，合并冲突是日常
集成测试：每次提交触发 CI，集成测试套件要跑 2 小时
部署耦合：改一行代码要发布整个应用，任何模块出问题都影响全站
扩展困难：用户模块需要扩容，但因为和订单模块在同一进程，只能全站扩容
技术锁定：想引入新技术（如 Go 重写部分模块），但单体架构不支持
```

### 为什么不一开始就用微服务

这个问题 CTO 被问过很多次。他的回答是：「因为那时候团队只有 5 个人，代码量 2 万行，单体架构是最快的开发方式。微服务带来的运维复杂度和团队协作成本，在小团队阶段是负担，不是优势。」

**微服务拆分需要两个条件**：

1. **业务复杂度足够高**：当单体应用变得臃肿到影响开发效率时，才需要拆分
2. **团队规模足够大**：当团队超过 10 人时，单体架构的协作成本开始超过拆分收益

## 第一年：准备工作（2018 年）

### 步骤 1：建立 DevOps 基础设施

微服务拆分的前提是 DevOps 能力跟得上。如果连单体应用的 CI/CD 都没做好，贸然拆分只会引入更多问题。

这一年团队做了以下基础设施建设：

```yaml

kind: Deployment
metadata:
  name: order-service
  labels:
    app: order-service
spec:
  replicas: 3
  selector:
    matchLabels:
      app: order-service
  template:
    metadata:
      labels:
        app: order-service
    spec:
      containers:
        - name: order-service
          image: registry.example.com/order-service:v1.0.0
          ports:
            - containerPort: 8080
          resources:
            requests:
              memory: "512Mi"
              cpu: "200m"
            limits:
              memory: "1Gi"
              cpu: "500m"
          readinessProbe:
            httpGet:
              path: /health
              port: 8080
            initialDelaySeconds: 30
            periodSeconds: 10
          livenessProbe:
            httpGet:
              path: /health
              port: 8080
            initialDelaySeconds: 60
            periodSeconds: 15
```

### 步骤 2：业务边界梳理（DDD 建模）

微服务拆分最关键的一步是**划好服务边界**。边界划错了，后期改造成本巨大。

团队用了 DDD（Domain-Driven Design，领域驱动设计）的方法进行业务边界梳理：

```markdown
# 电商系统 DDD bounded contexts

## 客户上下文（Customer Context）
- 客户注册、登录、认证
- 客户积分、等级
- 客户收货地址
- 依赖：无
- 被依赖：订单上下文（引用客户信息）、物流上下文（配送地址）

## 商品上下文（Product Context）
- 商品管理、上下架
- 商品分类、属性
- 商品搜索
- 依赖：无
- 被依赖：订单上下文（购买商品）、推荐上下文（商品数据）

## 订单上下文（Order Context）
- 订单创建、修改、取消
- 订单支付状态
- 订单履约流程
- 依赖：客户上下文、商品上下文、支付上下文
- 被依赖：物流上下文（配送）、客服上下文（售后）

## 支付上下文（Payment Context）
- 支付渠道接入
- 支付状态管理
- 依赖：无
- 被依赖：订单上下文（支付订单）

## 物流上下文（Logistics Context）
- 配送调度
- 物流状态追踪
- 依赖：客户上下文（收货地址）、订单上下文（配送信息）
- 被依赖：无

## 客服上下文（Support Context）
- 售后、退款、投诉
- 依赖：订单上下文（订单数据）、客户上下文（客户信息）
- 被依赖：无
```

关键原则：**上下文之间的依赖应该是单向的，尽量避免循环依赖**。如果 A 依赖 B，B 又依赖 A，说明边界划分有问题。

## 第二年：试点拆分（2019 年）

### 第一个拆分的模块：用户模块

用户模块是第一个被拆分的，因为它的依赖最少、风险最低，而且有独立的技术需求——需要支持短信验证码、第三方登录等，而其他模块不需要这些功能。

**迁移策略：绞杀者模式（Strangler Pattern）**

不是一次性拆完，而是在原有系统旁边构建新服务，逐步把流量从旧系统切换到新系统。

```
迁移前（单体）：
┌──────────────────────────────────────────────┐
│                  单体应用                      │
│  ┌────────┐ ┌────────┐ ┌────────┐ ┌────────┐ │
│  │用户模块│ │商品模块│ │订单模块│ │支付模块│ │
│  └────────┘ └────────┘ └────────┘ └────────┘ │
└──────────────────────────────────────────────┘

迁移中（双轨运行）：
┌────────────────────┐   ┌──────────────────────────────────────┐
│   用户微服务        │   │            单体应用                    │
│   ┌────────┐       │   │  ┌────────┐ ┌────────┐ ┌────────┐   │
│   │用户模块│       │ ←→ │  │商品模块│ │订单模块│ │支付模块│   │
│   └────────┘       │   │  └────────┘ └────────┘ └────────┘   │
│   (新系统)         │   │  (旧系统，保留用户模块但不再接收流量)   │
└────────────────────┘   └──────────────────────────────────────┘
                                    ↓
迁移后（纯微服务）：
┌──────────┐ ┌──────────┐ ┌──────────┐ ┌──────────┐
│用户服务  │ │商品服务  │ │订单服务  │ │支付服务  │
└──────────┘ └──────────┘ └──────────┘ └──────────┘
```

### 数据库拆分

用户模块拆分中，最复杂的部分是**数据库拆分**。

用户数据在单体时存在主库中，拆分的挑战是：如何把用户数据迁移到独立数据库，同时保持和旧系统双写期间的数据一致性？

```java
// 双写迁移实现
@Service
public class UserMigrationService {
    @Autowired private UserMapper userMapper;
    @Autowired private UserJdbi userJdbi;  // 新的用户数据库

    @Transactional
    public void saveUser(User user) {
        // 1. 写入新库
        userJdbi.insert(user);

        // 2. 写入旧库（通过 ORM）
        userMapper.insert(user);
    }

    @Transactional
    public void updateUser(User user) {
        // 1. 更新新库
        userJdbi.update(user);

        // 2. 更新旧库
        userMapper.update(user);
    }
}
```

数据校验是关键：

```java
// 增量数据校验
@Service
public class DataConsistencyValidator {
    @Autowired private UserMapper userMapper;
    @Autowired private UserJdbi userJdbi;

    // 定期对比新旧库的数据差异
    public void validateConsistency() {
        long lastId = 0L;
        int pageSize = 1000;
        int inconsistencyCount = 0;

        while (true) {
            // 分批查询
            List<User> newUsers = userJdbi.findByIdGreaterThan(lastId, pageSize);

            if (newUsers.isEmpty()) {
                break;
            }

            for (User newUser : newUsers) {
                // 从旧库查询对应记录
                User oldUser = userMapper.findById(newUser.getId());

                if (!equals(newUser, oldUser)) {
                    inconsistencyCount++;
                    log.warn("数据不一致：userId={}, new={}, old={}",
                        newUser.getId(), newUser, oldUser);
                }

                lastId = newUser.getId();
            }
        }

        if (inconsistencyCount > 0) {
            alertService.alert("用户数据不一致，数量：{}", inconsistencyCount);
        }
    }

    private boolean equals(User a, User b) {
        if (a == null || b == null) return false;
        return Objects.equals(a.getName(), b.getName())
            && Objects.equals(a.getEmail(), b.getEmail())
            && Objects.equals(a.getPhone(), b.getPhone());
    }
}
```

### 流量切换

当新服务的功能和性能都验证通过后，开始切换流量。

```yaml
# API 网关路由配置
spring:
  cloud:
    gateway:
      routes:
        # 用户模块：先切换 1% 流量到新服务
        - id: user-service-v1
          uri: lb://user-service-new
          predicates:
            - Path=/api/user/**
            - Header=X-Canary, true  # 带 canary header 的请求走新服务

        # 用户模块：其余流量走旧服务
        - id: user-service-legacy
          uri: http://legacy-monolith:8080
          predicates:
            - Path=/api/user/**
```

灰度策略：

```
第 1 周：1% 流量（带 canary header 的请求）
第 2 周：10% 流量
第 3 周：50% 流量
第 4 周：100% 流量
每周观察：错误率、延迟、业务指标
```

### 用户模块拆分总结

```
拆分周期：6 个月（2019 年 1 月 - 6 月）
拆分范围：用户注册、登录、认证、地址管理
拆分难度：低（依赖最少）
拆分问题：
- 双写期间出现了 3 次数据不一致，排查了 2 周
- 新服务的 session 管理方案选型花了 1 个月（最初用 JWT，后来改成 Redis Session）
拆分后效果：
- 发布频率：从每月 1 次提升到每周 3 次
- 团队协作：用户团队可以独立发布，不再需要协调其他团队
```

## 第三年：全面推进（2020 年）

### 艰难的教训：商品模块拆分

用户模块拆分的成功经验，让团队信心大增。但在拆分商品模块时，问题来了。

商品模块和订单模块之间有一个紧密的业务耦合：**订单创建时需要实时查询商品库存**。

```
单体时的调用方式：
订单模块 → 直接调用商品模块的 Java 方法（进程内调用）
      ↓ 同步获取商品信息

微服务后的调用方式：
订单服务 → HTTP 调用商品服务（跨进程调用）
      ↓ 网络延迟、可能的超时
```

商品服务响应变慢（从 1ms 到 5ms），订单的创建成功率从 99.9% 降到了 98.5%。在高峰期，这 1.4% 的失败率转化成了每天几百个失败的订单。

### 解决方案一：异步化库存校验

```java
// 优化：订单创建异步化
@Service
public class OrderService {

    @Autowired private KafkaTemplate kafkaTemplate;

    public String createOrderAsync(OrderCreateRequest request) {
        // 1. 生成订单号
        String orderId = generateOrderId();

        // 2. 发送创建订单事件到 Kafka
        OrderCreatedEvent event = new OrderCreatedEvent();
        event.setOrderId(orderId);
        event.setUserId(request.getUserId());
        event.setItems(request.getItems());
        kafkaTemplate.send("order-created", orderId, event);

        // 3. 立即返回订单号（订单状态为「创建中」）
        return orderId;
    }
}

// 库存服务消费事件
@Service
public class InventoryService {
    @KafkaListener(topics = "order-created")
    public void handleOrderCreated(OrderCreatedEvent event) {
        // 扣减库存（可能因为库存不足而失败）
        boolean success = inventoryService.reserve(event.getOrderId(), event.getItems());

        if (success) {
            // 库存扣减成功，发送订单确认事件
            kafkaTemplate.send("order-confirmed", event.getOrderId(),
                new OrderConfirmedEvent(event.getOrderId()));
        } else {
            // 库存不足，发送订单取消事件
            kafkaTemplate.send("order-cancelled", event.getOrderId(),
                new OrderCancelledEvent(event.getOrderId(), "库存不足"));
        }
    }
}
```

### 解决方案二：引入本地消息表

异步方案引入了消息队列的依赖，团队担心消息丢失的风险。最终采用了一个更可靠的方案：**本地消息表**。

```java
// 本地消息表：保证消息可靠发送
@Service
public class OrderService {
    @Autowired private OrderMapper orderMapper;

    @Transactional
    public void createOrder(Order order) {
        // 1. 插入订单
        orderMapper.insert(order);

        // 2. 插入消息记录（和订单在同一事务）
        LocalMessage message = new LocalMessage();
        message.setId(generateId());
        message.setTopic("order-created");
        message.setPayload(serialize(order));
        message.setStatus("PENDING");
        message.setRetryCount(0);
        orderMapper.insertMessage(message);
    }
}

// 消息投递器（定时任务）
@Service
public class MessageDispatcher {
    @Autowired private OrderMapper orderMapper;
    @Autowired private KafkaTemplate kafkaTemplate;

    @Scheduled(fixedDelay = 1000)
    public void dispatch() {
        List<LocalMessage> messages = orderMapper.selectPendingMessages(100);

        for (LocalMessage msg : messages) {
            try {
                kafkaTemplate.send(msg.getTopic(), msg.getPayload());
                msg.setStatus("SENT");
                orderMapper.updateMessage(msg);
            } catch (Exception e) {
                msg.setRetryCount(msg.getRetryCount() + 1);
                if (msg.getRetryCount() >= 3) {
                    msg.setStatus("FAILED");
                }
                orderMapper.updateMessage(msg);
            }
        }
    }
}
```

**核心设计**：消息记录和业务数据在同一个数据库实例中，在同一个事务里写入。即使 Kafka 投递失败，消息记录不会丢失，定时任务会重试。

## 最终结果

三年拆分的最终效果：

| 指标 | 拆分为 | 变化 |
| --- | --- | --- |
| 发布频率 | 每周 3 次（原来每月 1 次） | +300% |
| 平均故障恢复时间 | 8 分钟（原来 45 分钟） | -82% |
| 团队独立发布能力 | 5 个独立团队 | 新增 |
| 系统可用性 | 99.95%（原来 99.5%） | +0.45% |
| 基础设施成本 | 3 倍（K8s + 服务网格 + 日志 + 监控） | +200% |

### 付出的代价

CTO 后来做了一个总结，列出了三年中付出的隐性成本：

```
三年拆分总成本估算：
- 基础设施投入：4 人年（K8s、日志、监控、CI/CD）
- 拆分开发投入：8 人年（5 个服务 × 1.5 人年）
- 踩坑修复投入：3 人年（数据一致性、分布式事务、网络问题）
- 故障损失：约 6 次 P1 级故障，每次损失约 10 万
总投入：约 15 人年 + 60 万直接损失
```

## 总结：微服务拆分的核心认知

回顾这次三年拆分，有几个关键认知：

**第一，拆分前先建好基础设施**。Kubernetes、CI/CD、服务治理这些能力没有建立好之前，贸然拆分只会引入更多问题。第一年花在基础设施上的时间，是后面两年顺利拆分的基础。

**第二，边界划分是微服务拆分最关键的一步**。用户模块依赖少，拆分顺利；商品模块和订单模块耦合深，拆分的难度和代价远超预期。在动手之前，应该花足够的时间做 DDD 建模，画清楚上下文边界。

**第三，分布式事务是拆分中最难解决的问题**。服务拆分后，原来在同一个事务里的操作变成了跨服务的调用，本地事务变成了分布式事务。本地消息表、Saga、TCC 等方案各有优劣，需要根据业务场景选择。

**第四，微服务拆分是手段，不是目的**。拆分的目的是解决单体架构的协作效率和扩展性问题。如果你的团队不超过 10 人，代码量不超过 10 万行，单体架构可能是更好的选择。微服务带来的运维复杂度和分布式系统的挑战，不是小团队能轻松承担的。
