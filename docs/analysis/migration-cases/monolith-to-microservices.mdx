# 从单体到微服务

你的单体应用越来越臃肿，团队协作困难，部署一次影响全局。

单体到微服务的迁移是架构演进的重要里程碑。但迁移不是银弹，盲目拆分只会带来更多问题。

## 迁移实施步骤

### 步骤 1：领域建模

```java
// DDD 领域建模：识别 bounded context
/**
 * 电商系统的 bounded contexts：
 *
 * - 客户上下文（Customer Context）
 *   客户注册、认证、积分、等级
 *
 * - 商品上下文（Product Context）
 *   商品管理、库存、分类、搜索
 *
 * - 订单上下文（Order Context）
 *   订单创建、支付、履约、取消
 *
 * - 物流上下文（Logistics Context）
 *   配送、运费、物流追踪
 *
 * - 客服上下文（Support Context）
 *   售后、退款、投诉
 */

/**
 * 上下文映射关系：
 *
 * Customer Context ────> Order Context
 *     (订单引用客户)        (客户服务调用)
 *
 * Product Context ────> Order Context
 *     (订单引用商品)        (商品服务调用)
 *
 * Order Context ────> Logistics Context
 *     (物流服务调用)        (履约完成回调)
 */
```

// 领域事件驱动数据同步
@Service
public class OrderEventPublisher {
    @Autowired private ApplicationEventPublisher eventPublisher;

    public void createOrder(Order order) {
        // 保存到 order 数据库
        orderRepository.save(order);

        // 发布领域事件
        eventPublisher.publishEvent(
            new OrderCreatedEvent(order)
        );
    }
}

// 监听事件，同步数据
@Service
public class InventorySyncListener {
    @Autowired private InventoryService inventoryService;

    @EventListener
    public void onOrderCreated(OrderCreatedEvent event) {
        // 扣减库存
        inventoryService.reserve(event.getOrderId(), event.getItems());
    }
}

### 步骤 3：API 迁移

```java
// API 网关路由配置
@Configuration
public class GatewayRouteConfig {
    @Bean
    public RouteLocator routeLocator(RouteLocatorBuilder builder) {
        return builder.routes()
            // 客户服务
            .route("customer",
                r -> r.path("/api/customer/**")
                    .filters(f -> f.stripPrefix(1))
                    .uri("lb://customer-service"))

            // 商品服务
            .route("product",
                r -> r.path("/api/product/**")
                    .filters(f -> f.stripPrefix(1))
                    .uri("lb://product-service"))

            // 订单服务
            .route("order",
                r -> r.path("/api/order/**")
                    .filters(f -> f.stripPrefix(1))
                    .uri("lb://order-service"))

            // 仍然保留在单体的 API
            .route("legacy",
                r -> r.path("/api/legacy/**")
                    .filters(f -> f.stripPrefix(1))
                    .uri("http://legacy-monolith:8080"))
            .build();
    }
}

// 防腐层（Anti-Corruption Layer）
// 当新服务需要调用旧模块时
@Service
public class CustomerAdapter {
    @Autowired private CustomerFeignClient customerFeign;

    // 如果新服务可以直接调用客户微服务
    public Customer getCustomer(Long customerId) {
        return customerFeign.getById(customerId);
    }
}
```

### 步骤 4：数据迁移

```java
// 增量数据迁移
@Service
public class DataMigrationService {
    @Autowired private JdbcTemplate sourceJdbc;
    @Autowired private JdbcTemplate targetJdbc;
    @Autowired private OffsetStore offsetStore;

    @Transactional
    public void migrateOrders(LocalDateTime lastSyncTime) {
        // 1. 查询增量数据
        String sql = """
            SELECT * FROM orders
            WHERE updated_at > ?
            ORDER BY updated_at
            LIMIT 1000
            """;

        List<Order> orders = sourceJdbc.query(sql,
            new Object[]{lastSyncTime},
            orderMapper);

        if (orders.isEmpty()) {
            return;
        }

        // 2. 写入目标数据库
        for (Order order : orders) {
            saveToTarget(order);
        }

        // 3. 记录偏移量
        LocalDateTime maxUpdateTime = orders.stream()
            .map(Order::getUpdatedAt)
            .max(LocalDateTime::compareTo)
            .orElse(lastSyncTime);

        offsetStore.saveOffset("orders", maxUpdateTime);
    }
}
```

## 总结

单体到微服务的迁移是架构演进的重要里程碑。

**迁移时机**：

- 团队维度：超过 10 人，代码冲突频繁
- 性能维度：不同模块负载差异大
- 技术维度：需要不同技术栈
- 业务维度：业务边界清晰

**迁移策略**：

- 绞杀者模式：逐步替换
- 分支模式：新功能微服务，老功能单体

**实施步骤**：

1. 领域建模：识别 bounded context
2. 数据库拆分：每个服务独立数据库
3. API 迁移：网关路由 + 防腐层
4. 数据迁移：增量同步 + 校验

**核心认知**：微服务不是银弹，拆分带来的是复杂度而不是简单。拆分之前，先确保有足够的 DevOps 能力和自动化测试覆盖。
