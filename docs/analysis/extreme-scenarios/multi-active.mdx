# 跨机房多活

你的服务只部署在一个机房，机房断电后整个系统不可用。

跨机房多活是高可用架构的终极形态，但实现成本高、复杂度大。

## 数据同步

```java
// 异地数据同步
@Service
public class CrossRegionSyncService {
    @Autowired private KafkaTemplate kafkaTemplate;

    // 1. 同城同步：直接复制
    @KafkaListener(topics = "order-events")
    public void syncOrderToSameRegion(OrderEvent event) {
        // 同步到同城备机房
        kafkaTemplate.send("order-events-backup", event);
    }

    // 2. 异地同步：异步复制 + 补偿
    @Async
    public void syncToRemoteRegion(OrderEvent event) {
        try {
            // 发送到异地 Kafka
            kafkaTemplate.send("order-events-remote", event);
        } catch (Exception e) {
            // 记录失败，稍后重试
            saveToFailedQueue(event);
        }
    }

    // 3. 补偿任务
    @Scheduled(fixedDelay = 60000)
    public void compensateFailedEvents() {
        List<OrderEvent> failed = getFailedEvents();
        for (OrderEvent event : failed) {
            try {
                kafkaTemplate.send("order-events-remote", event);
                deleteFromFailedQueue(event);
            } catch (Exception e) {
                // 继续重试
            }
        }
    }
}
```

## 流量切换

```yaml
# DNS 智能解析
# 用户就近接入

apiVersion: v1
kind: Service
metadata:
  name: myapp-global
  annotations:
    # 主机房
    clouddns.example.com/geo:
      - region: cn-north
        weight: 100
      # 备机房
    clouddns.example.com/geo:
      - region: cn-east
        weight: 0

---
# 健康检查 + 自动切换
# 主机房故障时，自动将权重切换到备机房
```

```java
// 健康检查 + 流量切换
@Service
public class TrafficSwitchService {
    @Autowired private HealthCheckService healthCheck;
    @Autowired private DnsService dnsService;

    @Scheduled(fixedDelay = 5000)
    public void monitorAndSwitch() {
        // 检查主机房健康状态
        boolean mainHealthy = healthCheck.isHealthy("main-region");

        if (!mainHealthy && !isInSwitch) {
            // 触发切换
            switchToBackup();
            isInSwitch = true;
        } else if (mainHealthy && isInSwitch) {
            // 主机房恢复，等待一段时间后切回
            if (recoverWaitCount++ > 12) {  // 等待 1 分钟
                switchToMain();
                isInSwitch = false;
                recoverWaitCount = 0;
            }
        } else {
            recoverWaitCount = 0;
        }
    }

    private void switchToBackup() {
        log.warn("切换到备机房");
        dnsService.updateWeight("main-region", 0);
        dnsService.updateWeight("backup-region", 100);
        alertService.alert("主备切换已完成");
    }
}
```

## 总结

跨机房多活是高可用架构的终极形态。

**架构等级**：

- 冷备：手动切换，成本低
- 热备：自动切换，分钟级 RTO
- 同城双活：自动切换，秒级 RTO
- 异地多活：单元化架构，最高成本

**核心挑战**：

- 数据一致性：CAP 理论的权衡
- 切换速度：RTO 目标
- 成本控制：业务价值 vs 投入

**核心认知**：多活架构是高可用的高级形式，但成本极高。不要盲目追求多活，先评估业务对可用性的真实需求。
