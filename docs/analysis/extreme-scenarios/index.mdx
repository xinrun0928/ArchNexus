# 极限场景处理

当流量是平时的数百倍，当数据量达到 PB 级，当需要跨地域提供服务——普通架构已经不够用了。

本分类聚焦互联网公司面临的最极端场景，通过秒杀系统、高并发抢购、海量数据处理、跨机房多活四大场景，帮助你理解在极限压力下系统应该如何设计。

## 核心内容

**秒杀系统设计**：

- 秒杀特点：瞬时峰值、持续时间短、读多写少
- 秒杀架构：CDN → 网关 → 服务 → 库存
- 库存扣减：Redis Lua 脚本原子操作
- 请求拦截：限流、风控、防重复

**高并发抢购**：

- 抢购 vs 秒杀：持续时间、强调点不同
- 分布式锁：Redisson 实现
- 公平性保障：队列排队机制
- 实时库存：Redis 原子扣减

**海量数据处理**：

- 数据分层：ODS → DWD → DWS → ADS
- 存储选型：HDFS、Elasticsearch、ClickHouse、HBase
- 离线计算：Hive SQL、Flink
- 查询优化：分区裁剪、预聚合、数据压缩

**跨机房多活**：

- 多活等级：冷备 → 热备 → 同城双活 → 异地多活
- 同城双活：流量自动分配
- 异地多活：单元化架构、数据分类
- 流量切换：健康检查、DNS 切换

## 推荐阅读顺序

1. **秒杀系统设计**：理解高并发写入的架构
2. **高并发抢购**：理解公平性和性能的平衡
3. **海量数据处理**：理解大数据架构
4. **跨机房多活**：理解高可用架构的终极形态

:::tip 场景选择

极限场景的成本很高，不是所有系统都需要设计成这样。先评估业务的真实需求，再决定投入多少资源。
:::
