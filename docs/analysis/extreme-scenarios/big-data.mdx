# 海量数据处理

你的日志每天增长 1TB，用户需要查询任意时间范围的数据。

海量数据处理是数据平台的核心挑战，涉及存储、计算、查询等多个维度。

## 离线计算

```sql
-- Hive SQL 示例：用户行为日志分析
-- 按日统计活跃用户数

-- 1. 数据清洗
WITH cleaned_logs AS (
    SELECT
        user_id,
        event_type,
        event_time,
        device_id,
        -- 去除异常数据
        CASE
            WHEN length(user_id) > 50 THEN NULL
            WHEN event_time IS NULL THEN event_time
            ELSE event_time
        END as event_time
    FROM ods.user_events
    WHERE dt = '${bizdate}'
      AND event_time IS NOT NULL
)

-- 2. 用户维度聚合
SELECT
    dt,
    COUNT(DISTINCT user_id) as dau,
    COUNT(DISTINCT device_id) as device_count,
    SUM(CASE WHEN event_type = 'purchase' THEN 1 ELSE 0 END) as purchase_count,
    SUM(CASE WHEN event_type = 'cart' THEN 1 ELSE 0 END) as cart_count
FROM cleaned_logs
GROUP BY dt
```

## 实时计算

```java
// Flink 实时计算示例
public class UserActionStatistics {
    public static void main(String[] args) throws Exception {
        StreamExecutionEnvironment env =
            StreamExecutionEnvironment.getExecutionEnvironment();

        // 启用 Checkpoint
        env.enableCheckpointing(60000);
        env.getCheckpointConfig().setCheckpointStorage("hdfs://namenode/checkpoint");

        // 数据源：Kafka
        DataStream<String> rawStream = env.addSource(
            new FlinkKafkaConsumer<>(
                "user-actions",
                new JSONDeserializationSchema(),
                kafkaProps
            )
        );

        // 解析数据
        DataStream<UserAction> actions = rawStream.map(JSON::parseObject);

        // 实时统计每分钟 DAU
        DataStream<MinuteStats> minuteStats = actions
            .keyBy(action -> action.getUserId())
            .window(SlidingEventTimeWindows.of(Time.minutes(1), Time.minutes(1)))
            .aggregate(new MinuteDauAggregator());

        // 输出到 Kafka
        minuteStats.addSink(
            new FlinkKafkaProducer<>(
                "minute-dau-stats",
                new JSONSerializationSchema(),
                kafkaProps
            )
        );

        env.execute("User Action Statistics");
    }
}

public class MinuteDauAggregator
    implements AggregateFunction<UserAction, Set<String>, MinuteStats> {

    @Override
    public Set<String> createAccumulator() {
        return new HashSet<>();
    }

    @Override
    public Set<String> add(UserAction value, Set<String> accumulator) {
        accumulator.add(value.getUserId());
        return accumulator;
    }

    @Override
    public MinuteStats getResult(Set<String> accumulator) {
        return new MinuteStats(
            accumulator.size(),
            accumulator
        );
    }

    @Override
    public Set<String> merge(Set<String> a, Set<String> b) {
        a.addAll(b);
        return a;
    }
}
```

## 查询优化

```sql
-- ClickHouse 优化示例

-- 1. 分区裁剪
SELECT * FROM events
WHERE event_date = '2024-01-01'  -- 利用分区键

-- 2. 预聚合
CREATE MATERIALIZED VIEW events_pv
ENGINE = SummingMergeTree()
ORDER BY (event_date, event_type)
AS SELECT
    event_date,
    event_type,
    COUNT() as cnt,
    UNIQ(user_id) as uv
FROM events
GROUP BY event_date, event_type;

-- 3. 主键设计
-- ORDER BY (user_id, event_date, event_time)
-- 让高频查询的字段在前面

-- 4. 数据压缩
-- 使用 CODEC 压缩减少存储
ALTER TABLE events ADD COLUMN
    user_id String CODEC(ZSTD(5));
```

## 数据治理

```yaml
# 数据生命周期管理
# 自动冷热分层

# 热数据（30 天内）
# 存储：SSD / 本地盘
# 格式：Zstd 压缩
storage:
  tier: hot
  retention: 30d
  compression: zstd

# 温数据（30-90 天）
# 存储：普通硬盘
# 格式：Zstd 压缩
storage:
  tier: warm
  retention: 90d
  compression: zstd

# 冷数据（90 天以上）
# 存储：对象存储
# 格式：列式压缩
storage:
  tier: cold
  retention: 730d
  compression: lz4
  storage: s3://bucket/cold-data
```

## 总结

海量数据处理需要在成本和性能之间找到平衡。

**核心架构**：

- 数据分层：ODS → DWD → DWS → ADS
- 存储选型：按查询特点选择存储引擎
- 计算分层：离线批处理 + 实时流处理

**优化策略**：

- 分区裁剪：利用分区键减少扫描量
- 预聚合：物化视图加速查询
- 数据压缩：降低存储成本

**核心认知**：海量数据的核心是成本控制和查询性能的平衡。通过分层存储、数据压缩、预聚合，可以在有限成本下实现快速查询。
