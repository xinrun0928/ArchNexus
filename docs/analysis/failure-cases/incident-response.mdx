# 应急响应流程

故障发生了。告警响了，用户的投诉电话进来了，管理层在问什么时候能恢复。

这个时刻，最不需要的是「我们还在排查」。故障发生时，每一分钟都是代价——用户流失、收入损失、品牌受损。应急响应的目标只有一个：**用最短的时间把服务恢复到可接受状态**。

这一节用一个完整的真实案例——某电商平台「双十一前夜」的服务雪崩——来讲解应急响应的全流程。

## 事故背景

某中型电商平台，DAU 约 500 万，核心技术栈：Java Spring Cloud + MySQL + Redis + Kafka。距离双十一大促还有 3 天，团队刚刚完成了一轮系统压测，信心满满。

但他们不知道的是，压测只测了数据库，没有测缓存层。压测用的 Redis 是单节点配置，而生产环境用的是集群模式——这个差异，让他们错过了一个隐藏的连接池配置错误。

## 响应全流程：分钟级复盘

### T+0，21:47——发现阶段

监控告警：

```
[CRITICAL] order-service 错误率 > 15%
[CRITICAL] gateway-01 超时率 > 20%
[WARN] redis-cluster 连接数接近上限
```

21:47 刚过，值班工程师小李的手机响了。第一反应是点开监控大盘，看到订单服务的错误率曲线突然从 0.3% 跳升到 15%，而且还在继续上升。

**关键动作**：小李第一时间在故障群里发了第一条消息：

```
【故障确认】21:47 订单服务错误率突增，疑似服务雪崩。
当前状态：排查中
当前影响：订单创建接口不可用
影响范围：部分用户
值班人员：@小李
```

这条消息的目的是**同步信息、通知相关方**。在故障发生时，最怕的是信息不对称——工程师在排查，其他人不知道情况，在群里反复问「怎么了」「修好了吗」。

### T+5，21:52——止血阶段

小李快速看了一眼监控，发现错误日志里大量出现：

```
RedisConnectionException: Unable to connect to redis://redis-01:6379
RedisConnectionException: Unable to connect to redis://redis-02:6379
```

直觉告诉他：**Redis 出了问题**。他立即查看 Redis 集群的监控，发现：

- 集群中 3 个节点的连接数都在 8000 以上，接近 maxClients 配置（10000）
- 但连接数并没有完全打满，所以 Redis 本身没有拒绝连接
- 真正的问题是：**应用的 Redis 连接池配置了 maxTotal=50，而生产集群模式下每个节点的并发连接数需求远高于这个数字**

压测时用的是单节点 Redis，50 个连接足够；生产集群模式下，每个请求可能需要访问多个节点，连接数需求翻倍。他快速做了一个决定：**先扩容 Redis 连接池，防止情况继续恶化**。

```yaml
# 紧急修改：通过配置中心热更新连接池参数
spring:
  redis:
    lettuce:
      pool:
        max-active: 200   # 原来是 50，立即扩容到 200
        max-idle: 100
        min-idle: 20
```

5 分钟后，21:57，他再次查看监控：错误率从 15% 降到了 8%。但还没有完全恢复，说明还有其他问题。

**这个阶段的核心原则**：**先止血，再排查**。不要在止血的同时花时间找根因，止血优先于根因。

### T+15，22:07——定位阶段

错误率降到了 8%，但仍有大量超时。团队成员已经陆续上线（包括值班 leader 和 DBA），开始分工排查。

```
故障群动态：
22:02 @DBA老王：我来看数据库
22:04 @后端负责人老张：我来看订单服务的链路
22:06 @运维小赵：我来看 K8s 集群状态
22:08 @leader：止血进展如何？
22:08 @小李：Redis 连接池已扩容，错误率从 15% 降到 8%，但还有问题
```

22:10，老张发现了一个更严重的问题：链路追踪显示，订单服务中有 30% 的请求卡在「获取商品信息」这一步，而这个步骤的调用目标是商品服务。查看商品服务的监控，发现它也出问题了。

22:15，商品服务团队确认：商品服务依赖的 MySQL 数据库出现了**连接池泄漏**。

```java
// 根因定位：商品服务数据库连接泄漏
// 在 21:45 左右，有一次配置变更生效，其中一段代码引入了新的查询逻辑：
public List<Product> getProducts(List<Long> ids) {
    List<Product> result = new ArrayList<>();
    for (Long id : ids) {
        Connection conn = dataSource.getConnection(); // [!code error] 循环内获取连接
        try {
            result.add(query(conn, id));
        } catch (Exception e) {
            // [!code error] 异常路径没有关闭连接
            log.error("Query failed", e);
        }
        // [!code error] 连接从未关闭
    }
    return result;
}
```

这段代码在 21:45 随配置变更上线，每次调用会泄漏 1 个连接。连接池 maxSize 是 100，QPS 大约 30，到 22:10 左右连接池彻底耗尽，商品服务开始拒绝新请求。

**关键经验**：22:02 DBA 就开始看数据库了，但直到 22:15 才定位到根因——因为链路追踪显示的是「服务到服务的延迟」，而真正的问题在商品服务的数据库层。如果商品服务自己没有接链路追踪，DBA 就没法快速定位。

### T+30，22:17——修复阶段

根因确认后，修复方案很简单：回滚这次配置变更。

22:17，运维小赵执行回滚，2 分钟后变更生效。商品服务的数据库连接池开始逐步恢复正常。

```
22:17 @运维小赵：开始回滚配置变更
22:19 @运维小赵：回滚完成
22:20 @小李：订单错误率降到 1.2%，仍在观察
22:25 @小李：订单错误率恢复到 0.3%，服务恢复正常
22:25 @leader：故障恢复，持续观察 30 分钟
```

### T+60，23:00——验证与复盘

故障完全恢复。团队没有解散，而是在故障群里做了总结：

```
【故障恢复】22:25 订单服务恢复正常。
故障时长：约 38 分钟
影响用户：约 8 万用户在大约 20 分钟内无法正常下单
根因：商品服务配置变更引入数据库连接泄漏
触发条件：配置变更上线后，随着时间推移连接池耗尽
改进措施：
1. 回滚本次配置变更
2. Code Review 增加连接池使用检查项
3. 生产环境数据库连接池增加使用率监控
```

第二天上午 10 点，团队做了正式的事后复盘会议（复盘会议的具体方法在根因分析那一节有详细讲解）。

## 应急响应框架

回顾这次故障响应，可以提炼出一个标准化的应急响应框架：

```
发生 → 发现 → 止血 → 定位 → 修复 → 恢复 → 复盘
  │       │       │       │       │       │
  ↓       ↓       ↓       ↓       ↓       ↓
  故障    告警/   快速    找出    实施    确认    事后
  发生    用户    止损    根因    修复    正常    复盘
         反馈
```

每个阶段都有明确的目标和关键动作：

| 阶段 | 目标 | 关键动作 | 时间要求 |
| --- | --- | --- | --- |
| 发现 | 确认故障存在 | 监控告警 / 用户反馈 / 内部通知 | 越快越好 |
| 止血 | 防止影响扩大 | 限流 / 降级 / 隔离 / 回滚 | 黄金 5 分钟 |
| 定位 | 找出根因 | 日志 / 链路 / 指标 / 变更记录 | 30 分钟内 |
| 修复 | 消除根因 | 回滚 / 配置变更 / 扩容 | 找到即修 |
| 恢复 | 确认正常 | 监控验证 / 功能验证 | 持续观察 |
| 复盘 | 总结改进 | 根因分析 + 改进措施 | 48 小时内 |

## 止血手段详解

止血是应急响应中最关键的阶段。「止血」的本质是**快速隔离问题、减少损失**，而不是「修复问题」。

### 1. 限流

当系统负载超过处理能力时，优先限流，保护核心功能。

```bash
# 通过 Sentinel 控制台或命令行开启限流
# 将非核心接口限流到 10% 容量
curl -X POST "http://sentinel:8719/rules" \
  -d '[
    {
      "resource": "/api/recommendation/**",
      "grade": 1,
      "count": 100,
      "controlBehavior": 0
    },
    {
      "resource": "/api/search/**",
      "grade": 1,
      "count": 200,
      "controlBehavior": 0
    }
  ]'
```

### 2. 降级

关闭非核心功能，保证核心功能可用。

```yaml
# 通过配置中心热更新降级开关
feature-flags:
  recommendation: disabled   # 关闭推荐服务
  search: degraded            # 搜索降级，返回默认结果
  reviews: disabled           # 关闭评论服务
```

```java
// 代码层面的降级兜底
@Service
public class ProductService {
    @Value("${feature.recommendation:enabled}")
    private String recommendationEnabled;

    public ProductDetail getProductDetail(Long productId) {
        ProductDetail detail = getProductFromDB(productId);

        if ("disabled".equals(recommendationEnabled)) {
            // 关闭推荐，返回基础信息
            detail.setRecommendations(Collections.emptyList());
        } else {
            // 正常逻辑
            detail.setRecommendations(recommendationService.getRecommendations(productId));
        }
        return detail;
    }
}
```

### 3. 隔离

下线故障节点，防止影响扩散到健康节点。

```bash
# K8s 下线故障 Pod
kubectl cordon node-03
kubectl drain node-03 --ignore-daemonsets

# Eureka 下线故障实例
curl -X DELETE "http://eureka:8761/eureka/apps/ORDER-SERVICE/instance-id"
```

### 4. 回滚

版本问题最直接的止血手段是回滚。

```bash
# K8s 回滚
kubectl rollout undo deployment/order-service

# 配置变更回滚
# 如果是通过配置中心发布，找到上一次正常配置的版本，发布
apollo-cli publish --app=order-service --env=PROD --cluster=default \
  --namespace=application --release-title="回滚至 v1.2.3"
```

## 应急通信

故障发生时，通信和响应同样重要。通信的目标是**让所有相关方知道当前状态和进展**，避免信息不对称导致的不必要恐慌和干扰。

### 内部通信

```markdown title="故障群通知模板"
【故障通知】XX:XX 发生 [P0/P1/P2] 级故障

影响范围：[描述影响范围，如「订单创建接口不可用」]
当前状态：[止血中 / 定位中 / 修复中 / 已恢复]
预计恢复：[若能估计，提供时间]

值班人员：@xxx
响应人员：@xxx @xxx

【进展更新】XX:XX
[简短的进展描述，一句话]
```

### 外部通信（用户公告）

```
尊敬的 XX 用户：
您好！我们注意到目前部分功能无法正常使用，技术团队已第一时间介入处理。
预计恢复时间：XX:XX。
给您带来的不便，我们深表歉意。
感谢您的理解与支持。
```

### 通信原则

- **及时**：故障发生后 5 分钟内发出第一条通知
- **透明**：诚实告知当前状态，不隐瞒
- **简洁**：每次更新一句话，避免长篇大论
- **统一**：所有对外信息统一口径，由专人发布

## 响应级别与升级规则

不是所有故障都需要全公司介入。定义响应级别，让合适的人做合适的决策。

| 级别 | 触发条件 | 响应人员 | 要求 |
| --- | --- | --- | --- |
| P3 | 非核心功能部分不可用，影响少量用户 | 值班工程师 | 30 分钟内恢复或止血 |
| P2 | 非核心功能完全不可用，或核心功能部分不可用 | 值班工程师 + 团队负责人 | 15 分钟内止血，1 小时内恢复 |
| P1 | 核心功能完全不可用，影响大部分用户 | 全团队 | 5 分钟内止血，30 分钟内恢复 |
| P0 | 全站不可用，或造成重大经济损失 | CTO + 全团队 | 立即响应 |

**升级规则**：如果 15 分钟内没有找到止血手段，立即升级；30 分钟内没有定位到根因，立即升级。

## 应急演练

知道应急响应流程是一回事，真的遇到故障时能不能执行是另一回事。

这家公司每季度做一次**不打招呼的应急演练**，模拟真实的故障场景，让团队在「压力」下验证响应流程。

```
应急演练报告 - 2026-03-15

演练场景：订单服务数据库连接池耗尽（人工注入故障）
触发方式：演练前 5 分钟通知，不透露具体场景

响应评估：
- 发现时间：告警发出后 3 分钟（良好）
- 止血时间：15 分钟（合格，超过了 5 分钟目标）
- 止血方式：限流 + 降级（正确）
- 定位时间：28 分钟（偏慢，因为链路监控覆盖不全）

主要问题：
1. 止血阶段花了 10 分钟在「确认问题」而不是「止血」
2. 团队成员对降级开关不熟悉，操作延迟
3. 故障群通知格式不统一

改进计划：
1. 增加「先止血，再排查」的操作培训
2. 简化降级开关操作流程，增加快捷命令
3. 统一故障通知模板
```

## 总结

回顾这整次故障响应，最值得总结的经验有两点：

**第一，止血快慢决定了故障损失的大小**。这次故障中，Redis 连接池的紧急扩容在 5 分钟内完成，直接把错误率从 15% 压到了 8%。如果等 DBA 完全定位到根因再行动，这 10 分钟里受影响的用户数量会再翻一倍。

**第二，应急响应是需要练习的**。这家公司后来发现，真正故障时团队的表现比演练时要好——因为他们在每次演练后都做了复盘和改进，平时也在做「轻量级」的随机演练（每周随机选一个非核心服务，手动注入一次小故障，观察团队反应）。

故障不可避免，但**有准备的响应可以把故障的代价降到最低**。

到这里，故障案例分析的四个部分全部结束。从携程的数据库误删，到凌晨定时任务引发的连锁故障，到一年的防护体系建设，再到完整的应急响应流程——这些内容串联在一起，回答了一个核心问题：**故障发生了，我们怎么办；故障没发生，我们应该做什么准备**。
