# 应急响应流程

故障发生时，每一分钟都是损失。标准化的应急响应流程能让我们更快恢复服务。

故障不可避免，重要的是：能不能快速发现？能不能快速定位？能不能快速恢复？

## 响应流程详解

### 1. 发现阶段

```java
// 告警接收
public class IncidentDiscovery {
    // 告警渠道：监控告警 / 用户反馈 / 内部通知
    // 告警接收后立即进入响应流程

    public void onAlert(Alert alert) {
        // 1. 确认告警真实性（排除误报）
        if (!isRealAlert(alert)) {
            return;
        }

        // 2. 评估影响范围
        Impact assessment = assessImpact(alert);

        // 3. 记录事件
        Incident incident = createIncident(alert, assessment);

        // 4. 通知值班人员
        notifyOnCall(incident);

        // 5. 启动事件处理
        startIncidentResponse(incident);
    }
}
```

```bash
# 常用止血命令

# 1. 限流：开启限流规则
curl -X POST "http://sentinel:8719/rules" \
  -d '{"app":"order-service","rules":[{"resource":"/api/**","grade":1,"count":100}]}'

# 2. 降级：关闭非核心功能

  --patch '{"data":{"recommendation":"disabled"}}'

# 3. 隔离：下线故障节点

# 4. 回滚：回滚到上一版本
kubectl rollout undo deployment/order-service

# 5. 重启：重启服务
kubectl rollout restart deployment/order-service
```

### 3. 定位阶段

```java
// 定位问题常用手段

/**
 * 1. 日志分析
 */
public void analyzeLogs(Incident incident) {
    // 收集错误日志
    String query = "level:ERROR AND time:[" +
        incident.getStartTime() + " TO " + incident.getEndTime() + "]";

    List<LogEntry> errorLogs = elasticsearch.search(query);

    // 分析异常堆栈
    for (LogEntry log : errorLogs) {
        if (log.getException() != null) {
            logException("异常类型: {}", log.getException().getType());
            logException("异常信息: {}", log.getException().getMessage());
            logException("堆栈: {}", log.getException().getStackTrace());
        }
    }
}

/**
 * 2. 链路追踪分析
 */
public void analyzeTraces(Incident incident) {
    // 找出慢请求和错误请求
    List<Trace> slowTraces = traceService.findSlowTraces(
        incident.getStartTime(),
        incident.getEndTime(),
        Duration.ofSeconds(5)
    );

    // 分析调用链
    for (Trace trace : slowTraces) {
        Span rootSpan = trace.getRootSpan();
        log.info("慢请求: {}ms, 链路: {}",
            rootSpan.getDuration(),
            formatSpanTree(rootSpan));
    }
}

/**
 * 3. 指标分析
 */
public void analyzeMetrics(Incident incident) {
    // CPU / 内存 / GC
    Metrics cpu = metricsService.getCPU(incident);
    Metrics memory = metricsService.getMemory(incident);
    Metrics gc = metricsService.getGC(incident);

    // 找出异常时间点
    List<Anomaly> anomalies = metricsService.findAnomalies(
        incident.getStartTime(),
        incident.getEndTime()
    );

    for (Anomaly anomaly : anomalies) {
        log.info("指标异常: {} at {}, value: {}, threshold: {}",
            anomaly.getMetric(),
            anomaly.getTimestamp(),
            anomaly.getValue(),
            anomaly.getThreshold());
    }
}
```

### 4. 修复阶段

```java
// 修复决策
public class FixDecision {
    // 常用修复方案
    public static final String ROLLBACK = "回滚版本";
    public static final String HOTFIX = "热修复";
    public static final String CONFIG_CHANGE = "配置变更";
    public static final String SCALE_UP = "扩容";
    public static final String MANUAL_INTERVENTION = "人工介入";

    public String decideFix(Incident incident) {
        // 根据根因选择修复方案
        RootCause rootCause = incident.getRootCause();

        if (rootCause.isCodeIssue()) {
            return ROLLBACK;
        }

        if (rootCause.isConfigIssue()) {
            return CONFIG_CHANGE;
        }

        if (rootCause.isCapacityIssue()) {
            return SCALE_UP;
        }

        return MANUAL_INTERVENTION;
    }
}
```

## 总结

应急响应的目标是：快速发现、快速止血、快速定位、快速恢复。

**响应流程**：

- 发现：监控告警 / 用户反馈
- 止血：限流、降级、隔离、回滚
- 定位：日志、链路、指标分析
- 修复：选择合适方案
- 恢复：验证服务正常

**关键能力**：

- 7x24 值班
- 标准化的响应流程
- 有效的止血手段
- 快速定位的工具
- 及时的沟通机制

**核心认知**：故障不可避免，但可以通过反复演练，让团队形成肌肉记忆，在真实故障发生时能快速准确地响应。
