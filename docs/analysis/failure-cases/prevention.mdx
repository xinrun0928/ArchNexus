# 故障预防措施

最好的故障处理是让故障根本不发生。

预防胜于治疗，这句话在系统稳定性领域同样适用。与其事后救火，不如事前做好充分的预防措施。

## 代码质量保障

### 1. 代码审查

```java
// 代码审查 Checklist

/**
 * 代码提交前必须检查：
 *
 * 1. 异常处理

 *    [ ] 是否吞掉了异常？
 *    [ ] 是否有 finally 释放资源？

 * 2. 并发安全
 *    [ ] 共享变量是否加了锁？

 *    [ ] 是否存在死锁风险？
 *
 * 3. 资源管理
 *    [ ] 数据库连接是否正确关闭？
 *    [ ] 是否有连接泄漏风险？
 *    [ ] 是否有内存泄漏？
 *
 * 4. 性能
 *    [ ] 是否有 N+1 查询？
 *    [ ] 循环内是否有数据库操作？
 *    [ ] 大数据量是否有分页？
 *
 * 5. 安全
 *    [ ] 是否有 SQL 注入？
 *    [ ] 是否有 XSS 风险？
 *    [ ] 敏感数据是否脱敏？
 */
```

```java
// 单元测试覆盖率要求
public class TestCoverageRequirements {

    public void testEdgeCases() {
        // 空值测试

            calculate(null));

        // 边界值测试
        assertEquals(0, calculate(0));
        assertEquals(0, calculate(-1));

        // 超大值测试
        assertThrows(ArithmeticException.class, () ->
            calculate(Long.MAX_VALUE));
    }

    // 并发测试
    public void testConcurrency() {
        // 多个线程同时调用
        ExecutorService executor = Executors.newFixedThreadPool(10);
        List<Future> futures = new ArrayList<>();

        for (int i = 0; i < 100; i++) {
            futures.add(executor.submit(() ->
                counter.incrementAndGet()));
        }

        // 等待所有任务完成
        for (Future f : futures) {
            f.get();
        }

        assertEquals(100, counter.get());
    }
}
```

## 容错设计

### 熔断机制

```java
// 熔断器配置
@Configuration
public class CircuitBreakerConfig {
    @Bean
    public CircuitBreakerFactory circuitBreakerFactory() {
        return new Resilience4jCircuitBreakerFactory();
    }
}

// 服务调用配置熔断
@Service
public class OrderService {
    @CircuitBreaker(name = "inventoryService",
        fallbackMethod = "getInventoryFallback")
    public Inventory getInventory(Long productId) {
        return inventoryClient.get(productId);
    }

    // 降级方法
    public Inventory getInventoryFallback(Long productId,
                                         Throwable throwable) {
        log.warn("Inventory service failed: {}", throwable.getMessage());
        // 返回默认库存，允许下单
        return new Inventory(productId, Integer.MAX_VALUE);
    }
}
```

### 限流机制

```yaml
# 限流规则配置
sentinel:
  rules:
    # 接口限流
    flow:
      - resource: /api/order/create
        grade: 1  # QPS
        count: 1000
        controlBehavior: 0  # 直接拒绝

      - resource: /api/order/create
        grade: 1
        count: 10000
        controlBehavior: 2  # 排队等待

    # 熔断规则
    degrade:
      - resource: /api/inventory/get
        grade: 3  # 异常比例
        count: 0.3  # 30%
        timeWindow: 10  # 10 秒后恢复
```

### 兜底策略

```java
// 兜底策略示例
@Service
public class ProductService {
    @Autowired private ProductRepository productRepository;
    @Autowired private ProductCacheService cacheService;

    public ProductDTO getProduct(Long productId) {
        try {
            // 1. 先查缓存
            ProductDTO cached = cacheService.get(productId);
            if (cached != null) {
                return cached;
            }

            // 2. 查数据库
            Product product = productRepository.findById(productId);
            if (product == null) {
                return ProductDTO.EMPTY;  // 空对象兜底
            }

            ProductDTO dto = convertToDTO(product);

            // 3. 写缓存
            cacheService.set(productId, dto, Duration.ofMinutes(30));

            return dto;

        } catch (Exception e) {
            // 4. 异常兜底
            log.error("Get product failed", e);

            // 读取旧缓存（如果有）
            ProductDTO old = cacheService.get(productId);
            if (old != null) {
                return old;
            }

            return ProductDTO.EMPTY;
        }
    }
}
```

### 告警配置

```yaml
# 告警规则配置
alerting:
  rules:
    # CPU 使用率告警
    - name: cpu_high
      expr: cpu_usage > 80
      for: 5m
      severity: warning
      annotations:
        summary: "CPU 使用率过高"
        description: "CPU 使用率 {{ $value }}% 超过 80%"

    # 内存使用率告警
    - name: memory_high
      expr: memory_usage > 85
      for: 5m
      severity: warning

        summary: "内存使用率过高"
        description: "内存使用率 {{ $value }}% 超过 85%"

    # 接口错误率告警
```

## 总结

预防是最好的治疗。

**三层预防体系**：

- 第一层：代码质量（最基础）
- 第二层：容错设计（熔断、限流、兜底）
- 第三层：监控告警（发现问题）

**预防措施**：

- 代码审查：严格 Review
- 测试覆盖：单元 + 集成 + 性能
- 容错设计：熔断、限流、降级
- 监控告警：黄金指标 + 合理阈值
- 混沌工程：主动注入故障

**核心认知**：故障不可避免，但可以通过系统化的预防措施降低故障发生频率，减少故障影响范围，加快故障恢复速度。
