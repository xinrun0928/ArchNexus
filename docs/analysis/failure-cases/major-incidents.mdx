# 大型故障复盘

每次大型故障都是一堂昂贵的课。复盘不是为了追责，而是为了不再犯同样的错误。

故障复盘是技术团队成长的重要途径。一个团队对待故障的态度，决定了这个团队能走多远。

## 经典故障案例

### 案例一：数据库连接池耗尽

```
┌─────────────────────────────────────────────────────────────────┐
│                    数据库连接池耗尽故障                                   │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  故障现象：                                                     │
│  应用大量超时，数据库连接超时。                               │
│                                                                  │
│  影响范围：                                                     │
│  订单服务不可用 45 分钟。                                    │
│                                                                  │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │ 时间线：                                              │   │
│  ├─────────────────────────────────────────────────────────┤   │
│  │                                                         │   │
│  │  10:00  数据库慢查询监控告警                        │   │
│  │  10:05  多个服务开始响应超时                        │   │
│  │  10:10  数据库连接池打满                            │   │
│  │  10:15  故障扩散到关联服务                          │   │
│  │  10:30  重启数据库连接池                            │   │
│  │  10:45  服务恢复                                    │   │
│  │                                                         │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                  │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │ 根因分析：                                           │   │
│  ├─────────────────────────────────────────────────────────┤   │
│  │                                                         │   │
│  │  1. 慢查询未优化：                                │   │
│  │     SELECT * FROM orders WHERE status = 'pending'    │   │
│  │     → 没有 LIMIT，全表扫描                        │   │
│  │                                                         │   │
│  │  2. 连接池配置过小：                               │   │
│  │     最大连接数 100，但并发请求 200              │   │
│  │                                                         │   │
│  │  3. 缺乏熔断机制：                                 │   │
│  │     数据库慢 → 线程阻塞 → 连接耗尽               │   │
│  │                                                         │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

**复盘结论**：

- 慢查询必须加 LIMIT，有 LIMIT 的查询才能走索引
- 连接池大小需要根据并发量合理配置
- 服务必须实现熔断机制，防止故障扩散

### 案例二：缓存雪崩

```
┌─────────────────────────────────────────────────────────────────┐
│                    缓存雪崩故障                                          │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  故障现象：                                                     │
│  大量缓存同时过期，导致请求全部打到数据库。                     │
│                                                                  │
│  影响范围：                                                     │
│  所有接口响应时间从 50ms 上升到 5000ms+。                     │
│                                                                  │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │ 根因分析：                                           │   │
│  ├─────────────────────────────────────────────────────────┤   │
│  │                                                         │   │
│  │  1. 缓存 TTL 设置不当：                             │   │
│  │     大量缓存使用相同的过期时间                      │   │
│  │                                                         │   │
│  │  2. 缺乏缓存预热：                                   │   │
│  │     重启后缓存为空，请求全部打到 DB                 │   │
│  │                                                         │   │
│  │  3. 缺乏多级缓存：                                   │   │
│  │     只有 Redis，没有本地缓存作为缓冲               │   │
│  │                                                         │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

**复盘结论**：

- 缓存过期时间加随机值，避免同时过期
- 重启前必须预热缓存
- 使用本地缓存 + 远程缓存的多级缓存架构

### 案例三：消息队列消费积压

```
┌─────────────────────────────────────────────────────────────────┐
│                    消息队列消费积压故障                                │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  故障现象：                                                     │
│  订单完成通知延迟 2 小时才送达。                             │
│                                                                  │
│  影响范围：                                                     │
│  用户支付后长时间未收到订单完成通知。                        │
│                                                                  │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │ 根因分析：                                           │   │
│  ├─────────────────────────────────────────────────────────┤   │
│  │                                                         │   │
│  │  1. 消费者线程池配置不当：                           │   │
│  │     核心线程数 5，最大线程数 10                   │   │
│  │     但消息量是平时的 10 倍                          │   │
│  │                                                         │   │
│  │  2. 缺乏消费监控：                                   │   │
│  │     积压超过阈值未及时告警                          │   │
│  │                                                         │   │
│  │  3. 缺乏限流机制：                                   │   │
│  │     上游持续发消息，下游消费不过来                  │   │
│  │                                                         │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

**复盘结论**：

- 消费者线程池要能弹性扩展
- 消息积压必须设置告警阈值
- 上游限流，防止超过下游处理能力

## 总结

故障复盘是团队成长的重要途径。

**复盘的核心原则**：

- 不追责，只学习
- 追问到根因，不停在表面
- 措施必须具体可执行
- 定期跟踪验证

**常见故障模式**：

- 数据库：慢查询、连接池耗尽
- 缓存：雪崩、穿透、击穿
- 消息队列：积压、重复消费
- 网络：超时、DNS 故障

**改进方向**：

- 监控告警提前发现问题
- 熔断限流防止故障扩散
- 定期演练验证系统韧性
