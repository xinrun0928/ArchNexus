# 大型故障复盘

2015 年 5 月  27 日，携程官网页面全部变为空白，App 所有订单数据消失。用户看到的错误页面和官方后来的声明，让这件事在社交媒体上迅速发酵。舆论说「携程被黑」，但真相直到第二天才浮出水面。

这一节用这个真实事件作为主骨架，把大型故障的复盘方法讲清楚。

## 事故回放：那个改变携程的夜晚

### 背景：为什么携程那时候那么脆弱

2015 年的携程，技术架构处于一个尴尬的过渡期。老系统是 Perl 写的，一部分核心业务迁移到了 Java，但迁移工作还没完成。同时，业务高速增长带来的流量让老系统的压力持续增大，而新系统的边界又不够清晰——很多服务实际上还是共享同一个数据库实例。

当时的数据库架构是这样的：

- 生产库：Oracle RAC，两台 IBM 小型机做集群，用的是 IOE（IBM 小型机 + Oracle + EMC 存储）架构
- 备份：每天凌晨 2 点做一次全量备份，放到磁带库
- 访问控制：工程师通过跳板机连接生产库，有一定权限管控，但不够严格
- binlog：生产库没有开启——因为老 Oracle 用的是 Data Guard，不是 MySQL 的 binlog，所以没有这个概念

这个架构在 2015 年其实已经算是「大厂标配」，但问题在于：它的设计思路是「不要出事」，而不是「出了事怎么办」。没有多级备份、没有数据恢复演练、没有操作审计。

### 故障经过：时间线还原

| 时间 | 事件 |
| --- | --- |
| 22:10 | 运维工程师在后台执行一条 SQL，本意是删除测试数据 |
| 22:52 | 监控发现大量服务不可用，订单页面空白 |
| 22:55 | 值班 DBA 收到告警，开始排查 |
| 23:15 | 确认为数据删除导致，确认影响范围为航班相关信息表 |
| 23:30 | 尝试从备份恢复，发现备份磁带读取失败 |
| 00:30 | 改用第二个备份源，逐步恢复数据 |
| 02:00 | 部分服务恢复访问 |
| 04:00 | 主体业务基本恢复 |
| 次日 10:00 | 携程发布官方声明 |

从故障发生到服务基本恢复，花了大约 5 个小时。而官方声明中的「不明攻击」，事后被证明是对外沟通的一种策略——在还没有完全确认根因的时候，安抚用户比展示混乱更重要。

### 根因分析：一条 SQL 怎么搞垮了整个系统

事后复盘，这场故障的直接原因非常简单：**一条错误的 `DELETE` 语句**。

```sql
-- 本意：删除测试环境的数据
DELETE FROM flight_order WHERE test_flag = 1;

-- 实际执行时，连接到了生产库
-- 且表名写错了，删掉的是真实用户的航班订单表
```

但「一条 SQL」只是表象。真正的问题在于整个系统的脆弱链条：

**第一环：权限管控形同虚设**

工程师能够直接连接生产库并执行 `DELETE` 语句，没有二次确认，没有高危操作审批流程。在很多公司，这条约束靠的是「工程师自觉」，而不是系统强制。

```
生产环境访问 = 跳板机 + 数据库账号 → 就能执行任意 SQL
```

正确的方式应该是：

```
生产环境访问 = 跳板机 + 数据库账号 + 操作审批 + 高危语句阻断
```

**第二环：没有开启 binlog**

Oracle 的 Data Guard 是物理复制，而 binlog 是逻辑复制。前者可以恢复块级别的数据变更，但需要完整的物理备份；后者可以精确到每一条 SQL。携程的生产库既没有开启 Data Guard 的实时同步，也没有开启任何形式的逻辑变更日志，导致**数据被删了之后，没有任何增量记录可以追溯或回放**。

```
备份策略 = 每天全量备份 + 磁带库

缺失的层：
- 没有实时增量备份
- 没有逻辑变更日志（binlog）
- 没有跨机房备份
```

**第三环：备份恢复从未演练**

磁带库的备份能不能正常恢复？没有人知道。直到故障发生那一刻，工程师才发现磁带读取有问题——不是因为磁带损坏，而是因为**读取磁带需要的硬件设备在这个时间段内不可用**（设备被其他部门借走了，没有登记）。

这不是设备的问题，是流程的问题——没有人知道备份多久验证一次、验证的标准是什么。

**第四环：应急响应混乱**

故障发生后，值班团队的响应流程是：

```
告警 → 值班 DBA 排查 → 确认数据问题 → 联系备份负责人 → 发现备份问题 → 重新找方案
```

每一步都卡在了信息不对称上。DBA 不知道备份的具体情况，备份负责人不在现场，现场工程师没有权限做决策。这直接导致了从 22:52 到 23:30 近 40 分钟的「摸黑时间」。

### 影响评估：故障的代价

| 维度 | 数字 |
| --- | --- |
| 业务中断时长 | 约 5 小时 |
| 影响用户数 | 携程注册用户超 2 亿，当晚活跃用户受损 |
| 直接经济损失 | 估算千万级别（含订单退款、补偿、流量损失） |
| 品牌损失 | 股价次日下跌约 3%，社交媒体负面讨论持续一周 |
| 数据丢失 | 约 5 小时内新增的航班订单数据永久丢失 |

## 数据库类故障的通用模式

携程的故障是数据库类故障的典型代表，但不是唯一的形式。数据库类故障通常有以下几种：

### 模式一：误操作

最常见的数据库故障形态。`DELETE` 没加 `WHERE`、`UPDATE` 写错了条件、`DROP TABLE` 敲错了表名。任何在生产环境执行过 SQL 的工程师，应该都心有余悸。

这类故障的特点是：**发生极快，破坏极大，恢复极难**。一条 SQL 执行只需要几毫秒，但数据可能涉及几十万、几百万行。而恢复？要么靠备份，要么靠 binlog——两样都没有的话，只能认栽。

### 模式二：慢查询拖垮数据库

一条 `SELECT *` 从订单表查所有历史记录，`JOIN` 了 5 张表，没有索引，跑了 3 分钟还没出来。在这段时间内，数据库连接池被打满，所有新请求都在排队，最终服务全线超时。

```sql
-- 典型的慢查询：全表扫描 + 隐式类型转换
SELECT * FROM orders WHERE order_no = 12345;
-- order_no 是 VARCHAR，但传入了数字
-- MySQL 做了隐式类型转换，这一列的索引完全失效
```

### 模式三：连接池耗尽

应用端连接池配置过小，而某个接口大量占用连接不释放（比如打开连接后忘记关闭，或者长事务占用连接）。当连接池打满，新的请求要么排队，要么直接报错。

```java
// 典型的连接泄漏：异常路径没有释放连接
public void queryOrder(Long orderId) {
    Connection conn = dataSource.getConnection();
    try {
        // 查询逻辑
    } catch (Exception e) {
        log.error("Query failed", e);
        // [!code error] 异常时直接抛出，conn 没有关闭
        throw e;
    }
    // [!code error] 如果有 return，也要在这之前关闭
    conn.close();
}

// 正确做法：用 try-with-resources
public void queryOrder(Long orderId) {
    try (Connection conn = dataSource.getConnection()) {
        // 查询逻辑
    } // 连接自动关闭
}
```

### 模式四：主从延迟导致的数据不一致

读写分离架构下，写入主库后立即读取从库，但从库还没同步完，返回了旧数据。在高并发场景下，这种「短暂的不一致」可能引发业务逻辑错误——比如超卖。

## 故障分级标准

知道了故障长什么样，还需要知道怎么度量故障。行业内通用的分级标准如下：

| 级别 | 名称 | 定义 | 响应要求 |
| --- | --- | --- | --- |
| P0 | 灾难级 | 核心业务完全不可用，影响全部用户，持续 `>` 30 分钟 | CEO + CTO 直接介入 |
| P1 | 严重级 | 核心业务部分不可用，影响大部分用户，持续 10~30 分钟 | 技术 VP + 业务负责人介入 |
| P2 | 中等级 | 非核心业务不可用，影响部分用户，持续 5~10 分钟 | 团队负责人介入 |
| P3 | 轻微级 | 非核心业务部分不可用，影响少量用户，持续 `<` 5 分钟 | 值班工程师处理 |

携程这次故障，按这个标准，属于 **P0 级别**。

## 复盘报告模板

故障复盘不是写流水账，而是一份结构化的事后分析报告，目的是让团队从这次故障中学到东西，并且确保下次有改进。

```markdown title="故障复盘报告 - INC-20150527-001"

- 故障时间：2015-05-27 22:52 - 2015-05-28 04:00

- 影响范围：携程官网、App 全部订单查询

- 订单影响：约 500 笔新订单数据永久丢失
- 收入影响：估算直接损失约 500 万元，间接损失（品牌）难以量化
- 数据影响：5 小时增量数据无法恢复

## 根因分析

### 直接原因
工程师在执行数据清理操作时，错误连接到生产库，导致生产数据被误删。

### 深层原因（5 Why 追问）

**Why 1**：为什么会连接到生产库？
→ 因为生产库和测试库使用的是同一个跳板机账号，没有网络层面的隔离。

**Why 2**：为什么不使用测试库？
→ 当时测试环境和生产环境共用一套 Oracle 实例，测试数据直接在生产库上。

**Why 3**：为什么会共用实例？
→ 数据库资源紧张，测试库服务器在年初报废后一直没有补充。

**Why 4**：为什么不补充测试库资源？
→ 申请流程复杂，且认为「反正测试库只是跑脚本，影响不大」。

**Why 5**：为什么认为测试库不重要？
→ 没有建立「测试环境等同于生产环境安全水位」的意识，没有人对这件事负责。

### 系统性根因
1. 访问控制缺失：没有高危操作审批和阻断机制
2. 备份体系不完整：binlog 未开启，备份从未演练
3. 环境隔离缺失：测试与生产共用数据库实例
4. 应急响应失序：没有明确的故障响应 SOP
```

## 复盘中的常见误区

写过复盘报告的人都知道，最难的不是找根因，而是**避免复盘变成走过场**。以下几个误区是故障复盘中最常见的：

**误区一：追责而非学习**

复盘会上最常听到的一句话是「这是谁的责任？」——这个问题看似合理，但会把复盘带偏。回答「是哪个工程师的问题」，对系统改进毫无意义。

正确的问题是：「我们有什么机制可以防止这类问题再次发生？」

**误区二：停在表面原因**

「数据库被误删了」是表面原因。「为什么工程师能直接操作生产库」才是真正的根因。「为什么测试库一直没有搭建」才是系统性问题。

每次复盘至少追问 3 层 Why，才能触及真正的改进点。

**误区三：措施不落地**

「加强监控」「优化流程」「提高意识」——这类措施写进复盘报告，看起来正确，但无法执行和验证。

好的措施必须满足：有人负责、有完成时间、可验证。比如：

- ❌ 加强数据库安全培训
- ✅ 每个工程师在 6 月底前完成数据库安全操作考核，通过率 100%

**误区四：一次复盘解决所有问题**

故障往往是多个问题叠加的结果。一次复盘能解决一个主要问题，就是成功的。不要试图在一篇复盘里把所有改进都列出来，那样反而哪件都做不好。

## 总结

携程 2015 年的这次故障，是一个典型的「小失误 + 大脆弱」组合拳。一条 SQL 的误执行之所以能造成 P0 级事故，根本原因在于整个数据安全体系有多处漏洞——访问控制、备份体系、环境隔离、应急响应，每一环都在这次故障中暴露了问题。

从这次故障中学到的最重要一课是：**数据安全不是数据库管理员一个人的事，它是整个研发和运维体系共同的责任。** 每一次对生产数据的操作，都应该被视作高风险动作；每一个数据备份，都应该被定期验证可用性。
