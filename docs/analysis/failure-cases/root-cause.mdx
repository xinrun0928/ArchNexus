# 故障根因分析

故障发生了，表面上看起来的原因往往不是真正的原因。

根因分析是故障复盘的核心环节。只有找到真正的根因，才能从根本上解决问题，防止同类故障再次发生。

## 根因分析方法

### 1. 5 Why 分析法

```
┌─────────────────────────────────────────────────────────────────┐
│                    5 Why 分析法                                       │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │ 例子：用户无法登录                                    │   │
│  ├─────────────────────────────────────────────────────────┤   │
│  │                                                         │   │
│  │  Why 1：为什么用户无法登录？                         │   │
│  │  → Session 服务不可用                                 │   │
│  │                                                         │   │
│  │  Why 2：为什么 Session 服务不可用？                   │   │
│  │  → Redis 集群宕机                                    │   │
│  │                                                         │   │
│  │  Why 3：为什么 Redis 集群宕机？                      │   │
│  │  → 运维重启 Redis 时指定了错误配置文件               │   │
│  │                                                         │   │
│  │  Why 4：为什么会指定错误配置文件？                    │   │
│  │  → 没有配置文件版本管理                               │   │
│  │                                                         │   │
│  │  Why 5：为什么没有配置文件版本管理？                 │   │
│  │  → 配置变更流程不规范，缺乏 Review                   │   │
│  │                                                         │   │
│  │  ┌─────────────────────────────────────────────────┐ │   │
│  │  │ 根因：配置变更流程不规范                          │ │   │
│  │  │ 改进：引入配置管理平台，所有变更 Review 后执行   │ │   │
│  │  └─────────────────────────────────────────────────┘ │   │
│  │                                                         │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                  │
```

```
│  │ 偏差一：确认偏误                                       │   │
```

## 总结

根因分析是故障复盘的核心。

**分析方法**：

- 5 Why：连续追问直到根本原因
- 鱼骨图：从多个维度系统分析
- 事件链：按时间线梳理因果关系

**常见根因类型**：

- 人员因素：操作失误、沟通不畅
- 流程因素：缺少 Review、测试、监控
- 技术因素：代码缺陷、配置错误、容量不足
- 组织因素：协作问题、技术债务

**避免偏差**：

- 不先入为主
- 追溯完整事件链
- 关注系统而非个人
- 对比分析找出差异
