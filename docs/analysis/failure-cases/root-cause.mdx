# 故障根因分析

你有没有发现一个规律：**故障似乎格外偏爱凌晨**。

定时任务的执行时间大多集中在凌晨：数据报表生成、缓存预热、数据库归档、版本发布窗口选择……这些看起来无害的后台任务，在某个深夜突然变成压垮系统的最后一根稻草。

这一节就用这个现象作为主线，讲解系统化的根因分析方法。

## 一个真实案例：凌晨 3 点的告警

某中型电商平台，技术团队在凌晨 3:07 收到了大量告警：订单服务超时，支付接口 502，网关错误率飙升到 15%。

值班工程师在 5 分钟内上线，开始排查。

### 第一轮排查：看到了什么

```
告警信息：
- 订单接口 p99 延迟 > 10s
- 数据库 CPU 使用率 95%
- Redis 连接数打满
- 支付回调队列积压 8000+ 条
```

从表象来看，这是一个典型的「数据库被压垮」的场景。但问题是：**数据库为什么会突然被打满？**

### 第二轮排查：时间线回溯

值班工程师开始沿着时间线往前查：

| 时间 | 事件 |
| --- | --- |
| 02:00 | 定时任务「数据归档」启动 |
| 02:30 | 数据库 CPU 开始上升，从 20% 缓慢爬升 |
| 02:55 | 定时任务「全站缓存预热」启动，同时触发了商品推荐的批量计算 |
| 03:00 | 缓存命中率从 95% 骤降到 60%，大量请求穿透到数据库 |
| 03:07 | 数据库连接池打满，告警触发 |

现在清楚了：这不是单一原因，而是**三个定时任务叠加的连锁反应**。

## 5 Why 分析法：追问到真正的根因

### Why 1：为什么三个定时任务会同时跑？

查看定时任务的配置发现：

- 数据归档：每天凌晨 2:00
- 缓存预热：每天凌晨 3:00
- 商品推荐计算：每天凌晨 3:00

数据归档是凌晨 2:00，理论上应该在 3:00 之前结束。但为什么 3 点的时候它还在跑？

### Why 2：为什么数据归档花了这么长时间？

查看归档任务的 SQL：

```sql
-- 数据归档任务
INSERT INTO order_archive
SELECT * FROM orders
WHERE status = 'completed'
AND created_at < DATE_SUB(NOW(), INTERVAL 1 YEAR);

DELETE FROM orders
WHERE status = 'completed'
AND created_at < DATE_SUB(NOW(), INTERVAL 1 YEAR);
```

问题浮现了：归档任务一次性处理了一年的历史订单，数据量超过 300 万行。而且在 `DELETE` 之前，`INSERT` 执行了全表扫描，没有分批、没有索引利用、没有事务控制——直接导致了长时间的表锁。

### Why 3：为什么归档任务没有优化？

回到组织层面：这套定时任务的代码是 3 年前写的，当时的数据量只有几万行，跑起来没问题。后来业务增长了，但**没有人回顾过这些定时任务的执行效率**。

### Why 4：为什么没有人回顾？

这个问题更复杂：

- 没有建立定时任务的质量门禁
- 定时任务的变更不需要性能评审
- 监控只关注「任务跑没跑完」，不关注「跑了多久、消耗了多少资源」

### Why 5：为什么没有质量门禁？

这是更深层的组织问题：团队长期处于「功能交付优先」的模式，所有非功能需求（性能、安全、可维护性）都被视为「以后再说」的事情。技术债务日积月累，最终在一个不起眼的凌晨引爆。

### 根因总结

| 层次 | 原因 |
| --- | --- |
| 直接原因 | 凌晨 3 点三个定时任务同时执行，归档任务长时间锁表 |
| 深层原因 | 归档任务代码 3 年未优化，单次处理量太大 |
| 系统性原因 | 定时任务没有性能评审机制，执行时间窗口没有统一规划 |
| 组织性原因 | 技术债务长期被忽视，团队「功能交付优先」的惯性 |

现在再回头看最初的问题——「为什么故障总在凌晨发生」——答案就很清晰了：**不是凌晨特别倒霉，而是凌晨是定时任务执行的窗口，多个任务叠加在一起，在没有充分评估的情况下，它们对系统的冲击往往超出预期。**

## 鱼骨图：从更多维度分析

5 Why 擅长纵向追问，但故障的原因往往是多维度的。鱼骨图（也叫因果图）可以帮助系统性地列出所有可能的原因维度。

以这次凌晨故障为例，鱼骨图画出来是这样的：

```
                              问题：凌晨 3 点数据库被打满
                                                    ↓
    ┌──────────────┐   ┌──────────────┐   ┌──────────────┐   ┌──────────────┐
    │    人员      │   │    流程      │   │    技术      │   │    工具      │
    ├──────────────┤   ├──────────────┤   ├──────────────┤   ├──────────────┤
    │ 值班人员      │   │ 没有任务      │   │ 归档 SQL     │   │ 没有定时任务  │
    │ 响应不及时   │   │ 协调机制      │   │ 全表扫描     │   │ 监控工具      │
    │              │   │              │   │              │   │              │
    │ 任务开发者    │   │ 没有变更      │   │ 缺少分批     │   │ 没有任务     │
    │ 缺少性能意识  │   │ 评审流程      │   │ 处理机制     │   │ 执行监控      │
    │              │   │              │   │              │   │              │
    │              │   │              │   │ 连接池缺少    │   │ 没有跨任务    │
    │              │   │              │   │ 限流保护      │   │ 资源协调      │
    └──────────────┘   └──────────────┘   └──────────────┘   └──────────────┘
```

从这张图可以看到，故障的原因不只是「代码没优化」，还涉及流程缺失（没有任务协调、没有变更评审）和工具缺失（没有任务级监控）。

### 鱼骨图的使用方法

1. **画鱼头**：把故障现象作为鱼头，放在右侧
2. **画主骨**：按人、流程、技术、工具、管理等维度画主骨
3. **填小刺**：在每个维度下填写具体的原因
4. **分层级**：小刺之间有层级关系，比如「代码没优化」是小刺，「没有性能评审流程」才是大刺

鱼骨图的价值在于**强迫你从多个角度思考问题，而不是只盯着代码本身**。

## 事件链分析：还原完整的事故传播路径

5 Why 追问的是「为什么」，事件链回答的是「怎么发生的」。时间轴上每一个节点，都是理解故障传播路径的关键。

回到凌晨 3 点的故障，完整的事件链是这样的：

```
T+0    02:00     数据归档任务启动
 │                   │
 │                   ↓
T+30   02:30     数据库 CPU 从 20% 缓慢爬升到 50%
 │                   │
 │                   ↓
T+55   02:55     缓存预热任务启动
 │     推荐计算任务同时启动
 │                   │
 │                   ↓
T+60   03:00     缓存命中率从 95% 骤降到 60%
 │     大量请求穿透到数据库
 │                   │
 │                   ↓
T+65   03:05     数据库连接池使用率 80% → 95%
 │                   │
 │                   ↓
T+72   03:07     连接池打满，新请求开始排队
 │     订单服务开始超时
 │                   │
 │                   ↓
T+73   03:07     告警触发，值班工程师收到通知
 │                   │
 │                   ↓
T+80   03:12     工程师定位到三个定时任务
 │                   │
 │                   ↓
T+85   03:15     人工终止归档任务和预热任务
 │                   │
 │                   ↓
T+100  03:30     数据库负载逐步下降，服务恢复
```

事件链分析的作用是**找出连锁反应中的关键节点**。在这个例子里，T+55 是一个关键节点——缓存预热任务和推荐计算任务同时启动，形成了第一次叠加冲击。如果当时这两个任务有时间上的错峰，可能就不会触发后续的连锁反应。

## 常见根因类型

做过多次故障复盘之后，你会发现大多数根因可以归到以下几个类别：

### 人员因素

操作失误是最常见的人员问题。一个字符的差异，可能导致完全不同的结果：

```sql
-- 生产库：删除指定日期的测试数据
DELETE FROM orders WHERE created_at < '2026-04-01' AND test_flag = 1;
-- 实际执行时：
DELETE FROM orders WHERE created_at < '2026-04-01';
-- test_flag 列不存在，整条语句被跳过...吗？
-- 不，这条语句直接报错，被 catch 住，但数据已经部分删除
```

还有一种常见的人员问题是**沟通不畅**。A 团队改了数据库结构，没有通知依赖这个表的 B 团队；发布窗口临时改了，没有通知 DBA 提前准备。故障发生之后才发现，这些团队之间根本没有建立有效的沟通机制。

### 流程因素

流程缺失是系统性故障的根本原因之一。很多团队不是「不想做好」，而是**根本没有流程约束**。

比如，没有发布评审流程，导致发布内容和测试环境验证的内容不一致；没有配置变更审批，导致一次配置修改引发了连锁反应；没有定时任务的性能基准线，导致任务随时间积累的效率问题无人知晓。

### 技术因素

代码缺陷、架构不合理、容量不足——这些都是技术层面的根因。

举一个很典型的例子：Redis 连接池配置：

```yaml
# 错误配置
redis:
  pool:
    max-active: 8       # 最大连接数 8，在高并发下完全不够
    max-idle: 8         # 最大空闲连接
    min-idle: 0         # 最小空闲连接为 0
    max-wait: 100ms     # 等待超时 100ms，在高并发下频繁超时
```

这套配置在线上只有 100 QPS 的时候完全正常，但业务增长到 1000 QPS 的时候，Redis 连接池就成了瓶颈——等待时间超过 100ms 的请求直接失败。

### 组织因素

这是最容易被忽视的根因类型，但也是最深层的。

- 技术债务长期不被重视，因为没有人为它负责
- 团队太忙，没有时间做基础设施改进
- 人员流动导致知识断层，新人不了解系统的历史设计
- 跨团队协作机制缺失，导致变更影响无法评估

组织性根因的可怕之处在于，它不会直接导致故障，但它会让故障发生的概率大大增加，而且很难通过技术手段根除。

## 根因分析的常见偏差

分析方法本身也会出错。以下是几种最常见的分析偏差：

### 偏差一：确认偏误

先入为主地认定了一个原因，然后只收集支持这个原因的证据，忽略反对的证据。

比如，团队已经怀疑是缓存的问题，所以复盘时只关注缓存相关的日志和监控，而忽略了数据库慢查询。实际上，缓存命中率下降恰恰是数据库慢导致的，而不是反过来。

**对策**：在确定根因之前，先列出所有可能的假设，然后逐一证伪，而不是逐一证实。

### 偏差二：近因效应

只看到故障发生前的最后一个事件，忽略了前面已经存在的问题。

比如，某个系统凌晨 2 点告警，故障前的最后一个事件是「定时备份任务启动」。但实际上，数据库的连接泄漏问题已经存在了一个星期，只是备份任务恰好成为了压垮系统的最后一根稻草。

**对策**：从故障发生的时间点往前查，至少回溯到「上一个业务高峰期」，而不是只看到故障前的最后一个事件。

### 偏差三：责任归咎

把问题归咎到个人，而不是分析系统和流程的缺陷。

「这是张三的失误」——这句话没有价值。真正有价值的问题是：「为什么张三会犯这个失误？系统的哪个环节让这个失误变得可能？」

**对策**：把「谁」的问题换成「系统哪里」的问题。分析的对象是系统设计和流程，而不是人。

### 偏差四：幸存者偏差

只分析发生了故障的情况，而没有分析为什么其他类似情况没有发生故障。

比如，团队发现每次定时任务重叠的时候都会出问题，但忽略了另一个事实：过去一年里，定时任务重叠的次数超过 20 次，只有 3 次真正触发了故障。真正的问题不在于「任务重叠」，而在于「任务重叠 + 其他条件同时满足」。

**对策**：对比分析，找出「故障发生」与「故障未发生」之间的差异点。

## 分析工具与数据收集

根因分析不是凭空猜测，而是基于数据的推理。在分析之前，需要收集以下几类数据：

### 日志数据

故障发生时，最重要的数据是**错误日志和时间戳**。

```bash
# 收集故障时间窗口内的 ERROR 日志
grep "ERROR" application.log \
  | awk '$1 >= "2026-04-05 02:55:00" && $1 <= "2026-04-05 03:10:00"' \
  > /tmp/incident-error.log

# 统计错误类型分布
awk '{print $NF}' /tmp/incident-error.log | sort | uniq -c | sort -rn
```

### 监控数据

```
需要重点关注的时间序列数据：
- CPU 使用率（是否有突增）
- 内存使用率（是否有持续增长）
- 数据库连接池使用率
- QPS 和响应时间（p50/p95/p99）
- 错误率（是否有突增）
```

### 链路追踪数据

如果系统接入了链路追踪（如 SkyWalking、Zipkin），需要找出故障时间窗口内的**慢请求和错误请求**，还原完整的调用链路。

```
慢请求特征分析：
- 请求入口是什么（哪个 API）
- 哪个环节耗时最长（数据库、缓存、外部服务）
- 是否有跨服务调用失败
```

### 配置变更记录

很多故障的触发点是一次配置变更。查清楚故障前 24 小时内有哪些配置修改，比查代码更有价值。

```bash
# 查看最近一次配置变更
git log --oneline -20 --since="2 days ago" -- config/

# 导出最近修改的配置文件
git diff HEAD~5 --name-only | grep -E '\.(yaml|properties|json|conf)$'
```

## 总结

回到最初的问题：「为什么故障总在凌晨发生？」

不是因为运维工程师特别倒霉，也不是因为凌晨的系统特别脆弱。真正的原因是：**凌晨是定时任务、数据归档、版本发布的天然时间窗口，多个变更和任务在这个窗口内叠加，它们的交互效应在没有充分评估的情况下，成为了故障的温床。**

所以，解决「凌晨故障」的方法，不只是加强值班，而是：

1. **建立定时任务的统一编排机制**：所有定时任务的执行时间窗口应该有全局视图，避免任务重叠
2. **对定时任务做性能基准线**：每个任务在执行前后应该记录执行时长，超过基准线自动告警
3. **重大任务做灰度验证**：数据归档这类高风险任务，先在非高峰期验证执行效率
4. **建立变更评审机制**：不只是代码变更需要评审，定时任务的新增和修改也需要评审

根因分析的核心，是学会**追问而不停留**。问一次不够，问两次还不够，要一直问到「这个问题应该由谁负责、有没有机制可以预防」为止。

下一节，我们从「预防」的角度，来看一个团队是怎么花了一年时间，把故障发生率降低 80% 的。
