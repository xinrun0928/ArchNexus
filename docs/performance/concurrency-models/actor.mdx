# Actor 模型

你的系统要处理大量并发状态变更，但多线程共享状态让你焦头烂额。

问题出在哪？**共享可变状态是并发编程的万恶之源**。

Actor 模型正是解决这个问题的方案：**每个 Actor 有自己的状态，消息传递代替共享内存**。

## Actor 模型的核心概念

### Actor vs 线程

```
┌─────────────────────────────────────────────────────────────────┐
│                    Actor vs 线程                                 │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │ 线程                                                       │   │
│  │ - 共享堆内存                                              │   │

│  │ Actor                                                      │   │
│  │ - 独立状态，无共享可变状态                                │   │
│  │ - 消息传递代替共享内存                                    │   │
```

## Actor 通信模式

### 消息传递

```java
// Actor 消息
public record OrderMessage(Long orderId, String action, Object payload) {}
public record OrderResponse(boolean success, Object result, String error) {}

// Actor 定义
public class OrderActor extends AbstractActor {
    private OrderState state = new OrderState();

    @Override
    public Receive createReceive() {
        return receiveBuilder()
            .match(OrderMessage.class, this::handleMessage)
            .matchEquals("GET_STATE", msg -> getSender().tell(state, getSelf()))
            .build();
    }

    private void handleMessage(OrderMessage msg) {
        switch (msg.action()) {
            case "CREATE" -> state = state.createOrder(msg.orderId(), msg.payload());
            case "UPDATE" -> state = state.updateOrder(msg.orderId(), msg.payload());
            case "CANCEL" -> state = state.cancelOrder(msg.orderId());
        }
    }
}
```

### 监督策略

```java
// 监督策略
public class SupervisorActor extends AbstractActor {
    @Override
    public SupervisorStrategy supervisorStrategy() {
        return new OneForOneStrategy(
            3, Duration.ofMinutes(1),
            Decider.match(OutOfMemoryError.class, e -> SupervisorStrategy.resume())
                .match(IllegalArgumentException.class, e -> SupervisorStrategy.restart())
                .match(Exception.class, e -> SupervisorStrategy.stop())
                .build());
    }

    @Override
    public Receive createReceive() {
        return receiveBuilder()
            .match(ActorRef.class, this::forwardToChild)
            .build();
    }

    private void forwardToChild(ActorRef child) {
        child.forward(getCurrentMessage(), getContext());
    }
}
```

## Akka Actor 示例

### 订单服务

```java
// 订单 Actor
public class OrderActor extends AbstractActorWithTimers {
    private Order order;

    @Override
    public Receive createReceive() {
        return receiveBuilder()
            .match(CreateOrder.class, this::createOrder)
            .match(CancelOrder.class, this::cancelOrder)
            .match(ProcessPayment.class, this::processPayment)
            .build();
    }

    private void createOrder(CreateOrder cmd) {
        order = new Order(cmd.orderId(), cmd.userId(), cmd.amount());
        getSender().tell(new OrderCreated(order), getSelf());
    }

    private void cancelOrder(CancelOrder cmd) {
        if (order != null) {
            order.cancel();
            getSender().tell(new OrderCancelled(order), getSelf());
        }
    }
}

// 工厂
public class OrderActorFactory {
    public static Props props() {
        return Props.create(OrderActor.class);
    }
}

// 创建 Actor
ActorSystem system = ActorSystem.create("order-system");
ActorRef orderActor = system.actorOf(OrderActorFactory.props(), "order-1");

// 发送消息
orderActor.tell(new CreateOrder(id, userId, amount), ActorRef.noSender());

// 询问并等待响应
AskableActorRef askable = AskableActorRef.of(orderActor, Duration.ofSeconds(5));
OrderResponse response = askable.ask(OrderMessage.class).get(5, TimeUnit.SECONDS);
```

### 集群分片

```java
// Cluster Sharding
public class ClusterShardingExample {
    public static void main(String[] args) {
        ClusterSharding sharding = ClusterSharding.get(system);

        ShardRegion shardRegion = ShardRegion.create(
            OrderActor.props(),
            "orders",
            messageExtractor(),
            ClusterShardingSettings.create(system),
            Option.empty()
        );
    }

    // 消息路由器
    private static MessageExtractor<ShardingEnvelope, OrderMessage> messageExtractor() {
        return new MessageExtractor<ShardingEnvelope, OrderMessage>() {
            @Override
            public String entityId(ShardingEnvelope msg) {
                return msg.orderId().toString();
            }

            @Override
            public String shardId(ShardingEnvelope msg) {
                long shardId = Math.abs(msg.orderId().hashCode() % 10);
                return String.valueOf(shardId);
            }
        };
    }
}
```

## 总结

Actor 模型是处理并发状态的有效方案。

**核心概念**：

- 每个 Actor 有自己的私有状态
- Actor 之间通过消息传递通信
- Actor 逐条处理消息，无并发竞争
- 支持层级结构和监督策略

**优势**：

- 无共享可变状态
- 轻量级，可创建百万级 Actor
- 容错机制（监督策略）
- 位置透明

**适用场景**：

- 高并发状态管理
- 分布式计算
- 事件驱动架构

**不适用场景**：

- CPU 密集型计算
- 简单 CRUD
- 团队不熟悉的场景
