# 吞吐量与带宽

你的服务器 CPU 使用率只有 30%，但 QPS 已经上不去了。

问题出在哪？**CPU 不是瓶颈**，可能是网络带宽、IO 或者数据库连接池。

理解吞吐量和带宽，是找到系统瓶颈的关键。

## 基本概念

### 吞吐量（Throughput）

```
定义：单位时间内系统处理的请求数量

常用单位：
- QPS（Queries Per Second）：每秒查询数
- TPS（Transactions Per Second）：每秒事务数
- RPS（Requests Per Second）：每秒请求数
- FPS（Frames Per Second）：每秒帧数

影响因素：
- CPU 处理能力
- IO 速度
- 并发能力
- 网络带宽
```

### 带宽（Bandwidth）

```
定义：单位时间内能传输的数据量

常用单位：
- bps（bits per second）
- Bps（Bytes per second）
- Mbps（Mega bps）
- Gbps（Giga bps）

关系：1 Byte = 8 bits
- 100 Mbps ≈ 12.5 MB/s
- 1 Gbps ≈ 125 MB/s
```

### 吞吐量与带宽的关系

```
┌─────────────────────────────────────────────────────────────────┐
│                    吞吐量 vs 带宽                                 │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  带宽：管道的粗细                                               │
│  吞吐量：实际通过的水量                                        │
│                                                                  │
│  带宽 100Mbps，系统处理能力 10Gbps → 吞吐受限于带宽             │
│  带宽 10Gbps，系统处理能力 100Mbps → 吞吐受限于处理能力         │
│                                                                  │
│  示例：                                                         │
│  接口响应体：1KB                                                 │
│  带宽：100Mbps = 12.5MB/s                                       │
│  最大 QPS：12.5MB / 1KB = 12,500 QPS                           │
│                                                                  │
│  如果系统处理能力 > 12,500 QPS                                   │
│  → 瓶颈在带宽                                                  │
│  如果系统处理能力 < 12,500 QPS                                   │
│  → 瓶颈在处理能力                                              │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

## 吞吐量的度量

### 计算公式

```java
// 吞吐量计算
public class ThroughputCalculator {

    // 基础公式
    // QPS = 并发数 / 平均响应时间
    public double calculateQPS(int concurrency, double avgLatencyMs) {
        return concurrency / (avgLatencyMs / 1000.0);
    }

    // 示例
    // 线程数：100
    // 平均响应时间：50ms
    // QPS = 100 / 0.05 = 2000
}
```

### Little 定律

```
┌─────────────────────────────────────────────────────────────────┐
│                    Little 定律                                   │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  L = λ × W                                                     │
│                                                                  │
│  L：系统中平均请求数（队列长度 + 正在处理的请求）               │
│  λ：到达率（每秒请求数）                                        │
│  W：平均处理时间（响应时间）                                     │
│                                                                  │
│  示例：                                                         │
│  - 平均队列长度 = 10                                            │
│  - 平均响应时间 = 100ms = 0.1s                                  │
│  - 系统中的请求数 = 10                                          │
│  - 到达率 = L / W = 10 / 0.1 = 100 QPS                         │
│                                                                  │
│  推论：                                                         │
│  - 想要提高 QPS → 减少 W（降低延迟）或增加 L（增加并发）        │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

### 瓶颈识别

```java
// 瓶颈识别工具
public class BottleneckDetector {
    public String identifyBottleneck(Metrics metrics) {
        double cpuUtil = metrics.getCpuUtilization();
        double ioWait = metrics.getIoWait();
        double memoryUtil = metrics.getMemoryUtilization();

        // CPU 瓶颈
        if (cpuUtil > 80 && ioWait < 10) {
            return "CPU_BOUND: 计算密集型，考虑算法优化或并行化";
        }

        // IO 瓶颈
        if (ioWait > 30) {
            return "IO_BOUND: IO 等待严重，考虑异步 IO 或增加缓存";
        }

        // 内存瓶颈
        if (memoryUtil > 85) {
            return "MEMORY_BOUND: 内存压力大，考虑减少对象分配或增加内存";
        }

        // 网络瓶颈
        if (metrics.getNetworkUtilization() > 70) {
            return "NETWORK_BOUND: 网络带宽瓶颈，考虑压缩或减少数据传输";
        }

        return "UNKNOWN";
    }
}
```

## 吞吐量优化

### 批量操作

```java
// 批量操作提升吞吐量
@Service
public class BatchOptimization {

    // 错误：逐条插入
    public void insertOrdersSlow(List<Order> orders) {
        for (Order order : orders) {
            jdbcTemplate.update("INSERT INTO orders VALUES (?, ?, ?)",
                order.getId(), order.getUserId(), order.getAmount());
        }
    }

    // 正确：批量插入
    public void insertOrdersFast(List<Order> orders) {
        jdbcTemplate.batchUpdate("INSERT INTO orders VALUES (?, ?, ?)",
            orders, orders.size(),
            (ps, order) -> {
                ps.setLong(1, order.getId());
                ps.setLong(2, order.getUserId());
                ps.setBigDecimal(3, order.getAmount());
            });
    }
}
```

### 异步处理

```java
// 异步处理提升吞吐量
@Service
public class AsyncOptimization {
    @Autowired
    private KafkaTemplate<String, Object> kafkaTemplate;

    // 同步：每个请求等待处理完成
    public void syncProcess(Order order) {
        processOrder(order);      // 10ms
        sendNotification(order); // 5ms
        updateAnalytics(order);  // 3ms
        // 总计：18ms × QPS = 55
    }

    // 异步：非核心操作异步化
    public void asyncProcess(Order order) {
        // 核心流程同步
        processOrder(order);      // 10ms

        // 非核心流程异步
        CompletableFuture.runAsync(() -> sendNotification(order));  // 异步
        CompletableFuture.runAsync(() -> updateAnalytics(order));    // 异步

        // 总计：10ms × QPS = 100
    }
}
```

### 并行化

```java
// 并行查询提升吞吐量
@Service
public class ParallelQueryOptimization {

    // 串行查询
    public UserDetail getUserDetailSlow(Long userId) {
        User user = userDao.findUser(userId);           // 5ms
        List<Order> orders = orderDao.findOrders(userId); // 10ms
        List<Favorite> favorites = favoriteDao.findFavorites(userId); // 8ms
        return new UserDetail(user, orders, favorites);
    }

    // 并行查询
    public UserDetail getUserDetailFast(Long userId) {
        CompletableFuture<User> userFuture = CompletableFuture
            .supplyAsync(() -> userDao.findUser(userId));
        CompletableFuture<List<Order>> ordersFuture = CompletableFuture
            .supplyAsync(() -> orderDao.findOrders(userId));
        CompletableFuture<List<Favorite>> favoritesFuture = CompletableFuture
            .supplyAsync(() -> favoriteDao.findFavorites(userId));

        CompletableFuture.allOf(userFuture, ordersFuture, favoritesFuture).join();

        return new UserDetail(
            userFuture.join(),
            ordersFuture.join(),
            favoritesFuture.join()
        );
        // 总延迟：max(5, 10, 8) = 10ms
    }
}
```

## 总结

吞吐量和带宽是衡量系统处理能力的关键指标。

**核心要点**：

- 吞吐量：QPS = 并发数 / 响应时间
- 带宽：单位时间传输的数据量
- Little 定律：`L = λ × W`

**常见瓶颈**：

- CPU 瓶颈
- 内存瓶颈
- IO 瓶颈
- 锁瓶颈
- 连接池瓶颈

**优化方向**：

- 批量操作
- 异步处理
- 并行化
- 减少锁竞争
- 优化算法
