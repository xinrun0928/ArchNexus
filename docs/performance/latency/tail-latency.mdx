# 长尾延迟问题

99% 的用户请求都很快，只有 1% 慢得像蜗牛。但你的系统有 1 亿用户——1% 就是 100 万用户。

这就是长尾延迟问题：**少数请求的极端延迟，伤害了用户体验**。

## 长尾延迟的成因

## 长尾延迟的解决方案

### 方案一：消除共享（Sharding）

```
┌─────────────────────────────────────────────────────────────────┐
│                    消除共享减少锁竞争                            │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  错误：全局共享计数器                                          │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │ synchronized + AtomicLong                              │   │
│  │ 多个线程竞争同一把锁                                    │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                  │
│  正确：分片计数器                                             │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │ ConcurrentHashMap<Long, AtomicLong>                    │   │
│  │ 每个线程操作自己的分片                                   │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

```java
// 分片计数器减少锁竞争
public class ShardedCounter {
    private final AtomicLong[] counters;
    private final int shardCount;

    public ShardedCounter(int shardCount) {
        this.shardCount = shardCount;
        this.counters = new AtomicLong[shardCount];
        for (int i = 0; i < shardCount; i++) {
            counters[i] = new AtomicLong(0);
        }
    }

    public void increment() {
        // 分片键：线程 ID
        int shard = (int) (Thread.currentThread().getId() % shardCount);
        counters[shard].incrementAndGet();
    }

    public long sum() {
        return Arrays.stream(counters)
            .mapToLong(AtomicLong::get)
            .sum();
    }
}
```

### 方案二：读写分离

```
┌─────────────────────────────────────────────────────────────────┐
│                    读写分离减少长尾延迟                            │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  写：强一致节点                                                │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │ Primary DB ──同步──▶ Replica DB1, DB2, DB3          │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                  │
│  读：弱一致节点（负载均衡到任意副本）                          │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │ LB ──▶ Replica 1 (正常: 5ms)                         │   │
│  │ LB ──▶ Replica 2 (慢: 200ms) ← 长尾！                │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                  │
│  问题：慢副本拉低平均值                                        │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

```java
// 读取超时策略
@Service
public class ReadTimeoutStrategy {
    private final List<DataSource> replicas;

    // 方案一：跳过慢副本
    public Connection getConnection() {
        List<DataSource> healthy = replicas.stream()
            .filter(this::isHealthy)
            .collect(Collectors.toList());

        // 随机选择，跳过慢副本
        return healthy.get(ThreadLocalRandom.current().nextInt(healthy.size()))
            .getConnection();
    }

    // 方案二：超时快速失败 + 重试
    public <T> T queryWithRetry(String sql) {
        for (int i = 0; i < 3; i++) {
            try {
                return query(sql, 100);  // 超时 100ms
            } catch (TimeoutException e) {
                log.warn("查询超时，重试: {}", i);
                replicas.remove(getCurrentReplica());  // 临时跳过
            }
        }
        // 最后尝试主库
        return queryFromPrimary(sql);
    }
}
```

### 方案三：hedged requests

```
┌─────────────────────────────────────────────────────────────────┐
│                    Hedged Requests 策略                          │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  原理：发起多个相同请求，取最快结果                            │
│                                                                  │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │ Request A ──▶ Replica 1 ── 等待 [T] ──▶ 返回 200ms   │   │
│  │ Request B ──▶ Replica 2 ── 等待 [T] ──▶ 返回 5ms ← 先到！│
│  └─────────────────────────────────────────────────────────┘   │
│                                                                  │
│  延迟：从 200ms 降到 5ms+T                                    │
│  代价：多消耗 1 倍资源                                        │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

```java
// Hedged Requests 实现
@Service
public class HedgedRequestService {
    private final RestTemplate restTemplate = new RestTemplate();

    public String hedgedGet(String url) {
        // 同时发起两个请求
        CompletableFuture<String> future1 = CompletableFuture
            .supplyAsync(() -> restTemplate.getForObject(url, String.class));
        CompletableFuture<String> future2 = CompletableFuture
            .supplyAsync(() -> restTemplate.getForObject(url, String.class));

        try {
            // 取最快结果
            return CompletableFuture.anyOf(future1, future2).get();
        } catch (Exception e) {
            throw new RuntimeException(e);
        }
    }

    // 优化：等待一定时间后再发第二个请求
    public String hedgedGetWithDelay(String url, long delayMs) {
        CompletableFuture<String> first = CompletableFuture
            .supplyAsync(() -> restTemplate.getForObject(url, String.class));

        // 延迟后发第二个请求
        CompletableFuture<String> second = CompletableFuture
            .supplyAsync(() -> {
                try {
                    Thread.sleep(delayMs);
                    return restTemplate.getForObject(url, String.class);
                } catch (Exception e) {
                    return null;
                }
            });

        return first.applyToEither(second, v -> v).join();
    }
}
```

```
│  - 长尾请求的特征（哪个接口、哪个用户、哪个节点）               │

│  监控告警：                                                    │
│  - P99 延迟 > 500ms 告警                                     │

│  - 长尾请求占比 > 5% 告警                                     │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

```java
// 长尾请求监控
@Service
public class TailLatencyMonitor {
    private final MeterRegistry meterRegistry;

    public TailLatencyMonitor(MeterRegistry registry) {
        // 分布统计：延迟直方图
        DistributionSummary.builder("request.latency")
            .publishPercentiles(0.5, 0.9, 0.95, 0.99, 0.999)
            .publishPercentileHistogram()
            .register(registry);
    }

    public void recordLatency(long latencyMs, String endpoint) {
        Timer.builder("request.latency")
            .tag("endpoint", endpoint)
            .register(meterRegistry)
            .record(latencyMs, TimeUnit.MILLISECONDS);
    }
}
```

## 总结

长尾延迟是分布式系统的本质问题。

**成因**：

- GC 停顿
- 锁竞争
- 缓存失效
- 网络抖动

**解决方案**：

| 方案 | 原理 | 代价 |
|---|---|---|
| Sharding | 消除共享，减少锁竞争 | 复杂度增加 |
| 读写分离 | 分散负载 | 可能有数据不一致 |
| Hedged Requests | 多请求竞争，取最快 | 多消耗资源 |
| 就近访问 | 减少网络延迟 | 需要多机房部署 |

**监控重点**：

- P99、P999 延迟
- 长尾请求占比
- 长尾请求特征分析
