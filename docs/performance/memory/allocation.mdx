# 内存分配策略

你的系统内存使用率很高，GC 频繁，但对象真的需要那么多内存吗？

问题出在哪？**没有理解 JVM 的内存分配策略**。

## JVM 内存结构

### 运行时数据区

```

│                    JVM 运行时数据区                               │
```

### TLAB（线程本地分配缓冲）

```java
// TLAB 分配
/*
TLAB = Thread Local Allocation Buffer

原理：
- 每个线程在 Eden 区预先申请一小块空间
- 线程分配对象时，在自己的 TLAB 中分配
- 避免多线程竞争，提高分配效率

参数：
-XX:+UseTLAB              // 启用 TLAB
-XX:TLABSize=512k         // TLAB 大小
-XX:TLABRefillWasteFraction=64  // 允许浪费的比例

适用场景：
- 高并发、多线程分配场景
- 默认已经开启
*/
```

## 内存分配策略

### 年轻代分配

```
┌─────────────────────────────────────────────────────────────────┐
│                    年轻代分配策略                                 │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  Eden : Survivor0 : Survivor1 = 8 : 1 : 1                     │
│                                                                  │
│  对象分配流程：                                                 │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │ 1. 新对象在 Eden 区分配                               │   │
│  │ 2. Minor GC 时，Eden 区存活对象复制到 S0              │   │
│  │ 3. 再次 Minor GC 时，Eden + S0 存活对象复制到 S1      │   │
│  │ 4. 年龄计数器 +1                                      │   │
│  │ 5. 年龄达到阈值（默认 15）→ 晋升到老年代              │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                  │
│  晋升年龄阈值：                                                │
│  -XX:MaxTenuringThreshold=15  // 最大晋升年龄                   │
│                                                                  │
│  动态计算：                                                    │
│  - 如果 S0/S1 相同年龄对象大小 > S0/S1 的 50%                  │
│  - 则大于等于该年龄的对象晋升到老年代                           │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

### 大对象直接进入老年代

```java
// 大对象分配
/*
大对象：需要大量连续内存空间的对象
- 大数组
- 大字符串
- 大的集合

参数：
-XX:PretenureSizeThreshold=1m  // 超过这个大小的对象直接进入老年代

适用场景：
- 避免大对象在年轻代反复 GC
- 避免内存碎片

注意：
- 这个参数只在 Serial 和 ParNew 收集器下有效
- G1 收集器使用 -XX:G1HeapRegionSize 代替
*/

// 运行时判断
/*
HotSpot 源码（伪代码）：
if (对象大小 > PretenureSizeThreshold) {
    在老年代分配
} else if (TLAB 有足够空间) {
    在 TLAB 分配
} else {
    在 Eden 区分配
}
*/
```

### 对象分配示例

```java
public class ObjectAllocation {
    public static void main(String[] args) {
        // 小对象：在 Eden 区分配
        List<String> list = new ArrayList<>();  // 小数组

        // 大对象：直接进入老年代（如果配置了 PretenureSizeThreshold）
        byte[] bigArray = new byte[1024 * 1024 * 2];  // 2MB

        // 长期存活对象：经过多次 Minor GC 后晋升老年代
        // 每次 Minor GC，年龄 +1
        // 达到阈值后晋升
    }
}
```

## 内存分配优化

### 合理设置年轻代大小

```
┌─────────────────────────────────────────────────────────────────┐
│                    年轻代大小设置                                 │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  太小的问题：                                                  │
│  - Minor GC 频繁                                              │
│  - 对象过早晋升到老年代                                        │
│  - Full GC 频繁                                              │
│                                                                  │
│  太大的问题：                                                  │
│  - 老年代空间变小                                              │
│  - Full GC 时间变长                                            │
│  - 影响吞吐量                                                  │
│                                                                  │
│  建议：                                                        │
│  - 年轻代大小 = 总堆的 1/3 到 1/2                             │
│  - 通过 -Xmn 设置                                             │
│  - 监控 GC 日志，调整参数                                      │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

### Survivor 区优化

```java
// Survivor 区优化
/*
参数：
-XX:SurvivorRatio=8  // Eden/Survivor = 8:1

如果 Survivor 区太小：
- 对象容易直接晋升到老年代
- 短期对象在老年代积累

如果 Survivor 区太大：
- 浪费年轻代空间

动态调整：
- 使用 -XX:+UseAdaptiveSizePolicy
- JVM 自动调整 Survivor 区大小

推荐配置：
年轻代较大时，可以增加 Survivor Ratio
年轻代较小时，可以减小 Survivor Ratio
*/

// 监控 Survivor 区使用情况
/*
GC 日志：
eden space 100M, 80% used -> [S0: 10M, 50% used] [S1: 10M, 0% used]

分析：
- S0 使用 50%，说明 S1 足以容纳 Survivor 区的对象
- 如果 S0 接近 100%，说明 Survivor 区太小
*/
```

### 内存分配代码优化

```java
// 内存分配优化示例

// 错误：频繁创建对象
public class BadExample {
    public String process(List<String> data) {
        String result = "";
        for (String item : data) {
            result += item + ",";  // 每一次 += 都会创建新 String
        }
        return result;
    }
}

// 正确：使用 StringBuilder
public class GoodExample {
    public String process(List<String> data) {
        StringBuilder sb = new StringBuilder();
        for (String item : data) {
            sb.append(item).append(",");
        }
        return sb.toString();
    }
}

// 正确：使用集合预分配容量
public class CollectionExample {
    // 错误：默认容量
    List<String> list1 = new ArrayList<>();

    // 正确：预估容量
    List<String> list2 = new ArrayList<>(data.size());
}
```

## 总结

理解 JVM 内存分配策略是性能优化的基础。

**内存结构**：

- 堆：对象分配
- 非堆：元空间、代码缓存
- 栈：线程私有

**分配策略**：

- TLAB：线程本地分配
- 大对象直接进老年代
- 长期存活对象晋升老年代

**优化方向**：

- 合理设置堆大小
- 合理设置年轻代比例
- 监控 Survivor 区使用
- 减少对象分配
