# 垃圾回收原理

你的系统经常发生 Full GC，导致系统卡顿。

问题出在哪？**没有理解垃圾回收的原理和收集器的选择**。

## 垃圾回收算法

### 引用计数法

### G1 收集器

```java
// G1 收集器
/*
G1（Garbage-First）特点：
- 堆被划分为多个大小相等的 Region
- 每个 Region 可以是 Eden、Survivor 或 Old
- 采用标记-整理算法
- 可预测的停顿时间模型

参数配置：
-XX:+UseG1GC                      // 启用 G1
-XX:MaxGCPauseMillis=200         // 最大停顿时间目标
-XX:G1HeapRegionSize=1m          // Region 大小（1MB-32MB）
-XX:InitiatingHeapOccupancyPercent=45  // 触发 GC 的堆使用率

GC 流程：
1. 初始标记（STW）：标记 GC Roots 直接引用的对象
2. 并发标记：遍历对象图
3. 最终标记（STW）：处理 SATB 记录的变动
4. 筛选回收：计算各 Region 的回收价值，按价值排序
*/

// G1 GC 日志分析
/*
[GC pause (G1 Evacuation Pause) (young)
  // GC 类型：年轻代转移暂停
  // 耗时：10.3ms
  // Eden: 100M(120M) -> 0M(120M)
  // Survivor: 10M(20M) -> 20M(20M)
  // Old: 200M(800M) -> 200M(800M)
```

### ZGC 收集器

```java
// ZGC 收集器（JDK 11+）
/*
ZGC（Z Garbage Collector）特点：
- 停顿时间不超过 10ms
- 停顿时间不随堆大小增加而增加
- 支持 TB 级堆内存（8MB - 16TB）
- 基于染色指针（Colored Pointers）

参数配置：
-XX:+UseZGC                      // 启用 ZGC
-XX:MaxGCPauseMillis=10         // 最大停顿时间
-XX:+UseLargePages               // 使用大页

工作原理：
┌─────────────────────────────────────────────────────────────┐
│  ZGC 三色标记                                              │
│  白（White）：未被访问的对象                              │
│  灰（Gray）：已访问但子节点未处理                         │
│  黑（Black）：已访问且子节点已处理                        │
│                                                             │
│  GC 过程：                                                 │
│  1. 初始标记（STW）：标记 GC Roots 直接引用的对象为灰色   │
│  2. 并发标记：遍历灰色对象，直到没有灰色对象              │
│  3. 再标记（STW）：处理并发标记期间变化的对象             │
│  4. 并发重定位：移动对象，更新引用                        │
└─────────────────────────────────────────────────────────────┘
*/
```

## 总结

理解垃圾回收原理是 JVM 调优的基础。

**核心算法**：

- 标记-清除：产生内存碎片
- 标记-整理：无碎片，需要移动对象
- 复制：无碎片，浪费空间

**收集器选择**：

| 收集器 | 停顿时间 | 吞吐量 | 适用场景 |
|---|---|---|---|
| Serial | 长 | 低 | 单核 |
| Parallel | 长 | 高 | 吞吐量优先 |
| CMS | 短 | 中 | 低延迟 |
| G1 | 可控 | 中 | 通用 |
| ZGC | 极短 | 高 | 超大内存 |

**优化策略**：

- 监控 GC 日志
- 选择合适的收集器
- 合理设置堆大小
- 优化对象生命周期
