# 内存泄漏排查

你的系统运行几天后就开始 Full GC，然后 OOM 崩溃。

问题出在哪？**内存泄漏在悄悄积累**。

## 内存泄漏的本质

### 内存泄漏 vs 内存溢出

```
┌─────────────────────────────────────────────────────────────────┐
│                    内存泄漏 vs 内存溢出                          │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  内存泄漏（Memory Leak）：                                      │
│  - 对象不再使用，但无法被 GC 回收                              │
│  - 原因：对象被错误引用，阻止了回收                            │
│  - 表现：内存持续增长，GC 后不释放                            │
│                                                                  │
│  内存溢出（OOM）：                                            │
│  - 无法分配足够的内存                                          │
│  - 原因：内存不足或内存泄漏                                    │
│  - 表现：程序崩溃                                             │
│                                                                  │
│  关系：                                                        │
│  内存泄漏 → 内存持续增长 → 最终 OOM                          │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

### 内存泄漏的常见原因

```
┌─────────────────────────────────────────────────────────────────┐
│                    内存泄漏常见原因                               │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  1. 集合类泄漏                                                │
│     - List/Map/Set 只添加不删除                               │
│     - 定时任务不断添加数据                                     │
│                                                                  │
│  2. 静态集合类                                                │
│     - 静态字段持有大量对象                                     │
│     - 单例模式持有大量数据                                     │
│                                                                  │
│  3. 监听器未注销                                              │
│     - Swing/awt 事件监听器                                    │
│     - Spring 事件监听器                                        │
│                                                                  │
│  4. 连接未关闭                                                │
│     - 数据库连接                                              │
│     - 网络连接                                                │
│     - 文件流                                                  │
│                                                                  │
│  5. ThreadLocal 未清理                                        │
│     - 线程池复用时 ThreadLocal 未清理                        │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

## 排查工具

### jmap 堆内存分析

```bash
# 查看堆内存使用
jmap -heap <pid>

# 生成堆转储文件（heapdump）
jmap -dump:format=b,file=heap.hprof <pid>

# 展示对象统计信息
jmap -histo <pid>

# 按实例数量排序
jmap -histo <pid> | head -100

# 按内存大小排序
jmap -histo <pid> | sort -k 3 -n -r | head -100
```

### MAT（Memory Analyzer Tool）

```bash
# 生成 heapdump 后，使用 MAT 分析
# 下载地址：https://eclipse.dev/mat/downloads

# 常用功能：
# 1. Histogram：查看对象数量和大小
# 2. Dominator Tree：查看支配树
# 3. Leak Suspects：自动检测内存泄漏
# 4. Top Consumers：查找最大的对象
```

### VisualVM

```bash
# 启动 VisualVM
jvisualvm

# 功能：
# - 监控 CPU、内存、线程
# - 堆转储分析
# - GC 分析
# - 线程分析
```

### Arthas

```java
// Arthas 常用命令
// 1. 查看内存使用
dashboard

// 2. 生成 heapdump
heapdump /tmp/heap.hprof

// 3. 查看对象统计
sc -d <class-name>

// 4. 查看类加载器
classloader

// 5. 监控内存变化
memory
```

## 常见泄漏场景与解决

### 场景一：静态集合类泄漏

```java
// 错误：静态集合类持有大量对象
public class StaticCollectionLeak {
    // 静态 List 不断增长
    private static final List<byte[]> CACHE = new ArrayList<>();

    public void addToCache() {
        // 每次调用都添加 1MB
        CACHE.add(new byte[1024 * 1024]);
    }
}

// 正确：使用 WeakHashMap
public class FixedStaticCollectionLeak {
    // WeakHashMap 在内存不足时会自动回收
    private static final Map<String, byte[]> CACHE = new WeakHashMap<>();

    public void addToCache(String key, byte[] value) {
        CACHE.put(key, value);
    }
}

// 正确：添加容量限制
public class BoundedCache {
    private static final int MAX_SIZE = 1000;
    private static final LinkedList<byte[]> CACHE = new LinkedList<>();

    public void addToCache(byte[] value) {
        if (CACHE.size() >= MAX_SIZE) {
            CACHE.removeFirst();  // 移除最老的
        }
        CACHE.addLast(value);
    }
}
```

### 场景二：ThreadLocal 泄漏

```java
// 错误：ThreadLocal 未清理
public class ThreadLocalLeak {
    private static final ThreadLocal<byte[]> LOCAL_CACHE = new ThreadLocal<>();

    public void process() {
        LOCAL_CACHE.set(new byte[1024 * 1024]);
        // 方法结束，但 ThreadLocal 未清理
    }
}

// 正确：使用完及时清理
public class FixedThreadLocal {
    private static final ThreadLocal<byte[]> LOCAL_CACHE = ThreadLocal.withInitial(
        () -> new byte[1024 * 1024]);

    public void process() {
        try {
            byte[] cache = LOCAL_CACHE.get();
            // 使用 cache
        } finally {
            LOCAL_CACHE.remove();  // 清理
        }
    }
}

// 正确：使用 InheritableThreadLocal 时注意
public class SafeInheritableThreadLocal {
    // 如果不需要子线程继承，优先使用普通 ThreadLocal
    private static final ThreadLocal<UserContext> CONTEXT = new ThreadLocal<>();
}
```

### 场景三：数据库连接泄漏

```java
// 错误：连接未关闭
public class ConnectionLeak {
    private DataSource dataSource;

    public User getUser(Long id) {
        Connection conn = null;
        try {
            conn = dataSource.getConnection();
            // 使用连接
            return query(conn, id);
            // 异常时连接未关闭！
        } catch (SQLException e) {
            throw new RuntimeException(e);
        }
        // 没有 finally { conn.close(); }
    }
}

// 正确：使用 try-with-resources
public class FixedConnectionLeak {
    private DataSource dataSource;

    public User getUser(Long id) {
        // 自动关闭
        try (Connection conn = dataSource.getConnection()) {
            return query(conn, id);
        } catch (SQLException e) {
            throw new RuntimeException(e);
        }
    }
}

// 正确：使用连接池 + 合理配置
@Configuration
public class DataSourceConfig {
    @Bean
    public DataSource dataSource() {
        HikariConfig config = new HikariConfig();
        config.setMaximumPoolSize(50);
        config.setMinimumIdle(10);
        config.setConnectionTimeout(30000);
        config.setIdleTimeout(600000);
        // 关键：启用连接验证
        config.setConnectionTestQuery("SELECT 1");
        return new HikariDataSource(config);
    }
}
```

### 场景四：监听器未注销

```java
// 错误：监听器未注销
public class ListenerLeak {
    private final EventBus eventBus = new EventBus();

    public void register() {
        // 每次调用都注册新监听器
        eventBus.register(new OrderListener());
    }
}

// 正确：注册时保存引用，注销时移除
public class FixedListenerLeak {
    private final EventBus eventBus = new EventBus();
    private final OrderListener orderListener = new OrderListener();

    public void register() {
        eventBus.register(orderListener);
    }

    public void unregister() {
        eventBus.unregister(orderListener);
    }
}

// Spring 事件监听器
@Service
public class SpringListenerExample {
    @Autowired
    private ApplicationEventPublisher publisher;

    // Spring 监听器无需手动注销
    // 但注意避免内部类持有外部引用
    @EventListener
    public void handleOrderCreated(OrderCreatedEvent event) {
        // 直接使用，不持有外部引用
    }

    // 如果需要持有外部引用，使用弱引用
    private static class WeakListener implements ApplicationListener<OrderCreatedEvent> {
        private final WeakReference<OuterClass> outerRef;

        public WeakListener(OuterClass outer) {
            this.outerRef = new WeakReference<>(outer);
        }

        @Override
        public void onApplicationEvent(OrderCreatedEvent event) {
            OuterClass outer = outerRef.get();
            if (outer != null) {
                outer.process(event);
            }
        }
    }
}
```

## 排查流程

### 步骤一：确认内存泄漏

```bash
# 1. 监控内存使用趋势
# 观察内存是否持续增长，GC 后是否释放

# 2. 查看 GC 日志
# 如果 Full GC 后堆内存使用量仍然很高，说明有问题

# 3. 生成多次 heapdump，对比
jmap -dump:format=b,file=heap1.hprof <pid>
# 等待一段时间
jmap -dump:format=b,file=heap2.hprof <pid>

# 4. 对比两次 dump 中对象数量变化
```

### 步骤二：定位泄漏点

```
┌─────────────────────────────────────────────────────────────────┐
│                    内存泄漏定位流程                               │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  1. 生成 heapdump                                              │
│  2. 使用 MAT 分析                                              │
│  3. 查看 Histogram，按对象大小排序                              │
│  4. 找到可疑对象（如 byte[]、HashMap、List）                 │
│  5. 查看 GC Roots 引用路径                                      │
│  6. 定位代码                                                  │
│                                                                  │
│  MAT 关键功能：                                                │
│  - Path to GC Roots：找到是什么阻止对象回收                   │
│  - Merge Shortest Paths：合并到 GC Roots 的最短路径          │
│  - Group by value：按值分组查看                               │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

### 步骤三：修复与验证

```bash
# 修复后，持续监控内存使用
# 观察：
# 1. 内存是否稳定
# 2. GC 频率是否降低
# 3. 堆使用量是否合理
```

## 总结

内存泄漏排查需要系统化的方法。

**常见原因**：

- 静态集合类
- ThreadLocal 未清理
- 资源未关闭
- 监听器未注销

**排查工具**：

| 工具 | 用途 |
|---|---|
| jmap | 堆转储、对象统计 |
| MAT | 堆分析、泄漏定位 |
| VisualVM | 监控、分析 |
| Arthas | 在线诊断 |

**排查流程**：

1. 确认泄漏（监控趋势）
2. 生成 dump
3. 分析对象引用
4. 定位代码
5. 修复验证
