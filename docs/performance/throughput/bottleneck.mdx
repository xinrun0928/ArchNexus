# 吞吐量瓶颈分析

你的系统 CPU 使用率只有 30%，但 QPS 已经到顶了。

问题出在哪？**CPU 不是瓶颈，真正的瓶颈藏在你没注意到的地方**。

## 吞吐量瓶颈的本质

```
┌─────────────────────────────────────────────────────────────────┐
│                    吞吐量的木桶效应                                │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  ┌───────┐  ┌───────┐  ┌───────┐  ┌───────┐               │
│  │ CPU   │  │ 内存   │  │  IO   │  │ 网络   │               │
│  │  80%  │  │  30%  │  │  90%  │  │  20%  │               │
│  └───────┘  └───────┘  └───────┘  └───────┘               │
│                    │                                           │
│                    ▼                                           │
│            瓶颈 = min(80, 30, 90, 20) = 20%                │
│            吞吐量受限于内存使用率最低的环节                     │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

## 常见瓶颈类型

### 1. CPU 瓶颈

```
┌─────────────────────────────────────────────────────────────────┐
│                    CPU 瓶颈                                      │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  特征：                                                        │
│  - CPU 使用率 > 80%                                            │
│  - user% 高，sys% 正常                                        │
│  - us + sy > 90%                                              │
│                                                                  │
│  原因：                                                        │
│  - 计算密集型任务                                             │
│  - 算法复杂度高                                               │
│  - 正则表达式、回溯等                                          │
│                                                                  │
│  解决方案：                                                    │
│  - 算法优化                                                   │
│  - JIT 编译                                                   │
│  - 并行化                                                      │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

### 2. 内存瓶颈

```
┌─────────────────────────────────────────────────────────────────┐
│                    内存瓶颈                                      │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  特征：                                                        │
│  - 内存使用率 > 85%                                           │
│  - Swap 使用率高（频繁换页）                                  │
│  - GC 频繁                                                    │
│                                                                  │
│  原因：                                                        │
│  - 对象分配过多                                               │
│  - 内存泄漏                                                   │
│  - JVM 堆内存不足                                             │
│                                                                  │
│  解决方案：                                                    │
│  - 减少对象分配                                               │
│  - 对象池复用                                                 │
│  - 选择合适的 GC                                               │
│  - 增加堆内存                                                 │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

### 3. IO 瓶颈

```
┌─────────────────────────────────────────────────────────────────┐
│                    IO 瓶颈                                      │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  特征：                                                        │
│  - iowait > 30%                                               │
│  - 磁盘 util > 80%                                            │
│  - 大量磁盘读写                                                │
│                                                                  │
│  原因：                                                        │
│  - 磁盘读写速度跟不上                                          │
│  - 随机 IO 过多                                               │
│  - 缺少缓存                                                    │
│                                                                  │
│  解决方案：                                                    │
│  - SSD 替代 HDD                                               │
│  - 顺序读写替代随机读写                                        │
│  - 增加缓存减少 IO                                             │
│  - 异步 IO                                                    │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

### 4. 网络瓶颈

```
┌─────────────────────────────────────────────────────────────────┐
│                    网络瓶颈                                      │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  特征：                                                        │
│  - 网络带宽使用率高                                            │
│  - 大量小请求                                                 │
│  - 连接建立频繁                                                │
│                                                                  │
│  原因：                                                        │
│  - 带宽饱和                                                   │
│  - TCP 连接建立开销                                            │
│  - 请求响应体过大                                              │
│                                                                  │
│  解决方案：                                                    │
│  - 数据压缩                                                    │
│  - HTTP/2、WebSocket                                          │
│  - 连接复用                                                   │
│  - 批量请求                                                   │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

## 瓶颈分析方法

### 方法一：监控分析

```bash
# CPU 瓶颈分析
top
# 查看高 CPU 进程
ps aux | sort -k 3 -r | head -10

# 内存瓶颈分析
free -h
# 查看内存泄漏
pmap -x <pid> | sort -k 3 -r | head -10

# IO 瓶颈分析
iostat -x 1
# 查看 IO 最高的进程
iotop

# 网络瓶颈分析
nload
# 查看连接状态
netstat -an | awk '/^tcp/ {++S[$NF]} END {for(a in S) print a, S[a]}'
```

### 方法二：链路分析

```
┌─────────────────────────────────────────────────────────────────┐
│                    瓶颈链路分析                                  │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  请求流程：                                                    │
│  Gateway(5ms) → Auth(10ms) → Order(50ms) → Response(5ms)      │
│                                                                  │
│  各环节占比：                                                 │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │ Gateway  ████░░░░░░░░░░░░░░░░░░░░░  5ms (7%)       │   │
│  │ Auth     ████████████░░░░░░░░░░░░░░░░ 10ms (14%)      │   │
│  │ Order    ██████████████████████████████ 50ms (71%)      │   │
│  │ Response ████░░░░░░░░░░░░░░░░░░░░░░  5ms (7%)       │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                  │
│  结论：Order 服务是瓶颈                                       │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

### 方法三：压测定位

```bash
# 逐步加压，找到瓶颈点
wrk -t4 -c100 -d30s http://example.com/api/orders
# QPS = 1000, CPU = 40%

wrk -t4 -c200 -d30s http://example.com/api/orders
# QPS = 1800, CPU = 70%

wrk -t4 -c400 -d30s http://example.com/api/orders
# QPS = 2000, CPU = 80%

wrk -t4 -c800 -d30s http://example.com/api/orders
# QPS = 2100, CPU = 82%

# 分析：
# QPS 不再增长，CPU 也不再增长
# 但资源还没饱和，说明有其他瓶颈
# 可能是连接池、线程池等
```

## 常见瓶颈场景

### 场景一：数据库连接池瓶颈

```java
// 瓶颈分析
public class ConnectionPoolBottleneck {
    // 线程池：200
    // 数据库连接池：50
    // 线程等待连接时间：100ms
    // 吞吐量受限

    // 瓶颈定位
    // 线程池满载时，只有 50 个线程能获取到连接
    // 其他 150 个线程在等待连接
    // 实际并发 = 50

    // 解决方案：增加连接池
    @Bean
    public DataSource dataSource() {
        HikariConfig config = new HikariConfig();
        config.setMaximumPoolSize(200);  // 与线程池一致
        config.setMinimumIdle(100);
        return new HikariDataSource(config);
    }
}
```

### 场景二：线程池瓶颈

```java
// 瓶颈分析
public class ThreadPoolBottleneck {
    // 请求 QPS：5000
    // 单请求处理时间：20ms
    // 需要的线程数 = 5000 × 0.02 = 100

    // 但线程池只有 50
    // 实际吞吐量 = 50 / 0.02 = 2500 QPS
    // 瓶颈在线程池

    // 解决方案
    @Bean
    public ExecutorService executor() {
        int coreSize = Runtime.getRuntime().availableProcessors() * 2;
        return new ThreadPoolExecutor(
            coreSize,          // 核心线程数
            coreSize * 2,      // 最大线程数
            60, TimeUnit.SECONDS,
            new LinkedBlockingQueue<>(1000),
            new ThreadPoolExecutor.CallerRunsPolicy()
        );
    }
}
```

### 场景三：锁瓶颈

```java
// 瓶颈分析
public class LockBottleneck {
    // 全局锁：所有线程竞争同一把锁
    // 线程数：100
    // 锁等待时间：10ms
    // 吞吐量 = 1000 / (10 + 1) = 91 QPS（理论 1000）

    // 解决方案：分段锁
    private final Striped64 counter = new Striped64();

    // 或使用无锁算法
    private final AtomicLong counter = new AtomicLong(0);
    public void increment() {
        counter.incrementAndGet();
    }
}
```

## 总结

吞吐量瓶颈需要系统分析。

**瓶颈类型**：

| 瓶颈 | 监控指标 | 特征 |
|---|---|---|
| CPU | `top` us+sy > 90% | CPU 使用率高 |
| 内存 | `free` 可用 < 20% | Swap 使用 |
| IO | `iostat` util > 80% | iowait 高 |
| 网络 | `nload` 带宽饱和 | 带宽使用率高 |
| 连接池 | 连接等待 | 线程阻塞在获取连接 |
| 锁 | jstack 大量 BLOCKED | 线程等待锁 |

**分析方法**：

- 监控分析：观察资源使用率
- 链路分析：分解请求链路
- 压测定位：逐步加压找到拐点

**解决思路**：

- 木桶效应：补齐最短板
- 拐点分析：找到扩展极限
- 针对性优化：瓶颈在哪优化哪
