# 性能优化

你的系统在压测时 QPS 只能到 500，但目标是 5000。

问题出在哪？**性能不是靠猜测优化的，而是靠系统化的分析和科学的方法**。

性能优化不是调参游戏，而是一套完整的知识体系。

## 性能指标体系

性能优化首先需要正确度量。

**核心指标**：

- QPS / TPS：系统吞吐量
- 延迟：响应时间
- 并发数：同时处理的请求数
- 资源利用率：CPU、内存、IO、网络

**关系**：`QPS = 并发数 / 平均响应时间`

**监控重点**：P50、P90、P99、P999

## 延迟分析

延迟是用户最能感知的指标。

**延迟来源**：

- 网络延迟（DNS、TCP、SSL）
- IO 延迟（磁盘、网络）
- 计算延迟（CPU、GC）

**长尾问题**：1% 的请求可能拖垮整个系统

**分析方法**：链路追踪、百分位分布、瓶颈定位

## 吞吐量优化

吞吐量是系统处理能力的核心。

**瓶颈分析**：

- CPU 瓶颈
- 内存瓶颈
- IO 瓶颈
- 连接池瓶颈

**优化策略**：

- 连接池优化
- 批处理
- 异步化

## 并发模型

选择合适的并发模型是高性能的基础。

**模型对比**：

- 同步多线程：简单但资源消耗大
- 协程：轻量级并发
- 异步：非阻塞 IO

**Java 选择**：

- JDK 21+：虚拟线程
- 高并发：NIO / Netty
- 生产级：选择成熟的方案

## 内存管理

内存是性能优化的核心领域。

**分配策略**：

- TLAB 分配
- 大对象直接进入老年代
- 对象晋升机制

**垃圾回收**：

- 算法：标记-清除、复制、标记-整理
- 收集器：CMS、G1、ZGC

**优化方向**：

- 减少对象分配
- 选择合适的 GC
- 对象池复用

## I/O 模型

IO 往往是最慢的环节。

**模型演进**：

- BIO：同步阻塞
- NIO：同步非阻塞 + 多路复用
- AIO：异步非阻塞

**核心技术**：

- I/O 多路复用（epoll）
- 零拷贝（sendfile、mmap）
- Direct Buffer

## JIT 编译

JIT 是 Java 性能的秘密武器。

**分层编译**：

- Level 0：解释执行
- Level 1-2：C1 编译
- Level 4：C2 编译

**优化技术**：

- 方法内联
- 逃逸分析
- 热点编译

**最佳实践**：

- JIT 友好的代码
- 合理的预热
- 避免反优化

## 总结

性能优化是一套系统化的知识体系。

**学习路径**：

1. 理解性能指标
2. 掌握分析方法
3. 深入原理层面
4. 积累实践经验

**核心原则**：

- 先测量，后优化
- 找到瓶颈再下手
- 权衡成本和收益
- 持续监控和调优
