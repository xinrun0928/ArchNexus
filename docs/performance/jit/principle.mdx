# JIT 编译原理

你的 Java 程序启动很慢，但运行一段时间后反而越来越快。

问题出在哪？**JIT 编译器在运行时进行优化，把字节码编译成机器码**。

理解 JIT 编译原理，是深度性能优化的基础。

## Java 编译流程

### 从源码到执行

```
┌─────────────────────────────────────────────────────────────────┐
│                    Java 编译流程                                   │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  源码 (.java)                                                    │
│       │                                                         │
│       ▼ javac                                                   │
│  字节码 (.class)                                                │
│       │                                                         │
│       ▼ 类加载器                                                │
│  JVM 内存                                                      │
│       │                                                         │
│       ▼                                                         │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │                    解释执行                            │   │
│  │                    或 JIT 编译                         │   │
│  └─────────────────────────────────────────────────────────┘   │
│       │                                                         │
│       ▼                                                         │
│  机器码                                                        │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

### 解释执行 vs 编译执行

```
┌─────────────────────────────────────────────────────────────────┐
│                    解释执行 vs 编译执行                           │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  解释执行（Interpreter）：                                      │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │ 字节码 ──▶ 解释器逐条翻译执行                          │   │
│  │                                                              │   │
│  │ 启动快，执行慢                                           │   │
│  │ 每条指令都需要解释                                       │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                  │
│  编译执行（JIT）：                                             │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │ 字节码 ──▶ JIT 编译器 ──▶ 机器码 ──▶ 直接执行       │   │
│  │                                                              │   │
│  │ 启动慢（需要编译时间），执行快                          │   │
│  │ 编译一次，多次执行                                       │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                  │
│  JVM 的策略：                                                  │
│  - 启动阶段：解释执行                                          │
│  - 稳定运行后：JIT 编译热点代码                               │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

## JIT 编译器架构

### 分层编译

```
┌─────────────────────────────────────────────────────────────────┐
│                    分层编译架构                                   │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  Tier 0：解释执行                                      │   │
│  │  - 程序启动时的初始阶段                                │   │
│  │  - 收集 profiling 信息                                 │   │
│  │  - 无编译开销                                          │   │
│  └─────────────────────────────────────────────────────────┘   │
│       │                                                         │
│       ▼                                                         │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  Tier 1：C1 编译（client 编译器）                    │   │
│  │  - 快速编译，不做激进优化                              │   │
│  │  - 编译时间短，适合启动阶段                            │   │
│  │  - 轻量级 profiling                                    │   │
│  └─────────────────────────────────────────────────────────┘   │
│       │                                                         │
│       ▼                                                         │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  Tier 2 / Tier 3：C1 编译（更多优化）                 │   │
│  │  - 更多的 profiling 信息                                │   │
│  │  - 稍激进的优化                                        │   │
│  └─────────────────────────────────────────────────────────┘   │
│       │                                                         │
│       ▼                                                         │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  Tier 4：C2 编译（server 编译器）                    │   │
│  │  - 激进优化，可能耗时较长                              │   │
│  │  - 充分利用 profiling 信息                            │   │
│  │  - 生成高度优化的机器码                                │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

## JIT 编译参数

### 基本参数

```bash
# 启用分层编译（JDK 8+ 默认启用）
-XX:+TieredCompilation
-XX:+TieredCompilation

# 禁用分层编译
-XX:-TieredCompilation

# C1 编译器配置
-XX:TieredCompilation

# C2 编译器配置
-XX:+UseOptoAssembly   # C2 汇编输出（调试用）

# 热点检测阈值
-XX:CompileThreshold=1500          # 方法编译阈值
-XX:OnStackReplacePercentage=150   # OSR 阈值

# 分层编译策略
-XX:+UseBamdedTieredCompilation    # 激进分层编译
```

### 监控参数

```bash
# 打印 JIT 编译日志
-XX:+PrintCompilation

# 输出示例：
# 1   java.lang.Object::toString (42 bytes)
#    |    |  |  方法名
#    |    |  参数大小
#    |    方法字节码大小
#    编译ID

# 打印内联日志
-XX:+PrintInlining

# 输出示例：
# @ 3   java.lang.String::charAt (29 bytes)   inline (hot)
# @ 7   java.lang.String::length (6 bytes)     inline (hot)

# 打印 GC 信息
-XX:+PrintGCDetails

# 监控编译队列
-XX:+LogCompilation
-XX:CompileCommand=print,*MyClass::myMethod
```

## JIT 优化技术

### 热点编译优化

```java
// JIT 热点编译优化示例

// 优化前
public class BeforeOptimization {
    public int sum(int[] array) {
        int sum = 0;
        for (int i = 0; i < array.length; i++) {  // [!code highlight]
            sum += array[i];  // [!code highlight]
        }
        return sum;
    }
}

// JIT 优化后（伪代码）
public class AfterOptimization {
    public int sum(int[] array) {
        // 范围检查消除：如果已知 array.length 是固定的
        // 循环可以展开和向量化

        // SIMD 向量化
        // sum = _mm256_add_epi64(sum, _mm256_loadu_si256(array));

        // 循环展开
        // for (int i = 0; i < array.length; i += 4) {
        //     sum += array[i] + array[i+1] + array[i+2] + array[i+3];
        // }
        return sum;
    }
}
```

### 常见优化技术

```
┌─────────────────────────────────────────────────────────────────┐
│                    JIT 常见优化技术                               │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  1. 方法内联（Method Inlining）                                │
│     - 将被调用方法的内容直接嵌入调用处                        │
│     - 减少方法调用开销                                          │
│     - 开启更多其他优化                                          │
│                                                                  │
│  2. 逃逸分析（Escape Analysis）                                │
│     - 分析对象是否逃逸出方法                                   │
│     - 栈上分配：如果对象不逃逸，可以在栈上分配                 │
│     - 标量替换：如果对象不逃逸，可以拆解为成员变量             │
│                                                                  │
│  3. 死代码消除（Dead Code Elimination）                        │
│     - 移除永远不会执行的代码                                   │
│     - 移除计算结果未被使用的代码                               │
│                                                                  │
│  4. 热点编译（Hot Spot Compilation）                          │
│     - 只编译热点代码                                           │
│     - 平衡编译开销和执行效率                                    │
│                                                                  │
│  5. 激进优化（Speculative Optimization）                       │
│     - 基于 profiling 信息的猜测性优化                          │
│     - 可能回退（deoptimization）                              │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

### 方法内联

```java
// 方法内联示例
public class InlineExample {
    // 内联前：两次方法调用
    public int addAndSquare(int a, int b) {
        return square(add(a, b));  // [!code highlight]
    }

    public int add(int a, int b) {
        return a + b;  // [!code highlight]
    }

    public int square(int x) {
        return x * x;  // [!code highlight]
    }
}

// JIT 内联后：伪代码
public int addAndSquare_inlined(int a, int b) {
    // add 被内联
    int temp = a + b;
    // square 被内联
    return temp * temp;
}
```

### 逃逸分析

```java
// 逃逸分析示例
public class EscapeAnalysisExample {
    // 不逃逸：在栈上分配
    public Point createPoint(double x, double y) {
        Point p = new Point(x, y);  // [!code highlight]
        return p;  // 返回引用，逃逸
    }

    // 不逃逸：可以栈上分配
    public double distance(double x1, double y1, double x2, double y2) {
        Point p1 = new Point(x1, y1);  // [!code highlight]
        Point p2 = new Point(x2, y2);  // [!code highlight]
        return p1.distance(p2);
    }

    // JIT 优化后（伪代码）
    public double distance_optimized(double x1, double y1, double x2, double y2) {
        // 不需要 new Point，直接使用 x1, y1, x2, y2
        double dx = x2 - x1;
        double dy = y2 - y1;
        return Math.sqrt(dx * dx + dy * dy);
    }
}
```

## JIT 反优化

### 反优化的场景

```
┌─────────────────────────────────────────────────────────────────┐
│                    反优化（Deoptimization）                       │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  触发条件：                                                    │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │ 1. 类型猜测错误                                        │   │
│  │    - JIT 假设对象类型，执行时发现类型不符              │   │
│  │    - 假设只执行 add(Integer) 一次，执行了 add(String)   │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                  │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │ 2. 栈上分配失败                                       │   │
│  │    - 对象在编译后逃逸                                  │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                  │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │ 3. 激进优化回退                                       │   │
│  │    - 编译时的假设不再成立                             │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                  │
│  处理方式：                                                    │
│  - 回退到解释执行                                              │
│  - 重新收集 profiling 信息                                     │
│  - 稍后再次编译                                               │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

### 反优化示例

```java
// 反优化示例
public class DeoptExample {
    // 第一次调用时，list 只有 Integer
    public static int sum(List<Integer> list) {
        int sum = 0;
        for (Integer i : list) {
            sum += i;  // [!code highlight]
        }
        return sum;
    }

    // 触发反优化
    public static void main(String[] args) {
        List<Integer> intList = new ArrayList<>();
        for (int i = 0; i < 10000; i++) {
            intList.add(i);
            sum(intList);  // JIT 编译了这个方法
        }

        // 突然传入 String，类型猜测错误
        List<String> strList = new ArrayList<>();
        strList.add("hello");
        sum(strList);  // 触发反优化！  // [!code highlight]
    }
}
```

## 总结

JIT 编译是 Java 性能优化的核心。

**编译流程**：

- 解释执行 → 收集 profiling → C1 编译 → C2 编译

**热点检测**：

- 方法调用次数达到阈值
- 循环执行次数达到阈值

**核心优化**：

- 方法内联：减少调用开销
- 逃逸分析：栈上分配、标量替换
- 死代码消除
- 热点编译

**注意事项**：

- 启动性能：分层编译平衡启动和运行性能
- 反优化：某些情况下会回退到解释执行
- JIT 日志：帮助诊断性能问题
