# 分层编译策略

你的 Java 应用启动后性能逐渐提升，但一开始很慢。

问题出在哪？**分层编译在不同阶段使用不同的优化策略**。

理解分层编译，是理解 JIT 优化的关键。

## 分层编译概述

### 为什么需要分层

```
┌─────────────────────────────────────────────────────────────────┐
│                    为什么需要分层编译                             │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  问题：                                                        │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │ 方式一：仅使用 C2 编译器                               │   │
│  │ - 编译耗时长，启动慢                                   │   │
│  │ - 但最终性能好                                          │   │
│  │                                                          │   │
│  │ 方式二：仅使用 C1 编译器                               │   │
│  │ - 编译快速，启动快                                      │   │
│  │ - 但优化不够，性能差                                    │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                  │
│  解决：分层编译                                               │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │ 启动阶段 → C1 快速编译 → 快速执行                      │   │
│  │                  ↓                                       │   │
│  │              收集 profiling                             │   │
│  │                  ↓                                       │   │
│  │ 稳定阶段 → C2 激进优化 → 极致性能                      │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

### 分层级别

```
┌─────────────────────────────────────────────────────────────────┐
│                    分层编译级别                                   │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  Level 0：解释执行                                           │
│  - 程序启动时的初始阶段                                       │
│  - 收集 profiling 信息                                         │
│  - 不会编译                                                   │
│                                                                  │
│  Level 1：C1 编译（轻量优化）                                 │
│  - 快速编译                                                   │
│  - 轻量级 profiling                                            │
│  - 适合启动阶段                                               │
│                                                                  │
│  Level 2：C1 编译（完整优化）                                 │
│  - 完整 profiling                                             │
│  - 比 Level 1 更多的优化                                       │
│                                                                  │
│  Level 3：C1 编译 + C2 编译（已废弃）                         │
│  - 历史遗留，与 Level 2 合并                                   │
│                                                                  │
│  Level 4：C2 编译（激进优化）                                 │
│  - 最深度的优化                                               │
│  - 可能耗时较长                                               │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

## 分层编译流程

### 状态转换

```
┌─────────────────────────────────────────────────────────────────┐
│                    代码编译状态转换                               │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│                    ┌────────────────┐                           │
│                    │ 解释执行 (L0) │                           │
│                    └───────┬────────┘                           │
│                            │                                      │
│                            │ 热点检测                            │
│                            ▼                                      │
│                    ┌────────────────┐                           │
│              ┌────│  C1 编译 (L1)  │                           │
│              │     └───────┬────────┘                           │
│              │             │ 调用次数增长                       │
│              │             ▼                                      │
│              │     ┌────────────────┐                           │
│              │     │  C1 编译 (L2)  │                           │
│              │     └───────┬────────┘                           │
│              │             │ profiling 充分                      │
│              │             ▼                                      │
│              │     ┌────────────────┐                           │
│              │     │  C2 编译 (L4)  │                           │
│              │     └───────┬────────┘                           │
│              │             │                                      │
│              │      反优化 │                                      │
│              └────────────┘                                      │
│                                                                  │
│  状态：                                                       │
│  - made not entrant：正在替换，不接受新调用                     │
│  - made zombie：已废弃，GC 可以清理                            │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

### 热点检测机制

```java
// 热点检测基于两个计数器
public class HotspotDetection {
    /*
    方法调用计数器（Method Counters）：
    - 每个方法有一个调用计数器
    - 每次调用 +1
    - 达到阈值触发 C1 编译

    回边计数器（Backedge Counters）：
    - 每次循环回边 +1
    - 达到阈值触发 OSR（On-Stack Replacement）
    */

    // JVM 参数控制阈值
    /*
    -XX:CompileThreshold=1500    // 方法编译阈值
    -XX:OnStackReplacePercentage=150  // OSR 阈值

    实际 OSR 阈值 = CompileThreshold * OnStackReplacePercentage / 100
    = 1500 * 150 / 100 = 2250
    */
}
```

## OSR（栈上替换）

### OSR 概念

```
┌─────────────────────────────────────────────────────────────────┐
│                    OSR（栈上替换）                               │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  问题：                                                        │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │ 解释执行 long for-loop() {                              │   │
│  │     for (int i = 0; i < 1000000000; i++) {            │   │
│  │         sum += i;  // 热点代码                          │   │
│  │     }                                                    │   │
│  │ }                                                        │   │
│  │                                                          │   │
│  │ 循环执行了 100 万次后才触发编译？                        │   │
│  │ 此时方法还在执行中，无法等待编译完成                      │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                  │
│  解决：OSR                                                   │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │ 在循环执行过程中，用编译后的代码替换正在执行的方法       │   │
│  │ - 解释器正在执行                                         │   │
│  │ - JIT 完成编译                                           │   │
│  │ - 跳转到编译后的代码继续执行                             │   │
│  │ - 无需等待下次调用                                       │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

### OSR 示例

```java
// OSR 触发场景
public class OsrExample {
    // 这个方法会被 OSR
    public static long sum() {
        long sum = 0;
        for (long i = 0; i < 1_000_000_000; i++) {
            sum += i;  // [!code highlight]
        }
        return sum;
    }

    public static void main(String[] args) {
        // 第一次调用：进入解释执行
        sum();

        // 循环中达到 OSR 阈值
        // JVM 在循环执行过程中进行编译
        // 完成后切换到编译后的代码

        // 第二次调用：直接执行编译后的代码
        sum();
    }
}
```

## 分层编译参数

### 基本参数

```bash
# 启用分层编译（JDK 8+ 默认启用）
-XX:+TieredCompilation
-XX:-TieredCompilation  # 禁用分层编译

# 分层级别控制
-XX:TieredStopAtLevel=1  # 仅编译到 C1 L1
-XX:TieredStopAtLevel=2  # 仅编译到 C1 L2
-XX:TieredStopAtLevel=3  # 仅编译到 C1 L2 + C2（已废弃）
# 默认：TieredStopAtLevel=4

# 热点阈值
-XX:CompileThreshold=1500           # 方法编译阈值
-XX:OnStackReplacePercentage=150    # OSR 阈值
```

### 调优参数

```bash
# 调整编译速度 vs 优化程度
-XX:+UseFastCompiler    # 加快编译（减少优化）
-XX:+UseSlowCompiler     # 慢速编译（增加优化）

# 编译队列大小
-XX:CICompilerCount=2     # 并行编译线程数
-XX:CICompilerCountPerCPU=true  # 根据 CPU 设置

# C1 编译优化级别
-XX:TieredCompilation     # 默认启用分层

# 阈值调整
-XX:MinTickInterval=100ms           # 最小区分时间
-XX:CompileThresholdScaling=1.0      # 阈值缩放因子
```

### 监控参数

```bash
# 打印编译日志
-XX:+PrintCompilation

# 输出示例
# 1   java.lang.Object::toString (42 bytes)
# %1   java.lang.String::charAt (29 bytes)   inline (hot)
#      |  |  |  方法信息
#      |  编译级别（1=C1, 2=C2）
#      tiered 编译标识

# 打印分层状态
-XX:+UnlockDiagnosticVMOptions
-XX:+PrintTieredCompilation

# 查看编译队列
jstat -printCompilation <pid>
```

## 分层编译调优

### 启动优化

```bash
# 场景：应用启动后立即处理大量请求
# 目标：缩短冷启动时间

# 方案一：调整分层级别
java -XX:+TieredCompilation -XX:TieredStopAtLevel=1 app.jar
# 仅使用 C1 L1 编译，启动快，但性能稍差

# 方案二：提前编译
java -XX:+TieredCompilation \
     -XX:CompileThreshold=500 \      # 降低阈值，提前编译
     -XX:PretenureSizeThreshold=1m   # 大对象直接进入老年代
     app.jar

# 方案三：AOT 编译（JDK 9+）
jaotc --output=app.so --jar=app.jar
java -XX:AOTLibrary=./app.so app.jar
```

### 稳定期优化

```bash
# 场景：长时间运行的服务
# 目标：最大化稳定期性能

# 方案一：等待 C2 编译完成
java -XX:+TieredCompilation \
     -XX:+TieredCompilation \
     -XX:CICompilerCount=4 \        # 增加编译线程
     -XX:+BackgroundCompilation      # 后台编译
     app.jar

# 方案二：调整热点阈值
java -XX:+TieredCompilation \
     -XX:CompileThreshold=10000 \    # 提高阈值，减少不必要的编译
     -XX:OnStackReplacePercentage=200
     app.jar
```

```java

        } catch (Exception e) {
            e.printStackTrace();
        }
    }
}
```

## 总结

分层编译是平衡启动性能和运行性能的利器。

**分层级别**：

| 级别 | 名称 | 特点 |
|---|---|---|
| L0 | 解释执行 | 无编译开销 |
| L1 | C1 L1 | 快速编译，轻量优化 |
| L2 | C1 L2 | 完整 profiling |
| L4 | C2 | 激进优化 |

**OSR**：

- 在执行过程中替换代码
- 解决热点循环的编译问题

**调优方向**：

- 启动优化：降低分层级别
- 稳定期优化：增加编译线程
- 合理阈值：平衡编译开销和性能
