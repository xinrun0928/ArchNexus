# 按业务拆分

你的服务拆分后，模块之间还是互相调用一团糟。

问题出在哪？**没有正确识别业务边界**。

按业务拆分是微服务拆分的第一步，也是最重要的一步。

## 业务边界识别

```
┌─────────────────────────────────────────────────────────────────┐
│                    业务边界识别                                    │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  业务边界（Business Boundary）：                                 │
│                                                                  │
│  「一个业务领域内，所有相关的业务逻辑和数据聚合在一起，        │
│    与其他业务领域的交互最小化」                                 │
│                                                                  │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │                                                         │   │
│  │  好的边界：                                            │   │
│  │                                                         │   │
│  │  用户模块 ──▶ 只管用户相关的所有事情                 │   │
│  │              - 用户注册登录                           │   │
│  │              - 用户信息管理                           │   │
│  │              - 用户认证授权                           │   │
│  │              - 用户数据存储                           │   │
│  │                                                         │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                  │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │                                                         │   │
│  │  好的边界：                                            │   │
│  │                                                         │   │
│  │  订单模块 ──▶ 只管订单相关的所有事情                 │   │
│  │              - 订单创建                               │   │
│  │              - 订单状态流转                           │   │
│  │              - 订单数据存储                           │   │
│  │              - 引用用户 ID，但不管理用户             │   │
│  │                                                         │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

## 业务拆分的原则

```
┌─────────────────────────────────────────────────────────────────┐
│                    业务拆分原则                                    │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │ 1. 单一职责原则（SRP）                                │   │
│  ├─────────────────────────────────────────────────────────┤   │
│  │                                                         │   │
│  │  每个微服务只负责一个业务能力                        │   │
│  │  服务应该「小而专」                                   │   │
│  │                                                         │   │
│  │  示例：                                                │   │
│  │  - 订单服务：只负责订单相关                          │   │
│  │  - 支付服务：只负责支付相关                          │   │
│  │  - 库存服务：只负责库存相关                          │   │
│  │                                                         │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                  │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │ 2. 高内聚低耦合原则                                   │   │
│  ├─────────────────────────────────────────────────────────┤   │
│  │                                                         │   │
│  │  相关的东西放在一起，不相关的分开                    │   │
│  │  服务间依赖要最小化                                    │   │
│  │                                                         │   │
│  │  示例：                                                │   │
│  │  - 用户服务内部高度内聚                              │   │
│  │  - 用户服务和订单服务之间低耦合                      │   │
│  │                                                         │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                  │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │ 3. 独立数据原则                                        │   │
│  ├─────────────────────────────────────────────────────────┤   │
│  │                                                         │   │
│  │  每个微服务拥有自己的数据库                          │   │
│  │  不直接访问其他服务的数据库                          │   │
│  │                                                         │   │
│  │  示例：                                                │   │
│  │  - 用户服务有自己的用户数据库                        │   │
│  │  - 订单服务有自己的订单数据库                        │   │
│  │  - 服务间通过 API 通信，不直接 SQL                   │   │
│  │                                                         │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                  │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │ 4. 团队所有权原则                                      │   │
│  ├─────────────────────────────────────────────────────────┤   │
│  │                                                         │   │
│  │  一个微服务由一个团队端到端负责                      │   │
│  │  团队对服务的全生命周期负责                          │   │
│  │                                                         │   │
│  │  示例：                                                │   │
│  │  - 订单团队拥有订单服务                              │   │
│  │  - 订单团队负责开发、测试、部署、运维               │   │
│  │                                                         │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

## 常见拆分模式

### 动词 vs 名词

```
┌─────────────────────────────────────────────────────────────────┐
│                    动词 vs 名词拆分                                 │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │ 按动词拆分（活动）                                     │   │
│  ├─────────────────────────────────────────────────────────┤   │
│  │                                                         │   │
```

### 领域事件驱动

```java
// 领域事件
public class OrderCreatedEvent {
    private String orderId;
    private String userId;
    private BigDecimal amount;
    private LocalDateTime createdAt;
}

// 订单服务发布事件
@Service
public class OrderService {
    @Autowired
    private ApplicationEventPublisher eventPublisher;

    public void createOrder(Order order) {
        // 创建订单
        orderRepository.save(order);

        // 发布领域事件
        eventPublisher.publishEvent(
            new OrderCreatedEvent(
                order.getId(),
                order.getUserId(),
                order.getAmount(),
                order.getCreatedAt()
            )
        );
    }
}

// 其他服务订阅事件
@Service
public class NotificationService {
    @EventListener
    public void handleOrderCreated(OrderCreatedEvent event) {
        // 发送通知
        sendNotification(event.getUserId(), "订单已创建");
    }
}

@Service
public class InventoryService {
    @EventListener
    public void handleOrderCreated(OrderCreatedEvent event) {
        // 释放预留库存
        releaseReservedInventory(event.getOrderId());
    }
}
```

## 总结

按业务拆分是微服务拆分的第一步。

**拆分原则**：

- 单一职责：一个服务做一件事
- 高内聚低耦合：相关的东西放一起
- 独立数据：不共享数据库
- 团队所有权：端到端负责

**分析方法**：

- DDD 限界上下文
- 业务能力分析

**常见模式**：

- 按名词拆分（推荐）
- 按动词拆分（不推荐）
- 领域事件驱动

**核心认知**：业务边界识别是微服务拆分中最重要的工作。边界画错了，后续改动代价很大。在动手拆分之前，花时间仔细分析业务边界是值得的。
