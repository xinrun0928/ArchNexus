# 服务粒度控制

你的服务要么拆得太细，几百个服务根本管不过来；要么拆得太粗，一个服务包含半个系统。

问题出在哪？**没有正确控制服务粒度**。

服务粒度是微服务设计中最难把握的问题。

## 粒度控制原则

```
┌─────────────────────────────────────────────────────────────────┐
│                    粒度控制原则                                    │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │ 原则一：一个服务应该能被一个团队维护                    │   │
│  ├─────────────────────────────────────────────────────────┤   │
│  │                                                         │   │
│  │  推荐：                                                │   │
│  │  - 3-5 人可以维护一个服务                              │   │
│  │  - 服务代码量在 1-3 万行以内                          │   │
│  │  - 服务可以独立发布                                    │   │
│  │                                                         │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                  │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │ 原则二：先粗后细，渐进演进                            │   │
│  ├─────────────────────────────────────────────────────────┤   │
│  │                                                         │   │
│  │  建议：                                                │   │
│  │  - 先拆成 5-10 个大服务                              │   │
│  │  - 运行一段时间后，再识别需要拆分的边界               │   │
│  │  - 不要一开始就追求完美粒度                           │   │
│  │                                                         │   │
│  │  错误做法：                                            │   │
│  │  - 一开始就设计几十个服务                            │   │
│  │  - 纸上谈兵式的服务边界设计                          │   │
│  │                                                         │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                  │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │ 原则三：数据内聚优先                                    │   │
│  ├─────────────────────────────────────────────────────────┤   │
│  │                                                         │   │
│  │  如果两个功能访问同一份数据，大概率应该放在一起        │   │
│  │  如果两个功能从不访问相同的数据，可以考虑分开         │   │
│  │                                                         │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                  │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │ 原则四：按变更频率分组                                  │   │
│  ├─────────────────────────────────────────────────────────┤   │
│  │                                                         │   │
│  │  变更频繁的功能放在一起，稳定的放一起                  │   │
│  │  这样独立发布时回归范围更小                            │   │
│  │                                                         │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

## 服务合并策略

```java
// 合并前的两个服务
// UserRegisterService
@Service
public class UserRegisterService {
    public void register(User user) {
        // 注册逻辑
    }
}

// UserLoginService
@Service
public class UserLoginService {
    public LoginResult login(String username, String password) {
        // 登录逻辑
    }
}

// 合并后的服务
@Service
public class UserService {
    // 合并后可以共享内部方法
    private UserRepository userRepository;

    public void register(User user) {
        // 注册逻辑
        // 可以直接调用内部方法
        validateUser(user);
        userRepository.save(user);
    }

    public LoginResult login(String username, String password) {
        // 登录逻辑
        // 可以直接访问数据
        User user = userRepository.findByUsername(username);
        validatePassword(user, password);
        return generateToken(user);
    }

    // 私有方法，合并后可以共享
    private void validateUser(User user) {
        // 验证逻辑
    }
}
```

## 总结

服务粒度是微服务设计中最难把握的问题。

**光谱**：

- 太粗：协作困难、扩展不灵活
- 太细：服务爆炸、运维困难

**粒度控制原则**：

- 一个团队能维护
- 先粗后细，渐进演进
- 数据内聚优先
- 按变更频率分组

**调整时机**：

- 拆分：代码量过大、需要独立扩展、变更频率不同
- 合并：总是同时变更、调用频繁、需要强一致性

**核心认知**：服务粒度没有标准答案，需要根据团队规模、业务特点、运维能力来动态调整。关键是保持「刚刚好」的粒度：服务足够小以至于独立，但又足够大以至于不会产生过多的协调成本。
