# 服务拆分

服务拆分是微服务架构中最重要的工作。本部分将帮助你理解如何按业务、按数据拆分，如何实施前后端分离，以及如何正确控制服务粒度。

## 内容结构

**按业务拆分**介绍了 DDD 限界上下文、业务能力分析等方法，以及单一职责、高内聚低耦合、独立数据、团队所有权等拆分原则。业务边界识别是服务拆分中最重要的工作。边界画错了，后续改动代价很大。

**按数据拆分**强调了「服务可以调用，但数据不能共享」的原则，以及数据边界的划分策略、数据库选型、数据同步方案。数据边界决定了服务边界。不共享数据库是微服务拆分的基本原则。

**前后端分离**介绍了前后端分离的架构、API 设计原则、API 网关和 BFF 模式。前后端分离不仅是技术分离，更是职责分离和团队分离。

**服务粒度控制**分析了粒度过粗和过细的问题，以及粒度控制的原则（一个团队能维护、先粗后细、数据内聚优先、按变更频率分组）。服务粒度是微服务设计中最难把握的问题，没有标准答案，需要根据具体情况动态调整。

## 关键洞察

服务拆分的核心是业务边界。找到一个业务边界清晰、数据内聚、团队所有权的服务划分方案，是拆分成功的前提。

服务粒度的「刚刚好」原则：服务足够小以至于独立，但又足够大以至于不会产生过多的协调成本。
