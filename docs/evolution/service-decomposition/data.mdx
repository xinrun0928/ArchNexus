# 按数据拆分

你的服务拆分后，数据还在一个库里，数据库成了瓶颈。

问题出在哪？**没有正确划分数据边界**。

数据边界是服务边界的重要组成部分。

## 数据边界的重要性

│                                                                  │

│  │                                                         │   │

│  │                                                         │   │

│                                                                  │
│  ┌─────────────────────────────────────────────────────────┐   │

│  │                                                         │   │
│  │  ┌─────────────────────────────────────────────────┐  │   │

## 数据边界的划分原则

```
┌─────────────────────────────────────────────────────────────────┐
│                    数据边界划分原则                                 │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │ 1. 每个服务拥有独立的数据存储                          │   │
│  ├─────────────────────────────────────────────────────────┤   │
│  │                                                         │   │
│  │  服务 A ──▶ 数据库 A                                 │   │
│  │  服务 B ──▶ 数据库 B                                 │   │
│  │  服务 C ──▶ 数据库 C                                 │   │
│  │                                                         │   │
│  │  服务间不共享数据库                                   │   │
│  │                                                         │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                  │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │ 2. 服务间通过 API 共享数据                            │   │
│  ├─────────────────────────────────────────────────────────┤   │
│  │                                                         │   │
│  │  ┌─────────────────────────────────────────────────┐  │   │
│  │  │                                                 │  │   │
│  │  │  服务 A 需要服务 B 的数据：                     │  │   │
│  │  │                                                 │  │   │
│  │  │  方式 1：调用服务 B 的 API                    │  │   │
│  │  │  方式 2：订阅服务 B 发布的数据事件             │  │   │
│  │  │                                                 │  │   │
│  │  │  禁止：直接连接服务 B 的数据库               │  │   │
│  │  │                                                 │  │   │
│  │  └─────────────────────────────────────────────────┘  │   │
│  │                                                         │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                  │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │ 3. 允许数据冗余                                        │   │
│  ├─────────────────────────────────────────────────────────┤   │
│  │                                                         │   │
│  │  为了性能，可以冗余一些数据                          │   │
│  │  但要保证数据同步和最终一致性                        │   │
│  │                                                         │   │
│  │  示例：                                                │   │
│  │  - 订单表冗余用户姓名（只读）                        │   │
│  │  - 订单表冗余商品名称（只读）                        │   │
│  │                                                         │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

### 数据冗余模式

```java
// 订单服务 - 冗余数据
@Entity
@Table(name = "orders")
public class Order {
    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    private Long userId;

    // 冗余字段（只读，不修改）
    private String userName;

    // 冗余字段（只读，不修改）
    private String userEmail;

    // 冗余字段（只读，不修改）
    private String productName;

    private BigDecimal amount;

    // 同步用户信息
    public void syncUserInfo(User user) {
        this.userName = user.getName();
        this.userEmail = user.getEmail();
    }

    // 同步商品信息
    public void syncProductInfo(Product product) {
        this.productName = product.getName();
    }
}

// 用户服务 - 订阅事件同步
@Service
public class UserDataSyncService {
    @EventListener
    public void handleUserUpdated(UserUpdatedEvent event) {
        // 通知所有依赖用户信息的服务更新缓存
        messagePublisher.publish(
            "user.data.sync",
            new UserDataSyncMessage(
                event.getUserId(),
                event.getName(),
                event.getEmail()
            )
        );
    }
}
```

## 数据同步策略

```java
// 数据同步 - 消息队列方式
@Service
public class DataSyncService {
    @Autowired
    private KafkaTemplate<String, Object> kafkaTemplate;

    // 用户服务发布数据变更
    public void publishUserChange(User user) {
        kafkaTemplate.send("user.changes", user.getId(), user);
    }

    // 订单服务订阅数据变更
    @KafkaListener(topics = "user.changes", groupId = "order-service")
    public void handleUserChange(User user) {
        // 更新本地冗余数据或缓存
        userCache.put(user.getId(), user);
    }
}

// 数据同步 - 定时任务方式
@Service
public class DataSyncScheduler {
    @Autowired
    private UserService userService;

    @Scheduled(cron = "0 */5 * * * *")  // 每 5 分钟
    public void syncUserData() {
        // 批量同步最新用户数据
        List<User> users = userService.getRecentUpdatedUsers(
            LocalDateTime.now().minusMinutes(5));

        for (User user : users) {
            userCache.put(user.getId(), user);
        }
    }
}
```

## 总结

数据边界是服务边界的重要组成部分。

**划分原则**：

- 每个服务拥有独立数据库
- 通过 API 而非直接 SQL 共享数据
- 允许必要的冗余

**拆分策略**：

- 按服务拆分数据库
- 冗余必要的只读数据
- 跨服务数据通过事件同步

**数据库选型**：

- 核心业务：RDBMS
- 灵活数据：Document
- 高性能缓存：KV
- 搜索分析：Search

**同步策略**：

- 消息队列：实时同步
- 定时任务：批量同步
- 订阅发布：事件驱动

**核心认知**：数据边界决定了服务边界。不共享数据库是微服务拆分的基本原则。
