# 技术债务的度量

你的系统有技术债务，但不知道有多少。

问题出在哪？**没有量化和跟踪技术债务**。

度量是管理技术债务的第一步。

```
│  │  │ 21-50： 高复杂度，需要重构                    │ │ │
│  │  │ > 50：  非常高，必须重构                      │ │ │
│  │  └─────────────────────────────────────────────────┘ │ │
│  │                                                         │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                  │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │ 方法长度                                              │   │
│  ├─────────────────────────────────────────────────────────┤   │
│  │                                                         │   │
│  │  建议：                                                │   │
│  │  - 理想：< 10 行                                     │   │
│  │  - 可接受：10-30 行                                  │   │
│  │  - 警告：30-60 行                                   │   │
│  │  - 必须重构：> 60 行                                 │   │
```

### 代码度量工具

```bash
# SonarQube 代码度量
# 配置质量配置
sonar.projectKey=my-project
sonar.sources=src
sonar.language=java

# 运行分析
mvn sonar:sonar

# 查看报告：SonarQube Dashboard
# - 圈复杂度
# - 代码重复率
# - 单元测试覆盖率
# - 规范违规
```

```java
// 使用 Java 自定义度量
public class CodeMetrics {
    // 计算方法长度
    public static int countMethodLines(MethodDeclaration method) {
        return method.getEndLine() - method.getStartLine() + 1;
    }

    // 计算圈复杂度
    public static int calculateCyclomaticComplexity(MethodDeclaration method) {
        int complexity = 1;  // 基础复杂度

        for (Statement stmt : method.getBody().getStatements()) {
            if (stmt instanceof IfStatement ||
                stmt instanceof ForStatement ||
                stmt instanceof WhileStatement ||
                stmt instanceof CaseStatement ||
                stmt instanceof ConditionalExpr) {
                complexity++;
            }
        }

        return complexity;
    }
}
```

### 测试覆盖率配置

```xml
<!-- Maven JaCoCo 配置 -->
<plugin>
    <groupId>org.jacoco</groupId>
    <artifactId>jacoco-maven-plugin</artifactId>
    <configuration>
        <rules>
            <rule>
                <element>BUNDLE</element>
                <limits>
                    <limit>
                        <counter>LINE</counter>
                        <value>COVEREDRATIO</value>
                        <minimum>0.80</minimum>
                    </limit>
                    <limit>
                        <counter>BRANCH</counter>
                        <value>COVEREDRATIO</value>
                        <minimum>0.70</minimum>
                    </limit>
                </limits>
            </rule>
        </rules>
    </configuration>
</plugin>
```

```
┌─────────────────────────────────────────────────────────────────┐
│                    技术债务量化                                     │

│                                                                  │

│  │                                                         │   │

│  │                                                         │   │

│  │  - 覆盖率不足：2 个模块 × 8 小时 = 16 小时         │   │

│  │                                                         │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                  │
│  ┌─────────────────────────────────────────────────────────┐   │

│  ├─────────────────────────────────────────────────────────┤   │
│  │                                                         │   │

│  │                                                         │   │
│  │  建议：                                                │   │
│  │  - < 5%：健康                                       │   │
│  │  - 5-10%：可接受                                    │   │
│  │  - 10-20%：需要关注                                 │   │
│  │  - > 20%：危险，必须偿还                           │   │
```

## 总结

度量是管理技术债务的第一步。

**度量维度**：

- 代码复杂度：圈复杂度、方法长度
- 测试覆盖率：行覆盖率、分支覆盖率
- 架构债务：耦合度、依赖深度
- 量化指标：债务时间、债务比率

**度量工具**：

- SonarQube：综合代码分析
- JaCoCo：测试覆盖率
- 自定义脚本：专项分析

**跟踪看板**：

- 按类别分组
- 评估修复成本
- 确定优先级

**核心认知**：度量不是为了追求完美数字，而是为了让债务可见、可管理。
