# 单体到微服务

系统架构不是一成不变的，它需要随着业务的发展而演进。本部分将帮助你理解单体架构和微服务架构各自的特点、拆分的时机与方式，以及拆分后会面临的挑战。

## 内容结构

**单体架构的特点**深入分析了单体架构的优势（开发简单、部署容易、性能好）和局限（团队协作困难、部署风险大、技术栈锁定）。理解单体架构是判断是否需要演进的前提。很多时候，单体架构是最合适的架构——如果你的团队规模小、业务简单、并发量低，单体架构可能比微服务更高效。

**微服务架构的特点**系统介绍了微服务的核心特征：服务拆分、独立部署、独立数据、轻量通信、去中心化。微服务不是银弹，它在带来独立演进、独立扩展、故障隔离等优势的同时，也引入了分布式复杂度、运维复杂度、数据一致性挑战等代价。理解这些是正确使用微服务的基础。

**拆分的时机与方式**提供了拆分决策的框架：红灯、黄灯、绿灯三种情况，以及绞杀者模式、功能开关、抽象抽取三种拆分策略。拆分是一个渐进的过程，不是一次性的革命。错误的时机或方式拆分，会让问题变得更糟。

**拆分后的挑战**列举了分布式事务、服务间通信、数据一致性、运维复杂度、接口版本管理等典型问题，以及每个问题的应对策略。微服务拆分不是终点，而是新挑战的起点。理解这些挑战，才能做好充分准备。

## 关键洞察

微服务架构的适用条件：团队规模大（> 30 人）、业务相对稳定、有完整的微服务基础设施和运维经验。如果团队小、业务不确定、没有微服务运维经验，贸然拆分只会带来更多问题。

架构演进的原则：「先粗后细，渐进演进」。不要一开始就追求完美的微服务拆分，而是先从模块化设计开始，在业务发展和团队成长的过程中逐步演进。
