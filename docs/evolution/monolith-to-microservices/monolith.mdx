# 单体架构的特点

系统越来越慢，团队越来越大，改一个 bug 要上线整个系统。

问题出在哪？**没有正确理解单体架构的特点和适用场景**。

单体架构不是落后的代名词，理解它才能判断何时该演进。

```

├─────────────────────────────────────────────────────────────────┤

│                                                                  │
```

## 单体架构的局限

```
┌─────────────────────────────────────────────────────────────────┐
│                    单体架构的局限                                   │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  ┌─────────────────────────────────────────────────────────┐   │

│  │                                                         │   │
│  │  - 代码库巨大，合并冲突频繁                           │   │
│  │  - 模块间耦合，改一个地方可能影响全局                  │   │
│  │  - 团队边界不清                                      │   │
│  │                                                         │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                  │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │ 2. 部署风险大                                          │   │
│  ├─────────────────────────────────────────────────────────┤   │
│  │                                                         │   │
│  │  - 改一行代码也要全量部署                             │   │
│  │  - 一个 bug 可能影响整个系统                          │   │
│  │  - 回滚成本高                                          │   │
│  │                                                         │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                  │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │ 3. 技术栈锁定                                          │   │
│  ├─────────────────────────────────────────────────────────┤   │
│  │                                                         │   │
│  │  - 所有模块必须用同一种语言/框架                       │   │
│  │  - 新技术难以引入                                      │   │
│  │  - 难以针对不同模块做技术选型                         │   │
│  │                                                         │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                  │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │ 4. 扩展性差                                            │   │

│  │  - 只能整体扩展，浪费资源                              │   │
│  │  - 热点模块无法独立扩展                               │   │
│  │  - 硬件利用率低                                        │   │
│  │                                                         │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                  │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │ 5. 故障隔离差                                          │   │
│  ├─────────────────────────────────────────────────────────┤   │
│  │                                                         │   │
│  │  - 一个模块 OOM 可能拖垮整个系统                       │   │
│  │  - 数据库慢查询影响全局                               │   │
│  │                                                         │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                  │
```


## 总结

单体架构不是落后的代名词，有其适用场景。

**优势**：

- 开发、部署、测试简单
- 同进程调用，性能好
- 端到端事务简单

**局限**：

- 团队协作困难
- 部署风险大
- 技术栈锁定
- 扩展性差
- 故障隔离差

**适用场景**：

- 小团队
- 业务初期
- 低并发系统
- 简单业务

**优化策略**：

- 模块化设计
- 插件化架构
- 读写分离
- 缓存优化

**核心认知**：单体架构最大的敌人不是技术，而是团队规模和业务复杂度。在合适的场景下，单体是最优解。
