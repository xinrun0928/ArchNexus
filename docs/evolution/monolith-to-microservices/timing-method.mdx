# 拆分的时机与方式

你的单体系统越来越慢，团队越来越大，该拆了。但什么时候拆？怎么拆？

问题出在哪？**没有正确的拆分时机判断和拆分策略**。

错误的时机或方式拆，会让问题变得更糟。

## 拆分策略

```
┌─────────────────────────────────────────────────────────────────┐
│                    拆分策略概览                                    │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │ 绞杀者模式（Strangler Pattern）                          │   │
│  ├─────────────────────────────────────────────────────────┤   │
│  │                                                         │   │
│  │  在单体周围逐步构建新服务，                            │   │
│  │  慢慢替换原有功能，最终让单体「死去」                 │   │
│  │                                                         │   │
│  │  ✓ 风险低，支持渐进式拆分                            │   │
│  │  ✓ 可以随时回退                                      │   │
│  │  ✗ 周期长                                            │   │
│  │                                                         │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                  │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │ 功能抽取模式（Feature Toggle Pattern）                    │   │
│  ├─────────────────────────────────────────────────────────┤   │
│  │                                                         │   │
│  │  用功能开关控制新、旧实现的切换，                      │   │
│  │  可以随时回退到旧实现                                  │   │
│  │                                                         │   │
│  │  ✓ 可以快速验证拆分效果                               │   │
│  │  ✓ 回退代价低                                         │   │
│  │  ✗ 代码可能有两套实现                                │   │
│  │                                                         │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                  │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │ 抽象抽取模式（Branch by Abstraction）                   │   │
│  ├─────────────────────────────────────────────────────────┤   │
│  │                                                         │   │
│  │  先引入抽象层，隔离新旧实现，                          │   │
│  │  然后逐步将实现迁移到新服务                            │   │
│  │                                                         │   │
│  │  ✓ 符合开放封闭原则                                   │   │
│  │  ✓ 拆分过程可控                                       │   │
│  │  ✗ 需要前期投入                                       │   │
│  │                                                         │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

## 功能开关实现

```java
// 功能开关
@Service
public class FeatureToggleService {
    private final Map<String, Boolean> toggles = new ConcurrentHashMap<>();

    // 加载开关配置
    @PostConstruct
    public void loadToggles() {
        // 从配置中心加载
        toggles.put("order-service-enabled", true);
        toggles.put("payment-service-enabled", false);
    }

    // 检查开关状态
    public boolean isEnabled(String feature) {
        return toggles.getOrDefault(feature, false);
    }

    // 动态切换
    public void setEnabled(String feature, boolean enabled) {
        toggles.put(feature, enabled);
        log.info("功能开关切换: {} = {}", feature, enabled);
    }
}

// 使用功能开关
@Service
public class OrderService {
    @Autowired
    private FeatureToggleService toggleService;

    @Autowired
    private OldOrderRepository oldOrderRepository;

    @Autowired
    private NewOrderService newOrderService;

    public Order getOrder(Long orderId) {
        if (toggleService.isEnabled("order-service-enabled")) {
            // 新服务
            return newOrderService.getOrder(orderId);
        } else {
            // 旧实现
            return oldOrderRepository.findById(orderId);
        }
    }
}
```

## 总结

拆分时机和方式决定了拆分成败。

**拆分时机**：

- 红灯：团队小、业务不确定、无经验
- 黄灯：规模中等、有瓶颈、谨慎拆分
- 绿灯：团队大、业务稳定、有基础设施

**拆分策略**：

- 绞杀者模式：渐进替换，风险低
- 功能开关：快速验证，随时回退
- 抽象抽取：符合原则，过程可控

**拆分顺序**：

1. 独立性强、风险低的模块
2. 性能瓶颈明显的模块
3. 高频变更的模块
4. 核心业务模块最后拆

**核心认知**：拆分不是一次性完成的，而是一个渐进的过程。保持耐心，控制风险。
