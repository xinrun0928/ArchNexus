# 拆分后的挑战

你的服务拆出来了，但问题也来了：调用链路变长、数据不一致、排查问题困难。

问题出在哪？**没有预料到拆分后会面临的挑战**。

微服务拆分不是终点，而是新挑战的起点。

## 分布式事务挑战

```
┌─────────────────────────────────────────────────────────────────┐
│                    分布式事务挑战                                  │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  单体中一个 @Transactional 搞定一切：                            │
│                                                                  │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │                                                         │   │
│  │  @Transactional                                        │   │
│  │  public void createOrder() {                          │   │
│  │      // 扣库存 ✓                                      │   │
│  │      // 创建订单 ✓                                    │   │
│  │      // 发送消息 ✓                                    │   │
│  │  }                                                     │   │
│  │                                                         │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                  │
│  拆分后变成了分布式事务：                                        │
│                                                                  │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │                                                         │   │
│  │  订单服务：创建订单 ──▶ 成功 ✓                       │   │
│  │  库存服务：扣减库存 ──▶ 网络超时 ✗                   │   │
│  │  消息服务：发送消息 ──▶ 成功 ✓                       │   │
│  │                                                         │   │
│  │  问题：订单创建了，但库存没扣减！                     │   │
│  │                                                         │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                  │
│  应对策略：                                                    │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │                                                         │   │
│  │  1. Saga 模式：把事务拆成一系列本地事务              │   │
│  │  2. TCC 模式：Try-Confirm-Cancel 三阶段              │   │
│  │  3. 最终一致性：接受短暂不一致，靠补偿机制纠正        │   │
│  │                                                         │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

## 服务间通信挑战

```
┌─────────────────────────────────────────────────────────────────┐
│                    服务间通信挑战                                  │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │ 1. 网络延迟                                            │   │
│  ├─────────────────────────────────────────────────────────┤   │
│  │                                                         │   │
│  │  同进程调用：0.01ms                                    │   │
│  │  跨服务调用：1-10ms                                   │   │
│  │                                                         │   │
│  │  N 个服务串联调用：总延迟 = sum(每个调用延迟)        │   │
│  │                                                         │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                  │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │ 2. 服务不可用                                         │   │
│  ├─────────────────────────────────────────────────────────┤   │
│  │                                                         │   │
│  │  依赖的服务挂了怎么办？                                │   │
│  │  不断重试会导致雪崩                                    │   │
│  │                                                         │   │
│  │  ┌─────────────────────────────────────────────────┐  │   │
│  │  │                                                 │  │   │
│  │  │  A → B → C → D                                 │  │   │
│  │  │         ↑                                       │  │   │
│  │  │         D 挂了                                  │  │   │
│  │  │         B 等待超时                              │  │   │
│  │  │         A 等待 B                               │  │   │
│  │  │         请求堆积 → 资源耗尽 → 系统崩溃          │  │   │
│  │  │                                                 │  │   │
│  │  └─────────────────────────────────────────────────┘  │   │
│  │                                                         │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                  │
│  应对策略：                                                    │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │                                                         │   │
│  │  1. 超时设置：每个调用都要设置合理的超时              │   │
│  │  2. 重试机制：失败时自动重试，但要有上限              │   │
│  │  3. 熔断器：快速失败，防止雪崩                       │   │
│  │  4. 限流：保护被调用服务                              │   │
│  │  5. 降级：服务不可用时返回兜底数据                   │   │
│  │                                                         │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

## 数据一致性挑战

```
┌─────────────────────────────────────────────────────────────────┐
│                    数据一致性挑战                                  │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  拆分后每个服务有自己的数据库，数据一致性问题复杂：             │
│                                                                  │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │                                                         │   │
│  │  场景：查询用户订单                                     │   │
│  │                                                         │   │
│  │  ┌─────────────────────────────────────────────────┐  │   │
│  │  │                                                 │  │   │
│  │  │         用户服务 ──▶ 用户数据库                 │  │   │
│  │  │             │                                   │  │   │
│  │  │             │ 查询用户信息                      │  │   │
│  │  │             ▼                                   │  │   │
│  │  │         订单服务 ──▶ 订单数据库                 │  │   │
│  │  │                                                 │  │   │
│  │  └─────────────────────────────────────────────────┘  │   │
│  │                                                         │   │
│  │  问题：如何保证用户和订单数据的一致性？               │   │
│  │                                                         │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                  │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │                                                         │   │
│  │  场景：聚合查询（用户 + 订单 + 商品）                 │   │
│  │                                                         │   │
│  │  单体：一个 SQL join 就搞定                          │   │
│  │  微服务：需要调用多个服务，再聚合                      │   │
│  │                                                         │   │
│  │  ┌─────────────────────────────────────────────────┐  │   │
│  │  │                                                 │  │   │
│  │  │  并行调用：                                      │  │   │
│  │  │  userService.getUser(id)                        │  │   │
│  │  │  orderService.getOrders(userId)                 │  │   │
│  │  │  productService.getProducts(order.productIds)   │  │   │
│  │  │                                                 │  │   │
│  │  │  延迟 = max(t1, t2, t3) 而不是 sum(t1,t2,t3)   │  │   │
│  │  │                                                 │  │   │
│  │  └─────────────────────────────────────────────────┘  │   │
│  │                                                         │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                  │
│  应对策略：                                                    │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │                                                         │   │
│  │  1. API 组合：服务返回聚合数据（推荐）                 │   │
│  │  2. CQRS：读写分离，读服务订阅写事件                  │   │
│  │  3. 允许数据短暂不一致                                │   │
│  │  4. 分布式追踪：Skywalking/Jaeger                      │   │
│  │                                                         │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

## 总结

拆分后会面临多重挑战。

**分布式事务**：从本地事务变为分布式事务，需要 Saga/TCC/补偿机制。

**服务通信**：网络延迟、不可用、雪崩，需要超时、重试、熔断、降级、限流。

**数据一致性**：聚合查询复杂，需要 API 组合/CQRS/允许短暂不一致。

**运维复杂度**：服务数量爆炸，需要容器化、CI/CD、链路追踪、监控系统。

**接口版本**：多版本共存，需要语义化版本、渐进废弃。

**核心认知**：微服务拆分后，技术挑战只是开始，真正的难题在于团队的工程能力和运维体系建设。
