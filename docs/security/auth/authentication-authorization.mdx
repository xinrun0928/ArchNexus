# 认证 vs 授权

你的系统把「你是谁」和「你能做什么」混为一谈，导致权限控制漏洞百出。

问题出在哪？**没有正确区分认证（Authentication）和授权（Authorization）**。

认证和授权是安全的两个不同维度，必须分开处理。

## 实现示例

```java
// Spring Security 认证与授权分离实现
@Configuration
@EnableWebSecurity
public class SecurityConfig {

    @Bean
    public SecurityFilterChain filterChain(HttpSecurity http) throws Exception {
        http
            // 认证配置
            .authenticationProvider(authenticationProvider())
            .authorizeHttpRequests(auth -> auth
                // 公开路径不需要认证
                .requestMatchers("/login", "/register").permitAll()
                // 其他路径需要认证
                .anyRequest().authenticated()
            )
            // 授权配置（在 Controller 层或 Service 层单独处理）
            .addFilterBefore(authenticationFilter(),
                UsernamePasswordAuthenticationFilter.class);

        return http.build();
    }
}

// 认证服务
@Service
public class AuthenticationService {
    public AuthenticationResult authenticate(String username, String password) {
        // 1. 验证凭证（认证）
        User user = userRepository.findByUsername(username);

        if (user == null || !passwordEncoder.matches(password, user.getPassword())) {
            throw new BadCredentialsException("用户名或密码错误");
        }

        // 2. 生成认证 Token
        String token = jwtTokenProvider.generateToken(user.getId());

        return new AuthenticationResult(token, user);
    }
}

// 授权服务（与认证分离）
@Service
public class AuthorizationService {
    public boolean hasPermission(Authentication auth, String resource, String action) {
        User user = (User) auth.getPrincipal();

        // 查询用户的角色
        Set<Role> roles = user.getRoles();

        // 检查角色是否有对应权限
        for (Role role : roles) {
            Set<Permission> permissions = role.getPermissions();
            for (Permission p : permissions) {
                if (p.matches(resource, action)) {
                    return true;
                }
            }
        }

        return false;
    }
}

// Controller 层：先认证，后授权
@RestController
@RequestMapping("/api/orders")
public class OrderController {

    @Autowired
    private AuthorizationService authorizationService;

    @DeleteMapping("/{id}")
    public ResponseEntity<?> deleteOrder(
            @PathVariable Long id,
            @AuthenticationPrincipal User user) {

        // 认证信息已由 Security Filter Chain 处理

        // 授权检查（单独进行）
        if (!authorizationService.hasPermission(
                SecurityContextHolder.getContext().getAuthentication(),
                "ORDER", "DELETE")) {
            return ResponseEntity.status(HttpStatus.FORBIDDEN)
                .body("没有删除订单的权限");
        }

        orderService.delete(id);
        return ResponseEntity.ok().build();
    }
}
```

## 安全设计原则

```

├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │ 原则一：认证和授权必须分离                           │   │
│  ├─────────────────────────────────────────────────────────┤   │
│  │                                                         │   │
│  │  - 使用独立的认证服务                                │   │

│  │                                                         │   │
│  │  - 默认情况下所有访问都是拒绝                        │   │
```

## 总结

认证和授权是安全的两个不同维度，必须正确区分。

**认证（Authentication）**：

- 证明「你是谁」
- 验证凭证的正确性
- 结果是身份标识

**授权（Authorization）**：

- 决定「你能做什么」
- 检查权限的有无
- 结果是允许或拒绝

**常见错误**：

- 登录成功 = 授权完成
- 一个 Token 做两件事
- 用户 = 权限

**设计原则**：

- 认证和授权必须分离
- 默认拒绝
- 最小权限
