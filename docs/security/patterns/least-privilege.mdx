# 最小权限原则

你的系统给所有用户管理员权限，某天一个普通用户账号被攻破，整个系统沦陷。

问题出在哪？**没有遵循最小权限原则**。

最小权限原则是安全设计的基石。

## 什么是最小权限原则

```
┌─────────────────────────────────────────────────────────────────┐
│                    最小权限原则的定义                                │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │                                                         │   │
│  │  最小权限原则（Principle of Least Privilege）：            │   │
│  │                                                         │   │
│  │  「每个用户、程序或进程都应该只被授予                      │   │
│  │    完成其工作所需的最小权限」                              │   │
│  │                                                         │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                  │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │                                                         │   │
│  │  为什么要遵循？                                         │   │
│  │                                                         │   │
│  │  1. 减少攻击面                                        │   │
│  │     权限越少，可被利用的漏洞越少                       │   │
│  │                                                         │   │
│  │  2. 限制损害范围                                        │   │
│  │     账户被攻破时，损失最小化                          │   │
│  │                                                         │   │
│  │  3. 符合合规要求                                        │   │
│  │     PCI-DSS、SOC2 等标准都要求最小权限               │   │
│  │                                                         │   │
│  │  4. 减少错误影响                                        │   │
│  │     程序错误只影响授权范围内的操作                    │   │
│  │                                                         │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

## 最小权限的维度

```
┌─────────────────────────────────────────────────────────────────┐
│                    最小权限的三个维度                                │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │ 1. 身份权限（谁可以做什么）                          │   │
│  ├─────────────────────────────────────────────────────────┤   │
│  │                                                         │   │
│  │  用户权限：只授予必要的操作权限                       │   │
│  │  - 普通用户：读、写自己的数据                       │   │
│  │  - 管理员：读、写、删、改所有数据                │   │
│  │  - 审计员：只读访问                               │   │
│  │                                                         │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                  │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │ 2. 数据权限（可以访问哪些数据）                       │   │
│  ├─────────────────────────────────────────────────────────┤   │
│  │                                                         │   │
│  │  数据范围：只访问必要的数据                          │   │
│  │  - 部门 A 员工：只能访问部门 A 的数据             │   │
│  │  - 客服人员：只能访问客户联系信息               │   │
│  │  - 财务人员：只能访问财务相关数据              │   │
│  │                                                         │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                  │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │ 3. 时间权限（在什么时间可以操作）                     │   │
│  ├─────────────────────────────────────────────────────────┤   │
│  │                                                         │   │
│  │  时间窗口：只在需要时授予权限                       │   │
│  │  - 运维人员：只在发布窗口期有写权限             │   │
│  │  - 审计人员：只在审计期间有访问权限            │   │
│  │  - 离职员工：最后一天后立即回收权限          │   │
│  │                                                         │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

## 数据库最小权限

```sql
-- 错误示范：给应用所有权限
GRANT ALL PRIVILEGES ON database.* TO 'app_user'@'%';
-- ← 极度危险！应用被攻破 = 数据库被完全控制

-- 正确做法：按需授权

-- 只读用户
CREATE USER 'app_readonly'@'%' IDENTIFIED BY 'xxx';
GRANT SELECT ON database.orders TO 'app_readonly'@'%';
GRANT SELECT ON database.products TO 'app_readonly'@'%';
GRANT SELECT ON database.users TO 'app_readonly'@'%';
FLUSH PRIVILEGES;

-- 读写用户（按表分组授权）
CREATE USER 'app_readwrite'@'%' IDENTIFIED BY 'xxx';
GRANT SELECT, INSERT, UPDATE ON database.orders TO 'app_readwrite'@'%';
GRANT SELECT, INSERT, UPDATE ON database.products TO 'app_readwrite'@'%';
-- 不允许 DELETE（数据只能通过业务流程删除）
FLUSH PRIVILEGES;

-- 管理员用户（谨慎使用）
CREATE USER 'app_admin'@'%' IDENTIFIED BY 'xxx';
GRANT ALL ON database.orders TO 'app_admin'@'%';
GRANT ALL ON database.products TO 'app_admin'@'%';
-- 不授予 ALL ON *.*（数据库级别的全权限）
```

## 应用层最小权限

```java
// Spring Security 最小权限配置
@Configuration
@EnableMethodSecurity(prePostEnabled = true)
public class LeastPrivilegeConfig {

    // 按角色配置最小权限
    @Bean
    public SecurityFilterChain filterChain(HttpSecurity http) throws Exception {
        return http
            .authorizeHttpRequests(auth -> auth
                // 公开接口：不需要权限
                .requestMatchers("/public/**", "/login", "/register").permitAll()

                // 订单模块：按操作授权
                .requestMatchers(HttpMethod.GET, "/api/orders/**")
                    .hasAnyRole("USER", "ADMIN")
                .requestMatchers(HttpMethod.POST, "/api/orders")
                    .hasRole("USER")
                .requestMatchers(HttpMethod.DELETE, "/api/orders/**")
                    .hasRole("ADMIN")

                // 用户管理：只有管理员
                .requestMatchers("/api/admin/**")
                    .hasRole("ADMIN")

                // 其他请求需要认证
                .anyRequest().authenticated()
            )
            .build();
    }
}

// Service 层细粒度权限控制
@Service
public class OrderService {

    @Autowired
    private AuthorizationService authorizationService;

    // 查询订单：只能查自己的（除非是管理员）
    public List<Order> queryOrders(Long userId, OrderQuery query) {
        // 防御：即使有查询权限，也只能查自己的数据
        if (!authorizationService.hasRole("ADMIN")) {
            query.setUserId(userId);  // 强制限定用户范围
        }
        return orderRepository.findByQuery(query);
    }

    // 取消订单：验证所有权
    public void cancelOrder(Long userId, Long orderId) {
        Order order = orderRepository.findById(orderId);

        // 防御：验证订单属于当前用户
        if (!order.getUserId().equals(userId)) {
            // 即使有 CANCEL_ORDER 权限，也不能取消别人的订单
            throw new ForbiddenException("无权取消此订单");
        }

        orderService.cancel(order);
    }
}
```

## Kubernetes 最小权限

```yaml
# 错误示范：使用默认 ServiceAccount（拥有集群管理员权限）
apiVersion: v1
kind: ServiceAccount
metadata:
  name: my-app
secrets:
  - name: my-app-token
# ← 不要使用默认的 cluster-admin 权限

# 正确做法：创建专用 RBAC

---
# 1. 创建只读 ServiceAccount
apiVersion: v1
kind: ServiceAccount
metadata:
  name: my-app
  namespace: production

---
# 2. 创建只读 Role（只允许读 pod、configmap）
apiVersion: rbac.authorization.k8s.io/v1
kind: Role
metadata:
  name: my-app-readonly
  namespace: production
rules:
  - apiGroups: [""]
    resources: ["pods", "services", "configmaps"]
    verbs: ["get", "list", "watch"]
  - apiGroups: ["apps"]
    resources: ["deployments"]
    verbs: ["get", "list"]

---
# 3. 绑定 Role 到 ServiceAccount
apiVersion: rbac.authorization.k8s.io/v1
kind: RoleBinding
metadata:
  name: my-app-readonly-binding
  namespace: production
subjects:
  - kind: ServiceAccount
    name: my-app
    namespace: production
roleRef:
  kind: Role
  name: my-app-readonly
  apiGroup: rbac.authorization.k8s.io
```

## 云资源最小权限

```json
// AWS IAM 最小权限策略示例

// 错误：过度授权
{
    "Version": "2012-10-17",
    "Statement": [{
        "Effect": "Allow",
        "Action": "*",      // ← 太宽泛！
        "Resource": "*"
    }]
}

// 正确：只读 S3 + 只写特定前缀
{
    "Version": "2012-10-17",
    "Statement": [
        {
            "Effect": "Allow",
            "Action": [
                "s3:GetObject",
                "s3:ListBucket"
            ],
            "Resource": [
                "arn:aws:s3:::my-bucket",
                "arn:aws:s3:::my-bucket/*"
            ]
        },
        {
            "Effect": "Allow",
            "Action": [
                "s3:PutObject"
            ],
            "Resource": "arn:aws:s3:::my-bucket/uploads/*"
        }
    ]
}

// 正确：只读 DynamoDB
{
    "Version": "2012-10-17",
    "Statement": [{
        "Effect": "Allow",
        "Action": [
            "dynamodb:GetItem",
            "dynamodb:Query"
        ],
        "Resource": "arn:aws:dynamodb:us-east-1:123456789:table/Users"
    }]
}
```

## 微服务间最小权限

```
┌─────────────────────────────────────────────────────────────────┐
│                    微服务间调用的最小权限                             │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │                                                         │   │
│  │  场景：订单服务调用用户服务获取用户信息                 │   │
│  │                                                         │   │
│  │  问题：                                                │   │
│  │  订单服务被攻破 → 可以访问所有用户数据                │   │
│  │                                                         │   │
│  │  解决方案：                                             │   │
│  │  1. 服务间使用专用的限权 Token                     │   │
│  │  2. 只授予必要的 API 权限                          │   │
│  │  3. 限制可访问的数据范围                            │   │
│  │                                                         │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

```java
// 服务间调用的限权 Token
@Configuration
public class ServiceTokenConfig {

    // 订单服务的限权 Token（只能调用用户服务的部分接口）
    @Bean
    public RestTemplate limitedUserServiceClient() {
        RestTemplate template = new RestTemplate();

        // 设置专用的限权 Token
        String serviceToken = generateLimitedToken(
            "order-service",
            Arrays.asList("/api/users/{id}/basic-info"),
            Arrays.asList("GET")
        );

        template.getInterceptors().add((request, body, execution) -> {
            request.getHeaders().add("X-Service-Token", serviceToken);
            return execution.execute(request, body);
        });

        return template;
    }

    // 用户服务验证服务 Token
    @Component
    public class ServiceTokenValidator {

        public boolean validateServiceToken(String token, String path, String method) {
            ServiceTokenInfo info = parseToken(token);

            // 验证服务身份
            if (!registeredServices.contains(info.getServiceId())) {
                return false;
            }

            // 验证权限范围
            return info.getAllowedPaths().contains(path)
                && info.getAllowedMethods().contains(method);
        }
    }
}
```

## 权限审计与回收

```java
// 权限审计服务
@Service
public class PrivilegeAuditService {
    @Autowired
    private AuditLogRepository auditLogRepository;

    // 定期审计权限分配
    @Scheduled(cron = "0 0 2 * * ?")  // 每天凌晨 2 点
    public void auditPrivileges() {
        // 1. 找出过度授权的账号
        List<User> overprivilegedUsers = findOverprivilegedUsers();
        for (User user : overprivilegedUsers) {
            alertSecurityTeam(user, "账号权限超出工作需要");
        }

        // 2. 找出长期未使用的权限
        List<Permission> unusedPermissions = findUnusedPermissions();
        for (Permission perm : unusedPermissions) {
            alertManager(perm, "权限长期未使用，建议回收");
        }

        // 3. 记录审计日志
        auditLogRepository.save(new AuditLog(
            "PRIVILEGE_AUDIT",
            "system",
            LocalDateTime.now(),
            "审计完成"
        ));
    }

    // 权限变更时自动审计
    @Transactional
    public void grantPermission(Long userId, String permission) {
        // 检查是否过度授权
        Set<String> existingPermissions = getUserPermissions(userId);
        if (existingPermissions.size() > MAX_RECOMMENDED_PERMISSIONS) {
            alertManager(userId, "即将被授予过多权限");
        }

        // 执行授权
        permissionRepository.grant(userId, permission);

        // 记录审计
        auditLogRepository.save(new AuditLog(
            "PERMISSION_GRANT",
            userId,
            LocalDateTime.now(),
            "授予权限: " + permission
        ));
    }
}
```

## 总结

最小权限原则是安全设计的基石。

**三个维度**：

- 身份权限：谁可以做什么
- 数据权限：可以访问哪些数据
- 时间权限：在什么时间可以操作

**实施要点**：

- 数据库：按表、按操作授权
- 应用层：RBAC + ABAC 细粒度控制
- Kubernetes：专用 ServiceAccount + RBAC
- 云资源：专用 IAM Policy
- 服务间：限权 Token

**最佳实践**：

- 默认拒绝，显式授权
- 权限定期审计
- 及时回收无用权限
- 权限变更记录审计日志

**核心认知**：最小权限不是一次性的工作，而是持续的安全实践。系统演进过程中，权限可能不断累积，定期审计和清理是必要的。
