# SSO 单点登录

你的公司有 10 个系统，用户每个都要登录一次。

问题出在哪？**没有实现单点登录**。

SSO 让用户一次登录，处处访问。

## 什么是 SSO

```
┌─────────────────────────────────────────────────────────────────┐
│                    SSO 简介                                            │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │                                                         │   │
│  │  SSO（Single Sign-On）：                               │   │
│  │                                                         │   │
│  │  「用户只需登录一次，就可以访问所有相互信任的系统」      │   │
│  │                                                         │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                  │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │                                                         │   │
│  │  没有 SSO：                                            │   │
│  │  - 登录系统 A → 输入用户名密码                        │   │
│  │  - 切换到系统 B → 再次输入用户名密码                  │   │
│  │  - 切换到系统 C → 又要输入用户名密码                   │   │
│  │  用户体验差，密码管理混乱                             │   │
│  │                                                         │   │
│  │  有 SSO：                                               │   │
│  │  - 登录系统 A → 输入用户名密码                        │   │
│  │  - 切换到系统 B → 自动登录                            │   │
│  │  - 切换到系统 C → 自动登录                            │   │
│  │  用户体验好，密码管理统一                             │   │
│  │                                                         │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

## SSO 工作原理

```
┌─────────────────────────────────────────────────────────────────┐
│                    SSO 工作原理                                        │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │                                                         │   │
│  │  核心机制：共享登录状态                              │   │
│  │                                                         │   │
│  │  1. 用户访问系统 A，未登录                          │   │
│  │  2. 系统 A 跳转到 SSO 登录页                        │   │
│  │  3. 用户在 SSO 登录页输入用户名密码                 │   │
│  │  4. SSO 验证通过，创建全局 Session               │   │
│  │  5. SSO 返回 Ticket 给系统 A                     │   │
│  │  6. 系统 A 验证 Ticket，建立本地 Session          │   │
│  │  7. 用户访问系统 B                                  │   │
│  │  8. 系统 B 发现未登录，跳转到 SSO                │   │
│  │  9. SSO 发现已登录，直接返回 Ticket 给系统 B      │   │
│  │  10. 系统 B 验证 Ticket，建立本地 Session         │   │
│  │                                                         │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

## CAS 协议

```
┌─────────────────────────────────────────────────────────────────┐
│                    CAS 协议                                            │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │                                                         │   │
│  │  CAS（Central Authentication Service）：                 │   │
│  │                                                         │   │
│  │  耶鲁大学开发的 SSO 解决方案                          │   │
│  │  业界最成熟的 SSO 协议之一                            │   │
│  │                                                         │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                  │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │                                                         │   │
│  │  CAS 流程：                                           │   │
│  │                                                         │   │
│  │  1. 用户访问系统 A                                   │   │
│  │  2. 系统 A 检查 Ticket，未登录                       │   │
│  │  3. 重定向到 CAS: /login?service=系统A回调URL        │   │
│  │  4. CAS 检查全局 Session，未登录                    │   │
│  │  5. 显示登录页面，用户输入凭证                       │   │
│  │  6. CAS 验证通过，创建 TGC（Ticket Granting Cookie）│   │
│  │  7. 重定向回系统 A，携带 Ticket                     │   │
│  │  8. 系统 A 用 Ticket 到 CAS 验证                     │   │
│  │  9. CAS 返回用户信息                                 │   │
│  │  10. 系统 A 创建本地 Session                        │   │
│  │                                                         │   │
│  │  后续访问系统 B：                                     │   │
│  │  1. 系统 B 重定向到 CAS（携带 service URL）          │   │
│  │  2. CAS 用 TGC 检查全局 Session，已登录            │   │
│  │  3. CAS 直接返回 Ticket 给系统 B                   │   │
│  │  4. 系统 B 验证 Ticket，建立本地 Session            │   │
│  │                                                         │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

### Spring Security SSO 实现

```java
// OAuth 2.0 登录配置（支持多身份提供商）
@Configuration
@EnableWebSecurity
public class MultiProviderSecurityConfig {

    @Bean
    public SecurityFilterChain filterChain(HttpSecurity http) throws Exception {
        return http
            .authorizeRequests(auth -> auth
                .requestMatchers("/login", "/oauth2/**", "/sso/**").permitAll()
                .anyRequest().authenticated()
            )
            // 微信登录
            .oauth2Login(oauth2 -> oauth2
                .loginPage("/oauth2/authorization/wechat")
                .authorizationEndpoint(authorization -> authorization
                    .baseUri("/oauth2/authorization/wechat")
                )
                .defaultSuccessUrl("/login/wechat", true)
            )
            // 企业微信登录
            .oauth2Login(".oauth2Login", oauth2 -> oauth2
                .loginPage("/oauth2/authorization/wxwork")
                .authorizationEndpoint(authorization -> authorization
                    .baseUri("/oauth2/authorization/wxwork")
                )
                .defaultSuccessUrl("/login/wxwork", true)
            )
            // OIDC 登录
            .oauth2Login(".oidcLogin", oauth2 -> oauth2
                .loginPage("/oauth2/authorization/oidc")
                .userInfoEndpoint(userInfo -> userInfo
                    .oidcUserService(oidcUserService())
                )
            )
            .build();
    }
}

// SSO 状态管理
@Service
public class SsoStateService {
    @Autowired
    private RedisTemplate<String, Object> redisTemplate;

    // 生成状态参数
    public String generateState(String targetSystem) {
        String state = UUID.randomUUID().toString();

        SsoState ssoState = new SsoState();
        ssoState.setTargetSystem(targetSystem);
        ssoState.setCreatedAt(LocalDateTime.now());
        ssoState.setExpireAt(LocalDateTime.now().plusMinutes(10));

        redisTemplate.opsForValue().set(
            "sso:state:" + state,
            ssoState,
            Duration.ofMinutes(10)
        );

        return state;
    }

    // 验证状态
    public boolean validateState(String state, String targetSystem) {
        String key = "sso:state:" + state;
        SsoState ssoState = (SsoState) redisTemplate.opsForValue().get(key);

        if (ssoState == null) {
            return false;
        }

        if (ssoState.getExpireAt().isBefore(LocalDateTime.now())) {
            redisTemplate.delete(key);
            return false;
        }

        if (!ssoState.getTargetSystem().equals(targetSystem)) {
            return false;
        }

        redisTemplate.delete(key);
        return true;
    }
}
```

## SSO 与 OAuth 2.0 的区别

```
┌─────────────────────────────────────────────────────────────────┐
│                    SSO vs OAuth 2.0                                   │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  SSO：同一组织内的多个系统共享登录状态                 │   │
│  ├─────────────────────────────────────────────────────────┤   │
│  │                                                         │   │
│  │  场景：企业内部系统（ERP、CRM、OA）                    │   │
│  │  用户：同一组织的员工                                │   │
│  │  登录页：统一的身份认证入口                          │   │
│  │                                                         │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                  │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  OAuth 2.0：第三方应用获取用户在其他平台的资源        │   │
│  ├─────────────────────────────────────────────────────────┤   │
│  │                                                         │   │
│  │  场景：第三方应用（用微信登录、小程序授权）           │   │
│  │  用户：各平台用户                                   │   │
│  │  登录页：各平台的授权页面                          │   │
│  │                                                         │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                  │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │                                                         │   │
│  │  实际场景可以结合：                                   │   │
│  │                                                         │   │
│  │  - 企业内部系统用 SSO                              │   │
│  │  - 第三方登录用 OAuth 2.0 / OIDC                  │   │
│  │  - 两者可以共用同一个身份提供商                    │   │
│  │                                                         │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

### Session 共享 SSO 配置

```java
// 所有子系统共享 Redis Session
@Configuration
@EnableRedisHttpSession(maxInactiveIntervalInSeconds = 1800)
public class SharedSessionConfig {
    // 所有子系统都使用相同的配置
    // Session 自动在 Redis 中共享
}

// 统一 Cookie 域配置
// application.yml（所有子系统相同配置）
server:
  servlet:
    session:
      cookie:
        name: SESSIONID
        domain: .example.com  # 共享 Cookie 的域
        http-only: true
        secure: true
        same-site: strict
```

## 单点退出

```java
// 单点退出服务
@Service
public class SingleLogoutService {
    @Autowired
    private ApplicationEventPublisher eventPublisher;

    // 全局登出
    public void globalLogout(HttpSession session) {
        Long userId = (Long) session.getAttribute("userId");

        // 1. 发布全局登出事件
        eventPublisher.publishEvent(new GlobalLogoutEvent(userId));

        // 2. 销毁本地 Session
        session.invalidate();

        // 3. 通知其他子系统（可选）
        notifyOtherSystems(userId);
    }

    // 处理来自其他子系统的登出通知
    @KafkaListener(topics = "logout-notifications")
    public void handleLogoutNotification(LogoutNotification notification) {
        if (notification.getType() == LogoutType.GLOBAL) {
            // 全局登出，销毁本地 Session
            HttpSession session = findSessionByUserId(notification.getUserId());
            if (session != null) {
                session.invalidate();
            }
        }
    }
}
```

## 总结

SSO 让用户一次登录，处处访问。

**SSO vs OAuth 2.0**：

- SSO：同一组织内共享登录状态
- OAuth 2.0：第三方应用获取资源授权

**实现方式**：

- CAS 协议：经典的 SSO 协议
- OAuth 2.0 / OIDC：现代 SSO 方案

**Session 共享**：

- Redis Session 共享：简单高效
- 统一 Cookie 域：跨域访问

**核心认知**：SSO 解决的是「减少重复登录」的问题，OAuth 2.0 解决的是「安全授权」的问题。两者关注点不同，但可以结合使用。
