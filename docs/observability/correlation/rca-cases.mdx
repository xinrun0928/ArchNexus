# 根因分析案例实战

理论需要与实践结合才能真正掌握。本节通过三个典型故障场景，展示根因分析的实际流程。这些案例来自真实生产环境，每个案例都包含完整的分析思路和关键决策点。

## 案例一：级联故障

**场景**：凌晨 3 点，监控系统触发大量告警：订单服务超时告警、支付服务不可用告警、数据库连接池耗尽告警同时触发。值班人员需要在 5 分钟内判断「这是几个问题？根因在哪？」

**分析过程**：首先通过告警聚合将 100+ 条告警收敛为 3 个告警组。然后按时间顺序分析：数据库连接池告警最早触发，发生在 02:58:15；订单服务超时在 02:58:20 开始出现；支付服务在 02:58:25 不可用。这条时间链清晰地指向根因：数据库是罪魁祸首。

**定位结论**：数据库连接池配置过小，夜间批量任务导致连接池打满，新请求全部等待，最终超时。通过扩大连接池容量和错开批量任务时间解决。

## 案例二：依赖服务故障

**场景**：外部支付渠道 API 延迟突然从 50ms 上升到 5 秒，导致订单系统超时告警。问题是：应该通知哪个团队？是订单系统的问题还是支付渠道的问题？

**分析过程**：通过链路追踪分析，发现订单系统对支付渠道的调用超时，但其他内部服务的延迟正常。进一步分析发现：支付渠道的延迟是全局性的，所有调用方都受影响。这排除了「订单系统代码问题」的可能性。

**定位结论**：外部依赖故障，属于支付渠道侧问题，非我方责任。但需要启动降级预案：暂时关闭实时支付，切换到备用渠道，避免用户下单失败。故障根因需要等支付渠道恢复后通过其故障报告确认。

## 案例三：配置变更引发的雪崩

**场景**：周二下午进行了配置变更后，系统 CPU 使用率持续 100%，响应时间从 50ms 上升到 500ms。变更内容包括：升级了某个 RPC 框架版本、调整了线程池大小。

**分析过程**：首先排查是哪个配置变更导致的问题。通过时间线分析，发现 CPU 飙升发生在 RPC 框架升级之后。通过链路分析，发现 GC 频率异常高，但堆内存使用率正常。进一步分析发现：新版本 RPC 框架的序列化方式变更，导致对象分配量增加，引发频繁 GC。

**定位结论**：RPC 框架升级引入了不兼容的序列化方式，导致对象分配过多。解决方案是回滚 RPC 框架版本，并增加序列化性能测试用例到 CI 流程。
