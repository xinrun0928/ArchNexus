# 漏桶算法

你需要确保系统以固定的速率处理请求，即使请求突增也不能超过这个速率。

问题出在哪？**漏桶正是解决这个问题的方案**。

```
│  │ 示例：容量 100，漏速 10/s                           │   │
```

## 漏桶的实现

### 基本实现

```java
// 漏桶实现
public class LeakyBucket {
    private final double leakRate;      // 每毫秒漏出多少请求
    private final int capacity;         // 桶容量
    private double water;              // 当前水量（排队请求数）
    private long lastLeakTime;        // 上次漏水时间

    public LeakyBucket(double leakRate, int capacity) {
        this.leakRate = leakRate;
        this.capacity = capacity;
        this.water = 0;
        this.lastLeakTime = System.currentTimeMillis();
    }

    public synchronized boolean tryAcquire() {
        leak();

        if (water < capacity) {
            water++;
            return true;
        }

        return false;
    }

    private void leak() {
        long now = System.currentTimeMillis();
        long elapsed = now - lastLeakTime;

        // 计算漏出的水量
        double leaked = elapsed * leakRate;
        water = Math.max(0, water - leaked);

        lastLeakTime = now;
    }
}
```

### 异步漏桶

```java
// 异步漏桶（用于消息队列限流）
public class AsyncLeakyBucket {
    private final double leakRate;
    private final int capacity;
    private final ConcurrentLinkedQueue<Request> bucket = new ConcurrentLinkedQueue<>();
    private volatile double water = 0;
    private final ScheduledExecutorService scheduler;
    private final ExecutorService processor;

    public AsyncLeakyBucket(double leakRate, int capacity) {
        this.leakRate = leakRate;
        this.capacity = capacity;
        this.processor = Executors.newFixedThreadPool(10);

        this.scheduler = Executors.newSingleThreadScheduledExecutor();
        scheduler.scheduleAtFixedRate(() -> {
            leakAndProcess();
        }, 1, 1, TimeUnit.MILLISECONDS);  // 每毫秒漏出一次
    }

    public boolean tryAcquire(Request request) {
        if (water < capacity) {
            bucket.offer(request);
            water++;
            return true;
        }
        return false;
    }

    private void leakAndProcess() {
        Request request = bucket.poll();
        if (request != null) {
            water = Math.max(0, water - 1);
            processor.submit(() -> process(request));
        }
    }
}
```

## 漏桶的应用场景

### 消息队列限流

```java
// 消息队列漏桶限流
@Service
public class LeakyBucketMessageProducer {
    private final LeakyBucket rateLimiter;
    private final KafkaTemplate<String, Object> kafkaTemplate;

    public LeakyBucketMessageProducer() {
        // 每秒最多发送 1000 条消息
        this.rateLimiter = new LeakyBucket(1.0, 1000);
    }

    public void sendMessage(String topic, Object message) {
        if (rateLimiter.tryAcquire()) {
            kafkaTemplate.send(topic, message);
        } else {
            // 消息被限流，可以选择重试、降级、或丢弃
            log.warn("消息被限流: {}", message);
            handleOverflow(message);
        }
    }
}
```

### 连接数限流

```java
// 连接数漏桶限流
public class ConnectionRateLimiter {
    private final LeakyBucket bucket;
    private final ExecutorService connectionExecutor;

    public ConnectionRateLimiter() {
        // 每秒最多建立 100 个连接
        this.bucket = new LeakyBucket(1.0, 100);
        this.connectionExecutor = Executors.newFixedThreadPool(100);
    }

    public CompletableFuture<Connection> connect(String host, int port) {
        if (bucket.tryAcquire()) {
            return CompletableFuture.supplyAsync(() -> {
                try {
                    return new Connection(host, port);
                } catch (Exception e) {
                    throw new RuntimeException(e);
                }
            }, connectionExecutor);
        } else {
            return CompletableFuture.failedFuture(
                new RateLimitException("连接数超限"));
        }
    }
}
```

## 漏桶与队列的区别

```
┌─────────────────────────────────────────────────────────────────┐
│                    漏桶 vs 队列                                  │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │ 队列                                                   │   │
│  ├─────────────────────────────────────────────────────────┤   │
│  │ - 请求积压，没有速率控制                              │   │
│  │ - 内存可能无限增长                                    │   │
│  │ - 处理速度取决于消费者                                │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                  │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │ 漏桶                                                   │   │
│  ├─────────────────────────────────────────────────────────┤   │
│  │ - 有最大积压容量（桶容量）                            │   │
│  │ - 有固定处理速率（漏速）                              │   │

│                                                                  │

│  - 需要平滑处理 + 最大积压限制：漏桶                          │
```

## 漏桶的优缺点

```
┌─────────────────────────────────────────────────────────────────┐
│                    漏桶优缺点                                    │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  优点：                                                        │
│  ✓ 流量完全平滑，以恒定速率输出                               │
│  ✓ 有最大积压容量，不会无限积压                              │
│  ✓ 超过容量的请求被明确拒绝                                   │
│                                                                  │
│  缺点：                                                        │
│  ✗ 不允许突发，即使系统空闲也只能按固定速率处理                │
│  ✗ 如果漏速设置太低，可能导致正常请求被拒绝                  │
│  ✗ 实现比令牌桶稍复杂                                        │
│                                                                  │
│  适用场景：                                                    │
│  - 需要严格流量整形的场景                                     │
│  - 下游系统只能承受固定速率                                   │
│  - 消息队列等需要平滑消费的中间件                             │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

## 总结

漏桶是流量整形的利器。

**核心特点**：

- 固定输出速率
- 最大积压限制
- 完全不允许突发

**与令牌桶的区别**：

| 特性 | 令牌桶 | 漏桶 |
|---|---|---|
| 突发能力 | 允许 | 不允许 |
| 输出速率 | 平均速率 | 固定速率 |
| 适用场景 | API 限流 | 流量整形 |

**选择建议**：

- API 限流：令牌桶（允许突发）
- 消息队列消费：漏桶（严格平滑）
- 下游保护：漏桶（严格速率）
