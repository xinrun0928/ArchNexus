# 健康检查类型

你的服务明明不健康，但健康检查却显示正常。

问题出在哪？**没有正确理解健康检查的类型**。

健康检查不只是简单的「服务在不在」，而是要检测服务是否「能正常工作」。

## 健康检查类型概览

```
┌─────────────────────────────────────────────────────────────────┐
│                    健康检查类型                                    │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │                    存活探针（Liveness）                  │   │
│  ├─────────────────────────────────────────────────────────┤   │
│  │ 服务是否存活（进程是否在运行）                       │   │
│  │ 失败时：重启服务                                      │   │
│  │ 用途：检测程序是否卡死、OOM 等                        │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                  │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │                    就绪探针（Readiness）                  │   │
│  ├─────────────────────────────────────────────────────────┤   │
│  │ 服务是否就绪（能否处理请求）                         │   │
│  │ 失败时：停止接收流量                                  │   │
│  │ 用途：检测依赖是否就绪、是否已完成启动               │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                  │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │                    启动探针（Startup）                   │   │
│  ├─────────────────────────────────────────────────────────┤   │
│  │ 服务是否已完成启动                                    │   │
│  │ 失败时：重启服务                                      │   │
│  │ 用途：检测启动时间较长的服务                         │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

## 存活探针（Liveness Probe）

### 作用

```
┌─────────────────────────────────────────────────────────────────┐
│                    存活探针的作用                                  │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  问题：程序卡死了，但还在运行                                  │
│                                                                  │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │                                                         │   │
│  │  程序正在运行                                           │   │
│  │    │                                                    │   │
│  │    ▼                                                    │   │
│  │  线程死锁 ──▶ 无法处理请求                           │   │
│  │    │                                                    │   │
│  │    ▼                                                    │   │
│  │  进程还在                                               │   │
│  │    │                                                    │   │
│  │    ▼                                                    │   │
│  │  健康检查返回「正常」                                 │   │
│  │    │                                                    │   │
│  │    ▼                                                    │   │
│  │  但用户请求都失败了！                                │   │
│  │                                                         │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

### 适用场景

```
┌─────────────────────────────────────────────────────────────────┐
│                    存活探针适用场景                               │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  ✓ 检测线程死锁                                              │
│  ✓ 检测死循环                                               │
│  ✓ 检测 OOM（内存耗尽）                                    │
│  ✓ 检测外部资源耗尽                                         │
│                                                                  │
│  ✗ 不适合检测依赖服务是否可用                                │
│  ✗ 不适合检测业务逻辑错误                                    │
│                                                                  │
│  关键：存活探针检测的是「程序能否自救」                      │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

## 就绪探针（Readiness Probe）

### 作用

```
┌─────────────────────────────────────────────────────────────────┐
│                    就绪探针的作用                                  │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  问题：服务启动了，但依赖还没准备好                          │
│                                                                  │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │                                                         │   │
│  │  服务启动                                               │   │
│  │    │                                                    │   │
│  │    ▼                                                    │   │
│  │  接收用户请求                                           │   │
│  │    │                                                    │   │
│  │    ▼                                                    │   │
│  │  调用数据库 ──▶ 连接还没建立！                       │   │
│  │    │                                                    │   │
│  │    ▼                                                    │   │
│  │  请求失败！                                            │   │
│  │                                                         │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                  │
│  就绪探针可以确保服务真正就绪后再接收流量                   │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

### 适用场景

```
┌─────────────────────────────────────────────────────────────────┐
│                    就绪探针适用场景                               │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  ✓ 检测依赖服务是否可用（数据库、缓存）                        │
│  ✓ 检测配置是否加载完成                                       │
│  ✓ 检测初始化是否完成                                         │
│  ✓ 检测负载是否过高                                           │
│                                                                  │
│  ✗ 不适合检测程序崩溃                                        │
│  ✗ 不适合检测死锁                                            │
│                                                                  │
│  关键：就绪探针检测的是「服务能否处理请求」                   │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

## 启动探针（Startup Probe）

### 作用

```
┌─────────────────────────────────────────────────────────────────┐
│                    启动探针的作用                                  │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  问题：服务启动很慢，存活探针在启动期间就失败了              │
│                                                                  │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │                                                         │   │
│  │  服务启动需要 60 秒                                    │   │
│  │  存活探针每 10 秒检查一次                             │   │
│  │  存活探针 3 次失败后重启                               │   │
│  │                                                         │   │
│  │  结果：服务永远无法启动                                 │   │
│  │                                                         │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                  │
│  启动探针：在启动期间替代存活探针                          │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

### 适用场景

```
┌─────────────────────────────────────────────────────────────────┐
│                    启动探针适用场景                               │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  ✓ 启动时间较长的服务                                         │
│  ✓ 需要加载大量数据的服务                                     │
│  ✓ 需要连接多个依赖的服务                                     │
│                                                                  │
│  特点：                                                       │
│  - 在服务启动期间使用                                         │
│  - 启动完成后，存活探针和就绪探针 开始工作                     │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

## 健康检查实现

### Spring Boot Actuator

```java
// Spring Boot 健康检查
@Component
public class CustomHealthIndicator implements HealthIndicator {

    @Override
    public Health health() {
        // 检查数据库连接
        if (!checkDatabase()) {
            return Health.down()
                .withDetail("database", "unavailable")
                .build();
        }

        // 检查 Redis
        if (!checkRedis()) {
            return Health.down()
                .withDetail("redis", "unavailable")
                .build();
        }

        // 检查内存
        if (isMemoryPressure()) {
            return Health.status("DEGRADED")
                .withDetail("memory", "high usage")
                .build();
        }

        return Health.up().build();
    }

    private boolean checkDatabase() {
        try {
            jdbcTemplate.execute("SELECT 1");
            return true;
        } catch (Exception e) {
            return false;
        }
    }

    private boolean checkRedis() {
        try {
            redisTemplate.opsForValue().get("health");
            return true;
        } catch (Exception e) {
            return false;
        }
    }

    private boolean isMemoryPressure() {
        MemoryMXBean memoryMXBean = ManagementFactory.getMemoryMXBean();
        long used = memoryMXBean.getHeapMemoryUsage().getUsed();
        long max = memoryMXBean.getHeapMemoryUsage().getMax();
        return (double) used / max > 0.9;
    }
}
```

### Kubernetes 健康检查配置

```yaml
# Kubernetes 健康检查配置
apiVersion: apps/v1
kind: Deployment
metadata:
  name: order-service
spec:
  template:
    spec:
      containers:
      - name: order-service
        # 启动探针（可选，用于启动慢的服务）
        startupProbe:
          httpGet:
            path: /actuator/health/startup
            port: 8080
          initialDelaySeconds: 10    # 启动后等待 10 秒
          periodSeconds: 5            # 每 5 秒检查一次
          failureThreshold: 12         # 失败 12 次（60 秒）后认为启动失败

        # 存活探针
        livenessProbe:
          httpGet:
            path: /actuator/health/liveness
            port: 8080
          initialDelaySeconds: 30    # 启动后等待 30 秒
          periodSeconds: 10           # 每 10 秒检查一次
          failureThreshold: 3         # 失败 3 次（30 秒）后重启

        # 就绪探针
        readinessProbe:
          httpGet:
            path: /actuator/health/readiness
            port: 8080
          initialDelaySeconds: 10    # 启动后等待 10 秒
          periodSeconds: 5            # 每 5 秒检查一次
          failureThreshold: 3         # 失败 3 次后停止接收流量
          successThreshold: 1         # 成功 1 次后恢复接收流量
```

### 分层健康检查

```java
// 分层健康检查
@RestController
@RequestMapping("/actuator/health")
public class LayeredHealthController {

    @GetMapping("/liveness")
    public ResponseEntity<Health> liveness() {
        // 存活检查：只检查进程是否存活
        if (isProcessAlive()) {
            return ResponseEntity.ok(Health.up().build());
        }
        return ResponseEntity.status(503).body(Health.down().build());
    }

    @GetMapping("/readiness")
    public ResponseEntity<Health> readiness() {
        // 就绪检查：检查依赖是否就绪
        Health.Builder builder = Health.up();

        // 检查数据库
        if (!checkDatabase()) {
            builder.down().withDetail("database", "unavailable");
        }

        // 检查 Redis
        if (!checkRedis()) {
            builder.down().withDetail("redis", "unavailable");
        }

        // 检查外部服务
        if (!checkExternalServices()) {
            builder.down().withDetail("external", "unavailable");
        }

        if (builder.build().getStatus() == Status.DOWN) {
            return ResponseEntity.status(503).body(builder.build());
        }

        return ResponseEntity.ok(builder.build());
    }

    @GetMapping("/startup")
    public ResponseEntity<Health> startup() {
        // 启动检查：检查启动是否完成
        if (isStartupComplete()) {
            return ResponseEntity.ok(Health.up().build());
        }
        return ResponseEntity.status(503).body(Health.down().build());
    }

    private boolean isProcessAlive() {
        return true;  // 简单检查：进程在运行
    }

    private boolean isStartupComplete() {
        return startupLatch.getCount() == 0;
    }
}
```

## 健康检查最佳实践

### 检查策略

```
┌─────────────────────────────────────────────────────────────────┐
│                    健康检查最佳实践                               │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  1. 存活探针：简单、快速                                       │
│     - 检查进程是否存活                                          │
│     - 不检查依赖                                               │
│     - 失败后重启                                               │
│                                                                  │
│  2. 就绪探针：全面、准确                                       │
│     - 检查所有依赖                                             │
│     - 检查资源状态                                             │
│     - 失败后停止接收流量                                       │
│                                                                  │
│  3. 启动探针：允许长启动时间                                   │
│     - 给服务足够的启动时间                                     │
│     - 启动完成前不执行存活探针                                 │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

### 避免误判

```java
// 避免误判：设置合理的超时
@Component
public class CarefulHealthIndicator implements HealthIndicator {

    @Override
    public Health health() {
        Health.Builder builder = Health.up();

        // 数据库检查：设置超时，避免长时间等待
        try {
            if (!checkDatabaseWithTimeout(2, TimeUnit.SECONDS)) {
                builder.down().withDetail("database", "timeout");
            }
        } catch (Exception e) {
            builder.down().withDetail("database", e.getMessage());
        }

        // Redis 检查：设置超时
        try {
            if (!checkRedisWithTimeout(1, TimeUnit.SECONDS)) {
                builder.down().withDetail("redis", "timeout");
            }
        } catch (Exception e) {
            builder.down().withDetail("redis", e.getMessage());
        }

        return builder.build();
    }

    private boolean checkDatabaseWithTimeout(long timeout, TimeUnit unit) {
        return CompletableFuture.supplyAsync(() -> checkDatabase())
            .timeout(timeout, unit)
            .orElse(false);
    }
}
```

## 总结

三种健康检查探针各有分工。

**存活探针（Liveness）**：

- 检测进程是否存活
- 失败时重启
- 检查应该简单快速

**就绪探针（Readiness）**：

- 检测是否能处理请求
- 失败时停止接收流量
- 检查依赖和资源

**启动探针（Startup）**：

- 检测启动是否完成
- 启动期间替代存活探针
- 适合启动慢的服务

**选择建议**：

| 场景 | 探针类型 |
|---|---|
| 程序崩溃 | Liveness |
| 线程死锁 | Liveness |
| 数据库不可用 | Readiness |
| 缓存不可用 | Readiness |
| 启动时间长 | Startup |
| 启动完成后检查 | Liveness + Readiness |
