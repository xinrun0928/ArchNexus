# 存活探针

你的服务已经卡死了，但 Kubernetes 还在往它发送流量。

问题出在哪？**存活探针配置不当**。

存活探针是防止将流量发送到「僵尸进程」的最后防线。

## 存活探针配置

### HTTP 探针

```yaml
# HTTP 存活探针
apiVersion: apps/v1
kind: Deployment
metadata:
  name: order-service
spec:
  template:
    spec:
      containers:
      - name: order-service
        livenessProbe:
          httpGet:
            path: /actuator/health/liveness
            port: 8080
            scheme: HTTP
          initialDelaySeconds: 30    # 启动后 30 秒开始检查
          periodSeconds: 10          # 每 10 秒检查一次
          failureThreshold: 3        # 连续 3 次失败后重启（30 秒）
          timeoutSeconds: 5          # 每次检查超时 5 秒
          successThreshold: 1        # 成功 1 次即认为健康
```

### TCP 探针


## 存活探针实现

### Spring Boot 实现

```java
// 存活探针实现
@RestController
@RequestMapping("/actuator/health")
public class LivenessProbeController {

    private final AtomicBoolean isAlive = new AtomicBoolean(true);

    @GetMapping("/liveness")
    public ResponseEntity<Health> liveness() {
        // 简单检查：进程是否响应
        return ResponseEntity.ok(Health.up()
            .withDetail("timestamp", System.currentTimeMillis())
            .build());
    }

    // 程序可以主动标记自己为不健康
    public void markAsDead() {
        isAlive.set(false);
    }

    public void markAsAlive() {
        isAlive.set(true);
    }
}
```

### 检测常见僵尸状态

```java
// 检测僵尸状态
@Component
public class ZombieStateDetector implements HealthIndicator {

    @Override
    public Health health() {
        // 检查线程状态
        if (hasDeadlock()) {
            return Health.down()
                .withDetail("reason", "deadlock detected")
                .build();
        }

        // 检查内存使用
        if (isOutOfMemory()) {
            return Health.down()
                .withDetail("reason", "out of memory")
                .build();
        }

        // 检查线程池
        if (isThreadPoolExhausted()) {
            return Health.down()
                .withDetail("reason", "thread pool exhausted")
                .build();
        }

        return Health.up().build();
    }

    private boolean hasDeadlock() {
        ThreadMXBean threadMXBean = ManagementFactory.getThreadMXBean();
        long[] deadlockedThreads = threadMXBean.findDeadlockedThreads();
        return deadlockedThreads != null && deadlockedThreads.length > 0;
    }

    private boolean isOutOfMemory() {
        MemoryMXBean memoryMXBean = ManagementFactory.getMemoryMXBean();
        MemoryUsage heapUsage = memoryMXBean.getHeapMemoryUsage();
        double usedRatio = (double) heapUsage.getUsed() / heapUsage.getMax();
        return usedRatio > 0.95;  // 95% 以上认为 OOM
    }

    private boolean isThreadPoolExhausted() {
        // 检查常用线程池
        ThreadPoolExecutor executor = (ThreadPoolExecutor) Executors.newFixedThreadPool(10);
        int activeCount = executor.getActiveCount();
        int poolSize = executor.getPoolSize();
        return activeCount >= poolSize;
    }
}
```

## 存活探针的最佳实践

### 配置原则

```
┌─────────────────────────────────────────────────────────────────┐
│                    存活探针配置原则                               │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  1. initialDelaySeconds                                        │
│     ┌─────────────────────────────────────────────────────────┐   │
│     │                                                         │   │
│     │  设置为服务启动时间 + buffer                           │   │
│     │  如果服务启动需要 30 秒，设置 35-40 秒              │   │
│     │                                                         │   │
│     │  太短：服务还没启动完就被认为不健康                  │   │
│     │  太长：僵尸状态持续时间长                             │   │
│     │                                                         │   │
│     └─────────────────────────────────────────────────────────┘   │
│                                                                  │
│  2. periodSeconds                                              │
│     ┌─────────────────────────────────────────────────────────┐   │
│     │                                                         │   │
│     │  一般设置 10-30 秒                                     │   │
│     │                                                         │   │
│     │  太短：增加负载                                        │   │
│     │  太长：检测延迟大                                     │   │
│     │                                                         │   │
│     └─────────────────────────────────────────────────────────┘   │
│                                                                  │
│  3. failureThreshold                                            │
│     ┌─────────────────────────────────────────────────────────┐   │
│     │                                                         │   │
│     │  一般设置 3-5                                          │   │
│     │                                                         │   │
│     │  配合 periodSeconds 计算重启时间                       │   │
│     │  period=10, threshold=3 → 30 秒后重启               │   │
│     │                                                         │   │
│     └─────────────────────────────────────────────────────────┘   │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

### 存活探针应该做什么

```
┌─────────────────────────────────────────────────────────────────┐
│                    存活探针应该做什么                              │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  ✓ 简单、快速                                                 │
│  ✓ 只检查进程是否存活                                          │
│  ✓ 不检查外部依赖                                             │
│  ✓ 不检查业务逻辑                                             │
│                                                                  │
│  ✗ 不要检查数据库连接                                         │
│  ✗ 不要检查 Redis 连接                                        │
│  ✗ 不要执行复杂检查                                           │
│                                                                  │
│  存活探针失败 = 重启 Pod                                     │
│  如果存活探针检查外部依赖，外部依赖不可用时会无限重启！        │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

## 总结

存活探针是防止流量发送到僵尸进程的机制。

**配置要点**：

- initialDelaySeconds：启动时间 + buffer
- periodSeconds：10-30 秒
- failureThreshold：3-5
- timeoutSeconds：3-5 秒

**存活探针应该**：

- 简单、快速
- 只检查进程存活
- 不检查外部依赖
- 不检查业务逻辑

**存活探针失败的结果**：

- Kubernetes 重启 Pod
- 可能导致服务短暂不可用
- 应该尽可能避免误判
