# 就绪探针

你的服务启动了，但依赖还没准备好，Kubernetes 却开始往它发送流量了。

问题出在哪？**就绪探针配置不当**。

就绪探针确保只有真正「准备好」的服务才会接收流量。

## 就绪探针配置

### HTTP 探针

```yaml
# HTTP 就绪探针
apiVersion: apps/v1
kind: Deployment
metadata:
  name: order-service
spec:
  template:
    spec:
      containers:
      - name: order-service
        readinessProbe:
          httpGet:
            path: /actuator/health/readiness
            port: 8080
            scheme: HTTP
          initialDelaySeconds: 10    # 启动后 10 秒开始检查
          periodSeconds: 5            # 每 5 秒检查一次
          failureThreshold: 3         # 连续 3 次失败后移出
          timeoutSeconds: 3           # 每次检查超时 3 秒
          successThreshold: 1         # 成功 1 次后加入
```

### 就绪探针应该检查的内容

```java
// 应该检查的内容
@RestController
@RequestMapping("/actuator/health")
public class ReadinessProbeController {

    @Autowired
    private DataSource dataSource;

    @Autowired
    private RedisTemplate<String, Object> redisTemplate;

    @Autowired
    private RestTemplate externalServiceTemplate;

    @GetMapping("/readiness")
    public ResponseEntity<Health> readiness() {
        Health.Builder builder = Health.up();

        // 检查数据库
        if (!isDatabaseReady()) {
            builder.down().withDetail("database", "not ready");
        }

        // 检查 Redis
        if (!isRedisReady()) {
            builder.down().withDetail("redis", "not ready");
        }

        // 检查外部服务（可选）
        if (!isExternalServiceReady()) {
            builder.down().withDetail("externalService", "not ready");
        }

        // 检查资源
        if (isResourcePressure()) {
            builder.down().withDetail("resource", "high pressure");
        }

        // 检查线程池
        if (isThreadPoolBusy()) {
            builder.down().withDetail("threadPool", "busy");
        }

        Health health = builder.build();
        if (health.getStatus() == Status.DOWN) {
            return ResponseEntity.status(503).body(health);
        }

        return ResponseEntity.ok(health);
    }

    private boolean isDatabaseReady() {
        try {
            jdbcTemplate.execute("SELECT 1");
            return true;
        } catch (Exception e) {
            return false;
        }
    }

    private boolean isRedisReady() {
        try {
            redisTemplate.opsForValue().get("health");
            return true;
        } catch (Exception e) {
            return false;
        }
    }

    private boolean isResourcePressure() {
        MemoryMXBean memoryMXBean = ManagementFactory.getMemoryMXBean();
        double usedRatio = (double)
            memoryMXBean.getHeapMemoryUsage().getUsed() /
            memoryMXBean.getHeapMemoryUsage().getMax();
        return usedRatio > 0.85;  // 85% 以上认为有压力
    }

    private boolean isThreadPoolBusy() {
        ThreadPoolExecutor executor = (ThreadPoolExecutor) Executors.newFixedThreadPool(100);
        double busyRatio = (double) executor.getActiveCount() / executor.getPoolSize();
        return busyRatio > 0.8;  // 80% 以上认为繁忙
    }
}
```

## 就绪探针失败的影响

```
┌─────────────────────────────────────────────────────────────────┐
│                    就绪探针失败的影响                              │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  探针失败后：                                                 │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │                                                         │   │
│  │  1. Pod 从 Service Endpoints 移除                      │   │
│  │  2. 新流量不再发送到该 Pod                             │   │
│  │  3. 已有连接可以继续处理                                │   │
│  │  4. Pod 不会被重启                                      │   │
│  │                                                         │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                  │
│  探针恢复后：                                                 │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │                                                         │   │
│  │  1. Pod 重新加入 Service Endpoints                     │   │
│  │  2. 新流量开始发送到该 Pod                             │   │
│  │                                                         │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

### 示例：优雅下线

```java
// 优雅下线
@Service
public class GracefulShutdownService {
    private final AtomicBoolean isReady = new AtomicBoolean(false);

    @PreDestroy
    public void shutdown() {
        // 标记为不就绪
        isReady.set(false);

        // 等待现有请求完成
        waitForRequestsToComplete();

        // 关闭连接池
        closeConnectionPools();
    }

    private void waitForRequestsToComplete() {
        try {
            Thread.sleep(10 * 1000);  // 等待 10 秒
        } catch (InterruptedException e) {
            Thread.currentThread().interrupt();
        }
    }
}

// Kubernetes 优雅下线配置
spec:
  terminationGracePeriodSeconds: 60  # 优雅下线时间
```

## 总结

就绪探针确保只有真正「准备好」的服务才会接收流量。

**配置要点**：

- initialDelaySeconds：5-15 秒
- periodSeconds：5-10 秒
- failureThreshold：3-5

**就绪探针应该检查**：

- 数据库连接
- Redis 连接
- 外部服务
- 资源状态

**就绪探针失败的结果**：

- Pod 从 Endpoints 移除
- 停止接收新流量
- Pod 不会被重启
- 恢复后自动加入

**与存活探针的区别**：

- 存活探针失败 = 重启
- 就绪探针失败 = 停止接收流量
