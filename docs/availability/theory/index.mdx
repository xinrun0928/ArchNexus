# 可用性理论

可用性是衡量系统稳定性的核心指标。一个高可用的系统，能够在各种故障场景下持续提供服务，保障用户体验和业务连续性。

## 可用性基础

可用性（Availability）指的是系统正常运行的概率。业界通常用「几个 9」来衡量：

| 可用性等级 | 年宕机时间 | 适用场景 |
|---|---|---|
| 99%（2 个 9） | 3.65 天 | 内部系统 |
| 99.9%（3 个 9） | 8.76 小时 | 面向用户的业务系统 |
| 99.99%（4 个 9） | 52.6 分钟 | 核心业务、金融系统 |
| 99.999%（5 个 9） | 5.26 分钟 | 电信级、生命线系统 |

每提升一个 9，需要付出的代价是指数级增长的。对于大多数业务系统，99.9% 是一个合理的起点。

## SLA 与 SLO

SLA（Service Level Agreement）和 SLO（Service Level Objective）是管理可用性的关键工具。SLA 是对外的承诺，SLO 是内部的目标，SLO 通常比 SLA 更严格。配合 Error Budget 机制，可以量化系统在达到 SLO 目标后「还能犯多少错」，帮助团队在稳定性和迭代速度之间找到平衡。

## 可靠性指标

MTTR（Mean Time To Repair）和 MTBF（Mean Time Between Failures）是衡量系统可靠性的两个核心指标。可用性可以表示为：可用性 = MTBF / (MTBF + MTTR)。降低 MTTR 比提高 MTBF 对可用性的提升效果更显著，这也是为什么快速检测、快速诊断、快速恢复是可靠性工程的重点。

## 高可用设计原则

构建高可用系统需要遵循五个核心原则：消除单点、冗余设计、故障隔离、快速失败、可恢复性。这些原则贯穿于架构设计、代码实现、运维操作的每一个环节。
