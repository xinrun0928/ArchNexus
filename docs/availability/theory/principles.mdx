# 高可用设计原则

你的系统上线后故障频发，每次都要紧急修复。

问题出在哪？**架构设计阶段没有考虑高可用**。

高可用不是事后打补丁，而是设计之初就需要遵循的原则。

### 消除单点

```java
// 消除单点：数据库主从复制
@Configuration
public class DataSourceConfig {
    @Bean
    public DataSource dataSource() {
        // 主从数据源
        HikariDataSource master = createMaster();
        HikariDataSource slave = createSlave();

        // 读写分离路由
        return new RoutingDataSource(master, slave);
    }
}

// 读写分离路由
public class RoutingDataSource extends AbstractDataSource {
    private final DataSource master;
    private final DataSource slave;

    public Object getConnection() {
        if (isReadOnly()) {
            return slave.getConnection();
        }
        return master.getConnection();
    }
}
```

## 原则三：故障隔离

### 隔离级别

```
┌─────────────────────────────────────────────────────────────────┐
│                    故障隔离层级                                   │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │ 线程隔离                                                 │   │
│  │ 不同服务使用不同的线程池                                 │   │
│  │ 某个服务线程池耗尽不影响其他服务                       │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                  │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │ 进程隔离                                                 │   │
│  │ 不同服务部署在不同进程                                 │   │
│  │ 微服务架构天然提供进程隔离                             │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                  │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │ 机房隔离                                                 │   │
│  │ 服务部署在不同机房                                     │   │
│  │ 单机房故障不影响其他机房                               │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                  │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │ 泳道隔离                                                 │   │

│  │ 灰度发布时不影响主流程                                 │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

### 线程池隔离

```java
// 线程池隔离
@Service
public class IsolatedService {
    // 每个依赖服务独立的线程池
    private final ExecutorService userServicePool = new ThreadPoolExecutor(
        10, 20, 60L, TimeUnit.SECONDS, new LinkedBlockingQueue<>(500));

    private final ExecutorService paymentServicePool = new ThreadPoolExecutor(
        20, 50, 60L, TimeUnit.SECONDS, new LinkedBlockingQueue<>(1000));

    private final ExecutorService notificationServicePool = new ThreadPoolExecutor(
        5, 10, 60L, TimeUnit.SECONDS, new LinkedBlockingQueue<>(200));

    public CompletableFuture<User> getUser(Long userId) {
        return CompletableFuture.supplyAsync(
            () -> userClient.get(userId),
            userServicePool);
    }

    public CompletableFuture<PaymentResult> pay(Order order) {
        return CompletableFuture.supplyAsync(
            () -> paymentClient.pay(order),
            paymentServicePool);
    }

    public CompletableFuture<Void> notify(Order order) {
        return CompletableFuture.supplyAsync(
            () -> notificationClient.send(order),
            notificationServicePool);
    }
}
```

## 原则四：快速失败

### 快速失败原则

```java
// 快速失败示例
@Service
public class FastFailService {
    // 错误：无限等待
    public User getUserWrong(Long userId) {
        User user = restTemplate.getForObject(
            "http://user-service/api/users/{id}",
            User.class,
            userId);
        // 如果 user-service 挂了，会一直等待
        return user;
    }

    // 正确：设置超时，快速失败
    public User getUserRight(Long userId) {
        try {
            return restTemplate.getForObject(
                "http://user-service/api/users/{id}",
                User.class,
                userId);
        } catch (RestClientException e) {
            // 快速失败，返回降级结果
            log.warn("调用 user-service 失败", e);
            return User.getDefault();
        }
    }
}
```

## 原则五：可恢复性

### 幂等设计

```java
// 幂等设计
@Service
public class IdempotentService {

    @RedisDistributedLock(key = "order:create:{orderId}")
    public Order createOrder(CreateOrderRequest request) {
        // 幂等检查
        if (idempotentCheck(request.getIdempotentKey())) {
            return getExistingOrder(request.getIdempotentKey());
        }

        // 创建订单
        Order order = doCreateOrder(request);

        // 记录幂等键
        saveIdempotentKey(request.getIdempotentKey(), order.getId());

        return order;
    }
}
```

### 自动恢复

```java
// 健康检查 + 自动重启
@Service
public class SelfHealingService {
    @Autowired
    private HealthIndicatorRegistry registry;

    @Scheduled(fixedRate = 30000)  // 每 30 秒检查
    public void healthCheck() {
        for (String name : registry.getNames()) {
            HealthIndicator indicator = registry.get(name);
            Health health = indicator.health();

            if (health.getStatus() == Status.DOWN) {
                log.warn("服务 {} 健康检查失败: {}",
                    name, health.getDetails());

                // 触发告警
                alertService.alert(name, health);

                // 某些情况下可以自动重启
                // kubernetes 会自动重启不健康的 Pod
            }
        }
    }
}
```

## 总结

高可用设计需要系统性的原则指导。

**五大原则**：

- 消除单点：关键组件不能只有一份
- 冗余设计：多副本部署，多机房容灾
- 故障隔离：限制故障影响范围
- 快速失败：尽早暴露问题
- 可恢复性：故障后能快速恢复

**实施路线**：

1. 先消除明显的单点
2. 实现基础的冗余部署
3. 添加熔断、限流保护
4. 建立监控告警体系
5. 持续优化和改进
