# MTTR 与 MTBF

你的系统一年宕机了 2 次，每次修复花了 2 天。但你能说清楚系统的可靠性到底如何吗？

问题出在哪？**可靠性需要量化，MTTR 和 MTBF 是两个核心指标**。

## 基本概念

```
┌─────────────────────────────────────────────────────────────────┐
│                    MTTR vs MTBF                                  │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  MTTR（Mean Time To Repair）：平均修复时间                      │
│  - 从故障发生到恢复的平均时间                                   │
│  - 包含检测、诊断、修复时间                                    │
│  - 越短越好                                                    │
│                                                                  │
│  MTBF（Mean Time Between Failures）：平均故障间隔               │
│  - 两次故障之间的平均时间                                       │
│  - 衡量系统的稳定性                                            │
│  - 越长越好                                                    │
│                                                                  │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │ 正常运行 ──▶ 故障 ──▶ 修复 ──▶ 正常运行 ──▶ 故障    │   │
│  │        MTBF 1        MTTR 1      MTBF 2      MTTR 2     │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

## 计算公式

```java
// MTTR 和 MTBF 计算
public class ReliabilityCalculator {

    // MTTR = 总修复时间 / 故障次数
    public static Duration calculateMTTR(List<RepairRecord> records) {
        long totalRepairTime = records.stream()
            .mapToLong(r -> r.getRepairDuration().toMillis())
            .sum();
        return Duration.ofMillis(totalRepairTime / records.size());
    }

    // MTBF = 总正常运行时间 / 故障次数
    public static Duration calculateMTBF(List<FailureRecord> records, Duration period) {
        int failureCount = records.size();
        // MTBF = (总时间 - 总修复时间) / (故障次数 + 1)
        long totalTime = period.toMillis();
        long totalRepairTime = records.stream()
            .mapToLong(r -> r.getRepairDuration().toMillis())
            .sum();
        return Duration.ofMillis((totalTime - totalRepairTime) / (failureCount + 1));
    }
}
```

## 与可用性的关系

```
┌─────────────────────────────────────────────────────────────────┐
│                    MTTR、MTBF 与可用性                           │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  可用性 = MTBF / (MTBF + MTTR)                                 │
│                                                                  │
│  示例：                                                         │
│  - MTBF = 720 小时（30 天）                                    │
│  - MTTR = 1 小时                                               │
│  - 可用性 = 720 / (720 + 1) = 99.86%                          │
│                                                                  │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │ MTBF       │ MTTR      │ 可用性                        │   │
│  ├────────────┼───────────┼──────────────────────────────┤   │
│  │ 30 天      │ 1 小时    │ 99.86%                       │   │
│  │ 30 天      │ 30 分钟   │ 99.93%                       │   │
│  │ 30 天      │ 10 分钟   │ 99.99%                       │   │
│  │ 90 天      │ 1 小时    │ 99.93%                       │   │
│  │ 90 天      │ 30 分钟   │ 99.97%                       │   │
│  └────────────┴───────────┴──────────────────────────────┘   │
│                                                                  │
│  关键洞察：                                                    │
│  - MTTR 影响比 MTBF 更显著                                     │
│  - 降低 MTTR 是提升可用性的高效手段                              │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

## MTTR 分解

```
┌─────────────────────────────────────────────────────────────────┐
│                    MTTR 分解                                     │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  MTTR = 检测时间 + 诊断时间 + 修复时间 + 验证时间               │
│                                                                  │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │ 故障发生                                               │   │
│  │    │                                                   │   │
│  │    ▼                                                   │   │
│  │ 告警触发 ──▶ 定位问题 ──▶ 制定方案 ──▶ 执行修复      │   │
│  │  (检测)      (诊断)        (决策)        (修复)         │   │
│  │    │                                                   │   │
│  │    ▼                                                   │   │
│  │ 验证通过 ──▶ 服务恢复                               │   │
│  │  (验证)                                               │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                  │
│  每个阶段的优化：                                               │
│  - 检测：完善的监控和告警                                       │
│  - 诊断：日志、链路追踪、快速定位工具                           │
│  - 修复：预案、自动化、团队技能                                  │
│  - 验证：灰度发布、回滚机制                                     │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

## 降低 MTTR 的实践

### 快速检测

```java
// 监控告警配置
@Configuration
public class MonitoringConfig {
    @Bean
    public AlertRule errorRateAlert() {
        return AlertRule.builder()
            .metric("error_rate")
            .threshold(0.01)  // 1% 错误率
            .window(Duration.ofMinutes(1))
            .severity(AlertSeverity.CRITICAL)
            .notificationChannels("pagerduty", "slack")
            .build();
    }
}
```

## MTTD、MTTI、MTTR 区别

```
┌─────────────────────────────────────────────────────────────────┐
│                    时间指标全览                                   │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  MTTD（Mean Time To Detect）：平均检测时间                      │
│  - 故障发生到被检测到的平均时间                                 │
│                                                                  │
│  MTTI（Mean Time To Identify）：平均识别时间                     │
│  - 检测到异常到确定根因的平均时间                                │
│                                                                  │
│  MTTR（Mean Time To Repair / Recover）：平均恢复时间             │
│  - 识别根因到服务恢复的平均时间                                  │
│                                                                  │
│  或细分为：                                                     │
│  MTTF（Mean Time To Failure）：平均故障时间（首次故障）        │
│  MTBF（Mean Time Between Failures）：平均故障间隔                │
│                                                                  │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │ 故障发生 ──▶ MTTD ──▶ MTTI ──▶ MTTR ──▶ 恢复      │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

## 总结

MTTR 和 MTBF 是衡量可靠性的核心指标。

**核心公式**：

- 可用性 = MTBF / (MTBF + MTTR)
- MTTR = 检测 + 诊断 + 修复 + 验证

**优化方向**：

- 降低 MTTR 比提高 MTBF 更有效
- 快速检测：完善监控告警
- 快速诊断：链路追踪、日志分析
- 快速修复：自动化、预案、回滚

**关键洞察**：

- 故障不可怕，可怕的是恢复太慢
- 每次故障都是改进的机会
