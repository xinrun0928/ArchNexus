# SLA 与 SLO

你的团队承诺了 99.9% 的 SLA，但团队内部说 SLO 是 99.99%。

问题出在哪？**SLA 和 SLO 是不同的概念，混淆会导致管理混乱**。

## 基本概念

```
┌─────────────────────────────────────────────────────────────────┐
│                    SLA vs SLO vs SLI                             │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  SLA（Service Level Agreement）：服务等级协议                    │
│  - 对外部客户的承诺                                             │
│  - 具有法律效力的合同                                           │
│  - 通常比内部目标更宽松                                         │
│                                                                  │
│  SLO（Service Level Objective）：服务等级目标                    │
│  - 内部管理的目标                                               │
│  - 团队努力达成的标准                                           │
│  - 通常比 SLA 更严格                                           │
│                                                                  │
│  SLI（Service Level Indicator）：服务等级指标                    │
│  - 实际测量到的指标值                                           │
│  - 用于判断是否达到 SLO                                         │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

## SLO 制定实践

### 制定原则

```java
// SLO 制定原则
public class SloPrinciples {
    /*
    原则一：基于业务需求
    - 不是越高越好
    - 满足用户需求即可

    原则二：可测量
    - 必须有明确 SLI
    - 必须能持续监控

    原则三：可达成
    - 参考历史数据
    - 留有 Error Budget

    原则四：透明
    - 对内对外一致
    - 定期报告状态
    */
}
```

### SLO 文档模板

```yaml
# SLO 文档示例
service_name: Order Service

# 可用性 SLO
availability:
  target: 99.95%          # 目标 99.95%
  window: rolling_30d     # 滚动 30 天窗口
  measurement: external_monitoring  # 外部监控测量

# 延迟 SLO
latency:
  target: p99 < 500ms     # P99 延迟 < 500ms
  window: rolling_1h       # 滚动 1 小时窗口
  measurement: api_gateway_metrics

# 错误率 SLO
errors:
  target: < 0.1%          # 错误率 < 0.1%
  window: rolling_1d
  measurement: error_rate_from_gateway

# Error Budget 政策
error_budget:
  policy: 1h_alert_2h_explore  # 1 小时超过触发告警，2 小时触发探索
  burn_rate_alert:
    fast: 14.4x  # 快速消耗告警（5% 预算/小时）
    slow: 6x    # 慢速消耗告警（2% 预算/小时）
```

## Error Budget

### Error Budget 概念

```
┌─────────────────────────────────────────────────────────────────┐
│                    Error Budget 概念                              │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  Error Budget = 允许的「错误」额度                               │
│                                                                  │
│  示例（99.9% SLO，30 天窗口）：                                 │
│  - 总请求数：30 天 × 24 小时 × 1000 QPS × 3600 秒              │
│  - 允许错误：总请求数 × 0.1%                                     │
│  - Error Budget：一定的「浪费」空间                             │
│                                                                  │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │ ████████████████████████████░░░░░░░░░░░░░░░░░░░░░░░░░ │   │
│  │         Error Budget 消耗              剩余              │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

### Error Budget 政策

```java
// Error Budget 计算
public class ErrorBudgetCalculator {

    public static double calculate(double sloTarget, Duration window) {
        return 1 - sloTarget / 100.0;
    }

    public static void main(String[] args) {
        // 99.9% SLO，30 天
        double errorBudget = calculate(99.9, Duration.ofDays(30));
        System.out.println("Error Budget: " + (errorBudget * 100) + "%");
        // 0.1%

        // 允许的错误时间
        Duration allowedErrors = Duration.ofMillis(
            (long)(Duration.ofDays(30).toMillis() * errorBudget));
        System.out.println("Allowed downtime: " + allowedErrors);
        // PT43M12S（约 43 分钟）
    }
}
```

### Error Budget 告警

```yaml
# Prometheus 告警规则
groups:
  - name: slo-error-budget
    rules:
      # 快速消耗告警（1 小时消耗 5% 预算）
      - alert: ErrorBudgetFastBurn
        expr: |
          sum(rate(http_requests_total{job="order-service",status=~"5.."}[1h]))
          / sum(rate(http_requests_total{job="order-service"}[1h])) > 14.4 * 0.001
        for: 1h
        labels:
          severity: critical
        annotations:
          summary: "Error Budget 快速消耗"

      # 慢速消耗告警（1 小时消耗 2% 预算）
      - alert: ErrorBudgetSlowBurn
        expr: |
          sum(rate(http_requests_total{job="order-service",status=~"5.."}[1h]))
          / sum(rate(http_requests_total{job="order-service"}[1h])) > 6 * 0.001
        for: 5h
        labels:
          severity: warning
        annotations:
          summary: "Error Budget 慢速消耗"
```

## 总结

SLA、SLO、SLI 是服务可用性管理的三个层次。

**概念区分**：

- SLA：对客户的承诺
- SLO：内部管理目标
- SLI：实际测量指标

**Error Budget**：

- SLO 内的「缓冲空间」
- 允许一定的失败
- 超过预算触发改进

**实践要点**：

- SLO 要可测量、可达成
- Error Budget 是改进的「燃料」
- 透明公开 SLO 状态
