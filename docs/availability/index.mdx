# 可用性

可用性是衡量系统质量的核心指标之一。一个高可用的系统，能够在各种故障场景下持续提供服务，保障用户体验和业务连续性。

## 章节概述

在分布式系统中，故障是常态而非例外。可用性工程的核心任务是：预防故障发生、限制故障影响、快速恢复服务。本章节将从理论到实践，系统介绍保障系统可用性的核心技术与策略。

## 内容结构

本章节包含七个主题：

**可用性理论**介绍了可用性的量化标准（SLA/SLO/MTTR/MTBF）和高可用设计的核心原则，是理解后续内容的理论基础。

**故障模型**深入分析了系统故障的类型、级联传播机制和雪崩效应，并通过故障注入测试（混沌工程）验证系统的韧性。

**熔断机制**讲解了熔断器模式的工作原理、状态机转换、阈值配置，以及半开状态的恢复策略，是防止故障扩散的关键机制。

**限流策略**介绍了各种限流算法（令牌桶、漏桶、滑动窗口）的原理和特点，以及如何选择合适的限流方案保护系统。

**降级策略**探讨了如何在系统压力下保障核心功能，包括降级原则、粒度控制和恢复策略，实现「优雅退化」。

**超时与重试**讲解了超时设置的科学方法、重试策略的设计，以及如何避免重试风暴和幂等性问题。

**健康检查**是 Kubernetes 判断服务是否健康的依据，包括存活探针、就绪探针和启动探针的正确配置。

## 核心原则

保障系统可用性需要遵循几个核心原则：

- **消除单点**：任何组件故障都不应导致系统不可用
- **冗余设计**：关键组件至少部署两份
- **故障隔离**：限制故障影响范围，防止扩散
- **快速失败**：尽早暴露问题，避免级联影响
- **可恢复性**：故障后能快速恢复到正常状态

## 实践路线

构建高可用系统是一个渐进的过程：

1. **基础阶段**：消除明显的单点，配置基本的超时和重试
2. **保护阶段**：添加熔断器和限流保护，建立监控告警
3. **韧性阶段**：实施降级策略，进行混沌工程实验
4. **优化阶段**：持续优化阈值配置，提升恢复效率
