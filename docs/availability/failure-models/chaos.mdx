# 故障注入测试

你的系统经过严格测试，但上线后还是故障了。

问题出在哪？**测试环境永远无法完全模拟生产环境**。

故障注入测试（混沌工程）正是解决这个问题的方案：**在生产环境中主动注入故障，验证系统的韧性**。

## 故障注入实践

### Spring Cloud Contract + Toxiproxy

```java
// 使用 Toxiproxy 进行故障注入测试
@SpringBootTest
@AutoConfigureMockMvc
public class FaultInjectionTest {
    private ToxiproxyClient client;
    private Proxy proxy;

    @BeforeEach
    public void setup() throws IOException {
        client = new ToxiproxyClient("localhost", 8474);
        proxy = client.createProxy("test-redis", "127.0.0.1:63790", "127.0.0.1:6379");
    }

    @AfterEach
    public void cleanup() {
        proxy.toxics().getAll().forEach(Toxic::remove);
    }

    @Test
    public void testRedisDelay() throws IOException {
        // 添加 3 秒延迟
        proxy.toxics().latency("latency", Toxic.Direction.DOWNSTREAM, 3000);

        // 测试超时处理
        assertThatThrownBy(() -> redisTemplate.opsForValue().get("key"))
            .isInstanceOf(Exception.class);
    }

    @Test
    public void testRedisFailure() throws IOException {
        // 模拟 Redis 不可用
        proxy.toxics().bandwidth("limit", Toxic.Direction.DOWNSTREAM, 0);

        // 验证降级逻辑
        String result = cacheService.getWithFallback("key");
        assertThat(result).isEqualTo("fallback");
    }
}
```

### 微服务故障注入

```java
// 使用 Resilience4j 进行故障注入
@SpringBootTest
public class ChaosEngineeringTest {
    @Autowired
    private CircuitBreakerRegistry circuitBreakerRegistry;

    @Test
    public void testCircuitBreakerOpensOnFailure() {
        CircuitBreaker circuitBreaker = circuitBreakerRegistry.circuitBreaker("testService");

        // 模拟故障
        AtomicInteger successCount = new AtomicInteger(0);
        AtomicInteger fallbackCount = new AtomicInteger(0);

        for (int i = 0; i < 100; i++) {
            try {
                circuitBreaker.executeSupplier(() -> {
                    throw new RuntimeException("Simulated failure");
                });
            } catch (Exception e) {
                fallbackCount.incrementAndGet();
            }
        }

        // 验证熔断器状态
        assertThat(circuitBreaker.getState()).isEqualTo(State.OPEN);
    }

    @Test
    public void testTimeoutPreventsHang() {
        // 模拟超时
        ExecutorService executor = Executors.newFixedThreadPool(10);

        long start = System.currentTimeMillis();
        try {
            CompletableFuture<String> future = CompletableFuture.supplyAsync(
                () -> slowService.call(),
                executor
            ).orTimeout(3, TimeUnit.SECONDS).join();
        } catch (Exception e) {
            // 超时处理
        }
        long duration = System.currentTimeMillis() - start;

        // 验证没有无限等待
        assertThat(duration).isLessThan(5000);
    }
}
```

## 故障注入实验设计

### 实验流程

```
┌─────────────────────────────────────────────────────────────────┐
│                    故障注入实验流程                               │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  1. 定义稳态假设                                              │
│     - 「系统在正常情况下应该...」                              │
│     - 定义可测量的稳态指标                                     │
│                                                                  │
│  2. 设计实验                                                  │
│     - 选择注入的故障类型                                       │
│     - 确定故障范围和持续时间                                   │
│     - 定义预期结果                                             │
│                                                                  │
│  3. 执行实验                                                  │
│     - 最小爆炸半径先行                                        │
│     - 监控系统指标                                             │
│     - 记录观察结果                                             │
│                                                                  │
│  4. 分析结果                                                  │
│     - 是否符合预期                                             │
│     - 是否需要改进                                             │
│                                                                  │
│  5. 自动化实验                                                 │
│     - 定期执行                                                │
│     - CI/CD 集成                                              │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

## 总结

故障注入测试是提升系统韧性的有效手段。

**工具选择**：

| 工具 | 适用场景 |
|---|---|
| Chaos Monkey | Kubernetes Pod 杀灭 |
| ChaosBlade | 多场景故障注入 |
| Toxiproxy | 网络故障模拟 |
| Chaos Mesh | Kubernetes 原生故障 |

**实验原则**：

- 最小爆炸半径
- 定义稳态假设
- 渐进式增加风险
- 持续自动化

**注意事项**：

- 故障注入是高风险操作
- 必须有回滚方案
- 从非生产环境开始
- 监控和告警到位
