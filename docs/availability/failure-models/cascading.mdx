# 级联故障

你的系统 A 服务故障了，然后 B 服务也跟着挂了，最后整个系统都崩了。

问题出在哪？**故障没有及时隔离，导致了级联扩散**。

级联故障是分布式系统最危险的故障模式之一。

## 级联故障的本质

```
┌─────────────────────────────────────────────────────────────────┐
│                    级联故障的本质                                  │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  级联故障 = 局部故障 → 连锁反应 → 系统崩溃                     │
│                                                                  │
│  触发条件：                                                    │
│  1. 某个服务变慢或不可用                                     │
│  2. 调用方等待时间增加                                        │
│  3. 资源（线程、连接、内存）被耗尽                           │
│  4. 其他依赖服务也变慢或不可用                                │
│  5. 循环往复，最终整个系统崩溃                               │
│                                                                  │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │                                                         │   │
│  │   服务 A ──▶ 服务 B ──▶ 服务 C ──▶ 服务 D             │   │
│  │     ↓                                                   │   │
│  │   故障 ──▶ 等待 ──▶ 资源耗尽 ──▶ 扩散              │   │
│  │                                                         │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

## 级联故障的形成过程

### 阶段一：初始故障

```
┌─────────────────────────────────────────────────────────────────┐
│                    阶段一：初始故障                              │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  服务 A 因为某种原因（硬件故障、Bug、流量激增）开始变慢或不可用 │
│                                                                  │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │                                                         │   │
│  │   正常状态：                                          │   │
│  │   ┌───────┐                                           │   │
│  │   │服务 A │ ──▶ 响应时间: 10ms                       │   │
│  │   └───────┘                                           │   │
│  │                                                         │   │
│  │   故障状态：                                          │   │
│  │   ┌───────┐                                           │   │
│  │   │服务 A │ ──▶ 响应时间: 10000ms (超时)             │   │
│  │   └───────┘                                           │   │
│  │                                                         │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

### 阶段二：资源耗尽

```
┌─────────────────────────────────────────────────────────────────┐
│                    阶段二：资源耗尽                               │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  调用方不断等待，导致资源被耗尽：                                │
│                                                                  │
│  1. 线程池耗尽                                                │
│     - 100 个线程都在等待服务 A                                 │
│     - 新请求无法处理                                           │
│                                                                  │
│  2. 连接池耗尽                                                │
│     - 50 个连接等待服务 A 的响应                               │
│     - 新请求无法获取连接                                       │
│                                                                  │
│  3. 内存压力增加                                                │
│     - 等待中的请求占用内存                                     │
│     - 可能触发 GC                                               │
│                                                                  │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │ ┌─────────────────────────────────────────────────────┐ │   │
│  │ │ 线程 1: 等待服务 A                                  │ │   │
│  │ │ 线程 2: 等待服务 A                                  │ │   │
│  │ │ 线程 3: 等待服务 A                                  │ │   │
│  │ │ ...                                                  │ │   │
│  │ │ 线程 100: 等待服务 A                                │ │   │
│  │ └─────────────────────────────────────────────────────┘ │   │
│  │ ← 所有线程都在等待，线程池耗尽 →                      │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

### 阶段三：故障扩散

```
┌─────────────────────────────────────────────────────────────────┐
│                    阶段三：故障扩散                               │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  服务 A 不可用导致调用它的服务 B 也变慢                        │
│  服务 B 变慢导致调用它的服务 C 也变慢                           │
│  循环往复，最终整个系统不可用                                   │
│                                                                  │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │                                                         │   │
│  │   t=0: 服务 A 变慢                                      │   │
│  │   t=1: 服务 B 线程池耗尽                                │   │
│  │   t=2: 服务 B 开始超时                                   │   │
│  │   t=3: 服务 C 调用服务 B，开始等待                      │   │
│  │   t=4: 服务 C 线程池耗尽                                 │   │
│  │   t=5: 服务 D、E、F 也开始变慢                          │   │
│  │   t=6: 整个系统崩溃                                     │   │
│  │                                                         │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

## 级联故障的触发场景

### 场景一：同步调用链过长

```java
// 同步调用链过长
@Service
public class LongCallChainService {
    // 调用链：Gateway → A → B → C → D → E
    // 任何一个服务慢，都会导致整条链超时

    public String process(Request request) {
        String a = serviceA.call(request);      // [!code highlight]
        String b = serviceB.call(request);      // [!code highlight]
        String c = serviceC.call(request);      // [!code highlight]
        String d = serviceD.call(request);      // [!code highlight]
        String e = serviceE.call(request);      // [!code highlight]
        return a + b + c + d + e;
    }

    // 正确做法：异步并行调用
    public String processAsync(Request request) {
        CompletableFuture<String> aFuture = serviceA.callAsync(request);
        CompletableFuture<String> bFuture = serviceB.callAsync(request);
        CompletableFuture<String> cFuture = serviceC.callAsync(request);
        CompletableFuture<String> dFuture = serviceD.callAsync(request);
        CompletableFuture<String> eFuture = serviceE.callAsync(request);

        return CompletableFuture.allOf(aFuture, bFuture, cFuture, dFuture, eFuture)
            .thenApply(v -> aFuture.join() + bFuture.join() + cFuture.join()
                + dFuture.join() + eFuture.join())
            .join();
    }
}
```

### 场景三：重试风暴

```java
// 重试风暴
@Service
public class RetryStormService {
    @Autowired
    private RestTemplate restTemplate;

    // 错误：所有客户端都无脑重试
    public String callService() {
        for (int i = 0; i < 3; i++) {  // [!code highlight]
            try {
                return restTemplate.getForObject("http://slow-service/api", String.class);
            } catch (Exception e) {
                log.warn("调用失败，重试: {}", i);
            }
        }
        throw new RuntimeException("服务不可用");
    }
}

// 如果有 1000 个并发请求，每个重试 3 次
// 原始请求：1000
// 第一次重试：1000
// 第二次重试：1000
// 总请求：3000 → 重试风暴！  // [!code highlight]
```

## 级联故障的防范

### 熔断器

```java
// 熔断器防止故障扩散
@Configuration
public class CircuitBreakerConfig {
    @Bean
    public CircuitBreakerRegistry circuitBreakerRegistry() {
        return CircuitBreakerRegistry.of(
            CircuitBreakerConfig.custom()
                .failureRateThreshold(50)              // 50% 失败率触发熔断
                .slowCallRateThreshold(80)             // 80% 慢调用触发熔断
                .slowCallDurationThreshold(Duration.ofSeconds(3))  // 3 秒算慢调用
                .waitDurationInOpenState(Duration.ofSeconds(30))  // 熔断 30 秒后尝试半开
                .slidingWindowSize(10)                  // 滑动窗口 10 次调用
                .build()
        );
    }
}

// 使用熔断器
@Service
public class ProtectedService {
    private final CircuitBreaker circuitBreaker;

    public ProtectedService(CircuitBreakerRegistry registry) {
        this.circuitBreaker = registry.circuitBreaker("slowService");
    }

    public String callService() {
        return circuitBreaker.executeSupplier(() -> {
            return restTemplate.getForObject("http://slow-service/api", String.class);
        });
    }
}
```

### 超时设置

```java
// 超时设置
@Configuration
public class TimeoutConfig {
    @Bean
    public RestTemplate restTemplate() {
        SimpleClientHttpRequestFactory factory = new SimpleClientHttpRequestFactory();
        factory.setConnectTimeout(Duration.ofMillis(1000));  // [!code highlight]
        factory.setReadTimeout(Duration.ofMillis(3000));     // [!code highlight]
        return new RestTemplate(factory);
    }
}
```

### 限流

```java
// 限流保护
@Configuration
public class RateLimitConfig {
    @Bean
    public FilterRegistrationBean<RateLimitFilter> rateLimitFilter() {
        FilterRegistrationBean<RateLimitFilter> registration = new FilterRegistrationBean<>();
        registration.setFilter(new RateLimitFilter(
            1000,  // 每秒最多 1000 请求
            100    // 突发最多 100 请求
        ));
        registration.addUrlPatterns("/api/*");
        return registration;
    }
}

// Resilience4j 限流
@RateLimiter(name = "backendService", fallbackMethod = "fallback")
public String callBackend() {
    return restTemplate.getForObject("http://backend/api", String.class);
}

public String fallback(Exception e) {
    return "服务繁忙，请稍后重试";
}
```

### 资源隔离

```java
// 线程池隔离
@Configuration
public class ThreadPoolIsolationConfig {
    @Bean
    public ThreadPoolExecutor userServiceExecutor() {
        return new ThreadPoolExecutor(
            10, 20, 60L, TimeUnit.SECONDS,
            new LinkedBlockingQueue<>(100),
            new ThreadPoolExecutor.CallerRunsPolicy());
    }

    @Bean
    public ThreadPoolExecutor paymentServiceExecutor() {
        return new ThreadPoolExecutor(
            20, 50, 60L, TimeUnit.SECONDS,
            new LinkedBlockingQueue<>(200),
            new ThreadPoolExecutor.CallerRunsPolicy());
    }
}

// 使用 @Async 隔离
@Service
public class AsyncIsolationService {
    @Async("userServiceExecutor")
    public CompletableFuture<String> callUserService() {
        return CompletableFuture.supplyAsync(() ->
            restTemplate.getForObject("http://user-service/api", String.class));
    }
}
```

## 总结

级联故障是分布式系统最危险的故障模式。

**形成过程**：

- 初始故障 → 资源耗尽 → 故障扩散 → 系统崩溃

**触发场景**：

- 同步调用链过长
- 共享资源耗尽
- 重试风暴

**防范措施**：

- 熔断器：快速失败，防止故障扩散
- 超时设置：避免无限等待
- 限流：保护系统不被压垮
- 资源隔离：限制故障影响范围
