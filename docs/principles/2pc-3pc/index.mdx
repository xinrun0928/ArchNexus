# 两阶段与三阶段提交

在分布式系统中，事务是一个棘手的问题。在单机系统中，我们可以使用数据库的 ACID 事务来保证数据的一致性。但在分布式系统中，事务涉及多个节点，如何保证所有节点要么全部成功，要么全部失败？

两阶段提交（2PC）和三阶段提交（3PC）正是为解决这个问题而设计的分布式事务协议。它们通过协调者的角色，将分布式事务的提交过程分为多个阶段，确保所有参与节点达成一致的提交决策。

## 本章节内容

本章节将深入探讨两阶段提交和三阶段提交协议：

- **两阶段提交 (2PC)**：分布式事务的基础协议
- **三阶段提交 (3PC)**：解决 2PC 阻塞问题的改进方案
- **2PC vs 3PC 对比**：两种协议的权衡与选择

## 为什么需要分布式事务协议？

在分布式系统中，事务面临几个挑战：

- **跨节点操作**：一个事务可能涉及多个节点的数据修改
- **网络不可靠**：消息可能丢失、延迟、乱序
- **节点故障**：部分节点可能在事务执行过程中故障
- **协调者故障**：协调者故障可能导致事务无法完成

分布式事务协议正是为了解决这些问题而设计的。它们通过明确的阶段划分和故障处理机制，确保分布式事务的原子性。

## 两阶段提交 vs 三阶段提交

两阶段提交是最基础的分布式事务协议，它通过"准备"和"提交"两个阶段，确保所有参与节点达成一致的决策。

三阶段提交是对两阶段提交的改进，它在"准备"和"提交"之间增加了一个"预提交"阶段，以解决两阶段提交的阻塞问题。

两种协议各有优缺点，适用于不同的场景。通过本章节的学习，你将理解这两种协议的原理和权衡，能够在实际项目中做出正确的选择。
