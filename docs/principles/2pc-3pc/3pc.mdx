# 三阶段提交 (3PC)

两阶段提交（2PC）的阻塞问题一直是个痛点。想象一下：协调者在发送 Commit 后崩溃，所有参与者都在等待，资源被锁定，系统吞吐量急剧下降。这个问题能解决吗？

答案是：可以，但代价是增加一个阶段。三阶段提交（3PC）正是通过增加一个"预提交"阶段，将 2PC 的阻塞问题降到了最低。

## 3PC 的核心改进

### 2PC 的问题回顾

2PC 的阻塞问题源于**参与者不确定协调者的最终决策**。

```
问题场景：
1. 协调者发送 Prepare，参与者全部响应 Vote-Commit
2. 协调者发送 Commit，但在发送完成前崩溃
3. 参与者 A 收到 Commit，执行提交
4. 参与者 B 未收到 Commit，一直等待

参与者 B 的困境：
- 如果执行 Commit，可能与 A 的状态不一致
- 如果执行 Abort，可能与 A 的状态不一致
- 唯一的选择是：等待
```

### 3PC 的解决思路

3PC 的核心改进是**引入第三阶段：预提交**。

```
2PC 的两阶段：
准备阶段 ──── 提交/回滚

3PC 的三阶段：
准备阶段 ──── 预提交阶段 ──── 提交/回滚
```

**关键区别**：

- **2PC**：参与者不确定协调者是否会提交，在收到 Commit 之前一直等待
- **3PC**：参与者知道协调者已经决定提交，在收到预提交后可以预期最终会收到提交

## 3PC 的工作流程

### 三个阶段详解

#### 第一阶段：CanCommit

**协调者询问**：协调者向所有参与者发送 CanCommit，询问是否可以提交。

**参与者响应**：

- 如果可以提交，返回 Yes
- 如果不能提交，返回 No

```java
// 参与者 CanCommit 处理
public CanCommitResponse canCommit(CanCommitRequest request) {
    try {
        // 检查是否可以提交
        // 比如：检查资源是否可用、是否有锁冲突
        boolean canCommit = checkCanCommit(request);

        if (canCommit) {
            return new CanCommitResponse(true);
        } else {
            return new CanCommitResponse(false, "Cannot commit");
        }
    } catch (Exception e) {
        return new CanCommitResponse(false, e.getMessage());
    }
}
```

**CanCommit 阶段的目的**：

- 快速排除无法参与事务的节点
- 减少后续阶段的参与者数量
- 降低协调者崩溃的风险

#### 第二阶段：PreCommit

如果所有参与者都返回 Yes，协调者进入 PreCommit 阶段。

**协调者发送 PreCommit**：

协调者向所有参与者发送 PreCommit 消息。

**参与者执行**：

- 参与者收到 PreCommit 后，执行事务操作，写入预写日志
- 返回 Ack

```java
// 参与者 PreCommit 处理
public AckResponse preCommit(PreCommitRequest request) {
    try {
        // 锁定资源
        lockResources(request.transactionId);

        // 写预提交日志（表示协调者已决定提交）
        writePreCommitLog(request.transactionId);

        // 执行事务（但不提交）
        executeTransaction(request.transactionId);

        return new AckResponse(true);
    } catch (Exception e) {
        return new AckResponse(false, e.getMessage());
    }
}
```

**PreCommit 阶段的意义**：

参与者知道协调者已经决定提交，进入"提交就绪"状态。

#### 第三阶段：DoCommit

如果协调者收到所有参与者的 Ack，进入 DoCommit 阶段。

**协调者发送 DoCommit**：

协调者向所有参与者发送 DoCommit 消息。

**参与者提交**：

- 参与者收到 DoCommit 后，提交事务，释放锁
- 返回 Ack

```java
// 参与者 DoCommit 处理
public AckResponse doCommit(DoCommitRequest request) {
    try {
        // 提交事务
        commitTransaction(request.transactionId);

        // 释放锁
        releaseLocks(request.transactionId);

        // 写提交日志
        writeCommitLog(request.transactionId);

        return new AckResponse(true);
    } catch (Exception e) {
        // 提交失败，仍然需要重试
        return new AckResponse(false, e.getMessage());
    }
}
```

## 3PC 如何解决 2PC 的阻塞问题？

### 协调者崩溃的场景

```
场景：协调者发送 PreCommit 后崩溃

协调者                      参与者 A                    参与者 B
   │                            │                           │
   │─────── PreCommit ─────────▶│                           │
   │                            ▼                           │
   │                      [预提交完成]                       │
   │◀─────── Ack ──────────────│                           │
   │                            │                           │
   │  X（崩溃）                  │                           │
   │                            │                           │
   │                            │─────── PreCommit ─────────▶│
   │                            │                           │
   │                            │                      [预提交完成]
   │                            │                           │
   │                            │◀─────── Ack ──────────────│
   │                            │                           │
```

**参与者的行为**：

参与者 A 和 B 都收到了 PreCommit，它们知道协调者已经决定提交。

- 如果超时未收到 DoCommit，参与者可以**继续提交**
- 不会出现 2PC 中的"不知道该做什么"的情况

### 参与者超时机制

3PC 的另一个改进是**参与者的主动行为**。

```java
// 参与者超时处理
public class Participant {
    private Timer timer;

    // 等待 CanCommit 响应
    public void onCanCommitTimeout() {
        // 协调者可能崩溃，可以安全地中止
        abort();
    }

    // 等待 PreCommit
    public void onPreCommitTimeout() {
        // 协调者可能崩溃，但已经收到 PreCommit
        // 可以继续提交（因为预提交日志表明协调者已决定）
        doCommit();
    }

    // 等待 DoCommit
    public void onDoCommitTimeout() {
        // 继续提交
        doCommit();
    }
}
```

**超时决策**：

| 阶段 | 超时行为 | 原因 |
| --- | --- | --- |
| CanCommit | 中止 | 协调者还没决定，可以安全回滚 |
| PreCommit | 提交 | 协调者已经决定提交，等待协调者确认 |
| DoCommit | 提交 | 协调者已经决定提交，继续完成 |

## 3PC 的局限性

### 网络分区问题

3PC 在网络分区时仍然存在问题。

```
场景：网络分区，一部分节点收到 DoCommit，一部分没有

分区 1                    分区 2
参与者 A（收到）            参与者 B（未收到）
    │                        │
    ▼                        ▼
[提交事务]              [等待 DoCommit]
    │                        │
    │                        ▼
    │                  [超时，提交事务]
    ▼                        ▼
  分区恢复后，两个节点状态一致
```

**分析**：

分区后，两个分区都能继续处理事务，最终都会提交。虽然结果一致，但**时序上不一致**。

### 仍然存在的问题

| 问题 | 说明 |
| --- | --- |
| 不保证线性一致性 | 3PC 不保证所有节点在同一时刻提交 |
| 仍可能有不一致 | 网络分区时，不同时刻提交的节点可能看到不同状态 |
| 复杂性增加 | 三阶段增加了协议复杂性 |

### FLP 不可能定理

回顾 FLP 不可能定理：在纯异步网络中，即使只有一个节点可能故障，也不存在确定性的共识算法。

3PC 并没有打破 FLP 不可能定理。它只是**降低了阻塞的概率**，而不是完全消除。

## 2PC vs 3PC 对比

### 协议对比

| 维度 | 2PC | 3PC |
| --- | --- | --- |
| 阶段数 | 2 | 3 |
| 阻塞问题 | 严重 | 轻微 |
| 协调者崩溃 | 参与者阻塞 | 参与者可继续 |
| 网络分区 | 可能不一致 | 可能不一致 |
| 实现复杂度 | 低 | 中 |
| 性能 | 较高 | 略低 |

### 阻塞情况对比

| 场景 | 2PC | 3PC |
| --- | --- | --- |
| 协调者 Prepare 时崩溃 | 参与者可回滚 | 参与者可回滚 |
| 协调者发送 Commit 前崩溃 | 参与者阻塞 | 参与者可回滚 |
| 协调者发送 Commit 后崩溃 | 参与者阻塞 | 参与者可提交 |
| 参与者崩溃 | 协调者可能阻塞 | 协调者可能阻塞 |

### 性能对比

```
场景：1000 个事务，每个事务涉及 3 个参与者

2PC：
- 平均事务时间：100ms
- 系统吞吐量：~10,000 tps

3PC：
- 平均事务时间：150ms（多一个网络往返）
- 系统吞吐量：~7,000 tps
```

## 3PC 的实际应用

### 应用场景

3PC 在实际系统中的应用相对较少，主要原因是：

1. **2PC 已经够用**：在大多数场景下，2PC 的阻塞问题可以通过其他方式缓解
2. **3PC 的优势不明显**：3PC 只减少了阻塞概率，但增加了复杂性
3. **更好的替代方案**：Seata、Saga 等框架提供了更好的分布式事务方案

### 理论研究价值

3PC 的主要价值在于**理论研究**：

- 它展示了如何通过增加阶段来减少阻塞
- 它为后续的分布式事务协议提供了思路
- 它帮助我们理解分布式系统的基本限制

### 典型应用

虽然 3PC 直接应用较少，但其思想被应用在其他协议中：

1. **两阶段锁的改进**：在锁获取和释放之间增加缓冲阶段
2. **共识算法的优化**：Raft 的选举过程借鉴了类似的思路
3. **分布式缓存**：在缓存失效过程中使用多阶段确认

## 总结

三阶段提交（3PC）是对两阶段提交的改进，通过增加"预提交"阶段，降低了协调者崩溃时的阻塞概率。

**3PC 的核心改进**：

- **增加预提交阶段**：参与者知道协调者已经决定提交
- **超时主动行为**：参与者可以根据当前阶段决定超时行为
- **减少阻塞概率**：协调者崩溃时，参与者不再完全阻塞

**3PC 的局限性**：

- **仍然可能不一致**：网络分区时仍可能出现问题
- **不打破 FLP**：FLP 不可能定理仍然有效
- **性能略低**：多一个网络往返，吞吐量略低

**2PC vs 3PC 的选择**：

- 低并发场景：两者都可以，选择 2PC 更简单
- 高并发场景：两者都不适合，选择 Saga 等异步方案
- 对一致性要求高：选择 2PC + TCC 补偿
- 对可用性要求高：选择 Saga + 最终一致性
