# 两阶段提交 (2PC)

想象一个跨国转账场景：你从中国银行向美国花旗银行转账 100 万元。这个事务涉及两个银行系统的数据库。如果只在一个数据库扣款成功、另一个扣款失败，就出现了"钱凭空消失或凭空出现"的问题。

两阶段提交（2PC）正是为了解决这类问题而设计的。它通过引入一个"协调者"的角色，将分布式事务的提交分为两个阶段，确保所有参与者要么全部提交、要么全部回滚。

## 为什么需要两阶段提交？

### 分布式事务的难题

在单机系统中，事务由数据库的 ACID 机制保证。但在分布式系统中，事务涉及多个节点，这带来了挑战：

```
场景：分布式转账

节点 A（中国银行）：扣款 100 万
节点 B（美国花旗）：收款 100 万

问题：
- 如果 A 扣款成功，B 收款失败，如何处理？
- 如果 A 扣款成功，但 B 在收款时崩溃了，如何处理？
- 如果 A 和 B 都成功了，但网络中断，双方都不知道对方的状态？
```

### 两阶段提交的思路

两阶段提交的核心思想是**预占+确认**：

1. **第一阶段（准备阶段）**：协调者询问所有参与者"你们准备好了吗？"
2. **第二阶段（提交阶段）**：所有参与者都说"准备好了"，协调者才说"提交吧"

这种机制确保了分布式事务的原子性：**要么全部成功，要么全部失败**。

## 两阶段提交的工作流程

### 角色划分

两阶段提交涉及两类角色：

- **协调者（Coordinator）**：负责整个事务的管理，包括发起投票、决定提交或回滚
- **参与者（Participant）**：负责执行具体的数据库操作

```
架构示意：

            ┌─────────────┐
            │  协调者      │
            └──────┬──────┘
                   │
       ┌───────────┼───────────┐
       │           │           │
       ▼           ▼           ▼
   ┌───────┐   ┌───────┐   ┌───────┐
   │参与者 A│   │参与者 B│   │参与者 C│
   └───────┘   └───────┘   └───────┘
```

### 第一阶段：准备阶段

**协调者发送 Prepare**：

协调者向所有参与者发送 `Prepare` 消息，询问是否可以提交事务。

**参与者执行但不提交**：

参与者收到 `Prepare` 后，执行事务操作（锁定资源），但不提交。然后返回 `Vote-Commit` 或 `Vote-Abort`。

```java
// 参与者 Prepare 阶段处理
public VoteResponse prepare(PrepareRequest request) {
    try {
        // 1. 锁定相关资源
        lockResources(request.transactionId);

        // 2. 执行事务操作（但不提交）
        executeTransaction(request.transactionId);

        // 3. 写预写日志（Write-Ahead Log）
        writePrepareLog(request.transactionId);

        // 4. 返回可以提交
        return new VoteResponse(true, "Ready to commit");
    } catch (Exception e) {
        // 如果失败，回滚并返回不可提交
        rollback(request.transactionId);
        return new VoteResponse(false, e.getMessage());
    }
}
```

**流程图**：

```
协调者                      参与者 A                    参与者 B
   │                            │                           │
   │─────── Prepare ───────────▶│                           │
   │                            │                           │
   │                            ▼                           │
   │                      [锁定资源]                        │
   │                      [执行操作]                        │
   │                      [写预写日志]                      │
   │                            │                           │
   │◀──── Vote-Commit ─────────│                           │
   │                            │                           │
   │                            │─────── Prepare ──────────▶│
   │                            │                           │
   │                            │                      [锁定资源]
   │                            │                      [执行操作]
   │                            │                      [写预写日志]
   │                            │                           │
   │                            │◀──── Vote-Commit ─────────│
   │                            │                           │
   ▼                            ▼                           ▼
```

### 第二阶段：提交阶段

#### 提交分支

如果所有参与者都返回 `Vote-Commit`：

```
协调者                      参与者 A                    参与者 B
   │                            │                           │
   │◀──── Vote-Commit ─────────│◀──── Vote-Commit ─────────│
   │                            │                           │
   │─────── Commit ────────────▶│                           │
   │─────── Commit ─────────────────────────────────────────▶│
   │                            │                           │
   │                            ▼                           │
   │                      [提交事务]                         │
   │                      [释放锁]                          │
   │◀──── Ack ─────────────────│                           │
   │                            │                           │
   │                            │                      [提交事务]
   │                            │                      [释放锁]
   │                            │                           │
   │                            │◀──── Ack ─────────────────│
   ▼                            ▼                           ▼
```

```java
// 协调者提交阶段处理
public void commit(String transactionId) {
    // 1. 记录提交决定
    writeCommitLog(transactionId);

    // 2. 向所有参与者发送 Commit
    for (Participant participant : participants) {
        sendCommit(participant, transactionId);
    }

    // 3. 等待所有参与者的 Ack
    waitForAcks(participants);

    // 4. 标记事务完成
    markTransactionComplete(transactionId);
}
```

#### 回滚分支

如果任何一个参与者返回 `Vote-Abort`：

```
协调者                      参与者 A                    参与者 B
   │                            │                           │
   │◀──── Vote-Abort ──────────│◀──── Vote-Commit ─────────│
   │                            │                           │
   │─────── Abort ─────────────▶│                           │
   │─────── Abort ─────────────────────────────────────────▶│
   │                            │                           │
   │                            ▼                           │
   │                      [回滚事务]                         │
   │                      [释放锁]                          │
   │◀──── Ack ─────────────────│                           │
   │                            │                           │
   │                            │                      [回滚事务]
   │                            │                      [释放锁]
   │                            │                           │
   │                            │◀──── Ack ─────────────────│
   ▼                            ▼                           ▼
```

## 两阶段提交的关键机制

### 预写日志（Write-Ahead Log）

预写日志是 2PC 能够恢复的关键机制。

**为什么需要预写日志？**

如果协调者在发送 Commit 之前崩溃了，参与者需要知道如何恢复事务状态。

**预写日志的内容**：

```java
// 预写日志格式
public class WALEntry {
    String transactionId;
    WALOperation operation;  // PREPARE, COMMIT, ABORT
    long timestamp;
    List<Xid> participants;
}

// 预写日志示例
WALEntry {
    transactionId: "tx-123",
    operation: PREPARE,
    timestamp: 1699000000000,
    participants: ["participant-A", "participant-B"]
}
```

**恢复流程**：

```
协调者崩溃恢复：

1. 读取预写日志
2. 如果日志中有 "COMMIT"，重新发送 Commit
3. 如果日志中有 "ABORT"，重新发送 Abort
4. 如果日志中有 "PREPARE" 但没有决定，向参与者查询状态
```

### 阻塞问题

两阶段提交最大的问题是**阻塞**。

#### 阻塞场景

```
场景：协调者发送 Commit 后崩溃

协调者                      参与者 A                    参与者 B
   │                            │                           │
   │─────── Commit ────────────▶│                           │
   │                            ▼                           │
   │                      [提交中...]                       │
   │                            │                           │
   │  X（崩溃）                  │                           │
   │                            │                           │
   │                            │      （等待 Ack 确认）      │
   │                            │                           │
   │  （参与者 A 卡住了）          │                           │
   │                            │                           │
```

**阻塞的原因**：

参与者持有资源锁，等待协调者的 Commit 或 Abort 消息。在协调者恢复之前，这些资源无法释放。

#### 阻塞的危害

| 危害 | 说明 |
| --- | --- |
| 资源泄漏 | 其他事务无法获取被锁定的资源 |
| 系统停顿 | 大量事务堆积，系统吞吐量急剧下降 |
| 不可预测 | 阻塞时间取决于协调者恢复时间 |

## 两阶段提交的实际应用

### Seata 的 AT 模式

Seata 是阿里的分布式事务框架，其 AT 模式基于 2PC 实现。

```java
// Seata AT 模式示例
@GlobalTransactional
public void placeOrder(Order order) {
    // 1. 扣减库存
    inventoryService.deductStock(order.getProductId(), order.getCount());

    // 2. 创建订单
    orderService.createOrder(order);

    // 3. 扣减账户余额
    accountService.deductBalance(order.getUserId(), order.getAmount());
}
```

### JTA/XA 规范

Java 的 JTA（Java Transaction API）和 XA（eXtended Architecture）规范定义了 2PC 的标准接口。

```java
// JTA 分布式事务示例
@Resource
private UserTransaction userTransaction;

@Resource
private DataSource dataSource;

public void executeTransaction() throws Exception {
    userTransaction.begin();

    try {
        // 执行多个数据库操作
        Connection conn1 = dataSource1.getConnection();
        Connection conn2 = dataSource2.getConnection();

        // 操作 1
        conn1.prepareStatement("UPDATE account SET balance = ? WHERE id = ?")
             .executeUpdate();

        // 操作 2
        conn2.prepareStatement("UPDATE inventory SET stock = ? WHERE id = ?")
             .executeUpdate();

        userTransaction.commit();
    } catch (Exception e) {
        userTransaction.rollback();
        throw e;
    }
}
```

## 两阶段提交的优缺点

### 优点

| 优点 | 说明 |
| --- | --- |
| 原子性保证 | 确保所有参与者要么全部提交，要么全部回滚 |
| 实现简单 | 协议理解简单，实现相对直接 |
| 广泛应用 | 已被大多数数据库和中间件支持 |

### 缺点

| 缺点 | 说明 |
| --- | --- |
| 阻塞问题 | 协调者崩溃时，参与者会阻塞 |
| 单点故障 | 协调者是单点，一旦崩溃事务无法完成 |
| 资源锁定时间长 | 参与者锁定资源直到整个事务完成 |
| 不适合高并发 | 锁定时间过长，不适合高并发场景 |

## 两阶段提交的适用场景

### 适合的场景

- **低并发、对一致性要求高的场景**：如金融系统、库存管理
- **参与者数量较少的场景**：如 2-3 个服务
- **事务时间较短**：如简单的增删改操作

### 不适合的场景

- **高并发场景**：锁定时间过长，影响系统吞吐量
- **跨地域分布式系统**：网络延迟高，阻塞时间不可接受
- **长事务**：事务执行时间长，资源锁定代价大

## 总结

两阶段提交（2PC）是分布式事务的基础协议，通过"准备+提交"两个阶段，确保分布式事务的原子性。

**2PC 的核心机制**：

- **协调者-参与者架构**：协调者管理事务决策，参与者执行具体操作
- **预写日志**：保证事务可恢复
- **多数派确认**：确保决策的可靠性

**2PC 的局限性**：

- **阻塞问题**：协调者崩溃时，参与者会阻塞等待
- **单点故障**：协调者是单点
- **资源锁定时间长**：不适合高并发场景

正是为了解决 2PC 的阻塞问题，三阶段提交（3PC）被提了出来。在下一章中，我们将详细讨论 3PC 的设计原理和它如何克服 2PC 的缺陷。
