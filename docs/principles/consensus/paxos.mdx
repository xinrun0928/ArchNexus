# Paxos 算法详解

1990 年，Leslie Lamport 发表了一篇论文，提出了 Paxos 算法。这篇论文最初被 OSDI 学术会议拒绝——不是因为内容有错，而是因为评审们实在看不懂。后来，Lamport 把它改成了一个希腊小岛上"议会达成共识"的寓言故事，论文才得以发表。

这个寓言故事让 Paxos 变得家喻户晓，但也让很多人误以为它只是一个理论玩具。实际上，Paxos 是分布式共识领域的奠基之作，它证明了在异步网络模型中，共识问题是可解的——只是需要一个多数派机制来打破 FLP 不可能定理的诅咒。

## Paxos 的核心问题

Paxos 要解决的问题是：**在一组可能故障的进程集合中，如何对一个值达成共识？**

这个问题的精确定义包含三个要求：

1. **一致性**：所有提案中，最终只会有一个被选中
2. **合法性**：只有被提案的值才能被选中
3. **活性**：所有正确的进程最终都能做出决定

:::tip 回顾
在上一章中，我们介绍了 FLP 不可能定理：在一个纯异步网络中，如果至少有一个进程可能故障，那么不存在任何确定性的算法能保证活性。Paxos 通过引入超时和随机性，在工程上解决了活性的问题。
:::

## 两种 Paxos 的变体

Paxos 有两种主要变体，分别用于不同的场景：

### Single-Decree Paxos

Single-Decree Paxos 用于对一个值达成共识。它的核心思想是：**通过两阶段投票，确保只有多数派认可的提案才能被选中。**

### Multi-Paxos

Multi-Paxos 是 Single-Decree Paxos 的扩展，用于对一系列值达成共识。它通过选择一个 Leader 来优化性能，减少两阶段投票的次数。

在实际系统中，我们通常使用的是 Multi-Paxos 的变体。

## Single-Decree Paxos 的两阶段

Single-Decree Paxos 通过两个阶段来达成共识：

### 第一阶段：Prepare

**Prepare 阶段的目标**：选择一个提案编号，为后续的 Accept 阶段做准备。

**提案编号**：每个提案都有一个唯一的编号，编号越大优先级越高。编号必须是单调递增的。

**Prepare 消息**：

```
Proposer: 我想发起提案 N，请准备投票
Acceptor: 好，我记录下编号 N，并告诉你我已经批准过的最大编号的提案
```

**Acceptor 的响应规则**：

1. 如果 N 大于之前承诺过的任何提案编号，则承诺不再接受编号小于 N 的提案
2. 如果之前批准过某个提案，返回已批准的最大编号的提案

**Prepare 阶段示例**：

```
初始状态：
Acceptor P: 已批准提案 [无]
Acceptor Q: 已批准提案 [无]
Acceptor R: 已批准提案 [无]

Proposer 向 P、Q、R 发送 Prepare(1)：
- P: 承诺编号 1，返回已批准提案 [无]
- Q: 承诺编号 1，返回已批准提案 [无]
- R: 承诺编号 1，返回已批准提案 [无]

收到多数派（3 个）承诺，准备进入 Accept 阶段
```

### 第二阶段：Accept

**Accept 阶段的目标**：让多数派接受一个提案值。

**Accept 消息**：

```
Proposer: 基于你们返回的信息，我提议值 V，请批准
Acceptor: 如果编号 N 没有被更高级的提案抢占，我批准 V
```

**Acceptor 的接受规则**：

只有当提案编号 N 不小于已承诺的最大编号时，才接受该提案。

**Accept 阶段示例**：

```
继续上面的例子：
Proposer 收到 Q 和 R 的响应，都说"没有已批准的提案"
因此，Proposer 可以提议任何值，比如 V = "Hello"
Proposer 向 P、Q、R 发送 Accept(1, "Hello")：
- P: 接受，提案已批准
- Q: 接受，提案已批准
- R: 接受，提案已批准

收到多数派（3 个）接受，提案 "Hello" 被选中
```

## 多个 Proposer 的竞争

在实际系统中，可能有多个 Proposer 同时发起提案。这就是 Paxos 的活锁问题。

### 活锁场景

```
T1: Proposer A 发起 Prepare(1)，收到多数派承诺
T2: Proposer B 发起 Prepare(2)，收到多数派承诺（编号更大）
T3: Proposer A 发送 Accept(1, V1)，被拒绝（因为编号太小）
T4: Proposer A 发起 Prepare(3)，收到多数派承诺
T5: Proposer B 发送 Accept(2, V2)，被拒绝（因为编号太小）
T6: Proposer B 发起 Prepare(4)，收到多数派承诺
...（循环往复，永远无法结束）
```

这就是活锁：两个 Proposer 不断升级编号，但永远无法让提案被接受。

### 解决方案

解决活锁有几种常见方法：

1. **随机退避**：Proposer 在被拒绝后，随机等待一段时间再重试
2. **Multi-Paxos + Leader**：只允许一个 Leader 发起提案，避免竞争
3. **租约机制**：使用租约来保证一段时间内只有一个 Proposer 活跃

## Multi-Paxos 的优化

在实际系统中，我们通常使用 Multi-Paxos 来对一系列值达成共识，比如日志复制。

### Leader 的作用

Multi-Paxos 通过选择一个 Leader 来优化：

- 在 Leader 稳定期间，所有提案都由 Leader 提出
- 不需要每次都经过两阶段投票
- 只需要在 Leader 变更时重新走 Paxos 流程

**Leader 选举**：

```
如果 Follower 在选举超时时间内没有收到 Leader 的心跳：
- Follower 转换为 Candidate
- 增加当前任期号
- 向所有节点发送投票请求
- 如果获得多数派投票，成为新的 Leader
```

### 日志复制

Multi-Paxos 的核心应用是日志复制：

1. 客户端向 Leader 发送写请求
2. Leader 将请求追加到本地日志
3. Leader 并行向 Follower 发送日志条目
4. Follower 收到后追加到本地日志
5. 当多数派 Follower 确认后，Leader 提交日志

```
日志条目：
索引 1: <term=1, command="SET x=10">
索引 2: <term=1, command="SET y=20">
索引 3: <term=2, command="SET z=30">  // Leader 变更
```

### 优化效果

Multi-Paxos 的优化效果：

| 操作 | Basic Paxos | Multi-Paxos |
| --- | --- | --- |
| 正常情况 | 2 轮 RPC | 1 轮 RPC |
| Leader 变更 | 多轮 RPC | 多轮 RPC |
| 提案冲突 | 可能活锁 | 无冲突 |

## Paxos 在实际系统中的应用

### Chubby

Chubby 是 Google 内部的分布式锁服务，它使用 Paxos 来实现一致性。

**架构**：

- Chubby 集群由多个副本组成
- 副本之间通过 Paxos 协议同步数据
- 客户端连接到主副本进行读写操作

**应用场景**：

- 分布式锁
- 配置存储
- 选举服务

### Zookeeper (ZAB)

ZooKeeper 使用 ZAB 协议，ZAB 是 Paxos 的简化版本。

**ZAB 的特点**：

- 主备模式
- 简化了 Paxos 的两阶段
- 针对高吞吐量场景优化

**应用场景**：

- 服务发现
- 分布式锁
- 配置管理

### PostgreSQL 的共识协议

PostgreSQL 的流复制使用类 Paxos 的共识协议来保证主从一致。

## Paxos 的局限性

虽然 Paxos 是分布式共识的奠基之作，但它也有局限性：

### 性能问题

- 两阶段投票增加了延迟
- 活锁问题影响吞吐量和延迟
- 跨数据中心部署时延迟更高

### 实现复杂度

- 完整的 Paxos 实现非常复杂
- 需要处理各种边界情况
- 工程实现与理论描述有差距

### 难以理解

- Paxos 的论文非常晦涩
- Multi-Paxos 的优化增加了理解难度
- 很多实现实际上是 Paxos 的变体

## 总结

Paxos 是分布式共识领域的奠基之作，它证明了共识问题在异步网络中是可解的。Single-Decree Paxos 通过两阶段投票（Prepare + Accept）来达成共识，Multi-Paxos 通过 Leader 优化来提高性能。

Paxos 的核心思想是**多数派机制**：只有获得多数派支持的提案才能被选中。这个思想贯穿了 Raft、ZAB 等后续的共识算法。

但 Paxos 也有局限性：性能问题、实现复杂、难以理解。正是这些问题催生了 Raft 算法——一个以可理解性为设计目标的共识算法。
