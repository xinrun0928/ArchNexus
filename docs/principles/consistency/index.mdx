# 一致性模型

在上一章中，我们了解了 CAP 定理和 BASE 理论，知道了一致性和可用性之间的权衡。但一致性并不是一个非黑即白的概念，而是一个从强到弱的谱系。

不同的应用场景需要不同的一致性保证。金融系统需要强一致性，而社交媒体可以接受最终一致性。理解不同一致性模型的特点和适用场景，是架构师必备的能力。

## 本章节内容

本章节将深入探讨分布式系统中的各种一致性模型：

- **强一致性**：最严格的一致性保证
- **弱一致性**：最宽松的一致性保证
- **最终一致性**：BASE 理论的核心
- **因果一致性**：保证有因果关系的操作顺序一致

## 为什么需要多种一致性模型？

如果强一致性是最好的，为什么不所有系统都用强一致性？

答案很简单：**强一致的代价太高了**。

为了实现强一致性，系统需要：

- 同步复制数据到所有节点
- 等待确认后再返回成功
- 处理网络分区和节点故障

这会导致延迟高、性能差、可用性低。对于很多应用场景，这种代价是不必要的。

因此，我们需要多种一致性模型，以便根据业务需求选择最合适的一种。

## 一致性模型谱系

一致性模型可以看作一个谱系：

```
强一致性
    ↓
因果一致性
    ↓
会话一致性
    ↓
单调读
    ↓
读己所写
    ↓
最终一致性
    ↓
弱一致性
```

越往下，一致性越弱，但性能越好、可用性越高。
