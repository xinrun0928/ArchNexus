# ArchNexus 文章写作

根据用户提供的主题，生成符合 ArchNexus 规范的架构文档。

## 核心原则

1. **理论与实践结合**：每个概念配上代码示例（Java优先）或架构图描述
2. **权衡分析优先**：讲优点必须讲"为什么不是银弹"、局限性与适用场景
3. **问题驱动**：从"解决什么问题"出发，不罗列知识点
4. **真实项目视角**：贴近真实项目，加入踩坑案例、压测数据、故障复盘
5. **禁止套用模板**：开场必须结合主题自行构思，不得原封不动使用模板

## 工作流程

### 1. 确认主题与分类

对照 `topics/` 目录下的分类文件，确认文章属于哪个分类，选取对应的要点列表。

### 2. 设计开场

开场只提供一个思路，**禁止原封不动套用**。每篇必须结合主题自行构思真实、具体、不同的切入角度：

- 场景切入（故障排查）：构思真实感强的现场——时间、地点、异常现象、错误判断
- 痛点共鸣（常见误区）：从高频错误出发，让读者对号入座
- 核心追问（原理类）：问一个读者想过但没深究的问题
- 反差对比（打破认知）：找一个普遍误解，证明它是错的
- 悬念制造（新概念）：展示一个反直觉的结果，引发好奇
- 一行代码开场（最佳实践）：选一个看似简单但有深意的配置或代码片段
- 数字开场（性能分析）：找一个真实或有代表性的数据
- 历史演进（版本差异）：从起点出发，讲清楚为什么要演进

> 一篇文档只用一个开场，不要叠加。

### 3. 确定文章结构

根据主题类型选择结构：

**漏斗式展开**：问题背景 → 现有方案不够 → 核心设计思路 → 关键原理拆解 → 实际使用 → 常见问题 → 适用与不适用场景 → 总结。

**阶梯式推进**：最简版本（先跑通）→ 完整版本（加异常处理）→ 生产版本（加监控、限流、熔断）→ 原理版本（解释为什么）→ 调优版本。

### 4. 质量增强（按需选用）

以下技巧根据主题选取 3~4 项加入文章：

- **权衡矩阵表**：场景 vs 方案对比表，说明推荐与不推荐理由
- **反模式警示**：至少3个常见错误，包含：错误现象、为什么是错的、正确做法
- **Mermaid 图表**：至少1个核心流程图/时序图/状态图
- **量化数据**：性能对比提供吞吐量、延迟、资源消耗的具体数字，注明数据来源
- **真实案例引用**：引用公开技术博客/论文，注明现象、根因、解决方案、出处
- **思考题**：3个引导性问题，每个带 `<details>` 折叠参考答案

### 5. 写作时参考 rules/

`rules/` 目录包含所有格式和风格规范，写作时需严格参照。`index.mdx` 文件属于导览文件，不受规则限制。

- **rules/rspress.mdc**：Rspress 语法规范（代码块、容器、徽章、格式）
- **rules/notation.mdc**：符号转义规范（运算符、泛型、JSX 模板字面量）
- **rules/writing.mdc**：写作风格规范（开场、结构、节奏、表达）

### 6. 字数与深度要求

| 类型 | 字数 | 目标 |
|---|---|---|
| 工具型 | 不限制 | 快速解决问题 |
| 普通文章 | `>= 1500` | 建立完整知识体系 |
| 原理类 | `>= 3000` | 讲透底层机制与设计逻辑 |
| 架构设计类 | `>= 2000` | 讲清 trade-off、适用场景与边界 |

### 7. 结尾设计

好的结尾让读者继续思考。推荐：抛出新问题、延伸对比、回到开场形成闭环。

### 8. 质量检查

输出文章后，对照检查：

- [ ] 有明确痛点场景引入
- [ ] 至少1个 Mermaid 图
- [ ] 有可运行的 Java 代码示例
- [ ] 有权衡矩阵表
- [ ] 有反模式警示（至少3个常见错误）
- [ ] 有量化性能数据
- [ ] 有真实案例引用（附来源）
- [ ] 有3个思考题（含参考答案折叠）

缺失项补充后重新检查。

### 9. 输出路径

文章保存到 `docs/` 对应目录下，文件名为 kebab-case 的 `.mdx` 文件（如 `consistency-models/linearizability.mdx`）。不得出现 `.md` 后缀。站内链接不得包含 `/docs` 前缀，不得出现 `.mdx` 后缀。
