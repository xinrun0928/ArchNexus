# 设计模式篇写作指南

本文档是撰写「设计模式篇」文章的权威指南。设计模式是软件工程中的最佳实践总结，每一篇文章都应该以**讲透意图、展示实现、指导应用**为核心目标。

## 章节结构概览

设计模式篇包含以下 6 个主要章节：

1. **创建型模式** - 对象创建的最佳实践
2. **结构型模式** - 类与对象组合的艺术
3. **行为型模式** - 对象间通信的模式
4. **并发设计模式** - 并发编程中的专用模式
5. **分布式设计模式** - 分布式系统中的经典模式
6. **架构模式** - 系统架构层面的设计模式

## 核心写作原则

### 1. 意图优先，不是代码实现

设计模式的核心是设计意图，而不是代码模板。要让读者理解「为什么要这样设计」，而不仅仅是「如何写代码」。

**错误示例**：

```java
单例模式的实现如下：

public class Singleton {
    private static Singleton instance;
    private Singleton() {}
    public static Singleton getInstance() {
        if (instance == null) {
            instance = new Singleton();
        }
        return instance;
    }
}
```

**正确示例**：

```java
单例模式的核心意图是：**保证一个类仅有一个实例，并提供一个全局访问点**。这个需求在真实系统中无处不在：数据库连接池、线程池、配置管理器、日志对象等，都需要确保全局唯一。

但简单的实现就够了吗？在多线程环境下，两个线程可能同时进入 `if (instance == null)` 的判断，导致创建多个实例。这就需要我们引入线程安全的实现方式。
```

### 2. 问题驱动，不是模式罗列

每一篇文章都应该从**真实场景**出发，展示模式解决了什么问题。

**推荐的开场方式**：

- 痛点场景：描述开发中常见的代码问题
- 需求变化：展示需求变化带来的代码重构
- 设计困境：提出两难的选择，引出模式的权衡

**避免的开场方式**：

- ❌ 「工厂模式是创建型模式的一种」
- ❌ 「本文将介绍 23 种设计模式」
- ❌ 「观察者模式是常用的模式」

### 3. 图解思维，不是文字堆砌

设计模式的本质是结构关系，图解比文字更直观。

**应该包含的图解**：

- **类图**：展示类与类之间的关系（继承、实现、关联、依赖等）
- **时序图**：展示对象间的交互顺序
- **对比图**：模式应用前后的结构对比

**示例**：

```java
在观察者模式中，Subject（主题）维护一个 Observer（观察者）列表。当 Subject 状态发生变化时，会遍历所有观察者并调用它们的 `update()` 方法。这种解耦让主题和观察者可以独立变化。

```
Subject         Observer
    | <---------   |
    |   register   |
    |              |
    | ----notify-->|
    |   update()   |
```
```

### 4. 适用场景，不是万能药

每个模式都有它的适用场景和局限性，要明确告诉读者「什么时候用，什么时候不该用」。

**应该包含的内容**：

- **适用场景**：模式最擅长的场景
- **不适用场景**：模式会增加复杂度的场景
- **替代方案**：不使用模式时的替代做法
- **权衡分析**：使用模式的代价和收益

**示例**：

```java
策略模式适合「算法族」场景，如排序、压缩、支付方式等。但如果算法非常简单且不会变化，使用策略模式反而会增加不必要的抽象层。只有当算法可能频繁变化或需要在运行时切换时，策略模式的价值才真正体现。
```

### 5. 代码实战，不是伪代码

代码示例必须是可运行的、有上下文的完整代码，而不是脱离实际的伪代码。

**代码示例要求**：

- 完整的类结构，不是片段代码
- 有实际意义的业务场景，不是抽象的 `A`、`B`、`C`
- 包含测试代码或使用示例
- 添加必要的注释，说明关键设计点

## 各章节写作要点

### 创建型模式

#### 单例模式

**应该包含的内容**：

1. **单例的本质**：为什么需要单例，单例解决了什么问题
2. **多种实现方式**：饿汉式、懒汉式、双重检查锁、静态内部类、枚举
3. **线程安全性分析**：每种实现的线程安全性与性能对比
4. **反射与序列化的破坏**：如何防止反射和序列化破坏单例
5. **单例的误区**：Spring Bean 单例 vs GoF 单例

**写作建议**：

- 对比不同实现方式的优缺点，给出推荐方案
- 强调枚举单例的优势：简洁、线程安全、防反射
- 说明单例模式的隐藏陷阱：测试困难、隐藏依赖

#### 工厂方法模式与抽象工厂模式

**应该包含的内容**：

1. **简单工厂 vs 工厂方法 vs 抽象工厂**：三者的区别和关系
2. **开闭原则的体现**：如何通过工厂方法支持新增产品
3. **工厂模式的层次**：工厂的继承与组合
4. **实际应用**：JDBC 中的 `DriverManager`、Spring 的 `BeanFactory`

**写作建议**：

- 用具体产品族说明抽象工厂的价值，如数据库连接工厂
- 强调工厂模式的「延迟创建」特点
- 说明工厂模式与依赖注入的关系

#### 建造者模式

**应该包含的内容**：

1. **构建复杂对象**：为什么需要建造者模式
2. **Builder vs 构造器 vs Setter**：三种创建方式的选择
3. **Fluent API**：链式调用的实现技巧
4. **不可变对象的构建**：通过建造者模式实现不可变对象
5. **实际应用**：Lombok 的 `@Builder`、OkHttp 的 `Request.Builder`

**写作建议**：

- 对比 `new Person(name, age, gender, address, ...)` 的参数爆炸问题
- 说明建造者模式的「分步构建」特点
- 强调 Builder 本身的线程安全问题

### 结构型模式

#### 适配器模式

**应该包含的内容**：

1. **适配的本质**：让不兼容的接口协同工作
2. **类适配器 vs 对象适配器**：两种实现方式的选择
3. **适配器 vs 装饰器 vs 代理**：三者的区别
4. **实际应用**：Arrays.asList()、InputStreamReader

**写作建议**：

- 用「美标插头」转「国标插座」的比喻说明适配器原理
- 强调适配器的「补救」性质，而非设计阶段的选择
- 区分「单向适配」与「双向适配」场景

#### 装饰器模式

**应该包含的内容**：

1. **装饰 vs 继承**：为什么装饰优于继承
2. **装饰器的组合**：多个装饰器的嵌套应用
3. **透明装饰 vs 半透明装饰**：接口完整性选择
4. **实际应用**：Java I/O 流、Spring 的 `BeanWrapper`

**写作建议**：

- 用「穿衣」比喻说明装饰器的层层叠加
- 强调装饰器的「运行时扩展」特点
- 说明装饰器与 AOP 的关系

#### 代理模式

**应该包含的内容**：

1. **代理的本质**：控制对对象的访问
2. **静态代理 vs 动态代理**：实现方式与适用场景
3. **JDK 动态代理 vs CGLIB**：两种动态代理的区别
4. **代理的应用**：远程代理、虚拟代理、保护代理
5. **实际应用**：Spring AOP、RPC 调用

**写作建议**：

- 用「经纪人」比喻说明代理的作用
- 动态代理的代码示例要完整展示 `InvocationHandler`
- 强调代理模式的「控制」而非「增强」本质

### 行为型模式

#### 策略模式

**应该包含的内容**：

1. **算法族的封装**：如何将算法族独立封装
2. **策略的运行时切换**：如何动态选择策略
3. **策略模式的代价**：策略类的数量爆炸问题
4. **实际应用**：排序算法、支付方式、压缩算法

**写作建议**：

- 用「旅游出行方式」比喻策略选择
- 说明策略模式与多态的关系
- 提出策略类管理的解决方案（工厂 + 注册表）

#### 观察者模式

**应该包含的内容**：

1. **发布-订阅模型**：主题与观察者的关系
2. **推模型 vs 拉模型**：两种通知方式的对比
3. **同步 vs 异步通知**：通知方式的选择
4. **观察者的移除**：避免内存泄漏的关键
5. **实际应用**：Vue 的响应式、事件总线、MQ

**写作建议**：

- 用「订阅报纸」比喻发布-订阅
- 强调观察者模式的「一对多」特点
- 说明观察者模式与事件驱动的关系

#### 责任链模式

**应该包含的内容**：

1. **责任的传递**：请求在链上的传递过程
2. **纯责任链 vs 不纯责任链**：责任的处理方式
3. **链的构造**：如何动态构建责任链
4. **实际应用**：Servlet Filter、Spring Interceptor、Netty Pipeline

**写作建议**：

- 用「请假审批流程」比喻责任链
- 强调责任链的「灵活组合」特点
- 说明责任链与 AOP 拦截器的区别

### 并发设计模式

#### 生产者-消费者模式

**应该包含的内容**：

1. **解耦生产与消费**：为什么要解耦
2. **缓冲区的选择**：有界队列 vs 无界队列
3. **线程安全问题**：如何保证线程安全
4. **实际应用**：消息队列、线程池工作队列

**写作建议**：

- 用「餐厅后厨」比喻生产者与消费者
- 对比 `ArrayBlockingQueue`、`LinkedBlockingQueue`、`SynchronousQueue`
- 说明生产者-消费者模式与事件循环的关系

#### 读写锁模式

**应该包含的内容**：

1. **读多写少的优化**：为什么需要读写锁
2. **读写锁的实现原理**：读共享、写独占
3. **读写锁的降级**：从写锁降级到读锁
4. **实际应用**：ReentrantReadWriteLock、StampedLock

**写作建议**：

- 对比 `synchronized` 和读写锁的性能差异
- 说明读写锁的「公平性」选择
- 强调读写锁降级的安全性问题

### 分布式设计模式

#### Saga 模式

**应该包含的内容**：

1. **长事务的挑战**：为什么 2PC 不适合长事务
2. **Saga 的思想**：将长事务拆分为多个本地事务
3. **补偿机制**：如何回滚已执行的本地事务
4. **Saga 的两种实现**：编排式与协同式
5. **实际应用**：电商订单处理、微服务分布式事务

**写作建议**：

- 用「旅游行程」比喻 Saga 的顺序执行与补偿
- 对比编排式（中央协调）与协同式（事件驱动）的优缺点
- 说明 Saga 的「最终一致性」特点

#### CQRS 模式

**应该包含的内容**：

1. **读写分离的动机**：为什么读模型和写模型不同
2. **命令与查询的职责分离**：CQRS 的本质
3. **CQRS 的架构**：命令端与查询端的设计
4. **事件溯源与 CQRS**：两者的结合使用
5. **实际应用**：复杂读写的业务系统

**写作建议**：

- 强调 CQRS 的「复杂性代价」，不是所有系统都适用
- 说明 CQRS 与读写分离（数据库层面）的区别
- 展示 CQRS 的完整架构图

### 架构模式

#### 分层架构

**应该包含的内容**：

1. **分层的思想**：为什么需要分层
2. **经典的三层架构**：表现层、业务层、数据层
3. **分层的原则**：上层依赖下层，下层不依赖上层
4. **跨层调用问题**：如何处理跨层需求
5. **实际应用**：DDD 分层架构、Spring MVC

**写作建议**：

- 用「建筑分层」比喻软件分层
- 说明分层架构的「可测试性」优势
- 对比分层架构与管道架构的区别

#### 事件驱动架构

**应该包含的内容**：

1. **事件的本质**：什么是事件，事件与消息的区别
2. **事件驱动 vs 请求响应**：两种架构模式的对比
3. **事件流的处理**：事件的产生、传递、消费
4. **最终一致性**：事件驱动架构的一致性保证
5. **实际应用**：Kafka、事件溯源、CQRS

**写作建议**：

- 用「新闻推送」比喻事件传播
- 说明事件驱动架构的「异步解耦」优势
- 强调事件驱动架构的「调试困难」问题

## 写作格式规范

### 文件命名

- 使用 `kebab-case` 格式，如 `singleton-pattern.mdx`
- 文件名要能清晰表达文章主题

### 文章结构

推荐采用以下结构：

1. **开场** - 痛点场景，引发共鸣
2. **模式定义** - 意图说明，建立认知
3. **结构分析** - 图解类图，理解关系
4. **实现示例** - 完整代码，展示应用
5. **适用场景** - 场景分析，权衡选择
6. **常见误区** - 警示说明，避免误用
7. **总结延伸** - 收束全文，引发思考

### 代码示例

- 使用 Java 作为主要示例语言（可根据需要补充其他语言）
- 代码示例应该是完整的、可运行的
- 代码示例后要有解释说明「为什么这样做」

### 符号转义

- 比较运算符（`=`、`<=`、`>=` 等）在正文中需用反引号包裹
- 泛型类型（`<T>`）需用反引号包裹
- 路径不得包含 `/docs` 前缀和 `.mdx` 后缀

### 文章长度

- 模式介绍文章建议 `>= 2500` 字
- 确保讲透模式意图和应用，不要停留在表面

## 质量检查清单

在发布文章前，请检查以下问题：

- [ ] 文章是否从真实场景出发，而不是模式定义？
- [ ] 模式意图是否讲透了，读者能否理解「为什么」？
- [ ] 是否有图解说明类与对象的关系？
- [ ] 代码示例是否完整可运行？
- [ ] 是否包含了适用场景和权衡分析？
- [ ] 是否对比了相关模式？
- [ ] 是否遵循了所有格式规范？
- [ ] 字数是否达到建议要求？

只有以上问题都得到肯定的回答，文章才能发布。

## 总结

设计模式是软件工程中的最佳实践总结。撰写这部分文章，核心目标是让读者建立起对设计模式的系统性理解，培养他们识别问题、选择模式、权衡利弊的能力。

记住：**好的设计模式文档不是代码模板，而是设计思维的传递**。每一篇文章都应该让读者在读完之后，不仅知道「怎么做」，更重要的是理解「为什么要这样设计、什么时候不该用」。
