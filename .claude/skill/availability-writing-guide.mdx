# 可用性篇写作指南

本文档是撰写「可用性篇」文章的权威指南。可用性是高可用系统的核心指标，每一篇文章都应该以**建立可用思维、传授防护策略、提供实战经验**为核心目标。

## 章节结构概览

可用性篇包含以下 7 个主要章节：

1. **可用性理论** - 可用性的量化标准与 SLA
2. **故障模型** - 故障的分类与传播
3. **熔断机制** - 系统的自我保护
4. **限流策略** - 流量的控制与分配
5. **降级策略** - 系统的优雅降级
6. **超时与重试** - 调用的可靠性保证
7. **健康检查** - 系统的健康监控

## 核心写作原则

### 1. 可用性是概率，不是保证

要让读者理解可用性是一个概率指标，而不是绝对的保证。

**错误示例**：

```java
我们的系统设计了熔断、限流、降级，所以一定能保证 99.99% 的可用性。
```

**正确示例**：

```java
即使我们设计了完善的熔断、限流、降级机制，也无法 100% 保证可用性。可用性是一个概率指标，99.99% 的可用性意味着一年中允许的宕机时间只有 52.6 分钟，但这 52.6 分钟可能出现在任何时候。

更重要的是，可用性是一个「系统级」的指标，不是「组件级」的指标。即使每个组件都有 99.9% 的可用性，100 个组件串联后，系统的可用性会降到 90.5%。这就是「可靠性串联」的残酷现实。
```

### 2. 防御思维，不是事后补救

要让读者理解高可用设计必须是前置的，而不是事后补救的。

**应该包含的内容**：

- **故障前置**：在故障发生前设计防护机制
- **故障假设**：假设一切都会出错的思维模式
- **故障隔离**：如何限制故障的影响范围

**示例**：

```java
高可用设计的核心是「防御思维」：假设一切都会出错，并为此做好准备。

- 假设网络会断：设计重试和超时机制
- 假设服务会挂：设计熔断和降级机制
- 假设数据库会慢：设计缓存和读写分离
- 假设流量会暴增：设计限流和扩容机制

记住：故障不可避免，但故障的影响范围可控。
```

### 3. 量化可用性，不是模糊描述

可用性必须用明确的指标来衡量，而不是模糊的「很稳定」「很可靠」。

**应该包含的内容**：

- **可用性等级**：不同等级的可用性要求
- **SLA 与 SLO**：服务等级协议与目标
- **MTTR 与 MTBF**：平均修复时间与平均故障间隔

**示例**：

```java
99.9% 的可用性意味着一年中允许的宕机时间是多少？

- 99% 可用性：每年 3.65 天的宕机时间
- 99.9% 可用性：每年 8.76 小时的宕机时间
- 99.99% 可用性：每年 52.6 分钟的宕机时间
- 99.999% 可用性：每年 5.26 分钟的宕机时间

这就是为什么从 99.9% 提升到 99.99% 需要 10 倍的投入。每提高一个 9，成本呈指数增长。
```

### 4. 系统级思维，不是单点优化

高可用是系统级的设计，不是单点的优化。

**应该包含的内容**：

- **可靠性模型**：串联、并联、混联的可靠性模型
- **故障传播**：故障如何在系统中传播
- **故障隔离**：如何限制故障的影响范围

**示例**：

```java
提高可用性有三个方向：

1. **减少故障概率**：提高代码质量、充分测试
2. **减少故障影响范围**：故障隔离、熔断降级
3. **减少故障恢复时间**：自动化部署、快速回滚

很多团队只关注第一个方向，但事实上，第二个和第三个方向往往更有效。即使故障不可避免，如果影响范围小、恢复时间快，系统的可用性依然很高。
```

### 5. 降级是常态，不是异常

要让读者理解降级是系统设计的常态，而不是异常情况。

**应该包含的内容**：

- **降级的本质**：用功能的可用性换取系统的可用性
- **降级的策略**：自动降级 vs 手动降级
- **降级的粒度**：页面级、接口级、数据级

**示例**：

```java
在流量高峰期，为了保证核心功能的可用性，我们需要对非核心功能进行降级。

- 推荐功能降级：从「个性化推荐」降级为「热门推荐」
- 搜索功能降级：从「全文搜索」降级为「关键词搜索」
- 评论功能降级：从「实时评论」降级为「异步评论」

记住：核心功能的可用性优先于所有功能。当资源有限时，优先保证核心功能的可用性。
```

## 各章节写作要点

### 可用性理论

#### 可用性等级

**应该包含的内容**：

1. **可用性的定义**：可用性 = MTBF / (MTBF + MTTR)
2. **可用性等级**：从 99% 到 99.999% 的可用性等级
3. **各等级的宕机时间**：每年、每月、每天允许的宕机时间
4. **可用性等级的成本**：每提高一个 9 的成本增长
5. **实际应用**：不同业务场景的可用性要求

**写作建议**：

- 给出可用性等级与宕机时间的对照表
- 强调「每提高一个 9，成本呈指数增长」
- 给出不同业务场景的可用性要求建议

#### SLA 与 SLO

**应该包含的内容**：

1. **SLA 的定义**：服务等级协议
2. **SLO 的定义**：服务等级目标
3. **SLA vs SLO**：两者的区别和关系
4. **SLO 的制定**：如何制定合理的 SLO
5. **SLI 的选择**：如何选择合适的服务等级指标

**写作建议**：

- 强调 SLO 是「对内承诺」，SLA 是「对外承诺」
- 说明 SLO 制定需要考虑历史数据和业务需求
- 给出常见的 SLI 指标

#### MTTR 与 MTBF

**应该包含的内容**：

1. **MTTR 的定义**：平均修复时间
2. **MTBF 的定义**：平均故障间隔
3. **如何缩短 MTTR**：监控告警、自动化部署、快速回滚
4. **如何延长 MTBF**：代码质量、充分测试、故障演练
5. **可用性与 MTTR/MTBF 的关系**

**写作建议**：

- 给出缩短 MTTR 的具体方法
- 强调 MTTR 对可用性的影响
- 说明如何通过演练提高 MTTR 的准确性

### 故障模型

#### 故障分类

**应该包含的内容**：

1. **硬件故障**：磁盘故障、网络故障、电源故障
2. **软件故障**：代码 Bug、内存泄漏、死锁
3. **网络故障**：网络分区、网络延迟、丢包
4. **依赖故障**：依赖服务挂掉、依赖服务变慢
5. **人为故障**：配置错误、误操作

**写作建议**：

- 给出每类故障的典型例子
- 说明各类故障的应对策略
- 强调「人为故障」是最常见的故障类型

#### 级联故障

**应该包含的内容**：

1. **级联故障的定义**：故障如何在系统中传播
2. **级联故障的成因**：资源竞争、请求重试、雪崩效应
3. **级联故障的预防**：熔断、限流、降级
4. **级联故障的恢复**：如何快速恢复

**写作建议**：

- 用图示说明级联故障的传播过程
- 强调「请求重试」是级联故障的重要成因
- 给出预防级联故障的策略

#### 雪崩效应

**应该包含的内容**：

1. **雪崩效应的定义**：一个微小的故障如何引发系统崩溃
2. **雪崩效应的成因**：资源耗尽、队列积压、连锁反应
3. **雪崩效应的预防**：熔断、限流、资源隔离
4. **雪崩效应的恢复**：如何快速恢复

**写作建议**：

- 用具体案例说明雪崩效应（如 2012 年 GitHub 宕机事件）
- 强调「雪崩效应」往往在「临界点」突然发生
- 给出预防雪崩效应的策略

### 熔断机制

#### 熔断器模式

**应该包含的内容**：

1. **熔断器的原理**：借鉴电路熔断器的思想
2. **熔断器的三种状态**：关闭、打开、半开
3. **熔断器的触发条件**：错误率、超时率、响应时间
4. **熔断器的恢复策略**：如何从打开状态恢复
5. **实际应用**：Hystrix、Sentinel、Resilience4j

**写作建议**：

- 用图示说明熔断器的状态机
- 强调熔断器的「自我保护」作用
- 给出熔断器的配置建议

#### 熔断状态机

**应该包含的内容**：

1. **关闭状态**：正常调用，监控指标
2. **打开状态**：快速失败，不再调用
3. **半开状态**：试探调用，检查恢复
4. **状态转换**：各状态之间的转换条件
5. **实际应用**：如何配置状态转换参数

**写作建议**：

- 用状态图说明熔断器的状态转换
- 给出各状态的转换条件示例
- 强调「半开状态」的作用

### 限流策略

#### 限流算法

**应该包含的内容**：

1. **固定窗口算法**：简单但存在边界问题
2. **滑动窗口算法**：更平滑的限流
3. **漏桶算法**：恒定的输出速率
4. **令牌桶算法**：允许突发流量
5. **分布式限流**：Redis + Lua、Redisson

**写作建议**：

- 用图示说明各种算法的工作原理
- 对比各种算法的优缺点
- 给出不同场景下的算法选择建议

#### 令牌桶算法

**应该包含的内容**：

1. **令牌桶的原理**：令牌的生成和消耗
2. **令牌桶的参数**：桶的大小、令牌生成速率
3. **令牌桶的优势**：允许突发流量
4. **令牌桶的实现**：Guava RateLimiter、Redis + Lua
5. **实际应用**：接口限流、API 网关限流

**写作建议**：

- 给出令牌桶的代码实现
- 说明令牌桶适合「允许突发流量」的场景
- 强调令牌桶的「平滑限流」特点

#### 固定窗口与滑动窗口

**应该包含的内容**：

1. **固定窗口的原理**：时间窗口的划分
2. **固定窗口的问题**：边界效应
3. **滑动窗口的原理**：更平滑的时间窗口
4. **滑动窗口的实现**：环形数组、Redis + ZSet
5. **性能对比**：固定窗口 vs 滑动窗口

**写作建议**：

- 用图示说明固定窗口的边界效应
- 对比固定窗口和滑动窗口的性能
- 给出不同场景下的选择建议

### 降级策略

#### 降级原则

**应该包含的内容**：

1. **降级的本质**：用功能的可用性换取系统的可用性
2. **降级的原则**：核心功能优先、降级粒度可控、降级可恢复
3. **降级的策略**：页面降级、接口降级、数据降级
4. **降级的时机**：自动降级 vs 手动降级

**写作建议**：

- 强调「核心功能的可用性优先于所有功能」
- 给出降级粒度的选择建议
- 说明降级与熔断的区别

#### 自动降级 vs 手动降级

**应该包含的内容**：

1. **自动降级**：根据系统指标自动触发降级
2. **手动降级**：运维人员手动触发降级
3. **两者的对比**：响应速度、准确度、灵活性
4. **混合策略**：自动降级 + 手动降级

**写作建议**：

- 强调自动降级的「响应速度」优势
- 说明手动降级的「准确度」优势
- 给出混合策略的实施方案

### 超时与重试

#### 超时设置原则

**应该包含的内容**：

1. **超时的作用**：避免请求长时间阻塞
2. **超时的设置原则**：根据历史数据设置、分级设置
3. **超时的危害**：设置过短或过长的代价
4. **超时的监控**：如何监控超时情况

**写作建议**：

- 给出超时设置的计算方法
- 强调「超时不是一成不变的」
- 说明超时与 SLA 的关系

#### 重试策略

**应该包含的内容**：

1. **重试的作用**：提高请求的成功率
2. **重试的策略**：固定间隔、指数退避、随机间隔
3. **重试的代价**：增加延迟、增加系统压力
4. **重试的最佳实践**：限制重试次数、限制重试时间

**写作建议**：

- 说明重试可能导致的「雪崩效应」
- 强调「不是所有请求都适合重试」
- 给出重试策略的选择建议

#### 指数退避

**应该包含的内容**：

1. **指数退避的原理**：重试间隔按指数增长
2. **指数退避的优势**：避免频繁重试
3. **指数退避的实现**：如何计算退避时间
4. **指数退避的参数**：初始间隔、最大间隔、退避因子

**写作建议**：

- 给出指数退避的计算公式
- 说明指数退避适合「网络请求」场景
- 强调指数退避的「避免惊群」作用

### 健康检查

#### 健康检查类型

**应该包含的内容**：

1. **存活探针**：检查服务是否存活
2. **就绪探针**：检查服务是否就绪
3. **启动探针**：检查服务是否启动成功
4. **自定义探针**：检查业务逻辑是否正常
5. **实际应用**：Kubernetes 的健康检查

**写作建议**：

- 说明各种探针的区别和应用场景
- 给出健康检查的代码示例
- 强调健康检查的「轻量级」要求

## 写作格式规范

### 文件命名

- 使用 `kebab-case` 格式，如 `circuit-breaker-pattern.mdx`
- 文件名要能清晰表达文章主题

### 文章结构

推荐采用以下结构：

1. **开场** - 真实故障案例，引发思考
2. **问题分析** - 故障原因，影响范围
3. **解决方案** - 防护策略，具体实现
4. **权衡分析** - 代价收益，理性选择
5. **实战案例** - 真实案例，验证效果
6. **最佳实践** - 经验总结，避坑指南
7. **总结延伸** - 收束全文，引发思考

### 代码示例

- 使用 Java 作为主要示例语言
- 代码示例应该是完整的、可运行的
- 代码示例后要有解释说明

### 量化数据

- 故障影响要有量化数据
- 方案效果要有量化对比
- SLA 要有明确指标

### 符号转义

- 比较运算符（`=`、`<=`、`>=` 等）在正文中需用反引号包裹
- 泛型类型（`<T>`）需用反引号包裹
- 路径不得包含 `/docs` 前缀和 `.mdx` 后缀

### 文章长度

- 可用性文章建议 `>= 2500` 字
- 确保讲透故障防护策略和实战经验

## 质量检查清单

在发布文章前，请检查以下问题：

- [ ] 文章是否有真实的故障案例？
- [ ] 是否有量化的可用性指标？
- [ ] 是否包含了权衡分析？
- [ ] 是否有完整的解决方案？
- [ ] 是否有实战案例验证？
- [ ] 是否遵循了所有格式规范？
- [ ] 字数是否达到建议要求？

只有以上问题都得到肯定的回答，文章才能发布。

## 总结

可用性是高可用系统的核心指标。撰写这部分文章，核心目标是让读者建立起系统化的可用性思维，培养他们前置设计、故障防护的能力。

记住：**好的可用性文档不是技术清单，而是可用思维的传递**。每一篇文章都应该让读者在读完之后，不仅知道「如何防护」，更重要的是理解「为什么要这样防护、什么时候该防护、什么时候不该防护」。
