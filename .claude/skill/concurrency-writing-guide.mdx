# 并发篇写作指南

本文档是撰写「并发篇」文章的权威指南。并发编程是后端开发的核心能力，每一篇文章都应该以**建立并发思维、理解并发模型、掌握并发工具**为核心目标。

## 章节结构概览

并发篇包含以下 6 个主要章节：

1. **并发控制** - 并发控制的理论与策略
2. **锁机制** - 各种锁的使用与原理
3. **乐观与悲观锁** - 两种并发控制策略的对比
4. **事务隔离** - 数据库事务的隔离级别
5. **死锁处理** - 死锁的预防、避免、检测
6. **并发数据结构** - 线程安全的数据结构

## 核心写作原则

### 1. 并发是工具，不是目标

要让读者理解并发是解决问题的工具，而不是盲目追求的目标。

**错误示例**：

```java
我们使用多线程来提高性能，让系统支持更高的并发。
```

**正确示例**：

```java
并发不是目标，而是解决问题的工具。如果你的系统是 CPU 密集型的，多线程可能反而降低性能（上下文切换的开销）。如果是 IO 密集型的，多线程才能真正发挥优势。

更重要的是，并发带来的问题（线程安全、死锁、竞态条件）往往比它能解决的问题更严重。在引入并发之前，先问自己：并发真的必要吗？

记住：过早优化是万恶之源。只有在确定并发能解决问题时，才引入并发。
```

 synchronized 方法看起来简单，但它隐藏了锁的代价：


### 4. 工具链思维

并发编程需要系统的工具链支持，而不是手工编写并发逻辑。

**应该包含的内容**：

- **并发工具**：Java 并发包（JUC）的工具
- **并发原语**：volatile、CAS、锁、条件变量
- **并发数据结构**：线程安全的集合和队列

**示例**：

```java

记住：自己编写的并发逻辑几乎一定有 bug。使用经过充分测试的工具，而不是自己造轮子。
```

## 各章节写作要点

### 并发控制

#### 并发控制理论

**应该包含的内容**：

1. **并发控制的定义**：什么是并发控制
2. **并发控制的挑战**：竞态条件、死锁、活锁
3. **并发控制的方法**：乐观控制、悲观控制
4. **并发控制的代价**：性能、复杂度

**写作建议**：

- 用具体例子说明并发控制的必要性
- 强调并发控制的核心是「保证正确性」

#### 乐观并发控制

**应该包含的内容**：

1. **乐观控制的原理**：先操作，再检查冲突
2. **乐观控制的实现**：CAS、版本号
3. **乐观控制的适用场景**：读多写少

**应该包含的内容**：

## 写作格式规范

### 文件命名

- 使用 `kebab-case` 格式，如 `deadlock-detection.mdx`
- 文件名要能清晰表达文章主题

### 文章结构

推荐采用以下结构：

1. **开场** - 真实问题，引发思考
2. **原理剖析** - 深入原理，理解本质
3. **实现方案** - 具体实现，代码示例
4. **问题分析** - 常见问题，踩坑经验
5. **最佳实践** - 经验总结，避坑指南
6. **总结延伸** - 收束全文，引发思考

### 代码示例

- 使用 Java 作为主要示例语言
- 代码示例应该是完整的、可运行的
- 代码示例后要有解释说明

### 并发问题

- 用具体例子说明并发问题的产生
- 给出并发问题的解决方案
- 强调并发问题的「隐蔽性」

### 符号转义

- 比较运算符（`=`、`<=`、`>=` 等）在正文中需用反引号包裹
- 泛型类型（`<T>`）需用反引号包裹
- 路径不得包含 `/docs` 前缀和 `.mdx` 后缀

### 文章长度

- 并发文章建议 `>= 2500` 字
- 确保讲透并发原理和实现

## 质量检查清单

## 总结

并发编程是后端开发的核心能力。撰写这部分文章，核心目标是让读者建立起系统化的并发思维，培养他们正确使用并发工具的能力。

记住：**好的并发文档不是 API 说明，而是并发思维的传递**。每一篇文章都应该让读者在读完之后，不仅知道「如何使用并发工具」，更重要的是理解「为什么需要并发、什么时候该用并发、什么时候不该用并发」。
